## 摘  要

针对微信生态内轻量化待办工具协作能力不足、而通用项目管理工具又偏复杂的问题，本文设计并实现了一套基于微信小程序的简易协同待办清单系统。系统以前端小程序与云端一体化开发为基础，前端采用 WXML、WXSS、JavaScript 和 Vant Weapp 组件库实现交互界面，后端依托微信云开发平台，结合云函数、云数据库和云存储完成业务处理与数据持久化。围绕个人任务管理与多人清单协作两类核心场景，系统划分为个人待办清单、共享清单协作和个人中心三个模块。

在功能实现上，个人待办清单模块支持任务新增、编辑、删除、完成状态切换、优先级设置、分类管理、关键词搜索、截止日期提醒以及周期任务生成，并提供列表视图与日历视图两种查看方式；共享清单协作模块支持共享清单创建、成员邀请、角色权限控制和操作记录追踪，其中成员可通过微信好友、分享链接和用户名搜索三种方式加入，权限按照创建者、编辑者、查看者三类角色进行管理；个人中心模块实现了用户信息维护、数据看板、通知消息和分类标签管理等功能。

系统完成后，本文从功能、性能和兼容性三个方面进行了测试。功能测试共设计30个测试用例，覆盖用户管理、任务管理、共享清单和个人中心四个模块，测试结果均通过；性能优化后，首页加载时间由2.8s降低至1.2s，日历页面加载时间由1.8s降低至0.9s，`getTaskList` 接口响应时间由1.2s降低至0.6s，数据看板加载时间由2.5s降低至1.0s，小程序主包体积控制在1.5MB。测试结果表明，该系统能够较稳定地满足个人用户和小型团队在任务记录、共享协作与进度跟踪方面的实际需求。

**关键词：** 微信小程序；云开发；待办清单；协同协作；任务管理；Serverless

## Abstract

To address the gap between lightweight WeChat-based to-do tools with limited collaboration features and full-scale project management tools that are often too complex for daily use, this paper designs and implements a simple collaborative to-do list system based on WeChat Mini Programs. The system adopts an integrated front-end and cloud development approach. The front end is built with WXML, WXSS, JavaScript, and the Vant Weapp component library, while the back end relies on WeChat Cloud Development, including cloud functions, cloud database, and cloud storage, to support business logic and persistent data management. The system is organized into three modules: Personal To-Do List, Shared List Collaboration, and Personal Center.

The Personal To-Do List module supports task creation, editing, deletion, completion switching, priority setting, category management, keyword search, due-date reminders, and recurring task generation, and it provides both list and calendar views. The Shared List Collaboration module supports shared list creation, member invitation, role-based permission control, and operation log tracing. Members can join through WeChat friends, share links, or username search, and permissions are managed through three roles: creator, editor, and viewer. The Personal Center module includes profile maintenance, dashboard statistics, notifications, and category management.

After implementation, the system was evaluated in terms of functionality, performance, and compatibility. A total of 30 functional test cases covering user management, task management, shared lists, and personal center all passed successfully. After optimization, the home page loading time was reduced from 2.8 s to 1.2 s, the calendar page loading time from 1.8 s to 0.9 s, the `getTaskList` response time from 1.2 s to 0.6 s, and the dashboard loading time from 2.5 s to 1.0 s, while the main package size was controlled at 1.5 MB. The results show that the system can reliably support task recording, shared collaboration, and progress tracking for individual users and small teams.

**Keywords:** WeChat Mini Program; Cloud Development; To-do List; Collaborative Cooperation; Task Management; Serverless

## 第1章 绪论

### 1.1 研究背景与意义

随着信息技术的快速发展，人们的工作和生活节奏不断加快，每天需要处理的事务的种类和数量日益增长。无论是个人日常生活中的购物清单、学习计划、健身目标，还是工作中的项目任务、会议安排、文档编写，各类事务都需要一种高效、便捷的工具来进行管理和追踪。待办清单（To-do List）作为一种经典的任务管理方法，早已在个人时间管理和团队协作领域得到广泛运用。
纸质待办清单虽然简单直观，但存在诸多局限性，例如不便于携带和保存、难以进行修改和排序、无法设置提醒和截止日期、不支持多人共享和协作。随着计算机技术和移动互联网的发展，各类电子化的任务管理工具应运而生。从桌面端应用的Microsoft To Do、Todoist，到移动端的滴答清单、奇妙清单等，这些工具在一定程度上解决了纸质清单的不足[8-9]。不过，这类应用通常要求用户额外下载安装，这样不仅会占用设备存储空间，而且跨平台的数据同步和社交分享功能往往需要依赖独立账号体系，使用门槛相对较高。
微信作为国内用户规模最大的社交平台，拥有超过12亿的月活跃用户[1]。2017年自微信小程序正式上线以来，凭借"无需下载安装、即开即用、用完即走、无需卸载"的特点，使其迅速成长为移动应用开发的重要载体[2]。借助微信生态，小程序天然集成了社交分享、消息推送、用户身份认证等基础能力，这为构建协作型应用创造了极为有利的条件。此外，微信云开发平台的上线让开发者不必自行搭建和运维后端服务器，就能快速构建数据存储、文件管理、云函数调用等后端能力，从而大大降低了开发周期与运维成本[3-4]。
在这一背景下，本文设计并实现了简易协同待办清单小程序。用户无需额外下载APP，只需在微信中即可使用。小程序不仅支持用户的个人任务管理，还支持团队的共享清单管理，实现任务的多人协作。系统在最基础的待办清单能力上，拓展实现了优先级、分类标签、截止日期、周期重复、提醒通知等功能，同时提供了列表与日历两种视图，以便于用户通过不同的视角查看任务相关情况。


### 1.2 国内外研究现状

#### 1.2.1 国外现状

任务管理工具的演进大致经历了纸质化、电子化、网络化到协同化四个阶段。国外在这一领域的探索起步较早，产品形态也相对成熟。
最开始，只是在纸上记录个人的一些待办事项，但是无法满足团队协作功能。当计算机发展之后，出现了电子化的任务管理工具，例如，Microsoft Outlook将邮件与任务管理整合，用户可以在桌面端设置日程提醒并跟踪进度。随着Web技术快速发展，出现了在线的任务管理工具，例如，Basecamp支持在线协同工作，使得用户可以在网页上即可管理任务；Trello采用看板（Kanban）模式，用户可以通过拖拽卡片来管理任务，使得用户能够直观地组织和推进任务；Asana提供了任务指派、进度跟踪、甘特图等更加精细的项目管理能力[8-9]。
当智能手机开始普及后，任务管理工具进一步从桌面延伸至移动端。手机具备便于携带、交互便捷等优势，使得Wunderlist（奇妙清单）、Todoist等移动端任务管理应用得到广泛使用。这类应用通常支持跨平台同步，用户可以在手机、平板、电脑等多种设备上无缝切换使用，无需考虑数据迁移成本[8-9]。

#### 1.2.2 国内现状

国内的任务管理工具在之前主要使用国外的任务管理工具，之后才开始发展出自己的任务管理工具。目前国内市场上主流的产品主要分为三类：第一类是综合型任务管理工具，如滴答清单、Todo 清单等，这类工具功能全面，不仅支持个人任务分类、优先级设置、截止日期提醒等功能，还支持多人协作共享，但是用户学习成本高，并且核心协作功能付费，难以满足普通用户对工具简易功能的需求；第二类是办公软件内置的待办模块，如飞书文档、钉钉待办、企业微信任务等，这类工具主要是为软件提供拓展的功能，依靠软件本身存在的生态，为办公用户提供了方便快捷的使用体验，但是，它们主要面向企业用户，主要适用于办公用户的团队协作，对于个人用户的日常事务管理的场景适配不足，并且不是软件的主要功能，存在功能缺失的情况，且需依托主办公软件运行，缺乏独立便捷性；第三类是微信生态内的简易工具，如微信自带的“待办”功能、各类轻量化小程序，这类工具无需额外下载、即开即用、操作便捷，但由于比较轻量化，导致存在协作功能薄弱、多端数据同步延迟较高、用户权限管理机制不完善等问题，无法满足多人协同场景下的高校任务管理需求。
在任务管理工具的发展中，微信小程序的推出为国内开发者开辟了新的赛道。小程序运行在微信客户端内部，用户无需下载，仅在微信内即可使用[2]。2017年至今，小程序生态内已有不少效率类应用，如"微信待办""小打卡""腾讯文档"等。这些产品充分利用微信的社交属性，实现了低门槛的分享与协作。与此同时，微信云开发平台的上线让个人开发者和小团队得以降低成本，直接调用云数据库、云存储和云函数构建完整应用[3]。在教育管理、医药辅助和高校招生服务等场景中，基于微信小程序及云开发模式的业务系统已经得到应用验证[10-12]。云开发的推出极大地简化了小程序开发流程，使得个人开发者和小型团队也能开发功能完善的小程序。

#### 1.2.3 存在的问题

尽管市场上已有众多任务管理工具，但在微信小程序生态中，能够同时满足以下需求的产品仍然较少：
轻量级与功能的兼顾：大多数微信小程序端的待办工具功能较为简单，仅支持基本的个人任务创建和勾选，缺乏任务优先级管理、为任务添加分类标签、设置周期性任务等高级功能，并且只支持个人任务的管理，不支持共享清单的创建。
个人使用与团队协作的兼顾：部分工具仅仅聚焦于个人任务管理，不支持多人协作；而专业的项目管理工具又过于复杂，学习成本较高，不适合日常简单的协作场景，未出现将两者兼顾的任务管理工具。
社交化的协作体验：现有工具的成员邀请和协作流程往往不够便捷，只是工具内部实现的独立机制，未能充分利用微信的社交优势，以及微信内部提供的邀请机制。
本文所设计的系统正是针对上述不足，力求在微信小程序平台上实现一个功能丰富、操作便捷、同时支持个人任务管理与团队协作管理、安全可靠的待办清单管理工具。

## 第2章 相关技术介绍

### 2.1 微信小程序技术概述

微信小程序是腾讯于2017年正式发布的一种轻量级应用形态，其运行环境嵌入在微信客户端内部。小程序采用与Web相近的技术栈进行开发，但与传统Web应用不同，它并不是运行在浏览器，而是运行在微信内部，可以直接调用微信提供的原生API，简化开发流程，实现接近原生应用的用户体验[2]。
微信小程序的技术架构采用双线程模型，分别是渲染层（View Layer）和逻辑层（App Service Layer），它们分别运行在不同的线程中。渲染层使用WebView进行页面渲染，逻辑层使用JSCore运行JavaScript代码。两者通过微信客户端（Native）进行通信，这种架构设计有助于降低逻辑计算对界面渲染的阻塞，保障应用的流畅运行[2]。
小程序的开发框架提供了完整的组件系统和API接口。组件系统包括视图容器、表单组件、导航组件、媒体组件等多种类型，开发者可以通过组合这些组件快速构建用户界面。API接口提供了诸如网络请求、本地缓存、文件操作、界面交互等方面的能力，为开发者提供了丰富的功能支持[2]。

### 2.2 微信云开发平台

微信云开发是微信团队提供的一站式后端云服务，为小程序开发者提供了云数据库、云存储、云函数三大核心能力[3]。借助该服务，开发者无需搭建服务器，即可在小程序端直接调用云端实现的代码，实现各类后端功能。
云数据库是一个基于JSON格式的NoSQL数据库，提供了类似MongoDB的文档型数据存储能力。每个数据库由多个集合（Collection）组成，每个集合包含多个记录（Document），每个记录是一个JSON对象，字段可以按业务需要灵活扩展。云数据库支持丰富的查询操作符和聚合操作，能够满足大多数应用场景的数据存储和查询需求[3][17]。
云函数是运行在云端的Node.js函数，开发者只需编写实现业务逻辑的云函数，将云函数上传到服务器后，即可在小程序端直接调用云函数来执行相关操作。云函数运行在独立的Node.js环境中，天然具备微信鉴权能力，可以直接获取调用者的 OpenID 等用户信息，省去了开发者自行实现用户认证的繁琐工作[3]。
云存储提供了文件存储和管理能力，支持图片、视频、文档等多种类型文件的上传、下载和管理，无需用户另外管理文件相关资源。云存储与云数据库和云函数无缝集成，开发者可以在云函数中直接操作云存储中的文件[3]。

### 2.3 WXML与WXSS

WXML（WeiXin Markup Language）是微信小程序的标记语言，用于描述页面的结构。WXML的语法与HTML类似，但提供了一些特有的功能，包括数据绑定、条件渲染、列表渲染、模板引用等[2]。数据绑定通过双花括号语法（ {{}} ）实现，可将逻辑层的数据动态渲染到页面上。条件渲染（wx:if）和列表渲染（wx:for）语法使得开发者能够根据数据状态灵活控制页面元素的显示[2]。
WXSS（WeiXin Style Sheets）是微信小程序的样式语言，用于定义页面的视觉呈现。WXSS在CSS的基础上进行了扩展，新增了rpx（responsive pixel）响应式尺寸单位，能够根据屏幕宽度自动进行适配，使得页面在不同尺寸的设备上都能保持良好的显示效果，保证跨设备的一致性体验。WXSS支持大部分CSS特性，包括选择器、盒模型、弹性布局（Flexbox）等，同时支持样式的导入（@import）功能，便于样式的模块化管理[2]。
在本系统的开发中，WXML用于构建各个页面的结构，例如任务列表、日历视图、清单详情、个人中心等页面。WXSS用于定义页面的样式，确保系统在不同设备上呈现一致的视觉效果。

### 2.4 JavaScript与ES6+

JavaScript是微信小程序逻辑层的编程语言，用于编写页面逻辑和业务处理代码，实现与前端界面的交互，例如处理用户操作、数据加工、网络请求等业务逻辑。微信小程序的JavaScript运行环境支持ES6+（ECMAScript 2015及以上版本）的大部分语法特性，包括箭头函数、模板字符串、解构赋值、Promise、async/await、模块化等[15]。

### 2.5 Vant Weapp组件库

Vant Weapp是有赞团队开源的微信小程序 UI 组件库，提供了丰富的高质量组件，包括按钮、表单、弹窗、标签页、日历、操作菜单等[16]。该组件库遵循微信小程序的设计规范，组件风格简洁美观，交互体验流畅自然。
本系统使用Vant Weapp组件库构建用户界面，主要使用了以下组件：Button（按钮）用于各类操作触发；Tab/Tabs（标签页）用于任务分类切换；Checkbox（复选框）用于任务状态切换；Tag（标签）用于优先级和分类标识；Dialog（对话框）用于确认操作和信息提示；ActionSheet（操作菜单）用于提供多选项操作。
选择Vant Weapp的原因在于：组件丰富且质量高，能够满足系统的界面需求，保证界面一致性；文档完善，使用方便，方便学习与组件使用方法查询；社区活跃，维护更新及时，保证 bug 少，提高系统稳定性；组件体积小，不会显著增加小程序的包大小，减少用户内存负担[16]。

### 2.6 NoSQL数据库技术

与传统的关系型数据库相比，NoSQL 数据库具有灵活的数据模型、良好的水平扩展能力和较高的读写性能[17]。常见的 NoSQL 数据库类型包括文档型（如 MongoDB）、键值型（如 Redis）、列族型（如 HBase）和图型（如 Neo4j）[17]。
微信云数据库属于文档型数据库，其数据模型与 MongoDB 类似。文档型数据库以 JSON 文档作为基本存储单元，每个文档可以包含不同的字段和嵌套结构，无需预先定义固定的表结构（Schema-free）。这种灵活的数据模型特别适合微信小程序这种快速迭代开发的场景，开发者可以根据业务需求随时调整数据结构，而无需进行复杂的数据库迁移操作[17-18]。
本系统的数据库设计充分利用了 NoSQL 数据库的灵活性，无需要求同一集合中的文档包含完全一致的字段。例如，任务文档中可以包含可选的附件数组、重复设置等数据，无需为这些可选字段创建额外的关联表。同时，通过合理的索引设计，确保了查询操作的高效性[17]。

### 2.7 Serverless架构

Serverless（无服务器）架构是一种新兴的云计算执行模型，在该模型中，开发者只需编写并部署业务逻辑代码即可，由云服务提供商负责底层服务器的配置、扩缩容和运行维护[4-5]。Serverless 架构的核心组件是函数即服务（FaaS），开发者将业务逻辑封装为独立的函数，这些函数在被触发时自动执行，执行完毕后自动释放资源[4]。
Serverless 架构的主要优势包括：降低运维成本，开发者无需管理服务器；能够根据请求量自动调整计算资源；按实际用量计费，只为实际使用的计算资源付费；加快开发速度，开发者可以更加专注于业务逻辑的实现。这种按需调度和弹性分配的特征也有助于控制资源消耗并提升服务效率[4-5]。
微信云开发的云函数就是 Serverless 架构在小程序场景中的典型实现[3-4]。本系统的后端业务逻辑全部通过云函数实现，包括任务管理、清单管理、用户管理、通知管理等功能模块。每个云函数模块负责处理一类业务逻辑，通过 action 参数区分不同的操作，实现了清晰的模块化设计。

### 2.8 本章小结

本章详细介绍了系统开发所涉及的关键技术。微信小程序提供了轻量级的前端应用平台，微信云开发提供了完整的后端云服务，WXML和WXSS用于构建页面结构和样式，JavaScript用于实现业务逻辑，Vant Weapp提供了高质量的UI组件，NoSQL数据库提供了灵活的数据存储方案，Serverless架构简化了后端的开发和运维。这些技术的有机结合，为本系统的设计与实现提供了坚实的技术基础。

在任务管理方面，该模块贯穿了任务从创建到完成的全生命周期。用户可以创建待办任务并设置标题（必填）、描述、截止日期（必填）、优先级、分类标签、所属清单等多项属性。系统支持四级优先级设置（重要且紧急、重要不紧急、紧急不重要、不重要不紧急），帮助用户按照时间管理矩阵进行任务排序。用户可以随时编辑任务属性或删除任务（需二次确认），通过复选框快速切换任务的完成状态。系统支持按关键词模糊搜索任务，以及通过自定义分类标签筛选查看任务。在高级功能方面，系统支持创建周期性重复任务（每天、每周、每月三种模式），自动预生成未来30天的任务实例，且仅限在截止日期当天标记完成以确保打卡真实性；用户可以为任务设置提醒时间，系统通过云函数定时检查并发送通知；用户还可以为任务添加图片附件（单个任务最多9个），通过云存储进行管理。

在视图展示方面，系统提供列表视图和日历视图两种展示方式。列表视图将任务按状态分为进行中、已逾期、已完成三组显示，组内按截止日期排序，顶部提供分类标签栏和搜索功能。日历视图以月历网格形式展示任务时间分布，有任务的日期显示蓝色圆点标记，用户点击日期可查看当天任务列表，支持月份切换和周期任务在重复日期上的标记显示。

共享清单协作模块

共享清单协作模块是本系统区别于普通个人待办工具的核心特色功能，旨在为小型团队提供便捷的任务协作能力。该模块涵盖了清单的创建与管理、成员的邀请与管理、基于角色的权限控制、操作记录追溯以及邀请流程管理等功能。

在清单创建与管理方面，用户可以创建个人清单和共享清单。创建清单时需要设置清单名称（必填）和描述（选填），并选择清单类型——个人清单仅创建者可见，共享清单可以邀请其他用户加入。共享清单还支持设置可见性属性（公开或私密），公开清单允许通过链接直接加入，私密清单需要创建者审批后方可加入。用户可以编辑已创建的清单信息，也可以删除不再需要的清单（删除清单将同时删除清单下的所有任务）。

在成员邀请方面，系统充分利用微信的社交生态，提供三种邀请方式。第一种是微信好友邀请，创建者可以将邀请信息直接分享给微信好友或群聊，好友点击分享卡片即可接受邀请加入清单。第二种是链接邀请，系统生成包含唯一邀请码的邀请链接，创建者可以设置邀请角色（编辑者或查看者）、有效期（1天、3天、7天或永久）以及是否需要审批，将链接分享给目标用户。第三种是用户搜索邀请，创建者通过输入用户昵称搜索系统中的注册用户，选择目标用户后直接发送邀请。

在权限控制方面，系统实现了基于角色的访问控制（RBAC）机制，定义了三种角色及其对应的权限等级，如表3.2所示。

在邀请管理方面，清单创建者可以在邀请管理页面查看所有已发出的邀请记录，包括邀请状态（待接受、已接受、已拒绝、已过期、待审批）、邀请方式、邀请时间等信息。创建者可以对邀请进行撤销、重新发送提醒等操作，对于需要审批的邀请可以进行批准或拒绝。

个人中心模块

个人中心模块为用户提供个人信息管理、数据统计分析、消息通知管理、分类标签管理和系统设置等功能，是用户了解自身任务情况和管理系统配置的综合入口。

在用户信息管理方面，个人中心页面顶部显示用户的头像和昵称，用户可以点击进入编辑页面修改个人信息。昵称作为用户在共享清单中的显示名称，同时也是其他用户通过搜索邀请功能查找协作伙伴的依据。对于未登录的用户，页面显示登录引导，引导用户完成微信授权登录。

### 3.4 非功能性需求分析

#### 3.4.1 性能需求

（1）页面加载时间：首页加载时间不超过2秒，其他页面加载时间不超过1.5秒。

（2）接口响应时间：云函数接口响应时间不超过1秒（正常网络环境下）。

（3）数据缓存：系统应采用缓存优先策略，优先展示本地缓存数据，后台异步更新最新数据，提升用户感知速度。

#### 3.4.2 可用性需求

（1）界面设计：界面简洁直观，操作流程清晰，符合微信小程序的设计规范。

（2）错误处理：系统应对各类异常情况进行友好的错误提示，避免出现白屏或无响应的情况。

（3）离线支持：在网络不可用时，系统应能展示缓存数据，并在网络恢复后自动同步。

#### 3.4.3 安全性需求

（1）用户认证：基于微信OpenID进行用户认证，确保用户身份的唯一性和安全性。

（2）权限控制：严格执行基于角色的权限控制，确保用户只能访问和操作其有权限的数据。

（3）数据验证：前端和云函数双重验证数据的合法性，防止非法数据的写入。

（4）邀请安全：邀请链接设置有效期，过期自动失效；支持审批机制，防止未授权用户加入清单。

#### 3.4.4 兼容性需求

系统需要兼容iOS和Android两大移动操作系统平台，适配不同屏幕尺寸的设备，确保在主流机型上的显示和交互一致性。

## 第4章 系统总体设计

### 4.1 系统架构设计

本系统采用前后端分离的架构模式，整体架构分为表现层、业务逻辑层和数据存储层三个层次。前端基于微信小程序框架开发，后端基于微信云开发平台构建，采用Serverless无服务器架构，无需搭建和维护传统的后端服务器。系统的整体架构如图4.1所示。

**图4.1 系统整体架构图**（源文件：diagrams/architecture.puml）

<!-- 渲染方式：将puml文件内容复制到 https://www.plantuml.com/plantuml/uml 渲染，导出PNG后插入Word -->

（1）表现层

表现层即小程序前端，负责页面渲染、用户交互和本地缓存管理。前端使用WXML构建页面结构，WXSS定义页面样式（通过rpx单位实现响应式布局），JavaScript处理交互逻辑和数据绑定。界面组件采用Vant Weapp开源组件库，提供按钮、标签页、复选框、对话框、操作菜单等高质量UI组件，确保界面风格统一和交互体验流畅。系统通过底部TabBar导航将前端页面划分为"任务"、"日历"、"清单"和"我的"四个主要入口，共包含22个页面。前端采用缓存优先策略，页面加载时优先展示本地缓存数据，同时在后台异步请求云端最新数据进行更新，提升用户感知速度。

（2）业务逻辑层

业务逻辑层基于微信云开发的云函数实现，使用Node.js编写。云函数运行在独立的服务器环境中，天然具备微信鉴权能力，可通过cloud.getWXContext()直接获取调用者的OpenID，无需开发者自行实现用户认证逻辑。系统共设计5个云函数模块：taskFunctions（任务管理，13个接口+2个定时触发器）负责任务的创建、编辑、删除、状态切换、搜索、周期任务管理等；listFunctions（清单管理，22个接口）负责清单的创建与管理、成员邀请、权限控制、操作记录等；profileFunctions（个人中心，12个接口）负责用户信息管理、数据统计、通知消息等；categoryFunctions（分类管理，5个接口）负责分类标签的增删改查；quickstartFunctions（快速启动）负责系统初始化。每个云函数模块采用统一的入口函数和action路由模式，通过switch语句将不同的操作分发到对应的处理函数，返回统一格式的结果对象（包含code、message和data三个字段）。

（3）数据存储层

数据存储层基于微信云开发平台，包括云数据库和云存储两部分。云数据库是基于JSON的NoSQL文档型数据库，提供类似MongoDB的数据存储能力，支持丰富的查询操作符和聚合操作。系统设计了8个数据集合（users、tasks、lists、categories、list_members、list_invites、operations、notifications），分别存储用户信息、任务数据、清单数据、分类标签、成员关系、邀请记录、操作日志和通知消息。云存储提供文件存储和管理能力，用于存储任务附件（图片文件）等资源。云数据库与云函数深度集成，云函数通过wx-server-sdk直接操作数据库，数据操作便捷高效。

系统的数据流转过程为：用户在小程序前端进行操作，前端通过wx.cloud.callFunction调用云函数，云函数执行业务逻辑并操作云数据库，将处理结果返回给前端，前端更新页面显示和本地缓存。

### 4.3 数据库设计

（1）用户实体

用户表属性有用户ID、微信OpenID、昵称、头像、通知开关、创建时间、更新时间。用户通过微信授权登录系统，OpenID由微信平台生成，具有唯一性和不可伪造性。如图4.3所示。

### 4.5 安全性设计

系统的安全性设计主要包括以下几个方面：

（1）用户认证：系统基于微信OpenID进行用户认证。云函数通过cloud.getWXContext()获取调用者的OpenID，确保每个请求都能准确识别用户身份。OpenID由微信平台生成，具有唯一性和不可伪造性。

（2）权限验证：系统实现了verifyListPermission权限验证函数，在每个涉及共享清单的操作中进行权限检查。权限验证逻辑首先检查用户是否为清单成员，然后根据成员角色判断是否具有执行当前操作的权限。角色权限等级为：创建者（1）> 编辑者（2）> 查看者（3）。这种设计属于典型的基于角色的访问控制（RBAC）思想，能够较好地适配协同场景下的分级授权需求[6-7]。

（3）数据验证：云函数对所有输入数据进行合法性验证，包括必填字段检查、数据类型验证、数据范围验证等。例如，创建任务时验证标题非空、截止日期有效；创建分类时验证名称不重复。

（4）邀请安全：邀请链接采用16位随机字符串作为邀请码，具有足够的随机性；邀请链接设置有效期（默认7天），过期自动失效；支持审批机制，创建者可以选择是否需要审批才能加入清单。

### 4.6 本章小结

本章从系统架构设计、功能模块划分、数据库设计、接口设计和安全性设计五个方面对系统进行了总体设计。系统采用前后端分离的三层架构，前端基于微信小程序，后端基于微信云开发。数据库设计经历了概念结构设计（E-R模型）、逻辑结构设计（集合字段定义）和物理结构设计（二维关系表与索引）三个阶段，确保了数据结构的合理性。接口设计采用统一的调用方式和返回格式，安全性设计涵盖了认证、授权、验证和邀请安全等方面。

// taskFunctions/index.js - 周期任务状态切换
async function toggleTaskStatus(openid, data) {
  const userId = await getUserId(openid);
  const { taskId, status } = data;
  const { data: tasks } = await db.collection('tasks')
    .where({ _id: taskId }).get();
  const oldTask = tasks[0];
  // 权限验证
  if (oldTask.creatorId !== userId && oldTask.listId) {
    const { hasPermission } = await verifyListPermission(
      userId, oldTask.listId, 2);
    if (!hasPermission) return { code: -1, message: '无权限' };
  }
  // 周期任务：只能在截止日期当天完成
  if (status === 1 && oldTask.repeatType > 0) {
    const today = new Date(); today.setHours(0,0,0,0);
    const due = new Date(oldTask.dueDate); due.setHours(0,0,0,0);
    if (due.getTime() !== today.getTime()) {
      return { code: -1, message: '周期任务只能在当天完成' };
    }
  }
  // 更新状态
  await db.collection('tasks').doc(taskId)
    .update({ data: { status, updatedAt: db.serverDate() } });
  return { code: 0, message: '操作成功' };
}

// calendar.js - 日历生成算法
generateCalendar: function () {
  const { currentYear, currentMonth } = this.data;
  const firstDay = new Date(currentYear, currentMonth - 1, 1);
  const lastDay = new Date(currentYear, currentMonth, 0);
  const daysInMonth = lastDay.getDate();
  const firstDayOfWeek = firstDay.getDay();
  const calendarDays = [];
  const today = new Date();
  const todayStr = this.formatDate(
    today.getFullYear(), today.getMonth() + 1, today.getDate());
  // 填充上月末尾日期
  const prevMonth = new Date(currentYear, currentMonth - 1, 0);
  for (let i = firstDayOfWeek - 1; i >= 0; i--) {
    calendarDays.push({
      day: prevMonth.getDate() - i,
      isCurrentMonth: false, isToday: false
    });
  }
  // 填充当月日期，计算每日任务数
  for (let i = 1; i <= daysInMonth; i++) {
    const dateStr = this.formatDate(currentYear, currentMonth, i);
    calendarDays.push({
      day: i, date: dateStr,
      isCurrentMonth: true,
      isToday: dateStr === todayStr,
      periodicCount: this.calculatePeriodicCount(dateStr)
    });
  }
  // 填充下月开头日期至42格
  const remaining = 42 - calendarDays.length;
  for (let i = 1; i <= remaining; i++) {
    calendarDays.push({ day: i, isCurrentMonth: false });
  }
  this.setData({ calendarDays });
}

// taskFunctions/index.js - 清单权限验证
async function verifyListPermission(userId, listId, requiredRole = 3) {
  // 角色等级：1-创建者，2-编辑者，3-查看者
  const { data: members } = await db.collection('list_members')
    .where({ listId, userId }).get();
  if (members.length === 0) {
    // 检查是否是清单创建者
    const { data: lists } = await db.collection('lists')
      .where({ _id: listId, creatorId: userId }).get();
    if (lists.length === 0) {
      return { hasPermission: false, role: null };
    }
    return { hasPermission: true, role: 1 };
  }
  const role = members[0].role;
  return { hasPermission: role <= requiredRole, role };
}
```

// listFunctions/index.js - 生成邀请链接
async function generateInviteLink(openid, data) {
  const { listId, role = 3, expireDays = 7 } = data || {};
  const userId = await getUserId(openid);
  // 验证创建者权限
  const { data: lists } = await db.collection('lists')
    .where({ _id: listId, creatorId: userId }).get();
  if (lists.length === 0) return { code: -1, message: '无权限' };
  // 生成16位随机邀请码
  const chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789';
  let inviteCode = '';
  for (let i = 0; i < 16; i++) {
    inviteCode += chars.charAt(Math.floor(Math.random() * chars.length));
  }
  // 计算过期时间并创建邀请记录
  const expireAt = expireDays > 0
    ? new Date(Date.now() + expireDays * 86400000) : null;
  await db.collection('list_invites').add({
    data: { listId, inviterId: userId, role, inviteType: 'link',
      inviteCode, status: 0, expireAt, createdAt: db.serverDate() }
  });
  return { success: true, inviteCode };
}
```

// profileFunctions/index.js - 数据看板并行查询
async function getDashboardData(openid) {
  const { data: users } = await db.collection('users')
    .where({ openid }).get();
  const userId = users[0]._id;
  const todayStart = new Date();
  todayStart.setHours(0, 0, 0, 0);
  // 使用Promise.all并行查询，提升性能
  const [completed, uncompleted, overdue] = await Promise.all([
    db.collection('tasks').where({
      creatorId: userId, status: 1,
      repeatType: _.not(_.gt(0))
    }).count(),
    db.collection('tasks').where({
      creatorId: userId, status: 0,
      dueDate: _.gte(todayStart), repeatType: _.not(_.gt(0))
    }).count(),
    db.collection('tasks').where({
      creatorId: userId, status: 0,
      dueDate: _.lt(todayStart), repeatType: _.not(_.gt(0))
    }).count()
  ]);
  return {
    code: 0, data: {
      pieChart: { completed: completed.total,
        uncompleted: uncompleted.total, overdue: overdue.total }
    }
  };
}

周期任务连续完成天数的计算实现如下：

```javascript
// taskFunctions/index.js - 连续完成天数计算
async function getPeriodicTaskStats(openid, data) {
  const userId = await getUserId(openid);
  const { taskId } = data;
  // 分批查询所有周期实例（突破100条限制）
  let allInstances = [];
  let batchIndex = 0;
  while (true) {
    const { data: batch } = await db.collection('tasks')
      .where(_.or([{ _id: taskId }, { parentTaskId: taskId }]))
      .orderBy('dueDate', 'asc')
      .skip(batchIndex * 100).limit(100).get();
    allInstances = allInstances.concat(batch);
    if (batch.length < 100) break;
    batchIndex++;
  }
  // 筛选已到期实例，计算连续完成天数
  const now = new Date(); now.setHours(0,0,0,0);
  const dueInstances = allInstances.filter(t => {
    const d = new Date(t.dueDate); d.setHours(0,0,0,0);
    return d <= now;
  }).sort((a, b) => new Date(a.dueDate) - new Date(b.dueDate));
  let currentStreak = 0;
  for (let i = dueInstances.length - 1; i >= 0; i--) {
    if (dueInstances[i].status === 1) currentStreak++;
    else break;
  }
  return { code: 0, data: { currentStreak, totalCount: dueInstances.length } };
}
```

#### 6.3.2 优化措施

针对性能测试中发现的问题，采取了以下优化措施：

（1）缓存优先策略：在任务列表、分类列表、日历数据等高频访问的数据上实施缓存优先策略。页面加载时优先展示本地缓存数据，同时在后台异步请求最新数据。这一策略将首页的感知加载时间从2.8秒降低到1.2秒。

（2）并行查询优化：在数据看板等需要多次数据库查询的场景中，使用Promise.all将多个独立的查询操作并行执行，避免串行等待。这一优化将数据看板的加载时间从2.5秒降低到1.0秒。

（3）分页加载：对任务列表、通知列表、操作记录等可能数据量较大的列表采用分页加载策略，每次只加载一页数据（默认10-20条），用户滚动到底部时再加载下一页，减少单次请求的数据量。

（4）索引优化：根据实际查询模式，为数据库集合添加合理的索引。例如，为tasks集合添加(creatorId, status)复合索引，为notifications集合添加(userId, isRead)复合索引，显著提升了查询效率。

（5）数据预处理：在云函数中对返回数据进行预处理，减少前端的计算量。例如，在getTaskList接口中直接返回格式化后的日期字符串，避免前端重复进行日期格式化。

（6）分批查询：对于可能超过云数据库单次查询100条限制的场景（如周期任务统计），采用分批查询策略，通过循环skip和limit实现全量数据的获取。

## 第7章 总结与展望

本文主要针对基于微信小程序的简易协同待办清单系统，完成了系统的设计与实现。从当前移动互联网环境下个人任务管理和团队协作的实际需求入手，对国内外任务管理工具的发展现状进行了调研。然后对系统整体架构、技术框架、开发工具、需求分析、可行性分析进行说明后，开始搭建系统框架，设计数据库，着手进行详细设计与编码实现。经过多方面的功能测试、性能测试与兼容性测试分析，目前系统已经能够稳定运行，各项功能均达到预期目标，同时在微信云开发平台的稳定支持下，系统能够为用户提供可靠的服务。

经过数月的毕业设计，从一开始面对课题时的不知从何下手，到回顾已有知识、学习微信小程序和云开发等新技术，接着是进行系统的需求分析和总体设计，然后开始各个功能模块的详细设计与编码实现，包括个人待办清单模块、日历视图模块、共享清单协作模块和个人中心模块，再将各模块进行集成与测试，最后撰写毕业论文。这样的一个过程让我深刻体会到了软件工程实践的完整流程。一开始觉得分析一个系统的需求很难，后来觉得编写系统代码更难，最后觉得将整个设计与实现过程系统地整理成一篇完整的论文同样不易。每一个阶段开始时都充满挑战，但当真正开始去面对、去解决时，这些困难又逐一被克服了。

回顾本系统的开发过程，从最初的技术选型到最终的系统测试，每一步都是一次学习和成长的过程。在前端开发方面，通过使用WXML、WXSS和JavaScript构建了22个页面，结合Vant Weapp组件库实现了良好的用户界面和交互体验；在后端开发方面，基于微信云开发的Serverless架构，设计并实现了5个云函数模块共52个接口和2个定时触发器，完成了任务管理、清单协作、权限控制、通知推送等核心业务逻辑；在数据库设计方面，设计了8个数据集合，合理利用了NoSQL文档型数据库的灵活性。特别是在共享清单协作功能的实现过程中，对基于角色的访问控制（RBAC）模型、多种成员邀请方式以及操作记录追溯等功能的设计与实现，让我对协同系统的架构设计有了更深入的理解。通过本次毕业设计的经历，对于微信小程序开发、云开发技术、数据库设计以及软件工程实践都有了全新的认识和更深层次的掌握。

虽然最终顺利完成了系统的设计与实现，但由于自身能力和开发时间有限，系统中仍有不少需要改进的地方。例如，当前系统采用请求-响应模式进行数据交互，共享清单中的数据更新需要用户手动刷新，未来可以引入实时数据推送能力提升协作体验；系统在离线状态下仅能展示缓存数据，不支持离线编辑操作；通知推送目前仅在应用内展示，尚未接入微信订阅消息实现外部推送；数据统计功能也较为基础，还可以增加更丰富的统计维度和数据导出功能。对于能够真正投入到实际广泛使用的软件系统，还需要在实时同步、离线支持、安全加固、用户体验优化等方面积累更多的经验，希望在今后的学习和工作中能够持续改进和完善。
