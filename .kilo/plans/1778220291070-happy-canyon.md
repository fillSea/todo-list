# 邀请链接机制简化方案

## 已确认的产品决策
保留“邀请链接”的高级设置能力：
- 保留 `有效期`
- 保留 `需要审核`
- 但交互上收进 `pages/list-invite/list-invite` 内的轻量设置区
- 不再让用户为了生成链接跳转到独立页面作为主流程

## 目标
将当前“邀请链接”从独立页面/独立操作流，简化为与“微信好友邀请”一致的使用体验：
- 在 `pages/list-invite/list-invite` 内完成配置与发起
- 直接拉起微信分享，而不是跳转到 `pages/list-invite-link/list-invite-link`
- 接收端仍统一落到 `pages/list-invite-accept/list-invite-accept`
- 尽量复用现有 `list_invites` 数据结构与 `getInviteInfo` / `acceptInvite` / `applyJoinList` 能力

## 现状结论

### 前端现状
1. `miniprogram/pages/list-invite/list-invite.js`
- “微信好友邀请”在当前页内完成：先选角色，再调用 `createWechatInvite` 生成 `inviteCode`，最后通过 `open-type="share"` 分享。
- “邀请链接”则跳转到独立页 `pages/list-invite-link/list-invite-link`。

2. `miniprogram/pages/list-invite-link/list-invite-link.js`
- 独立维护 `role` / `expireDays` / `needApproval`。
- 页面加载即调用 `generateInviteLink`。
- 用户每次改角色、有效期、审核开关，都会重新生成一条新的公开模板邀请，并使旧链接失效。
- 页面还承载了二维码、小程序码、复制链接、重新生成、统计等较重 UI。

3. `miniprogram/pages/list-invite-accept/list-invite-accept.js`
- 接收端已统一支持 `inviteType=wechat/link/search/recent`。
- `needApproval`、已处理状态、注册跳转、公开模板邀请转用户记录等逻辑已存在，不需要重新设计。

### 后端现状
1. `cloudfunctions/listFunctions/index.js`
- `createWechatInvite`：创建一条 `inviteType: 'wechat'` 的公开邀请模板，默认 7 天有效。
- `generateInviteLink`：创建一条 `inviteType: 'link'` 的公开邀请模板，支持 `role` / `expireDays` / `needApproval`，并会废弃同清单旧的 active link template。
- `getInviteInfo` / `acceptInvite` / `applyJoinList`：对公开模板邀请已经做了统一处理，理论上不依赖入口页长什么样。

2. 数据模型
- 两类邀请都落在 `list_invites`。
- 关键差异主要是：
  - `inviteType` 不同
  - link 支持 `needApproval`
  - link 支持自定义 `expireDays`
  - link 维护 `activeTemplateCount` 等统计

## 实施方案

### 方案 A：保留能力，简化入口
把“邀请链接”保留为一种邀请类型，但 UI 和交互改成接近“微信好友邀请”的单页模式。

#### 交互设计
1. 在 `pages/list-invite/list-invite` 中点击“邀请链接”后，不再 `navigateTo` 独立页面。
2. 复用现有角色弹窗，并在确认后进入“链接邀请已准备”面板。
3. 该面板与当前“微信邀请已准备”面板合并或并列，展示：
- 当前角色
- 是否需要审核
- 有效期
- 分享按钮
- 复制邀请链接按钮
- 可选：重新生成按钮

4. 用户默认只需要：
- 选角色
- 点击分享 或 复制链接

5. 高级配置入口做折叠/轻量化：
- `needApproval` 开关
- `expireDays` 选择
- 默认使用 `7 天`、`无需审核`
- 仅在用户点击“高级设置”时展开

#### 技术实现
1. 前端 `list-invite`
- 新增统一分享状态，建议替换当前零散的 `wechatInvite*` 字段，收敛为：
  - `shareMode: '' | 'wechat' | 'link'`
  - `shareInviteCode`
  - `shareInviteRole`
  - `shareInviteLink`
  - `shareNeedApproval`
  - `shareExpireDays`
  - `shareExpireText`
  - `shareReady`
  - `shareStatusText`
  - `shareHelpText`
- 同时增加 link 专属配置态：
  - `showLinkAdvancedSettings`
  - `linkDraftExpireDays`
  - `linkDraftExpireText`
  - `linkDraftNeedApproval`

2. 邀请入口改造
- `onWechatInvite()`：保持“点入口 -> 选角色 -> 创建 invite -> 准备分享”。
- `onLinkInvite()`：改为“点入口 -> 选角色 -> 显示/应用轻量设置 -> 调用 `generateInviteLink` -> 准备分享”，不再跳页。
- `onConfirmRole()` 中按 `inviteType` 分流：
  - `wechat` -> `createWechatInvite`
  - `link` -> `generateInviteLink`
  - 其他邀请类型维持现状

3. `onShareAppMessage` 统一化
- 不再只依赖 `wechatInviteCode`。
- 改为从 `shareMode` 和 `shareInviteCode` 取值。
- 保证当用户刚准备的是 link invite 时，分享出去的就是 link 的 `inviteCode`，避免误用旧微信邀请 code。

4. `list-invite.wxml` 改造
- 将现有 `wechat-share-panel` 升级为通用 `share-panel`。
- 面板内容：
  - 标题：`微信好友邀请已准备` / `邀请链接已准备`
  - 角色信息
  - 若 `shareMode === 'link'`，额外展示：
    - `有效期`
    - `需要审核`
  - 按钮：
    - `发送给微信好友`（`open-type="share"`）
    - `复制邀请链接`（仅 link）
    - `重新生成`（可选，link 更适合保留）
- 在“邀请链接”方式卡片下方或分享面板中加入“高级设置”折叠区。

5. 复制链接能力
- 复用 `generateInviteLink` 返回的 `inviteLink`。
- 在 `list-invite.js` 中新增 `onCopyInviteLink()`。

6. 后端处理
- 首版不改协议，继续复用：
  - `createWechatInvite`
  - `generateInviteLink`
- 可选整理：抽公共 helper 合并两者的“创建公开邀请模板”逻辑，减少重复；但不是首要任务。

7. 独立页面处理策略
- 第一阶段：
  - 保留 `pages/list-invite-link/list-invite-link` 文件与页面注册，避免一次性清理过大。
  - 仅移除从 `list-invite` 到该页的入口。
- 第二阶段：
  - 确认无其他依赖后，从 `app.json` 移除并删除该页面文件。

## 不推荐方案

### 方案 B：彻底砍掉高级设置
做法：点击“邀请链接”时也只生成和微信好友邀请一致的默认邀请，只支持角色选择。

问题：
- 会丢掉 `needApproval` 和 `expireDays`。
- 这与已确认的产品决策不符。

## 具体实施步骤
1. 重构 `pages/list-invite/list-invite.js`
- 引入统一分享状态模型，避免 `wechatInvite*` 与未来 `linkInvite*` 双套状态冲突。
- 保留最小范围改动，不动搜索用户、最近协作、待处理邀请逻辑。

2. 改造“邀请链接”入口
- `onLinkInvite()` 改为当前页内流程。
- 在 `inviteType === 'link'` 时，使用当前选择的：
  - `selectedRole`
  - `linkDraftExpireDays`
  - `linkDraftNeedApproval`
  调用 `generateInviteLink`。

3. 改造 `list-invite.wxml`
- 将微信分享面板改成统一分享面板。
- 增加 link 的轻量高级设置 UI。
- 保持页面结构简洁，避免复制 `list-invite-link` 全页的重 UI。

4. 可复用/可迁移的旧页面逻辑
- 从 `list-invite-link.js` 迁移：
  - `expireDays -> 文案转换`
  - `复制链接`
  - `重新生成链接`
- 不迁移或延后迁移：
  - 小程序码展示
  - 统计卡片
  - 大量独立页面样式

5. 验证接收端兼容性
- 不需改 `list-invite-accept` 主逻辑。
- 仅确认邀请方式文案、审核逻辑、过期逻辑都仍正确。

6. 第二阶段清理（可选）
- 确认 `list-invite-link` 不再被使用后：
  - 从 `app.json` 移除页面注册
  - 删除 `list-invite-link.{js,wxml,wxss,json}`

## 验证清单
1. 从邀请页发起微信好友邀请，分享卡片仍可正常打开邀请接受页。
2. 从邀请页发起邀请链接，不跳新页，能直接：
- 分享给微信好友
- 复制链接
3. 链接打开后，`list-invite-accept` 能正确展示：
- 邀请方式
- 角色
- 是否需审核
4. 修改链接配置后，旧 link invite 失效，新 link invite 生效。
5. 邀请管理页中的 inviteType 展示仍正确（微信/链接/搜索）。
6. 未登录用户、已是成员、已申请、已拒绝、已过期等状态不回退。

## 建议的实现顺序
1. 在 `list-invite` 中完成统一分享状态改造
2. 接入 link invite 的页内生成与复制
3. 添加 link 的轻量高级设置区
4. 调整通用分享面板文案与按钮
5. 自测邀请接受流程
6. 视结果决定是否删除 `list-invite-link` 页面

## 执行注意点
1. `onShareAppMessage` 必须统一使用当前激活的分享状态，避免把旧邀请码分享出去。
2. `generateInviteLink` 当前每次改设置都会使旧链接失效；页内设置变更时要避免无意义重复生成，建议只有“确认邀请”或“重新生成”时才真正调用云函数。
3. 首版优先简化流程，不必强行把二维码、小程序码、统计面板也塞回邀请页，避免页面过重。
4. 若后续删除 `list-invite-link` 页面，需要同步检查 `app.json` 和任何残留跳转引用。
