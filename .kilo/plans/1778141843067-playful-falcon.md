# 清单模块权限收敛修改方案

## 背景
用户已确认第 1 个问题不作为缺陷处理：系统默认通过微信 `openid` 识别用户，资料页仅用于补充信息。

本次仅处理第 2 个问题：限制清单编辑权限，避免 `role = 2` 的编辑者修改 `isShared`、`visibility`，以及触发共享/个人切换带来的成员清理与邀请清理。

## 目标
1. 后端强制收紧 `updateList` 权限边界。
2. 前端按角色隐藏或禁用不允许编辑的字段，避免误操作。
3. 保持创建者 `role = 1` 的现有能力不变。
4. 保持编辑者 `role = 2` 仍可修改安全字段：`name`、`description`、`color`。

## 影响范围
- `cloudfunctions/listFunctions/index.js`
- `miniprogram/pages/list-edit/list-edit.js`
- `miniprogram/pages/list-edit/list-edit.wxml`
- `miniprogram/pages/checklist/checklist.js`
- `miniprogram/pages/checklist/checklist.wxml`
- `miniprogram/pages/list-detail/list-detail.js`

## 现状问题
### 后端
`updateList` 当前允许 `role = 1` 和 `role = 2` 统一修改：
- `name`
- `description`
- `isShared`
- `visibility`
- `color`

这会导致编辑者可：
- 把共享清单改成个人清单
- 触发移除其他成员
- 清理待处理邀请

### 前端
前端当前有两套编辑入口，且都会让编辑者提交敏感字段：
- `checklist` 长按菜单允许 `myRole === 2` 看到“编辑清单”，并通过本页弹窗 `openEditDialog` / `onDialogConfirm` 直接提交 `isShared`、`visibility`
- `list-detail` 详情页通过 `onEditList()` 跳转 `list-edit`
- `list-edit` 页面默认展示全部字段，保存时无差别提交 `name`、`description`、`isShared`、`visibility`、`color`

这意味着即使只改后端，前端仍会暴露错误操作入口，用户提交后才会失败，体验较差；如果只改 `list-edit`，`checklist` 弹窗编辑仍会绕过前端限制。

## 修改策略

### 一、后端作为最终权限兜底
在 `cloudfunctions/listFunctions/index.js` 的 `updateList` 中做字段级权限控制。

#### 方案
1. 保留基础访问限制：仅 `role = 1` 或 `role = 2` 可调用更新。
2. 新增敏感字段判断：
   - `isShared`
   - `visibility`
3. 当调用者不是创建者时：
   - 若请求中包含 `isShared` 且值与当前值不同，直接拒绝
   - 若请求中包含 `visibility` 且值与当前值不同，直接拒绝
4. 编辑者允许继续修改：
   - `name`
   - `description`
   - `color`
5. 共享/个人切换相关的后续逻辑，仅在创建者成功提交 `isShared` 变更时执行。

#### 推荐实现细节
- 先读取当前 `list`
- 计算：
  - `const canManageStructure = myRole === 1`
  - `const nextIsShared = isShared !== undefined ? isShared : list.isShared`
  - `const nextVisibility = visibility !== undefined ? visibility : list.visibility`
- 若 `!canManageStructure` 且请求试图改变结构字段，则返回明确错误：
  - `仅创建者可修改清单类型或可见性`
- 只有在 `canManageStructure` 条件下才写入 `updateData.isShared` / `updateData.visibility`
- 只有在 `canManageStructure` 且 `isShared` 实际发生变化时才执行成员/邀请清理逻辑

#### 这样处理的好处
- 防止前端绕过
- 防止旧版本小程序继续提交敏感字段
- 权限规则集中在云函数，安全边界明确

### 二、前端两个编辑入口都按权限收敛字段

#### A. `list-edit` 页面
在 `list-edit` 页面基于当前用户角色限制可编辑字段。

#### 方案
1. `loadListData` 从 `getListDetail` 返回值中读取：
   - `listInfo.creatorId`
   - `myRole`
2. 页面新增权限状态，例如：
   - `myRole`
   - `isCreator`
   - `canEditStructure`
3. 当 `canEditStructure === false` 时：
   - 不展示“清单类型”切换
   - 不展示“可见性”切换
4. 保存时，若不是创建者：
   - 请求参数中不传 `isShared`
   - 请求参数中不传 `visibility`
   - 仅提交 `name`、`description`、`color`
5. 若该页存在“创建模式”，创建模式不受此限制，因为创建者就是当前用户。

#### B. `checklist` 页弹窗编辑
在 `checklist` 页面为弹窗编辑增加同样的字段级权限状态，确保本页弹窗不能绕过限制。

#### 方案
1. `openEditDialog(listId)` 打开弹窗时，从当前 `list` 记录中同步权限态，例如：
   - `editingListRole`
   - `editingListCreatorId`
   - `canEditStructure`
2. 继续保留 `role = 2` 的“编辑清单”入口，因为其仍可修改安全字段。
3. 当 `canEditStructure === false` 时：
   - 弹窗内不展示“清单类型”单选
   - 弹窗内不展示“可见性”单选
4. 弹窗确认保存时，若是非创建者编辑：
   - `updateList` 请求中不传 `isShared`
   - `updateList` 请求中不传 `visibility`
   - 仅提交 `name`、`description`
5. 创建弹窗仍保持原逻辑，创建场景继续允许选择 `isShared`、`visibility`。

#### 说明
前端不传敏感字段不是安全措施本身，而是为了：
- 降低误操作概率
- 避免用户看到会失败的控件
- 与后端规则保持一致

### 三、入口行为保持一致

#### checklist 页
已确认长按菜单当前逻辑为：
- 创建者通过 `isCreator` 显示“编辑清单”
- 编辑者通过 `myRole === 2` 显示“编辑清单”

本次不移除该入口，只收敛弹窗内字段和提交参数。

#### list-detail 页
已确认详情页加载后会设置：
- `canEdit: myRole === 1 || myRole === 2`
- `canDelete: myRole === 1`
- `canManageMembers: myRole === 1`

因此详情页编辑入口展示逻辑已基本符合目标，不需要新增角色规则。

推荐补一个轻量本地兜底：
- `onEditList()` 内先判断 `canEdit`
- 若无权限则 `toast` 并直接返回

这样可以避免模板条件失效或旧状态下误跳转。

## 具体实施步骤
1. 修改 `cloudfunctions/listFunctions/index.js`
2. 在 `updateList` 中读取当前 `list` 后计算：
   - `isCreator`
   - `myRole`
   - `canManageStructure = myRole === 1`
   - `nextIsShared`
   - `nextVisibility`
3. 对 `name`、`description`、`color` 维持现有可编辑范围。
4. 对 `isShared`、`visibility` 增加字段级权限限制：
   - 非创建者仅在“未改变值”的情况下可忽略传入
   - 非创建者若试图改值，直接返回 `仅创建者可修改清单类型或可见性`
5. 仅在 `canManageStructure` 时写入 `updateData.isShared` / `updateData.visibility`。
6. 仅在创建者实际改变 `isShared` 时执行：
   - 共享转个人时的成员清理
   - 邀请清理
   - 个人转共享时的创建者成员补录
7. 修改 `miniprogram/pages/list-edit/list-edit.js`
8. 在编辑模式加载完成后保存权限状态：`myRole` / `isCreator` / `canEditStructure`。
9. 修改 `onSave()` 参数构造：
   - 创建模式仍提交 `isShared`、`visibility`
   - 编辑模式且 `canEditStructure === false` 时只提交 `name`、`description`、`color`
10. 修改 `miniprogram/pages/list-edit/list-edit.wxml`
11. 当 `isEditing && !canEditStructure` 时隐藏：
   - 清单类型模块
   - 可见性模块
12. 修改 `miniprogram/pages/checklist/checklist.js`
13. 在页面 `data` 中补充弹窗编辑权限态，例如：
   - `editingListRole`
   - `editingListCreatorId`
   - `canEditStructure`
14. `openEditDialog()` 根据当前列表项的 `myRole` 与 `creatorId` 设置上述状态。
15. `onDialogConfirm()` 在编辑模式下按 `canEditStructure` 构造请求：
   - 创建者提交 `name`、`description`、`isShared`、`visibility`
   - 编辑者仅提交 `name`、`description`
16. 维持创建弹窗现有创建参数不变。
17. 修改 `miniprogram/pages/checklist/checklist.wxml`
18. 编辑模式且 `!canEditStructure` 时隐藏：
   - 清单类型单选
   - 可见性单选
19. 检查 `miniprogram/pages/list-detail/list-detail.js`
20. 保持现有 `canEdit` 展示逻辑；视实现复杂度补充 `onEditList()` 本地权限兜底。
21. 回归检查 `checklist` 弹窗编辑与 `list-edit` 独立页在 `role = 2` 下的行为一致性。

## 建议修改点

### 后端 `updateList`
- 当前风险点是 `updateData` 在权限分流前就无条件接收 `isShared`、`visibility`。
- 建议改成先判断“是否请求了敏感字段变更”，再决定是否写入：

```js
const canManageStructure = myRole === 1;
const nextIsShared = isShared !== undefined ? isShared : list.isShared;
const nextVisibility = visibility !== undefined ? visibility : list.visibility;
const isChangingShared = isShared !== undefined && isShared !== list.isShared;
const isChangingVisibility = visibility !== undefined && visibility !== list.visibility;

if (!canManageStructure && (isChangingShared || isChangingVisibility)) {
  return { code: -1, message: '仅创建者可修改清单类型或可见性' };
}

if (canManageStructure && isShared !== undefined) {
  updateData.isShared = isShared;
}

if (canManageStructure && visibility !== undefined) {
  updateData.visibility = visibility;
}
```

### 前端 `list-edit`
- 不新增接口，直接复用 `getListDetail` 已返回的 `myRole` 和 `listInfo.creatorId`。
- 创建模式默认 `canEditStructure = true`，避免误伤创建流程。
- 事件 `onTypeChange` / `onVisibilityChange` 继续保留当前 `dataset` 写法，不要切成别的事件模型。

### 前端 `checklist`
- 该页当前弹窗没有颜色字段，因此本次不需要为该弹窗额外引入 `color` 控件。
- 由于长按菜单里创建者判断使用 `userInfo._id || userInfo.openid`，而 `creatorId` 真实更接近用户表 `_id`，这里不建议再扩散本地创建者判断逻辑；编辑权限最终以后端和 `list.myRole` 为准。
- 方案上更稳的是：
  - 菜单入口维持现状
  - 弹窗内部按 `list.myRole === 1` 决定 `canEditStructure`
  - 提交参数按 `canEditStructure` 收敛

## 验证方案
### 用例 1：创建者编辑共享清单
预期：
- 可修改名称、描述、颜色、类型、可见性
- 从共享改为个人时，成员与待处理邀请按现有逻辑清理

### 用例 2：编辑者编辑共享清单
预期：
- 可进入编辑页
- 在 `list-edit` 页只能修改名称、描述、颜色
- 在 `checklist` 弹窗只能修改名称、描述
- 页面不显示或不可操作类型/可见性控件
- 提交成功
- 不会触发成员或邀请变更

### 用例 3：编辑者通过旧前端或手工构造请求提交 `isShared`/`visibility`
预期：
- 云函数拒绝
- 返回明确错误信息
- 数据库中清单类型、成员、邀请均不变化

### 用例 4：查看者 `role = 3`
预期：
- 不出现编辑入口，或进入编辑时被后端拒绝

### 用例 5：个人清单编辑
预期：
- 创建者可正常改为共享
- 编辑者若存在该场景，也不能更改类型/可见性

## 风险点
1. `list-edit` 页面如果被其他页面复用，需要确认创建模式和编辑模式都兼容。
2. `list-detail` 页可能也有独立的权限变量，需要避免前端多个页面规则不一致。
3. 若某些老数据的 `visibility` 缺失，需要沿用现有默认值逻辑，不要在本次改动中顺带重构。
4. `checklist` 弹窗和 `list-edit` 页面字段并不完全一致，回归时要分别验证两个入口，而不是只测其中一个。

## 本次不处理
1. `openid` 登录态与资料完善态的文案区分
2. 搜索分页优化
3. 清单编辑入口统一为弹窗或独立页的交互重构

## 预期结果
修改完成后：
- 创建者仍拥有完整清单结构管理能力
- 编辑者仅能修改安全字段
- 非创建者无法再通过任何路径把共享清单改为个人清单，或修改可见性，或触发成员/邀请清理
- 前后端权限规则一致
