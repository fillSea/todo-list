# 任务完成状态切换性能优化方案

## 目标

降低用户点击“完成/取消完成”后的体感延迟，并减少云函数在状态切换链路中的同步数据库操作。优先保证交互即时反馈，其次降低后端真实耗时，最后避免重复刷新和缓存失效带来的额外开销。

## 现状瓶颈

1. 前端多数页面等待 `taskFunctions.toggleTaskStatus` 返回后才更新 UI，用户感知慢。
2. 周期任务切换成功后，`taskToggle.js` 会因为 `resultData.newPeriodicTasks || resultData.isRepeatTask` 触发 `reloadTasks()`，导致状态切换后又完整拉取列表。
3. `list-detail` 页面切换成功后执行 `app.clearTaskCaches()` 和 `loadListDetail(false)`，不是局部更新。
4. 云函数 `toggleTaskStatus` 同步串行执行任务查询、权限校验、状态更新、操作日志、周期实例补齐、清单版本更新和清单统计全量重算。
5. `touchListVersion()` 内部调用 `recalculateListStats()`，每次清单任务状态变化都会读取该清单全部任务并重新计算统计。
6. `touchListVersion()` 更新 `list_versions` 后可能触发前端实时监听，再次 `loadTasks({ forceRefresh: true })`，造成重复刷新。

## 优化原则

1. 状态切换的主链路只保证当前任务状态写入成功。
2. 用户点击后尽快本地更新 UI，失败时回滚。
3. 统计、日志、周期实例补齐、版本广播等非关键工作尽量不阻塞主响应。
4. 优先做最小改动，保持现有逾期确认、非当天周期任务限制、权限校验逻辑不变。
5. 避免破坏任务缓存行为，因为缓存是当前业务逻辑的一部分。

## 阶段一：前端乐观更新，解决体感慢

### 1. 改造 `miniprogram/utils/taskToggle.js`

在 `handleTaskStatusToggle()` 中增加乐观更新流程：

1. 保留现有本地前置确认逻辑：
   - 逾期任务完成前确认。
   - 日历页非当天周期任务拦截。
   - 取消完成仍需根据后端返回确认。
2. 调用云函数前先执行 `updateLocalTaskStatus(newStatus)`。
3. 记录旧状态，用于失败或取消确认时回滚。
4. 云函数返回失败时执行 `updateLocalTaskStatus(oldStatus)` 并提示错误。
5. 后端返回 `needConfirmCompleteNotToday` 时回滚，并按现有逻辑提示或跳转。
6. 后端返回 `needConfirmUncheck` 时先回滚到旧状态，用户确认后再次调用；确认成功后再本地切到 `newStatus`。
7. 周期任务成功后不要阻塞等待 `reloadTasks()`，改为先保持本地状态，后台触发刷新或延迟刷新。

建议新增可选参数：

```js
optimistic = true
silentRefreshAfterSuccess
```

其中 `silentRefreshAfterSuccess` 用于周期任务或复杂状态成功后后台刷新，不阻塞 checkbox 响应。

### 2. 首页 `pages/index/index.js`

当前已提供 `updateLocalTaskStatus`，可复用。需要补充：

1. 更新 `_rawTasks` 后同步更新缓存中的当前任务状态，避免页面切换后短时间内读到旧缓存。
2. 对搜索结果 `searchResults` 也做同样局部更新，否则搜索模式下可能状态不同步。
3. 周期任务成功后可以触发一次非阻塞 `loadTasks({ forceRefresh: true })`，但不要 `await`。

### 3. 日历页 `pages/calendar/calendar.js`

当前已有 `updateLocalTaskStatus(taskId, status)`。需要确认该方法同时更新：

1. `_rawTasks`。
2. `selectedDateTasks`。
3. 日历索引与当天列表。
4. 月份缓存 `calendarTasks_${userId}_${year}_${month}`。

如果缓存未更新，需要补齐局部写缓存逻辑，避免状态切换后又从旧缓存恢复。

### 4. 清单详情页 `pages/list-detail/list-detail.js`

将状态切换成功后的：

```js
app.clearTaskCaches();
this.loadListDetail(false);
```

改为局部更新：

1. 更新 `taskList` 中对应任务的 `status`、`completedAt`、`completedBy`。
2. 重新计算 `completedCount`、`progressPercent`、`progressColor`。
3. 调用 `applyTaskFilter()` 更新当前筛选列表。
4. 只在周期任务、权限变化、后端返回特殊标记时后台刷新清单详情。
5. 保留 `DEBUG_MODE` 路径一致行为，避免只优化云路径。

## 阶段二：后端主链路瘦身，降低真实耗时

### 1. 拆分 `toggleTaskStatus()` 的关键与非关键工作

关键路径保留：

1. `getUserId(openid)`。
2. 查询任务。
3. 权限校验。
4. `evaluateTaskStatusTransition()` 业务规则。
5. 更新当前任务状态。
6. 返回当前任务新状态。

非关键路径改为不阻塞响应：

1. `recordOperation()`。
2. `ensurePeriodicTasks()`。
3. `touchListVersion()`。
4. `recalculateListStats()`。

可选实现方式：

1. 最小改动：状态更新成功后先构造返回值，然后使用 `Promise.allSettled([...])` 触发非关键任务，但不 `await`。需要评估云函数运行结束后异步任务是否可靠完成。
2. 更稳妥：新增轻量后台动作，例如 `postTaskStatusChanged`，由前端成功后非阻塞调用，或者由定时任务补偿处理。
3. 最可靠：引入队列/事件集合，例如写入 `task_status_events`，再由定时云函数消费。改动较大，可作为后续阶段。

推荐本项目先采用“状态切换接口立即返回 + 前端非阻塞调用后处理接口”的折中方案，避免云函数提前结束导致未 await 任务被中断。

### 2. 新增后处理接口

在 `taskFunctions` 增加内部动作，例如：

```js
action: 'afterTaskStatusChanged'
```

参数：

```js
{
  taskId,
  oldStatus,
  newStatus,
  listId,
  repeatType,
  parentTaskId
}
```

职责：

1. 重新读取任务，校验权限或使用当前 openid 获取 userId。
2. 写操作日志。
3. 完成周期任务时执行 `ensurePeriodicTasks()`。
4. 更新清单版本。
5. 必要时重算清单统计。

前端在 `toggleTaskStatus` 成功后 fire-and-forget 调用该动作，不影响 UI。

### 3. 避免 `ensurePeriodicTasks()` 重复 `getUserId()`

当前 `toggleTaskStatus()` 已获取 `userId`，但 `ensurePeriodicTasks(openid, completedTask)` 内部又调用 `getUserId(openid)`。

优化为：

```js
ensurePeriodicTasks(userId, completedTask)
```

减少一次用户查询。

### 4. 周期实例批量创建优化

`ensurePeriodicTasks()` 当前循环中逐条：

```js
await db.collection('tasks').add(...)
```

可改为并发受控或批量写入：

1. 生成 `tasksToCreate` 数组。
2. 使用 `Promise.all()` 并发 add，数量上限当前最多 20，可接受。
3. 或使用数据库批量能力，如果当前云开发 SDK 支持。

注意返回的 `newPeriodicTasks` 只需要 `_id` 和 `dueDate`，并发 add 后再组装。

## 阶段三：清单统计优化

### 1. 短期方案：状态切换时不阻塞统计重算

把 `recalculateListStats()` 从 `toggleTaskStatus()` 主链路移出。即使统计短时间延迟，任务状态本身应立即生效。

### 2. 中期方案：普通任务增量更新统计

对非周期任务可以直接根据状态变化增量更新：

1. `0 -> 1`：`completedCount + 1`，`pendingCount - 1`。
2. `1 -> 0`：`completedCount - 1`，`pendingCount + 1`。
3. `taskCount` 不变。
4. `progress` 根据新值重算。

周期任务和逾期展示规则较复杂，仍可走后台全量重算。

### 3. 长期方案：统计字段来源统一

当前前端、云函数都存在周期任务展示过滤和统计逻辑。长期建议抽取统一的统计计算规则，至少保证：

1. 首页展示规则。
2. 清单详情统计。
3. 后端 `calculateListTaskStats()`。

三者对周期任务、逾期任务、已完成任务的计数一致。

## 阶段四：减少重复刷新和缓存抖动

### 1. 前端刷新防抖

对 `handleRealtimeListChange()` 加短时间防抖，例如 500-1000ms，避免连续版本事件触发多次 `loadTasks({ forceRefresh: true })`。

### 2. 忽略本机刚提交事件

在状态切换成功时记录：

```js
this._recentLocalTaskStatusChangeAt = Date.now();
```

实时监听收到 `task_status_change` 且距离本机提交很近时，可以跳过或延迟合并刷新。

### 3. 缓存局部更新优先于清空

状态切换成功后，不要默认 `app.clearTaskCaches()`。改为：

1. 首页任务缓存局部 patch。
2. 日历月份缓存局部 patch。
3. 清单详情本地列表局部 patch。
4. 只有任务结构变化、周期实例新增、删除任务、编辑截止日期等影响列表归属的操作才清缓存。

## 验证方案

### 1. 手工验证路径

1. 首页普通任务：完成、取消完成，UI 应立即变化，刷新后状态正确。
2. 首页逾期任务：弹确认框，取消时状态不变，确认后立即变化。
3. 首页周期任务：当天任务可完成，非当天任务仍提示不可完成或跳转。
4. 日历页普通任务：完成后选中日期列表和日历计数正常。
5. 日历页周期任务：当天完成后 UI 立即变化，后台刷新后未来实例正常。
6. 清单详情页普通任务：完成后进度条和筛选列表立即更新。
7. 清单详情页查看者权限：仍不可修改。
8. 取消已完成任务：保留确认逻辑，取消确认时回滚状态。

### 2. 性能日志验证

在云函数临时增加分段耗时日志，确认优化前后耗时：

1. 查询任务耗时。
2. 权限校验耗时。
3. 更新状态耗时。
4. 操作日志耗时。
5. 周期补齐耗时。
6. 清单统计重算耗时。
7. 总耗时。

优化目标：

1. 前端点击后 100ms 内有视觉状态变化。
2. `toggleTaskStatus` 主接口仅包含必要逻辑后，普通任务云函数耗时显著下降。
3. 清单任务不再因为清单任务总数增加而线性拖慢主响应。

### 3. 回归风险

1. 乐观更新失败回滚不完整，可能造成本地与云端短暂不一致。
2. 周期任务后台补齐延迟，可能导致刚完成后未来实例短暂不可见。
3. 清单统计延迟更新，可能导致其他成员短时间看到旧进度。
4. 实时监听去重过度可能漏掉其他成员的更新，需要只忽略极短时间窗口内的本机事件。

## 推荐实施顺序

1. 实施 `taskToggle.js` 乐观更新和失败回滚。
2. 改造首页、日历、清单详情的局部状态更新与缓存 patch。
3. 将 `list-detail` 状态切换成功后的完整刷新改为局部更新。
4. 增加耗时日志，在真实环境验证主要后端耗时。
5. 拆出 `afterTaskStatusChanged` 后处理接口，把日志、周期补齐、清单版本/统计从主链路移出。
6. 优化 `ensurePeriodicTasks()`，减少重复查询并并发创建实例。
7. 对 `handleRealtimeListChange()` 增加防抖和本机事件合并。
8. 评估是否将清单统计改成增量更新或后台定时重算。

## 第一轮建议落地范围

第一轮只做低风险改动：

1. 前端乐观更新。
2. 普通任务不完整 reload。
3. 清单详情局部更新。
4. 周期任务成功后后台刷新，不阻塞 UI。
5. 加云函数分段耗时日志用于确认后端瓶颈。

暂不在第一轮引入事件队列或大规模统计重构，避免一次改动过大。
