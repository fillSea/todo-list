# 登录功能分析与退出登录方案

## 一、现状梳理

### 1. 登录的真实含义
该项目当前没有传统意义上的服务端会话登录，也没有 token / session。

实际“登录态”由两部分组成：
1. 微信云开发天然身份：云函数通过 `cloud.getWXContext().OPENID` 识别用户。
2. 前端本地状态：`userInfo`、`isLoggedIn`、`loginTime` 存在本地 storage 中，用来决定页面是否把用户当作“已登录/已完成资料”。

因此目前更准确地说，这是“用户资料注册 + 本地登录标记”，不是完整账号系统。

### 2. 核心链路

#### 启动阶段
- `miniprogram/app.js`
- `onLaunch -> checkLoginStatus()`
- 读取本地 `userInfo`、`isLoggedIn`、`loginTime`
- 若 30 天过期则调用 `clearLoginStatus()`
- 未过期则更新 `globalData`，再调用 `syncUserInfo()` 从云端拉最新资料

#### 注册/登录页
- `miniprogram/pages/register/register.js`
- 通过头像 + 昵称 + 通知开关提交 `profileFunctions.registerOrUpdateUser`
- 成功后写入：
  - `userInfo`
  - `isLoggedIn = true`
  - `loginTime`
- 并更新 `app.globalData`

#### 我的页
- `miniprogram/pages/profile/profile.js`
- 每次 `onLoad/onShow` 从 storage 读取 `isLoggedIn/userInfo`
- 已登录则继续拉：
  - 用户资料
  - 统计数据
  - 看板数据
  - 未读通知数

#### 其他依赖登录态的页面
- `pages/index/index.js`：依赖 `isLoggedIn` 决定是否拉任务，且读取 `cachedTasks/cachedCategories`
- `pages/calendar/calendar.js`：依赖 `isLoggedIn` 决定是否拉任务，且读取 `calendarTasks_${year}_${month}`、`cachedCategories`
- `pages/checklist/checklist.js`：页面初始化直接请求 `getMyLists`，没有前置登录拦截
- 多个清单协作页面直接读取 `userInfo`

### 3. 已存在但不完整的退出逻辑
- `miniprogram/pages/register/register.js` 已有 `logout()`
- 但页面 UI 中没有看到明确可用入口
- 且只清理了：
  - `userInfo`
  - `isLoggedIn`
  - `loginTime`
- 没有统一清理任务/分类/日历/通知设置等缓存，也没有清理 TabBar 红点

## 二、当前登录体验上的问题

### 1. “登录”概念不清晰，用户理解成本高
表现：
- 页面文案写的是“登录”“已登录”，但实际行为是“完善个人信息”。
- 用户并没有经历明显的授权登录或账号切换流程，只是填写头像昵称后被标记为已登录。

影响：
- 用户会误以为这是完整账号登录。
- 退出登录后仍会被同一微信身份识别，容易产生“为什么退出后还能识别我”的认知落差。

涉及位置：
- `pages/register/register.wxml`
- `pages/register/register.js`
- `pages/profile/profile.js`

### 2. 登录入口弱，首次使用路径不清晰
表现：
- “我的”页点击用户卡片才进入资料页。
- 其他主 Tab 页未登录时并没有明显 CTA 引导去完善资料。
- 首页/日历页未登录时大多只显示空数据，不说明原因。

影响：
- 用户会以为“系统没有数据”而不是“我还没完成登录/资料初始化”。
- 降低首次转化率。

涉及位置：
- `pages/profile/profile.wxml`
- `pages/index/index.wxml`
- `pages/calendar/calendar.wxml`
- `pages/checklist/checklist.wxml`

### 3. 未登录态与空状态混淆
表现：
- 首页未登录时显示“暂无任务”风格内容。
- 日历未登录时显示“该日期暂无任务”。
- 清单页即使未登录也会直接请求云函数，失败后只 toast “加载失败，请重试”。

影响：
- 用户无法判断是数据为空、网络失败，还是未登录。
- 交互反馈不准确。

### 4. 登录态判断分散，页面行为不一致
表现：
- `app.js` 有 `checkLoginStatus/clearLoginStatus/setLoginStatus`
- `profile.js`、`register.js`、`index.js`、`calendar.js` 都各自重复读取 storage
- `checklist.js` 基本没有显式登录拦截

影响：
- 退出登录后各页面刷新行为可能不一致。
- 后续新增页面容易漏掉登录校验。

### 5. 本地缓存与登录态解绑不彻底，退出后容易残留旧数据
表现：
- 任务缓存：`cachedTasks`
- 分类缓存：`cachedCategories`
- 日历缓存：`calendarTasks_${year}_${month}`
- 通知设置缓存：`notificationSettings`
- 我的页还会设置 TabBar badge

影响：
- 退出后重新进入首页/日历，可能短暂看到上一个账号的数据残影。
- 视觉上像“没有真正退出”。

### 6. 过期处理静默发生，体验突兀
表现：
- `app.js` 在 30 天过期后直接清理本地状态
- 没有 toast / 弹窗提示“登录已过期，请重新完善资料”

影响：
- 用户只会发现状态突然丢失，不知道原因。

### 7. 注册页已有退出能力，但入口位置不合理
表现：
- 退出功能放在 `register.js`，而不是主要账户中心“我的”页。
- 若用户不知道这个页存在“退出”，就几乎找不到该能力。

影响：
- 退出操作可发现性很差。

### 8. 弱网降级逻辑会写入不完整登录态
表现：
- `register.js` 提交失败时会走本地兜底：写入 `userInfo` 和 `isLoggedIn`
- 但异常分支没有写 `loginTime`
- 且可能产生只有本地资料、云端并未成功注册的状态

影响：
- 登录状态定义不一致。
- 30 天过期判断可能失效或行为异常。
- 后续某些云函数可能按 openid 查不到预期资料。

## 三、退出登录功能的实现目标

### 目标
1. 让退出入口在“我的”页可见、易理解。
2. 退出后彻底清掉本地登录态与用户相关缓存。
3. 退出后所有 Tab 页展示明确的未登录态，而不是旧数据或空数据混淆。
4. 不修改云端用户数据，不做服务端“注销账号”，只做本地退出。
5. 为后续统一登录态管理预留一个中心化方法。

### 非目标
1. 不做微信账号切换能力。
2. 不做删除账号。
3. 不做服务端 token/session 登出，因为当前架构没有这层。

---

## 四、推荐实现方案

### 方案原则
优先做“小改动但闭环完整”的实现：
- 复用 `app.js` 作为登录态中心
- 在“我的”页增加退出入口
- 将原本散落的清理逻辑统一成一个全局方法
- 补齐几个主 Tab 的未登录态展示和刷新逻辑

### 方案步骤

#### 1. 在 `app.js` 中集中封装退出方法
建议新增统一方法，例如：
- `logout(options = {})`
- 或扩展现有 `clearLoginStatus()`，让它负责“清理全部本地用户态”而不是只清 3 个 key

建议统一清理内容：
- `userInfo`
- `isLoggedIn`
- `loginTime`
- `cachedTasks`
- `cachedTasksTime`
- `cachedCategories`
- `notificationSettings`
- `jumpToDate`（可选，一并清掉）
- `calendarTasks_${year}_${month}` 这类按月份缓存

同时执行：
- `this.globalData.userInfo = null`
- `this.globalData.isLoggedIn = false`
- `wx.removeTabBarBadge({ index: 3 })`

说明：
- 日历缓存 key 是动态前缀，不能只删固定 key，需遍历 storage keys 后清理前缀为 `calendarTasks_` 的项。
- 这样能避免退出后重新打开仍看到旧任务。

#### 2. “我的”页增加退出登录入口
修改：
- `miniprogram/pages/profile/profile.wxml`
- `miniprogram/pages/profile/profile.js`

推荐交互：
- 在功能列表下方增加一个明显但不过度突兀的按钮/单元格：`退出登录`
- 点击后弹出二次确认：
  - 标题：`确认退出`
  - 内容：`退出后将清除当前设备上的登录状态和本地缓存，不会删除云端数据。`

确认后执行：
- 调用 `app.logout()` 或 `app.clearLoginStatus({ clearCache: true })`
- 当前页立即 `setData` 重置：
  - `isLogin: false`
  - `userInfo: null`
  - 统计/看板/通知角标清零
- 视情况 `wx.switchTab({ url: '/pages/profile/profile' })` 或停留当前页刷新未登录态
- toast：`已退出登录`

说明：
- 退出入口应放在“我的”页，不建议继续藏在 `register` 页。

#### 3. 注册页复用统一退出逻辑或移除重复实现
修改：
- `miniprogram/pages/register/register.js`

做法：
- 不再保留独立的手写 storage 清理逻辑
- 若仍保留该页面的退出入口，则直接调用 `app.logout()`

收益：
- 避免两套退出逻辑未来不一致。

#### 4. 为主 Tab 页面补齐“未登录态”展示
优先覆盖：
- `pages/index/index`
- `pages/calendar/calendar`
- `pages/checklist/checklist`
- `pages/profile/profile`

建议表现：
- 明确提示：`登录后可查看你的任务/清单/统计数据`
- 提供按钮：`去完善资料`
- 点击跳转 `/pages/register/register`

细节建议：
- `index`：未登录时不显示“暂无任务”，而显示专属未登录空态
- `calendar`：未登录时不显示“该日期暂无任务”，而显示登录引导
- `checklist`：未登录时不要直接请求 `getMyLists`，先显示登录引导
- `profile`：未登录时用户卡片文案应改成“点击完善个人资料”而不是假装已有账户卡片

#### 5. 调整页面 onShow/checkLoginStatus 退出后的重置行为
重点页面：
- `profile.js`
- `index.js`
- `calendar.js`
- `checklist.js`

建议：
- 当检测到 `isLoggedIn = false` 时，不只是不加载云端数据，还要主动把当前页面数据清空/重置。

例如：
- `profile`：清空 `statistics`、`dashboardData`、`unreadCount`、功能角标
- `index`：清空 `tasks/inProgressTasks/overdueTasks/completedTasks/category*Tasks`，分类恢复到默认 `全部`
- `calendar`：清空 `tasks/selectedDateTasks/inProgressTasks/overdueTasks/completedTasks`，分类恢复默认
- `checklist`：清空 `lists/filteredLists`，空态改成登录引导

原因：
- 仅删除 storage 不足以让当前已渲染页面立即消失旧数据。

#### 6. 统一登录态帮助方法，减少重复读取 storage
可选但推荐的小优化：
- 在 `app.js` 增加 `getLoginState()`，统一返回：
  - `isLoggedIn`
  - `userInfo`
  - `loginTime`

页面层调用统一方法，而不是每个页面自己读多个 storage key。

收益：
- 降低状态不一致风险。
- 为后续扩展“登录过期提示”“首次登录引导”提供统一出口。

#### 7. 优化过期提示
修改：
- `app.js`

建议：
- 30 天过期时，在下次进入相关页面或 App 启动后给予轻提示：`登录状态已过期，请重新完善资料`

注意：
- 小程序 `App.onLaunch` 里直接 toast 不一定体验最佳，可考虑在首个展示页检查到“刚被清理”标记后再提示。
- 这项可以放在第二优先级，不阻塞退出登录功能主链路。

#### 8. 处理注册失败本地兜底逻辑的不一致问题
修改：
- `register.js`

建议至少补齐：
- 异常兜底分支也写入 `loginTime`
- 或更严格：提交失败时不直接标记 `isLoggedIn = true`，而只提示保存失败

从体验和一致性角度，更推荐：
- 若云端保存失败，不要把用户视为真正已登录

但这是行为改动，可能影响当前离线容错策略。
如果后续实现阶段追求最小改动，可以先只补 `loginTime`，并记录为后续优化项。

## 六、推荐的交互设计

### 1. 我的页未登录态
用户卡片区域文案建议改为：
- 主标题：`立即完善资料`
- 副标题：`登录后可同步任务、清单和通知设置`

### 2. 退出按钮样式
建议：
- 放在“我的”页底部功能区之后
- 使用明显但非危险色的大按钮或 cell
- 文案：`退出登录`

### 3. 退出确认文案
建议使用更准确描述：
`退出后将清除当前设备上的登录状态、本地任务缓存和通知角标，云端数据不会被删除。`

这样可降低用户对“数据丢失”的担忧。

## 七、实现时的注意点

1. 不能把“退出登录”理解成删除云端 `users` 记录。
2. 不能只清 `userInfo/isLoggedIn/loginTime`，否则任务/分类/日历缓存会残留。
3. `checklist.js` 现在没有明确登录拦截，退出方案必须覆盖它。
4. `profile` 页的通知 badge 退出时必须主动移除。
5. `calendarTasks_` 是动态 key，需要按前缀清理。
6. 由于多个页面都在 `onShow` 读取 storage，退出后要保证它们能在重新展示时正确重置，而不是继续沿用旧 `data`。
7. 若保留 `register.js` 的退出入口，必须改为调用统一方法，避免分叉。

## 八、验证清单

### 功能验证
1. 已登录状态下进入“我的”页，可以看到退出登录入口。
2. 点击退出，出现确认弹窗。
3. 确认退出后：
   - `userInfo/isLoggedIn/loginTime` 被清除
   - `cachedTasks/cachedCategories/notificationSettings/calendarTasks_*` 被清除
   - “我的”Tab 红点消失
4. 退出后停留在“我的”页，页面立即变为未登录态。
5. 退出后切到“任务”页，不应出现上一个用户的任务列表。
6. 退出后切到“日历”页，不应出现上一个用户的日历任务缓存。
7. 退出后切到“清单”页，不应继续请求用户清单并报模糊错误，应显示登录引导。
8. 再次进入资料页并保存后，登录态恢复正常。

### 边界验证
1. 连续点击退出不会报错。
2. 未登录状态下再次进入“我的”页，不会展示旧昵称/头像。
3. 登录过期后重新打开小程序，不会出现旧缓存数据残留。
4. 若通知 badge 原本存在，退出后会立刻清除。

## 十、结论

当前项目的登录体验问题，核心不在“技术上能不能登录”，而在于：
- 登录语义不清
- 未登录态提示不足
- 退出能力不可发现
- 本地缓存与登录态解绑不彻底

最合适的退出登录实现方式，是把它定义为“本地退出当前用户资料状态”，通过 `app.js` 统一清理登录态与缓存，并在“我的”页提供明显入口，同时补齐三个主业务 Tab 的未登录态展示。

这套方案改动范围可控，且能一次性解决“退出后仍像没退出”的主要体验问题。
