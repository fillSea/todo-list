# 清单成员头像不可见修复方案

## 问题结论

清单详情页和成员管理页能拿到成员记录，但后端返回的 `avatarUrl` 仍是用户表中的原始值。注册流程会把头像上传到云存储并保存为 `cloud://...` fileID，前端 `<image>` 直接渲染该值时，对其他成员头像可能无权限或无法解析，因此只能稳定看到自己的头像。

项目中 `cloudfunctions/listFunctions/index.js` 已有 `resolveAvatarUrl(avatarUrl)`，可以把 `cloud://...` 转成临时访问 URL，但当前清单成员相关接口没有统一使用它。

## 修改目标

- 清单详情页成员头像正常显示。
- 成员管理页成员头像正常显示。
- 清单列表/我的清单里的成员头像摘要也保持一致。
- 不修改数据库字段结构，不改变前端页面结构。

## 修改范围

主要修改 `cloudfunctions/listFunctions/index.js`。

涉及函数：

- `getListMemberSummary(listId)`：清单列表成员头像摘要。
- `getListDetail(openid, data)`：清单详情页成员列表。
- `getListMembers(openid, data)`：成员管理页成员列表。

可选检查函数：

- 邀请/申请列表中使用 `inviteeInfo.avatarUrl`、`applicantInfo.avatarUrl` 的返回逻辑。如果这些页面也展示云存储头像，后续可同样接入 `resolveAvatarUrl()`。

## 实施步骤

1. 在 `getListMemberSummary` 中，将 `memberAvatars` 的构建改为异步 `Promise.all`，对 `user.avatarUrl` 调用 `resolveAvatarUrl()`。

示例目标形态：

```js
const memberAvatars = await Promise.all(members.slice(0, 3).map(async (m) => {
  const user = memberUsers.find(u => u._id === m.userId);
  return {
    avatarUrl: user ? await resolveAvatarUrl(user.avatarUrl) : '',
    nickname: user ? user.nickname : ''
  };
}));
```

2. 在 `getListDetail` 中，将 `members = memberRecords.map(...)` 改为 `await Promise.all(memberRecords.map(async ...))`，返回前将头像转换为临时 URL。

示例目标形态：

```js
members = await Promise.all(memberRecords.map(async (m) => {
  const user = memberUsers.find(u => u._id === m.userId);
  return {
    _id: m._id,
    userId: m.userId,
    role: m.role,
    nickname: user ? user.nickname : '未知用户',
    avatarUrl: user ? await resolveAvatarUrl(user.avatarUrl) : ''
  };
}));
```

3. 在 `getListMembers` 中，将 `membersWithInfo = members.map(...)` 改为异步 `Promise.all`，同样转换 `user.avatarUrl`。

示例目标形态：

```js
const membersWithInfo = await Promise.all(members.map(async (member) => {
  const user = userMap[member.userId] || {};
  return {
    ...member,
    nickname: user.nickname || '未知用户',
    avatarUrl: await resolveAvatarUrl(user.avatarUrl || '')
  };
}));
```

4. 保留 `resolveAvatarUrl()` 当前行为：

- 空值返回 `''`，前端继续使用 `/images/default-avatar.png`。
- `http://` / `https://` 原样返回。
- `cloud://` 调用 `cloud.getTempFileURL()`。
- 非法格式返回 `''`。

5. 不建议前端逐个调用 `wx.cloud.getTempFileURL()`：

- 头像数据来自云函数聚合结果，后端统一转换更集中。
- 可避免多个页面重复写转换逻辑。
- 后续如果头像来源变更，只需改云函数。

## 验证方案

由于该项目主要通过微信开发者工具验证，建议部署 `listFunctions` 后在小程序中验证：

1. 使用共享清单，准备至少 2 个不同用户且头像均为云存储头像。
2. 打开清单详情页，确认顶部成员头像都显示，不再只显示自己。
3. 进入“成员管理”，确认创建者、编辑者、查看者分组中的头像都显示。
4. 返回清单列表/我的清单，确认成员头像摘要也显示。
5. 检查控制台是否还有 `转换头像临时链接失败` 或图片加载失败日志。

## 风险与注意事项

- 临时 URL 有有效期，这是微信云存储的正常行为。页面每次请求云函数时重新获取即可。
- 如果云存储文件权限、文件 ID 已失效或头像文件被删除，转换后仍会为空，前端会显示默认头像。
- `getListDetail` 和 `getListMembers` 成员数通常较少，逐个转换头像性能影响可接受。
- 如果未来成员数较多，可将多个 fileID 批量传给 `cloud.getTempFileURL()` 做优化，但当前优先保持最小改动。
