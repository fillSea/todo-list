# 周期任务统计改造方案

## 目标

修复 `miniprogram/pages/periodic-stats` 的统计失真问题，使周期统计页能够基于真实业务操作生成统计结果，而不是只依赖 `tasks` 表当前快照。

核心目标：

1. 完成一个周期任务后，统计页能立即正确反映该周期点已完成。
2. 取消完成后，统计页能回退对应统计。
3. 多人协作清单下，统计页能明确按照“任务真实状态”统计，必要时同时保留“谁完成的”维度。
4. 最近记录、月度统计、连续完成等指标都来自同一套可解释的数据口径。

## 现状与根因

### 当前前端

周期统计页 `miniprogram/pages/periodic-stats/periodic-stats.js` 通过云函数：

- `getPeriodicTaskStats`
- `getPeriodicTaskMonthDetail`

读取后端聚合结果，再展示：

- 完成率
- 已到期/已完成/未完成
- 最近 7/30 次
- 本月应做/已做/漏做
- 月历点位
- 最近记录

### 当前后端统计口径

`cloudfunctions/taskFunctions/index.js` 中：

- `buildPeriodicStatsPayload`
- `mapOccurrencesWithInstances`
- `getPeriodicTaskStats`
- `getPeriodicTaskMonthDetail`

它的判断方式是：

1. 先根据周期规则生成理论周期点 `occurrences`
2. 再按 `dueDate` 把 `tasks` 实例映射到周期点
3. 如果某个周期点存在 `status === 1` 的实例，则认为该点已完成
4. 否则认为未完成

### 失真根因

当前统计只看 `tasks` 表的“当前状态”，没有可靠记录“完成事件”本身：

1. `toggleTaskStatus` 只更新 `status` 和 `updatedAt`
2. 没有稳定写入 `completedAt`
3. 没有稳定写入 `completedBy`
4. 虽然会写 `operations`，但目前周期统计页完全没有消费这些操作日志

这会带来几个问题：

1. 任务被完成再取消完成时，统计只能看到最终状态，缺少操作轨迹语义。
2. 共享清单中，非创建者可以完成任务，但统计页无法区分“任务已完成”与“是谁完成的”。
3. `recentRecords` 现在本质是“周期点状态列表”，不是“真实最近操作记录”。
4. 后续如果要支持“本月完成次数”“最近完成时间”“完成人分析”，当前数据模型支撑不足。

### 新发现：最近记录更新但统计详情不变

当前代码里，“最近记录”和“统计详情”虽然都来自 `getPeriodicTaskStats`，但底层依赖的数据并不一样：

1. `recentEventRecords` 来自 `operations` 操作日志，只要发生了完成/取消完成事件，就会立即出现。
2. `summary` / `monthlyStats` / `allRecords` 来自 `buildPeriodicStatsPayload`，它依赖理论周期点 `occurrences` 与 `tasks` 实例映射。

这导致当前会出现一个边界问题：

1. 今天完成了一个周期任务实例。
2. `recentEventRecords` 能正常记录这次完成操作，所以“最近记录”立即显示。
3. 但 `buildPeriodicStatsPayload` 使用的 `occurrenceEnd` 是 `todayDate`（UTC+8 当天零点），而不是“今天结束时刻”。
4. `generatePlannedOccurrences` 又保留了任务原始截止时间（例如今天 20:00）。
5. 如果今天的周期点截止时间晚于当天零点，那么这个周期点在 `occurrences` 中不会被生成为“今天的理论点”。
6. 最终 `completedDueCount`、`missedDueCount`、`thisMonthCompleted` 等统计不会把今天刚完成的这一点算进去。

换句话说，当前 bug 不是“最近记录假了”，而是：

**统计详情对“今天”的统计截止边界用了当天零点，导致今天带具体时间的周期点被错误排除。**

## 建议口径

### 统计口径拆分

把周期统计分为两层：

1. 周期点结果统计
2. 操作事件统计

### A. 周期点结果统计

用于：

- 完成率
- 已到期周期点
- 已完成/未完成
- 最近 7 次/30 次完成率
- 月度应做/已做/漏做
- 连续完成
- 月历打点

口径定义：

1. 一个周期点是否“已完成”，以该周期点对应实例的最终 `status` 为准。
2. 一个周期点是否“漏做”，指截至当前统计时点已到期且最终状态仍为未完成。
3. 连续完成按周期点顺序计算，不按操作次数计算。
4. “截至今天应完成的周期点”中的“今天”，必须按业务时区下的**当天结束**判断，而不是当天零点。

这部分仍可继续基于 `tasks` 当前状态快照生成，因为它表达的是“当前结果”。

### B. 操作事件统计

用于：

- 最近记录
- 最近完成时间
- 谁完成了该周期点
- 完成/取消完成次数
- 共享清单下按操作者观察真实行为

口径定义：

1. 每次从 `status:0 -> 1` 记为一次完成事件。
2. 每次从 `status:1 -> 0` 记为一次取消完成事件。
3. 最近记录应展示真实事件流，而不是静态推导结果。

## 推荐改造方案

### 方案选择

采用“最小但正确”的方案：

1. 保留现有基于周期点的结果统计框架。
2. 在任务状态切换时补齐稳定事件字段。
3. 在周期统计接口中增加“事件流”聚合。
4. 前端调整“最近记录”等展示为真实操作记录。

不建议直接把所有统计完全改为扫描 `operations` 重建全部状态，因为：

1. 当前 `operations.content` 结构不够严格，历史数据未必完整。
2. 全量事件回放实现成本高，且与现有周期点聚合能力重叠。
3. 对“完成率/漏做”这类结果型指标，用当前任务实例状态更直接、性能更稳定。

## 具体实施方案

### 1. 补齐任务完成元数据

修改 `cloudfunctions/taskFunctions/index.js` 中 `toggleTaskStatus`：

当状态切换时：

1. `0 -> 1`
   - 写入 `completedAt: serverNow`
   - 写入 `completedBy: userId`
2. `1 -> 0`
   - 清空 `completedAt`
   - 清空 `completedBy`

目的：

1. 让每个实例具备稳定的“最近一次完成信息”。
2. 共享清单下能知道最近是谁完成了该实例。
3. 周期统计页可直接展示最近完成者/最近完成时间，无需完全依赖 `operations`。

注意：

- 这是当前状态字段，不代表完整历史。
- 历史事件仍由 `operations` 承担。

### 1.1 修正统计边界时间

修改 `cloudfunctions/taskFunctions/index.js` 中周期统计相关日期边界，确保“今天的周期点”会进入结果统计。

建议新增辅助函数：

```js
function getEndOfDayInUTC8(date) {
  const dateOnly = toDateOnlyInUTC8(date);
  return new Date(dateOnly.getTime() + 24 * 60 * 60 * 1000 - 1);
}
```

并在以下位置修正：

1. `buildPeriodicStatsPayload`
   - 当前：`occurrenceEnd = latestInstanceDueDate < todayDate ? latestInstanceDueDate : todayDate`
   - 应改为：先按 UTC+8 日期比较，再将上界取到“目标日期的结束时刻”
2. `getPeriodicTaskMonthDetail`
   - 当前：`effectiveMonthEnd = monthRange.end < todayDate ? monthRange.end : todayDate`
   - 应改为：使用“今天结束时刻”作为当月统计上界

目的：

1. 让今天 08:00、18:00、23:59 的周期点都能被视为“今天应统计的点”。
2. 解决“最近记录已显示完成事件，但统计详情数字不变”的直接问题。
3. 保持结果型统计与用户感知一致：今天做完，今天就应该计入已完成。

### 2. 强化操作日志结构

继续使用现有 `recordOperation('task_update', ...)`，但在状态切换场景下补充明确字段，确保可查询性。

建议在 `content` 中稳定增加：

```js
{
  taskTitle,
  action: 'toggle_status',
  old: { status: 0 },
  new: { status: 1 },
  statusTransition: 'complete', // 或 'uncomplete'
  completedAt,
  completedBy,
  dueDate,
  parentTaskId,
  repeatType,
  isPeriodicInstance,
  seriesTaskId,
  listId
}
```

目的：

1. 降低后续从 `operations` 中识别真实完成事件的复杂度。
2. 让周期统计页的“最近记录”可以直接读取结构化事件。
3. 为共享清单场景预留“谁操作的”统计能力。

### 3. 为周期统计新增事件查询能力

在 `cloudfunctions/taskFunctions/index.js` 中新增辅助逻辑：

1. 基于 `parentTaskId` 获取整个周期系列任务 ID 集合。
2. 查询 `operations` 集合中：
   - `type = 'task_update'`
   - `targetId in seriesTaskIds`
   - `content.action = 'toggle_status'` 或通过 `old.status/new.status` 识别
3. 将事件标准化为：

```js
{
  taskId,
  dateKey,
  dueDate,
  eventType: 'complete' | 'uncomplete',
  operatorId,
  operatedAt,
  completedBy,
  completedAt
}
```

用途：

1. 组装“最近记录”列表
2. 支撑未来的操作维度统计
3. 当 `completedAt/completedBy` 缺失时，允许从日志补偿展示

说明：

- `recentEventRecords` 只负责回答“发生了什么操作”。
- `summary` / `monthlyStats` 仍然必须修正日期边界后，才能正确回答“截至今天完成了多少周期点”。

### 4. 明确两个返回数据块

调整 `getPeriodicTaskStats` 返回结构，分为：

1. `summary`
2. `monthlyStats`
3. `allRecords` / `calendarRecords`：周期点结果
4. `recentEventRecords`：真实操作事件
5. `seriesMembers` 或 `operatorMap`：可选，映射操作者昵称

建议保留兼容字段，避免一次性打断前端：

- `recentRecords` 暂时继续存在
- 新增 `recentEventRecords`
- 前端优先消费新字段

### 5. 前端页面展示口径调整

修改 `miniprogram/pages/periodic-stats/periodic-stats.js` 与 `.wxml` 的展示逻辑设计如下：

#### 保留当前指标

保留以下指标继续按“周期点结果”展示：

1. 完成率
2. 最近 7 次
3. 本月漏做
4. 累计应做
5. 已到期周期点
6. 已完成
7. 未完成
8. 最近 30 次
9. 本月应做/已做
10. 月历打点
11. 连续完成

#### 替换“最近记录”

将底部“最近记录”从：

- 当前的静态 `已完成/未完成` 列表

改为：

- 真实操作事件流

建议展示：

1. 日期
2. 操作类型：完成 / 取消完成
3. 操作者（共享清单时显示）
4. 可选显示对应周期点日期

示例：

- `05-07 09:12 张三 完成了 05-07 周期点`
- `05-07 10:04 张三 取消完成 05-07 周期点`

这样页面才真正体现“根据相应操作统计数据”。

### 6. 历史数据兼容策略

历史任务和历史操作可能没有：

- `completedAt`
- `completedBy`
- `content.action`

兼容策略：

1. 对结果型统计：继续按当前 `status` 聚合，不受影响。
2. 对事件型统计：
   - 优先使用新结构字段
   - 回退到 `content.old.status` 和 `content.new.status` 识别完成/取消完成
3. 对操作者名称：
   - 优先使用 `operations.userId`
   - 无法解析时显示“成员”或不显示姓名
4. 对当天带时间的周期点：
   - 统计时按 UTC+8 的“日期”归属
   - 不再因为具体截止时分晚于零点而被排除

## 修改点清单

### 后端

文件：`cloudfunctions/taskFunctions/index.js`

需要修改/新增：

1. `toggleTaskStatus`
   - 写入 `completedAt/completedBy`
   - 扩展 `recordOperation` 的状态切换内容
2. `buildPeriodicStatsPayload`
   - 保留现有结果型统计逻辑
   - 不强制改为事件回放
   - 修正“今天”的统计截止边界为当天结束时刻
3. 新增辅助函数

   - `normalizePeriodicOperationEvent(op, taskMap)`

5. `getPeriodicTaskMonthDetail`
   - 修正月份详情对“今天”的统计上界

### 前端

文件：

1. `miniprogram/pages/periodic-stats/periodic-stats.js`
2. `miniprogram/pages/periodic-stats/periodic-stats.wxml`

需要修改：

1. 兼容读取 `recentEventRecords`
2. `generateHistoryRecords` 改为优先处理事件流
3. 历史记录文案改为完成/取消完成事件
4. 共享清单场景下展示操作者信息

## 验证方案

### 用例 1：个人周期任务完成

1. 创建一个每日周期任务
2. 完成今天实例
3. 打开周期统计页
4. 预期：
   - `completedDueCount +1`
   - 本月已做 +1
   - 最近记录新增“完成”事件

补充：任务截止时间设为“今天晚上 20:00”时，统计也必须立即 +1，不能等到次日。

### 用例 2：取消完成

1. 对同一实例取消完成
2. 打开周期统计页
3. 预期：
   - `completedDueCount -1`
   - `missedDueCount +1` 或恢复未完成状态
   - 最近记录新增“取消完成”事件

补充：如果该周期点属于今天且带具体时间，取消完成后统计详情也必须当场回退。

### 用例 3：共享清单他人完成

1. A 创建共享清单周期任务
2. B 以编辑者身份完成任务
3. 打开该周期统计页
4. 预期：
   - 周期点结果显示已完成
   - 最近记录显示 B 完成了该周期点
   - 若页面展示最近完成者，应显示 B

### 用例 4：历史兼容

1. 用旧数据直接进入周期统计页
2. 预期：
   - 完成率、已完成、未完成等结果型统计仍正确
   - 最近记录在缺少新字段时仍可通过旧日志尽量回放

## 风险与注意事项

1. `operations` 查询可能增加云函数读取成本，需要限制最近记录条数，例如最近 20 或 50 条。
2. 系列任务很多时，`targetId in []` 查询需注意集合上限，必要时分批查询。
3. 当前 `recordOperation('task_update')` 也用于普通编辑任务，状态切换识别必须加明确标记，避免把标题修改等事件误认为完成事件。
4. `completedAt/completedBy` 是“最近一次完成状态”的辅助字段，不应替代完整历史。
5. 如果后续要求“严格按操作历史重建每个月的统计快照”，则需要第二阶段引入正式事件模型；本次方案先解决当前统计页真实数据问题。
6. 周期统计目前是按“业务日期”统计，不应被任务具体截止时分影响是否计入今天；修正时要避免把“今天零点”继续误当成“今天结束”。

## 推荐实施顺序

1. 修正 `buildPeriodicStatsPayload` 和 `getPeriodicTaskMonthDetail` 的“今天结束时刻”边界。
2. 修改 `toggleTaskStatus`，补齐 `completedAt/completedBy` 和结构化操作日志。
3. 新增周期系列事件查询辅助函数。
4. 调整 `getPeriodicTaskStats` 返回 `recentEventRecords`。
5. 修改前端周期统计页，替换最近记录展示。
6. 做个人任务、共享清单、取消完成三类回归验证，并追加“今天晚上截止”的边界测试。

## 预期结果

实施后，周期统计页将具备两种一致但不混淆的数据能力：

1. 结果统计：反映当前每个周期点的真实完成结果。
2. 事件统计：反映用户实际进行了哪些完成/取消完成操作。

这样“完成一个周期任务后，统计页没有正确显示”这一问题会被系统性修复，并为协作场景下的真实统计提供稳定基础。
