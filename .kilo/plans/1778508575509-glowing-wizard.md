# 修复申请审核头像显示

## 目标

只修复邀请管理页“申请审核”列表无法显示待审核用户头像的问题。

本次明确不处理问题 1：

- 不修改“待处理”Tab 的语义。
- 不修改成员页“邀请管理”入口徽标逻辑。
- 不把 `status=4` 的申请审核计入 `pendingInviteCount`。
- 不调整通知跳转到 `申请审核` Tab。
- 不新增入口提醒、红点或待办总数。

## 问题范围

### 需要处理

申请审核列表中待审核用户头像无法显示，尤其是用户头像存储为云文件 `cloud://...` 时。

相关文件：

- `cloudfunctions/listFunctions/index.js`
- `miniprogram/pages/list-invite-manage/list-invite-manage.js`
- `miniprogram/pages/list-invite-manage/list-invite-manage.wxml`

### 不处理

以下内容不在本次实现范围：

- 成员页入口 `pendingInviteCount` 只统计 `status=0` 的现状。
- 邀请管理页 `待处理` Tab 只显示 `status=0` 的现状。
- 邀请管理页 `申请审核` Tab 的计数、Tab 切换和默认 Tab。
- 通知页点击 `join_request` 后默认进入邀请管理页的现状。
- 任何待处理/申请审核合并展示。

## 根因分析

前端申请审核头像来源链路如下：

1. `miniprogram/pages/list-invite-manage/list-invite-manage.wxml` 渲染：

```xml
<image class="invite-avatar" src="{{item.applicantInfo.avatarUrl || '/images/default-avatar.png'}}" mode="aspectFill" />
```

2. `miniprogram/pages/list-invite-manage/list-invite-manage.js` 中 `formatApplications()` 映射：

```js
applicantInfo: invite.inviteeInfo || {}
```

3. 后端 `cloudfunctions/listFunctions/index.js` 的 `getInviteList()` 返回 `inviteeInfo.avatarUrl`：

```js
avatarUrl: invite.inviteeInfo?.avatarUrl || user?.avatarUrl || ''
```

问题点：

- `getInviteList()` 直接返回数据库里的头像字段。
- 用户头像可能是 `cloud://...` 云存储 fileID。
- 小程序 `<image>` 在该场景下需要可访问的 HTTP 临时 URL。
- 同文件已有 `resolveAvatarUrl()`，但 `getInviteList()` 没有调用。
- 其他接口如 `getListMembers()` 已经调用 `resolveAvatarUrl()`，说明这是项目内既有处理方式。

## 修复方案

### 1. 修改 getInviteList 的头像解析

修改文件：

- `cloudfunctions/listFunctions/index.js`

修改位置：

- `getInviteList()` 中构造 `enriched` 的逻辑。

当前逻辑：

```js
const enriched = filteredInvites.map(invite => {
  const user = invite.inviteeId ? userMap[invite.inviteeId] : null;
  return {
    ...invite,
    inviteeInfo: {
      ...(invite.inviteeInfo || {}),
      nickname: invite.inviteeInfo?.nickname || user?.nickname || '',
      avatarUrl: invite.inviteeInfo?.avatarUrl || user?.avatarUrl || ''
    }
  };
});
```

目标逻辑：

```js
const enriched = await Promise.all(filteredInvites.map(async invite => {
  const user = invite.inviteeId ? userMap[invite.inviteeId] : null;
  const rawAvatarUrl = user?.avatarUrl || invite.inviteeInfo?.avatarUrl || '';

  return {
    ...invite,
    inviteeInfo: {
      ...(invite.inviteeInfo || {}),
      nickname: user?.nickname || invite.inviteeInfo?.nickname || '',
      avatarUrl: await resolveAvatarUrl(rawAvatarUrl)
    }
  };
}));
```

设计说明：

- 使用已有 `resolveAvatarUrl()`，不新增工具函数。
- HTTP/HTTPS 头像会原样返回。
- `cloud://...` 头像会转换为临时 HTTP URL。
- 无效头像会返回空字符串，前端继续使用默认头像。
- 优先使用用户表最新头像 `user?.avatarUrl`，再回退到邀请记录快照 `invite.inviteeInfo?.avatarUrl`。
- 昵称同样优先使用用户表最新昵称，回退到邀请记录快照。

### 2. 前端不需要改动

`list-invite-manage.wxml` 已经使用了兜底默认头像：

```xml
{{item.applicantInfo.avatarUrl || '/images/default-avatar.png'}}
```

`list-invite-manage.js` 的 `formatApplications()` 已经正确将 `inviteeInfo` 映射为 `applicantInfo`：

```js
applicantInfo: invite.inviteeInfo || {}
```

因此本次只需修复后端返回值，不需要修改前端页面结构或 Tab 逻辑。

## 影响范围

该修复会影响 `getInviteList()` 返回的所有邀请列表头像字段，包括：

- `待处理` Tab 的 `status=0` 邀请头像。
- `申请审核` Tab 的 `status=4` 申请头像。
- `已接受` Tab 的头像。
- `已拒绝` Tab 的头像。
- `已过期` Tab 的头像。

但不会改变：

- 任意邀请或申请的 `status`。
- 任意 Tab 的筛选逻辑。
- `pendingCount`、`applicationCount` 的统计语义。
- 成员页入口红点或徽标逻辑。
- 通知生成或跳转逻辑。

## 验证方案

由于该项目主要通过微信开发者工具验证，根目录无有效测试脚本，建议手动验证：

1. 准备一个需要审核的邀请链接。
2. 使用测试用户设置头像，头像字段最好覆盖两种情况：HTTP URL 和 `cloud://...` 云文件 ID。
3. 测试用户通过邀请链接提交加入申请，产生 `list_invites.status=4` 记录。
4. 创建者打开邀请管理页，进入 `申请审核` Tab。
5. 确认申请人头像正常显示。
6. 通过云函数返回数据确认 `inviteeInfo.avatarUrl` 为 HTTP/HTTPS 可访问地址，而不是原始 `cloud://...`。
7. 验证没有改变 `待处理` Tab：`status=4` 申请仍只在 `申请审核` Tab 显示，不出现在 `待处理` Tab。

## 风险与注意事项

- `getInviteList()` 每条记录会调用 `resolveAvatarUrl()`，当前列表限制为 50 条，性能风险可接受。
- 临时 URL 有有效期，页面每次加载重新获取即可。
- 如果用户表头像为空，而邀请记录快照也为空，前端仍显示默认头像，这是预期行为。
- 如果数据库中已有历史无效 HTTP URL，`resolveAvatarUrl()` 会原样返回，前端仍可能加载失败；该场景需要数据修复或前端 `binderror` 兜底，不在本次范围。

## 实施顺序

1. 修改 `cloudfunctions/listFunctions/index.js` 中 `getInviteList()` 的 `enriched` 构造逻辑。
2. 使用 `resolveAvatarUrl()` 处理 `inviteeInfo.avatarUrl`。
3. 保持所有前端邀请管理 Tab 和计数逻辑不变。
4. 在微信开发者工具中按验证方案手动验证。
