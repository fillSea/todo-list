# 清单模块邀请成员机制审查与修改方案

## 目标

审查 `清单 -> 邀请成员 -> 受邀人处理 -> 创建者审核/管理` 这一整条链路，判断是否存在逻辑错误、状态模型缺陷、前后端不一致和可优化点，并整理成可直接实施的修改方案。

## 现状梳理

### 入口与页面

- 邀请入口页：`miniprogram/pages/list-invite/list-invite.js`
- 邀请链接页：`miniprogram/pages/list-invite-link/list-invite-link.js`
- 邀请接受页：`miniprogram/pages/list-invite-accept/list-invite-accept.js`
- 邀请管理页：`miniprogram/pages/list-invite-manage/list-invite-manage.js`
- 成员页中的待处理邀请数量：`miniprogram/pages/list-members/list-members.js`

### 后端核心函数

位于 `cloudfunctions/listFunctions/index.js`：

- `inviteMember`
- `createWechatInvite`
- `generateInviteLink`
- `getInviteInfo`
- `acceptInvite`
- `applyJoinList`
- `rejectInvite`
- `getInviteList`
- `cancelInvite`
- `approveApplication`
- `rejectApplication`
- `clearInvites`

### 当前状态模型

`list_invites.status` 当前使用：

- `0` 待处理/待接受
- `1` 已接受
- `2` 已拒绝
- `3` 已过期
- `4` 待审批

同时还有：

- `inviteType`: `search | recent | wechat | link`
- `needApproval`: 是否需要审核
- `inviteeId`: 为空表示模板型公开邀请；有值表示用户级邀请记录
- `approvalSource`: `direct_invite | public_link`

### 当前机制摘要

#### 定向邀请

- `inviteMember` 直接创建一条 `inviteeId=目标用户` 的邀请记录。
- 受邀人访问后：
  - 无审核：`acceptInvite` 直接加成员并把状态改为 `1`
  - 需审核：`acceptInvite` 转调 `applyJoinList`，把状态改为 `4`
  - 拒绝：`rejectInvite` 把状态改为 `2`

#### 公开链接/微信邀请

- `generateInviteLink` / `createWechatInvite` 创建 `inviteeId=''` 的模板邀请。
- 用户访问时：
  - `getInviteInfo` 优先取自己已有记录，否则取模板邀请
  - 接受时若无审核，先复制一条用户级记录，再加成员并置为 `1`
  - 申请时若需审核，复制一条用户级记录并置为 `4`
  - 拒绝时复制一条用户级记录并置为 `2`

## 发现的问题

### 1. 公开邀请的“模板记录”被错误地当成待处理邀请展示

位置：

- `cloudfunctions/listFunctions/index.js:1446-1503`
- `miniprogram/pages/list-invite/list-invite.js:153-211`
- `miniprogram/pages/list-members/list-members.js:199-224`
- `miniprogram/pages/list-invite-manage/list-invite-manage.js:192-281`

表现：

- `getInviteList(status=0)` 直接查询 `listId + status=0`，会把 `inviteeId=''` 的链接/微信模板邀请一起返回。
- 前端“待处理邀请”列表默认按“待某个用户处理的邀请”理解，但模板邀请没有具体受邀人，`inviteeInfo` 为空，页面会显示“未知用户”。
- `pendingInviteCount` 也会把模板邀请计入，导致成员页和邀请页数字失真。

影响：

- 管理者看见“未知用户”的待处理邀请，语义错误。
- 链接型邀请与定向邀请混在一起，无法区分“待某人处理”与“公开入口仍有效”。

结论：属于实际逻辑错误，不只是体验问题。

### 2. 邀请管理页的“申请审核”与“待处理”计数不完整

位置：

- `miniprogram/pages/list-invite-manage/list-invite-manage.js`
- `miniprogram/pages/list-invite-manage/list-invite-manage.wxml`

表现：

- 页面初始化只加载当前 tab 对应数据。
- 但顶部 `pendingCount` / `applicationCount` 徽标依赖当前页内数据，不会同时准确加载两个计数。
- 从默认 `pending` tab 进入时，`applicationCount` 往往是 `0`，即使实际上存在待审批申请。

影响：

- 徽标信息不可信。
- 管理者容易漏掉待审批申请。

结论：前端状态管理缺陷。

### 3. `header-right` 上直接使用 `canClear`，而 `canClear` 实现为函数，WXML 无法按预期求值

位置：

- `miniprogram/pages/list-invite-manage/list-invite-manage.wxml:8`
- `miniprogram/pages/list-invite-manage/list-invite-manage.js:685-698`

表现：

- WXML 中使用 `wx:if="{{canClear}}"`。
- 但 `Page` 实例方法不是响应式 data 字段，也不会在模板中自动调用为函数结果。

影响：

- “清空”按钮显示逻辑可能异常，通常会始终不可靠。

结论：明确前端实现错误。

### 4. “重新生成链接后旧链接失效”的文案与后端行为不一致

位置：

- 前端文案：`miniprogram/pages/list-invite-link/list-invite-link.wxml:98,159`
- 后端行为：`generateInviteLink` 仅新增记录，不会使旧链接失效

表现：

- 页面提示“重新生成后，旧链接将失效”。
- 实际 `generateInviteLink` 每次只是新增一条模板邀请，没有取消同 list 的旧 link 邀请。

影响：

- 文案误导。
- 同一清单会长期累积多条同时有效的公开链接，后续统计与管理都会变脏。

结论：明确产品行为与实现不一致，属于逻辑缺口。

### 5. 邀请码生成无唯一性校验，理论上存在碰撞风险

位置：

- `cloudfunctions/listFunctions/index.js:120-127`

表现：

- `generateInviteCode()` 仅随机生成 16 位字符串。
- 创建前没有查询数据库确认未占用。

影响：

- 概率低，但一旦碰撞会把不同邀请串到同一个入口，影响严重。

结论：低概率高影响风险，值得修正。

### 6. 加成员与更新邀请状态不是原子操作，存在并发重复入会风险

位置：

- `acceptInvite`
- `approveApplication`
- `addMemberFromInvite`

表现：

- 先查 `existingMembers`，再 `addMemberFromInvite`，再更新邀请状态。
- 如果用户重复点击、网络重试、或两个审批动作并发，可能同时通过“不是成员”的检查，然后插入两条 `list_members`。
- 当前没有事务，也没有成员唯一约束的兜底处理。

影响：

- `list_members` 可能出现同一 `(listId, userId)` 重复记录。
- 成员数、权限判断、任务可见性等都会受污染。

结论：后端核心一致性风险。

### 7. `rejectInvite` 对“待审批申请”没有区分语义

位置：

- `cloudfunctions/listFunctions/index.js:2035-2080`

表现：

- 如果受邀人已有 `status=4` 记录，`getInviteRecordForUser` 会先命中，然后统一返回“您已处理过该邀请”。
- 对用户来说，如果他已经申请加入，应该看到更明确的“已申请，等待审批”，而不是泛化提示。

影响：

- 用户状态感知不清晰。

结论：属于提示语义问题，可顺手修正。

### 8. 邀请统计字段含义混乱

位置：

- `generateInviteLink` 返回的 `stats`
- `list-invite-link.wxml` 中“待审核”展示

表现：

- `stats.pendingCount = allInvites.filter(i => i.status === 0).length`
- 页面却把这个数字标为“待审核”。
- 实际 `status=0` 是待处理，`status=4` 才是待审核。

影响：

- 统计名实不符。
- 如果存在很多公开模板链接，数字更失真。

结论：前后端字段定义不清。

### 9. 公开链接点击次数统计未实现，但 UI 展示为真实数据

位置：

- `generateInviteLink` 中 `clickCount: 0`
- 页面展示“点击次数”

表现：

- 后端始终返回 `0`，未在 `getInviteInfo` 或其他入口中累积。

影响：

- 如果保留该指标，会误导用户。

结论：要么实现，要么移除/降级为占位指标。

### 10. 取消邀请只允许邀请人本人，和“创建者统一管理邀请”存在冲突

位置：

- `cancelInvite`: 先校验创建者权限，再额外要求 `invite.inviterId === userId`

表现：

- 目前因为只有创建者可以发起邀请，所以暂时没出错。
- 但函数语义是“邀请管理”，却把权限绑定成“必须是邀请人本人”。
- 如果未来允许编辑者发邀请，创建者将无法统一取消他人发出的邀请。

影响：

- 扩展性差。

结论：短期不算 bug，属于权限模型可维护性问题。

## 建议的修改原则

1. 把“公开邀请模板”和“用户级邀请记录”区分清楚。
2. 让前端展示围绕真实业务对象，而不是机械展示所有 `list_invites` 记录。
3. 对成员入会流程做幂等化，优先解决重复写入风险。
4. 统一统计口径与文案，避免字段含义漂移。
5. 尽量做小改动，不重构整张表结构。

## 推荐修改方案

### 方案总览

采用“保留现有表结构，小步修正查询语义和并发控制”的方案，而不是重建邀请模型。

理由：

- 当前逻辑已经围绕 `list_invites` 的模板记录 + 用户记录模式展开。
- 彻底拆表成本高，且会影响邀请页、审批页、通知、历史记录等多个链路。
- 现有问题主要集中在查询过滤、状态语义、统计口径和幂等控制，属于可局部修复的问题。

## 具体修改步骤

### 一、修正邀请列表查询语义

修改文件：`cloudfunctions/listFunctions/index.js`

改造 `getInviteList`，增加更清晰的筛选规则：

- 当 `status === 0` 时，仅返回“用户级待处理邀请”，即：
  - `status === 0`
  - `inviteeId` 有值
- 当 `status === 4` 时，仅返回待审批申请：
  - `status === 4`
  - `inviteeId` 有值
  - `needApproval === true`
- 当 `status === 1/2/3` 时，仍只返回用户级记录，避免模板记录混入历史。
- 如果需要展示公开链接模板，不复用 `getInviteList`，单独提供公开邀请链接查询接口，或在链接页单独统计。

预期结果：

- “待处理邀请”不再出现“未知用户”。
- 成员页待处理数量只统计真实待用户处理的邀请。
- 邀请管理页中的 pending/accepted/rejected/expired 都变成可解释的数据。

### 二、为邀请管理页补充独立计数接口或并行计数查询

优先推荐：在现有 `getInviteList` 基础上前端并行拉取两类计数。

修改文件：

- `miniprogram/pages/list-invite-manage/list-invite-manage.js`
- 可选：`cloudfunctions/listFunctions/index.js`

两种可行做法：

#### 做法 A：前端并行调用两次 `getInviteList`

- 页面进入时无论当前 tab 是什么，都并行请求：
  - `status: 0` 用于 `pendingCount`
  - `status: 4` 用于 `applicationCount`
- 当前 tab 对应列表再单独请求内容。

优点：改动小。

#### 做法 B：后端新增 `getInviteSummary`

返回：

- `pendingCount`
- `applicationCount`
- `acceptedCount`
- `rejectedCount`
- `expiredCount`
- `activeLinkInviteCount` 或 `publicInviteCount`

优点：后续扩展更稳。

本次建议：

- 先做 A，成本最小。
- 若实施过程中发现页面重复请求过多，再补 B。

### 三、修复 `canClear` 的前端实现错误

修改文件：`miniprogram/pages/list-invite-manage/list-invite-manage.js` / `.wxml`

改法：

- 不在 WXML 中直接引用实例方法。
- 改为在每次 `setData` 时同步维护布尔字段，例如：`canClearCurrentTab`。
- 或在切换 tab / 列表加载后调用 `updateDerivedState()`，统一更新：
  - `pendingCount`
  - `applicationCount`
  - `canClearCurrentTab`

预期结果：

- “清空”按钮展示稳定。

### 四、统一“重新生成链接”的行为

修改文件：

- `cloudfunctions/listFunctions/index.js`
- `miniprogram/pages/list-invite-link/list-invite-link.wxml`
- `miniprogram/pages/list-invite-link/list-invite-link.js`

建议采用“重新生成时使旧公开链接失效”的真实实现，而不是只改文案。

实施方式：

- 在 `generateInviteLink` 生成新链接前，把同一 `listId` 下仍有效的 `inviteType='link'` 且 `inviteeId` 为空的旧模板邀请批量置为 `status=3`，或新增 `isActive=false` 字段后屏蔽旧链接。
- `getInviteByCode` 只取当前有效模板邀请。
- 若已有 UI 显示“旧链接失效”，则后端行为必须对齐。

附加建议：

- `createWechatInvite` 是否也只保留最近一个有效微信模板邀请，需要统一规则。
- 若不希望过于激进，可只让 `link` 类型单实例生效，`wechat` 仍允许多条。

本次建议：

- 先修 `link`。
- `wechat` 保持现状，但在后续可与产品确认是否也应单实例有效。

### 五、为邀请码生成增加唯一性保障

修改文件：`cloudfunctions/listFunctions/index.js`

实现方式：

- 新增 `generateUniqueInviteCode()`：循环生成邀请码并查询 `list_invites` 是否已存在同码，存在则重试。
- 设置最大重试次数，例如 10 次，失败抛错。
- 所有创建邀请码的入口统一改用该函数：
  - `inviteMember`
  - `generateInviteLink`
  - `createWechatInvite`

预期结果：

- 消除低概率高危碰撞问题。

### 六、让入会流程具备幂等性，降低并发重复入会风险

修改文件：`cloudfunctions/listFunctions/index.js`

目标：至少保证同一个用户不会被同一清单重复加入。

推荐分层处理：

#### 第一层：收敛写入入口

新增辅助函数，例如：

- `ensureMemberJoinedFromInvite(invite, userId, joinType)`

内部流程：

1. 再次查询是否已有 `(listId, userId)` 成员。
2. 若已有成员：
   - 不重复写入 `list_members`
   - 直接把邀请状态修正为 `1`
   - 返回幂等成功
3. 若不存在成员：
   - 写入成员
   - 更新邀请状态

#### 第二层：尽量使用事务

如果微信云开发当前环境可稳定使用数据库事务：

- 把“查成员 -> 写成员 -> 更新邀请状态”放入事务。

如果不方便使用事务：

- 仍用单辅助函数封装顺序操作。
- 在成员新增后若更新邀请失败，至少可通过后续重试恢复状态。

本次建议：

- 优先做辅助函数封装。
- 若项目里事务使用成本可控，再升级为事务版。

### 七、优化用户提示语义

修改文件：`cloudfunctions/listFunctions/index.js`

细节建议：

- `rejectInvite` 若命中 `status=4`，返回“您已申请过，请等待审批”而不是“已处理过”。
- `acceptInvite` 若已是成员，返回更友好的成功型语义也可考虑，例如“您已是该清单成员”，前端可以直接跳转清单页。
- `getInviteInfo` 对“模板链接存在，但当前用户已有处理记录”保持现有优先级即可，这部分逻辑合理。

### 八、修正邀请链接统计口径和文案

修改文件：

- `cloudfunctions/listFunctions/index.js`
- `miniprogram/pages/list-invite-link/list-invite-link.wxml`

建议把统计拆成明确字段：

- `joinedCount`: `status === 1` 的用户级记录数
- `appliedCount`: `status === 4` 的用户级记录数
- `pendingUserInviteCount`: `status === 0 && inviteeId 有值` 的定向邀请待处理数
- `activeTemplateCount`: `status === 0 && inviteeId 为空` 的有效公开模板数

前端文案同步调整：

- 原“待审核”应绑定 `appliedCount`
- 原“点击次数”若未实现，先隐藏或改成“有效链接数/公开入口数”

本次建议：

- 若暂不做点击埋点，则先移除“点击次数”展示，避免假数据。

### 九、明确“公开模板邀请”与“用户级邀请记录”的字段注释和约束

修改文件：`cloudfunctions/listFunctions/index.js`

建议增加少量注释，明确：

- `inviteeId` 为空：公开模板邀请，不代表某个真实受邀人
- `inviteeId` 有值：某个用户的处理记录/邀请记录
- `status=4` 仅对 `needApproval + inviteeId 有值` 的记录有意义

目的：

- 降低后续维护者误用 `getInviteList(status=0)` 的概率。

## 验证清单

### 定向邀请

1. 创建者通过搜索邀请某用户。
2. 邀请页待处理列表显示该用户昵称，不出现“未知用户”。
3. 受邀人接受后进入成员列表，且只新增一条成员记录。
4. 受邀人拒绝后，创建者在“已拒绝”页可见。

### 公开链接邀请

1. 生成链接后，不在“待处理邀请”中显示为未知用户。
2. A 用户打开链接并接受，成功加入。
3. B 用户打开同链接并拒绝，只生成自己的拒绝记录，不影响 A。
4. 开启审核后，用户申请加入进入“申请审核”列表。
5. 审批通过后，成员列表只出现一条对应成员记录。

### 链接重新生成

1. 生成第一个链接。
2. 重新生成第二个链接。
3. 第一个链接再次访问时应提示失效/过期。
4. 第二个链接可正常使用。

### 计数与管理页

1. 有待处理邀请时，`pendingCount` 正确。
2. 有待审批申请时，`applicationCount` 正确。
3. 切换不同 tab，“清空”按钮仅在可清空 tab 且有数据时显示。

### 并发/幂等

1. 同一用户连续点击接受两次，不应产生两条 `list_members`。
2. 审批接口重复触发，不应重复加成员。

## 风险与注意事项

- `list_invites` 里历史上可能已经存在模板记录混入待处理列表的数据脏污，修改后 UI 会“少一些数据”，这是修正不是回归。
- 若要让旧链接失效，需确认是否接受“重新生成立即废弃旧链接”的产品语义；当前前端文案已经默认这个语义，后端补齐即可。
- 并发幂等若不使用事务，只能显著降低风险，不能像数据库唯一索引那样彻底消灭竞争窗口；但仍比现状稳定很多。

## 结论

当前邀请成员机制整体思路能工作，但存在 3 类核心问题：

- 查询语义错误：模板邀请被当成待处理邀请展示。
- 前端状态错误：计数和 `canClear` 实现不可靠。
- 后端一致性风险：邀请码无唯一校验，入会流程非原子，存在重复入会风险。

推荐按本方案进行小步修复，不重构表结构，优先修正查询过滤、链接失效语义、前端计数和加入幂等性。
