# 个人任务界面周期任务统计优化设计方案

## 目标

优化个人任务界面中周期任务的数据统计能力，重点解决以下问题：

1. 统计口径不够准确，尤其是“连续完成”对周/月周期任务的语义不严谨。
2. 后端统计查询需要全量拉取整个任务系列实例，实例数增长后性能会明显下降。
3. 前端统计页信息密度偏低，难以帮助用户理解“执行习惯”“漏做模式”“未来压力”。
4. 当前统计建立在“已预生成实例”之上，对未来实例覆盖范围受 `preGenerateDays` 限制，统计结果存在天然上限。
5. 个人任务首页虽然对周期任务做了“每个系列只显示最近未过期实例”的优化，但统计页仍是单独的全量扫描逻辑，缺少统一的数据模型。

## 当前实现概况

### 前端

1. 个人任务页 `miniprogram/pages/index/index.js`
   - 对周期任务做了展示级过滤：每个系列仅显示最近一个未完成且未过期实例。
   - 周期统计入口通过 `onViewStats` 跳转到 `periodic-stats` 页面。

2. 统计页 `miniprogram/pages/periodic-stats/periodic-stats.js`
   - 调用云函数 `getPeriodicTaskStats` 获取整包统计数据。
   - 只展示：完成率、已完成、未完成、已到期、当前/最长连续、按月日历、近 50 条历史。
   - `monthlyStats` 虽然后端有返回，但前端未使用。

3. 日历页 `miniprogram/pages/calendar/calendar.js`
   - 同样支持查看周期任务统计。
   - 页面内部已有“某日期周期任务是否应出现”的判定逻辑，可复用于未来统计口径统一。

### 后端

1. 云函数 `cloudfunctions/taskFunctions/index.js#getPeriodicTaskStats`
   - 通过 `_id === taskId` 或 `parentTaskId === taskId` 拉取整个系列所有实例。
   - 使用分页 `skip + limit` 全量扫描。
   - 统计口径：
     - `dueDate <= today` 视为已到期实例。
     - 已完成数 = `status === 1`
     - 未完成数 = `status === 0`
     - 完成率 = 已完成 / 已到期
   - 连续完成计算方式：仅按实例排序后，遇到未完成就中断，不区分日/周/月周期真实间隔。
   - 输出 `allRecords` 供前端渲染月历。

## 主要问题分析

### 1. 连续完成统计口径不准确

当前 `currentStreak` 和 `longestStreak` 的计算只是“连续已完成实例数”，默认相邻实例天然连续。这个算法对每天重复任务基本可接受，但对每周/每月任务存在问题：

1. 如果是“每周一、三、五”，连续 3 次完成不等于“连续 3 天”，更准确的语义应为“连续 3 个应执行周期点”。
2. 如果某些周期点因为未预生成实例而缺失，当前算法会把“数据缺口”误当作不存在，连续统计偏乐观。
3. 月周期如 `1,15`，连续完成次数应按计划执行点计算，而不是按自然日连续。

### 2. 统计性能会随实例数线性下降

当前每次进入统计页都：

1. 扫描整个系列所有实例。
2. 在内存中过滤“已到期/未来”。
3. 再组装 `allRecords` 给前端画月历。

问题：

1. 老用户长期使用后，一个周期系列可能累积上百到上千条实例。
2. `skip` 在云数据库大数据量分页下成本较高。
3. 为了画统计页月历，当前返回了比页面实际首屏展示更多的数据。
4. 每次重复计算，没有缓存或汇总层。

### 3. 统计模型依赖“实例存在”，缺少“计划点”视角

当前统计的本质是“对已生成任务实例做统计”，不是“对周期计划执行点做统计”。这会带来两个问题：

1. 未来统计只看得到已预生成的 30 天左右数据，`futureCount` 不是完整未来计划量。
2. 如果历史某些周期点因为早期逻辑或异常未生成实例，统计会失真。

### 4. 前端呈现偏结果，不够洞察

当前页面展示了基础 KPI，但缺少用户最关心的解释性指标：

1. 最近 7/30 个周期点完成率。
2. 最近一次漏做时间。
3. 本月应做/已做/漏做。
4. 一周内/一月内即将到来的周期点。
5. 周期分布说明，例如“每周一三五”或“每月 1/15 日”。

### 5. 前后端存在重复逻辑但未抽象统一

项目中已有多处周期判断逻辑：

1. 云函数中有 `calculateNextRepeatDate`、预生成逻辑。
2. 日历页中有 `isTaskRepeatingOnDate`。
3. 统计页则直接基于已生成实例。

如果继续叠加统计能力而不统一“周期命中规则”，后续容易出现：

1. 页面显示应有任务，但统计不算。
2. 统计日历显示空白，但日历页显示有周期点。

## 优化方案

建议分为三层优化：统计口径统一、查询性能优化、前端信息架构优化。

---

## 一、统计口径优化

### 方案 1：将“连续完成”改为“连续命中周期点”

重新定义：

1. `currentStreak`：从最近一个“已到期周期点”开始，连续完成的周期点数量。
2. `longestStreak`：历史上最长的连续完成周期点数量。

这里的“周期点”定义：

1. 每日任务：每天一个周期点。
2. 每周任务：由 `repeatValue` 指定的每个星期几都是一个周期点。
3. 每月任务：由 `repeatValue` 指定的每个月几号都是一个周期点。

实现上不直接依赖“实例是否生成”，而是根据父任务的 `dueDate + repeatType + repeatValue` 生成理论周期点序列，再与实例状态做映射。

收益：

1. 日/周/月统计语义统一。
2. 不会因为预生成深度不同导致 streak 偏差。
3. 后续可直接扩展“最近 7 个周期点完成率”等指标。

### 方案 2：区分“计划点统计”和“实例统计”

返回两类指标：

1. `scheduleStats`：基于理论周期点。
2. `instanceStats`：基于真实生成任务实例。

页面主展示使用 `scheduleStats`，开发排查或兼容场景保留 `instanceStats`。

建议主页面对用户只展示计划点统计，避免概念混乱。

### 方案 3：增加时间窗口型指标

新增指标建议：

1. `last7Rate`：最近 7 个已到期周期点完成率。
2. `last30Rate`：最近 30 个已到期周期点完成率。
3. `thisMonthCompleted / thisMonthPlanned / thisMonthMissed`
4. `lastMissedDate`
5. `nextPlannedDate`
6. `upcomingCount7d`：未来 7 天内待执行次数。

这些指标比总完成率更能体现近期执行状态。

## 二、查询与计算性能优化

### 方案 1：限制首屏数据范围，分层返回

将当前一次性返回 `allRecords` 的方式拆成两层：

1. 首屏统计接口返回：
   - 任务基础信息
   - 核心 KPI
   - 最近 N 条历史记录
   - 最近 6~12 个月月度摘要
2. 日历明细接口按月份加载：
   - 入参：`taskId + month`
   - 返回：该月每日状态点

收益：

1. 降低首屏云函数返回体积。
2. 减少前端首次渲染和月历构建成本。
3. 页面可以做“按需展开月份”。

### 方案 2：弃用深分页 `skip`，改为游标分页或区间查询

当前全量扫描通过 `skip(batchIndex * 100)` 分页，后续可改为：

1. 使用 `dueDate` 作为游标，按时间范围拉取。
2. 统计已到期数据时，只查 `dueDate <= endOfToday`。
3. 历史记录只查最近 `N` 条已到期实例。
4. 未来计划无需查全部未来实例，理论周期点可以按规则实时生成。

收益：

1. 避免大 `skip` 带来的扫描退化。
2. 查询行为更贴近统计场景实际需要。

### 方案 3：构建轻量汇总字段或独立统计表

这是中期优化方案，优先级高于“立即改 UI”，低于“先统一口径”。

可选实现：

1. 在父任务文档冗余统计字段，例如：
   - `stats.totalDueCount`
   - `stats.completedDueCount`
   - `stats.currentStreak`
   - `stats.longestStreak`
   - `stats.lastMissedDate`
   - `stats.updatedAt`
2. 或新增集合 `task_repeat_stats`，每个周期系列一条汇总记录。

推荐优先使用独立集合 `task_repeat_stats`，原因：

1. 避免父任务文档被高频更新，降低写冲突。
2. 统计结构可以独立演进。
3. 后续可以补充月度桶、近 30 次窗口统计等字段。

更新时机：

1. 创建周期任务时初始化。
2. 完成/取消完成周期任务时增量更新。
3. 删除单次或删除整个周期时重算或局部修正。
4. 定时任务每天补偿校验一次，防止统计漂移。

### 方案 4：理论周期点生成与实例状态映射分离

后端新增统一函数，例如：

1. `generatePlannedOccurrences(task, startDate, endDate)`
2. `mapOccurrencesWithInstances(occurrences, instances)`
3. `calculateRepeatStats(occurrenceStates)`

收益：

1. 统计、日历、未来提醒共用同一规则。
2. 降低不同页面各自实现重复周期判断的风险。
3. 便于补测试。

## 三、前端展示优化

### 方案 1：调整统计页信息架构

建议页面结构改为：

1. 顶部卡片：任务标题、周期说明、优先级、起始日期。
2. 核心指标卡：
   - 总完成率
   - 最近 7 次完成率
   - 当前连续完成
   - 本月漏做次数
3. 趋势卡：按月完成率或近 6 个月柱状/分段统计。
4. 日历卡：只展示最近 3~6 个月，支持“查看更多月份”。
5. 历史卡：最近 20 条，支持按“已完成/漏做”筛选。

### 方案 2：展示可解释的周期规则文案

把 `repeatType + repeatValue` 转换为清晰文案：

1. 每天：`每天`
2. 每周：`每周一、三、五`
3. 每月：`每月 1 日、15 日`

并增加“统计说明”：

1. `连续完成按周期执行点计算，不按自然日计算。`
2. `总完成率只统计截至今天应完成的周期点。`

这样可以减少用户对 streak 和完成率的误解。

### 方案 3：月历按需加载，不一次性渲染全历史

当前 `generateCalendarMonths(records)` 会从最早月份渲染到最晚月份。若用户有几年历史，页面会非常重。

建议：

1. 默认只加载最近 6 个月。
2. 提供“查看更多历史月份”。
3. 按月请求明细而不是全量组装。

### 方案 4：用 `monthlyStats` 做趋势展示

后端已经返回了 `monthlyStats`，但前端未使用。可先做低成本优化：

1. 展示最近 6 个月：应做数、完成数、漏做数。
2. 用简单进度条或柱形条即可，不必引入重图表库。

这是一个投入小、感知收益高的首轮优化点。

## 四、推荐落地路径

建议分三期实施，避免一次性大改周期逻辑。

### 第一期：低风险快收益

目标：先优化展示和部分统计结果，不改核心存储模型。

包含内容：

1. 复用现有接口，前端开始使用 `monthlyStats`。
2. 周期规则显示文案化。
3. 页面首屏只展示最近几个月，不渲染全量月份。
4. 历史记录条数从固定 50 改为分页或“查看更多”。
5. 后端补充最近 7/30 周期点完成率等轻量指标。

风险：低。

收益：用户能明显感知统计页更有价值，接口改动有限。

### 第二期：统计口径统一

目标：把统计从“实例统计”升级为“计划点统计”。

包含内容：

1. 抽象统一的周期点生成函数。
2. streak、月统计、未来待执行等全部改用周期点口径。
3. 区分 `scheduleStats` 与 `instanceStats`，前端以 `scheduleStats` 为主。
4. 补充日/周/月周期任务的边界测试。

风险：中。

收益：统计正确性显著提升，后续扩展更稳。

### 第三期：汇总层与按月明细接口

目标：解决长期性能问题。

包含内容：

1. 新增 `task_repeat_stats` 汇总集合。
2. 周期任务状态变更时做增量更新。
3. 统计页改为“首屏汇总接口 + 月明细接口”。
4. 加定时补偿任务，校验汇总数据一致性。

风险：中高。

收益：大幅降低统计页全量扫描成本，适合长期使用场景。

## 数据结构建议

### 方案 A：保持现有 `tasks` 不变，新增汇总集合

建议结构：

```json
{
  "_id": "parentTaskId",
  "taskId": "parentTaskId",
  "creatorId": "userId",
  "repeatType": 2,
  "repeatValue": "1,3,5",
  "lastCalculatedAt": "2026-05-07T00:00:00.000Z",
  "scheduleStats": {
    "totalDueCount": 42,
    "completedDueCount": 35,
    "missedDueCount": 7,
    "completionRate": 83,
    "currentStreak": 5,
    "longestStreak": 12,
    "last7Rate": 86,
    "last30Rate": 80,
    "lastMissedDate": "2026-04-29T09:00:00.000Z",
    "nextPlannedDate": "2026-05-08T09:00:00.000Z",
    "upcomingCount7d": 3
  },
  "monthlyBuckets": {
    "2026-03": { "planned": 13, "completed": 11, "missed": 2 },
    "2026-04": { "planned": 14, "completed": 12, "missed": 2 }
  }
}
```

优点：

1. 不侵入现有任务实例结构。
2. 更适合增量维护和单独重算。

### 方案 B：在父任务文档上冗余统计字段

优点是实现快；缺点是职责混杂、后续扩展受限。

建议结论：

1. 短期可先冗余少量字段。
2. 中长期应迁移到独立 `task_repeat_stats`。

## 接口设计建议

### 1. 周期任务统计概览接口

保留 `getPeriodicTaskStats`，但升级返回结构：

```json
{
  "taskInfo": {},
  "summary": {
    "completionRate": 83,
    "completedDueCount": 35,
    "missedDueCount": 7,
    "totalDueCount": 42,
    "currentStreak": 5,
    "longestStreak": 12,
    "last7Rate": 86,
    "last30Rate": 80,
    "nextPlannedDate": "2026-05-08T09:00:00.000Z",
    "lastMissedDate": "2026-04-29T09:00:00.000Z"
  },
  "monthlyStats": [],
  "recentRecords": []
}
```

### 2. 周期任务月历明细接口

新增如 `getPeriodicTaskMonthDetail`：

入参：

```json
{
  "taskId": "xxx",
  "month": "2026-05"
}
```

返回：

```json
{
  "month": "2026-05",
  "days": [
    { "date": "2026-05-01", "status": 1, "source": "instance" },
    { "date": "2026-05-03", "status": 0, "source": "instance" },
    { "date": "2026-05-05", "status": 2, "source": "planned" }
  ]
}
```

状态建议：

1. `1` 已完成
2. `0` 已到期未完成
3. `2` 未来待执行
4. `-1` 非周期点

## 风险与兼容性

### 风险 1：历史实例不完整导致重算结果与旧统计不一致

处理建议：

1. 在方案切换初期保留旧字段，仅前端展示新口径。
2. 加一段说明文案，避免用户误解数字波动。

### 风险 2：月周期边界复杂

如：

1. `31` 号在小月不存在。
2. 起始日期不是周期点首日。
3. 时区固定为 `+08:00`，需和现有 dueDate 逻辑保持一致。

处理建议：

1. 明确规则：不存在的日期当月跳过，不自动补到月末。
2. 所有统计都基于北京时间日界线处理。

### 风险 3：删除单次实例对 streak 的影响

如果用户删除“单次实例”，该周期点应如何统计需要统一规则。

建议：

1. 对统计口径而言，删除单次实例后，该周期点视为“已跳过”还是“未完成”，必须明确定义。
2. 推荐定义为：
   - 如果这是用户主动删除的单次任务，则该周期点不计入完成率分母，记为 `skipped`。
3. 如果暂不支持该语义，则第一期先沿用“没有实例即不统计”，第二期统一改造。

这是当前方案中唯一建议在实施前再确认的产品规则点。

## 测试建议

重点补以下测试场景：

1. 每日周期：连续 7 天完成，中间断一天。
2. 每周多选：每周一三五，连续完成 5 个周期点。
3. 每月多选：1 号和 15 号，跨月统计。
4. 月周期 31 号在 30 天月份跳过。
5. 未来预生成只有 30 天时，未来统计仍能正确展示未来 7 天计划点。
6. 单次删除、整组删除、取消完成对统计结果的影响。
7. 不同入口进入统计页：个人任务页、日历页结果一致。

## 推荐结论

可以优化，而且值得优化。

优先推荐路线：

1. 先做前端展示增强与 `monthlyStats` 利用，快速提升可见价值。
2. 再把统计核心改为“基于周期计划点”的统一模型，修正 streak 和完成率语义。
3. 最后引入 `task_repeat_stats` 汇总层与月明细接口，解决长期性能问题。

## 实施清单

1. 梳理并抽象统一的周期点计算函数。
2. 明确“连续完成”“漏做”“删除单次”的统计规则。
3. 升级 `getPeriodicTaskStats` 返回结构。
4. 新增按月明细接口。
5. 前端重构 `periodic-stats` 页面信息架构。
6. 增加边界测试与历史数据校验脚本。
