# 微信好友邀请简化方案设计

## 目标

将当前“微信好友邀请”简化为一条清晰、可落地、符合微信小程序能力边界的流程：

1. 邀请人通过微信分享发送邀请
2. 好友点击邀请进入小程序
3. 若未完善资料，先进入资料完善页
4. 资料完善完成后生成/补全用户信息并建立登录态
5. 回到邀请确认页，展示邀请内容
6. 好友选择“接受”或“拒绝”
7. 后端根据好友选择更新邀请状态，并在接受时加入成员

本方案明确放弃“强定向好友校验”目标，不再依赖 `activityId` / `wx.authPrivateMessage` 形成后端强鉴权闭环，而是回归当前项目已经成熟的 `inviteCode + 用户级邀请记录` 模型。

---

## 平台与现状结论

### 平台边界

- 微信小程序可以用 `open-type="share"` 发起分享
- 不能像通讯录选择器一样由代码直接选择某个微信好友
- 私密消息能力即使接入，也不适合作为当前版本的后端可信鉴权基础

### 现有代码基础

当前仓库已经具备实现简化流程所需的关键能力：

- `miniprogram/pages/register/register.js`
  - 已能完善资料
  - 已能创建/更新用户并写入 `userInfo`、`isLoggedIn`、`loginTime`
- `miniprogram/pages/list-invite-accept/list-invite-accept.js`
  - 已能通过 `code/scene` 进入邀请接收页
  - 已有接受/申请/拒绝交互
- `cloudfunctions/listFunctions/index.js`
  - 已有 `getInviteInfo`
  - 已有 `acceptInvite`
  - 已有 `rejectInvite`
  - 已有 `applyJoinList`
  - 已有 `createInviteRecordFromTemplate`
- `profile/notifications/notifications.js`
  - 当前邀请通知仍走 `?code=...` 跳转，必须继续兼容

因此，本次设计不需要重写邀请体系，只需要围绕“登录前拦截 + 资料完成回跳 + 弱化私密消息逻辑”做收敛。

---

## 推荐设计

### 一、邀请模型定义

将“微信好友邀请”定义为：

- 邀请类型仍可保留 `inviteType: 'wechat'`
- 但其本质是“通过微信分享发送的邀请码模板邀请”
- 首次点击者在完成资料后，再根据自己的操作生成用户级记录并流转状态

### 二、整体用户流程

#### 1. 发起邀请

邀请页中：

- 用户选择角色
- 前端调用 `createWechatInvite`
- 后端创建一条模板邀请记录：
  - `inviteeId: ''`
  - `status: 0`
  - `inviteType: 'wechat'`
  - `inviteCode`
- 前端通过 `open-type="share"` 把 `inviteCode` 带入分享路径

#### 2. 好友点击分享卡片

进入：

- `/pages/list-invite-accept/list-invite-accept?code=xxx`

接收页首先做两件事：

- 校验 `inviteCode` 是否存在
- 检查当前是否已登录/是否已有有效 `userInfo`

#### 3. 未登录或未完善资料时

不直接调用 `getInviteInfo`，而是：

- 缓存待处理邀请上下文
- 跳转到 `/pages/register/register`
- 资料完善完成后自动回到邀请接收页

#### 4. 完善资料完成后

`register` 页完成：

- 调用 `profileFunctions.registerOrUpdateUser`
- 写入本地登录态
- 更新 `app.globalData`
- 根据预存的邀请回跳信息返回邀请页

#### 5. 邀请接收确认

回到邀请页后：

- 调用 `getInviteInfo(inviteCode)`
- 展示邀请人、清单名、角色、是否需要审批
- 用户点击“接受”或“拒绝”

#### 6. 后端状态流转

- 接受：
  - 若模板邀请，则 `createInviteRecordFromTemplate(..., status=0)`
  - 然后加入 `list_members`
  - 再更新该用户级邀请状态为 `1`
- 拒绝：
  - 若模板邀请，则 `createInviteRecordFromTemplate(..., status=2)`
  - 或复制后立即更新状态为 `2`
- 若开启审批：
  - 走 `applyJoinList`
  - 状态置为 `4`
  - 由邀请人审批

## 实施范围

### 1. 前端：邀请接收页改造

文件：`miniprogram/pages/list-invite-accept/list-invite-accept.js`

#### 改造目标

- 把“是否已登录/是否已完善资料”的判断前置
- 未登录时不直接报“用户不存在”或“邀请失效”
- 支持“补资料后回到当前邀请页继续处理”

#### 计划改动

- 新增登录态检查方法，例如：
  - `checkLoginStatus()`
  - 使用 `app.getLoginState()` 或本地 `userInfo/isLoggedIn`
- 在 `onLoad` / `onShow` 中：
  - 若无登录态，则缓存待处理邀请信息并跳转注册页
  - 若已有登录态，则调用 `loadInviteInfo()`
- 清理当前页面中与私密消息强校验相关的主流程依赖：
  - `privateMessageResult` 可整体移除，或仅保留为调试字段但不参与业务流转
- 接受/拒绝/申请动作只传：
  - `inviteCode`

#### 建议行为细节

- 第一次进入且未登录：
  - 显示短提示：`请先完善资料后再处理邀请`
  - 跳转注册页
- 从注册页返回后：
  - 自动重新拉取邀请信息

### 2. 前端：注册页支持邀请回跳

文件：`miniprogram/pages/register/register.js`

#### 改造目标

完善资料后不要只是 `navigateBack()`，而是优先恢复邀请上下文。

#### 计划改动

支持两种实现方式，推荐第一种：

#### 方案 A：本地缓存回跳信息（推荐）

进入注册页前写入本地缓存：

- `pendingInviteAction`
- 内容示例：
  - `type: 'listInvite'`
  - `inviteCode`
  - `redirectUrl: /pages/list-invite-accept/list-invite-accept?code=xxx`

注册成功后：

- 读取缓存
- 若存在邀请回跳上下文，则 `wx.redirectTo` 到邀请页
- 成功跳转后清除缓存
- 若不存在，则保持当前的返回逻辑

#### 方案 B：通过 query 透传 redirect

例如：

- `/pages/register/register?redirect=%2Fpages%2Flist-invite-accept%2Flist-invite-accept%3Fcode%3Dxxx`

注册成功后解析并跳回。

#### 方案选择建议

优先使用方案 A，因为：

- 当前 register 页 `onLoad` 还未消费参数
- 本地缓存更容易兼容多入口（邀请页、通知页、未来二维码页）
- 可避免复杂的 URL encode/decode 和页面栈问题

### 3. 前端：邀请页简化文案与分享逻辑

文件：`miniprogram/pages/list-invite/list-invite.js`
`miniprogram/pages/list-invite/list-invite.wxml`
`miniprogram/pages/list-invite/list-invite.wxss`

#### 改造目标

保留“两段式分享”交互，但去掉“强私密校验”叙事，文案改为真实能力范围内表述。

#### 计划改动

- 保留：
  - 先选角色
  - 再点击 `open-type="share"` 按钮发送
- 弱化或删除：
  - “仅本次接收该消息的微信好友可加入”
  - “不可二次转发”作为核心安全承诺
- 改为更准确的文案，例如：
  - `邀请已准备好，点击发送给微信好友`
  - `好友打开后需完善资料并确认是否加入`

#### `createWechatInvite` 保留内容

- 仍创建 `inviteCode`
- 仍作为 `inviteType: 'wechat'` 的模板邀请

#### 可选简化

若希望进一步收敛复杂度，可在本次实现中直接停止使用：

- `wx.updateShareMenu({ isPrivateMessage: true, activityId })`
- `activityId`
- `shareMode: 'private_message'`
- `shareTicketRequired`

保守做法是：

- 前端先不删 `open-type="share"`
- 后端停止把 `activityId/shareMode/shareTicketRequired` 作为业务语义依赖
- 后续再决定是否彻底移除字段

### 4. 后端：微信邀请回归模板邀请模型

文件：`cloudfunctions/listFunctions/index.js`

#### 改造目标

让 `wechat` 邀请与 `link` 邀请在处理方式上尽可能统一，只保留展示类型差异。

#### 计划改动

##### `createWechatInvite`

- 保留创建模板邀请记录的能力
- 继续返回 `inviteCode`
- 如果要做最简化实现，建议移除对 `createPrivateMessageActivity()` 的硬依赖

推荐方向：

- 若 `activityId` 创建失败，不要让整个微信邀请失败
- 更进一步可以直接不再调用 `createPrivateMessageActivity()`

##### `getInviteInfo`

- 保持当前逻辑：
  1. 当前用户已处理记录
  2. 当前用户待处理记录
  3. 模板邀请 `getInviteByCode`
- 不依赖任何 `privateMessageResult`
- 当用户已注册登录后，允许正常查看模板邀请内容

##### `acceptInvite`

- 保持现有成熟逻辑：
  - 用户级待处理记录直接处理
  - 模板邀请则复制为用户级记录后再接受
- 接受成功后状态应明确落到 `1`
- 若当前函数仍依赖辅助函数收敛状态，需要顺带检查这点是否完整

##### `rejectInvite`

- 与 `acceptInvite` 保持一致
- 模板邀请先复制为用户级记录，再置 `2`

##### `applyJoinList`

- 保持现状即可
- 若产品希望“防止链接扩散造成误加群”，建议让微信邀请默认 `needApproval: true`

### 5. 通知与外部入口兼容

文件：`miniprogram/pages/profile/notifications/notifications.js`

#### 目标

确保以下入口都还能走同一条简化流程：

- 分享卡片 `?code=...`
- 通知跳转 `?code=...`
- 小程序码 `scene=invite=...`

#### 计划改动

- 不修改通知跳转结构
- 只确保邀请页在未登录情况下会自动跳注册，而不是报错

## 数据与状态设计

### 本地缓存建议

新增一个轻量缓存键，例如：

- `pendingInviteContext`

结构建议：

```json
{
  "type": "listInvite",
  "inviteCode": "abc123",
  "redirectUrl": "/pages/list-invite-accept/list-invite-accept?code=abc123",
  "createdAt": 1778217272892
}
```

用途：

- 邀请页检测到未登录时写入
- 注册完成后消费并跳回
- 消费后立即清除

### 邀请状态语义

沿用现有状态，不新增新状态：

- `0` 待处理
- `1` 已接受
- `2` 已拒绝
- `3` 已过期
- `4` 待审批

这能最大限度降低后端改动面。

## 推荐产品策略

### 默认是否启用审批

这里有两个可选方向：

#### 方向 A：微信邀请默认直接接受

优点：

- 路径最短
- 用户体验最好

风险：

- 链接被转发后，拿到 code 的人理论上都可以加入

#### 方向 B：微信邀请默认需审批（更推荐）

优点：

- 更适合“微信分享链路不可强绑定好友”的现实
- 即使链接扩散，也只是进入申请流程

影响：

- 好友点击后按钮从“接受邀请”变成“申请加入”

推荐结论：

- 若你更重视安全性，推荐方向 B
- 若你更重视体验，可先保持当前 `needApproval` 可配置，不强改默认值

## 实施步骤建议

1. 先清理邀请接收页的登录前置逻辑
   - 未登录不再直接拉邀请信息
   - 改为跳转注册页
2. 给注册页增加“邀请回跳”能力
   - 优先用 `pendingInviteContext` 本地缓存
3. 接收页在登录后重新进入并拉取 `getInviteInfo`
4. 接受/拒绝动作只传 `inviteCode`
5. 弱化微信私密消息相关依赖和文案
6. 视范围决定是否顺手移除 `activityId` 相关硬依赖

## 验证清单

### 主流程

1. A 发送微信邀请给 B
2. B 未登录时点击邀请
3. B 被引导到注册页
4. B 完善头像昵称并保存
5. B 自动回到邀请页
6. B 能看到邀请详情
7. B 点击接受后成功加入清单
8. B 点击拒绝后状态变为已拒绝

### 兼容流程

1. 从通知页打开 `?code=...` 仍可正常处理邀请
2. 从 `scene=invite=...` 进入仍可解析邀请码
3. `link/search/recent` 邀请链路不受影响
4. 已登录用户点击邀请时不应被重定向到注册页
5. 已处理邀请再次打开时展示正确状态

### 异常流程

1. 邀请码不存在时展示无效提示
2. 邀请已过期时展示过期提示
3. 注册完成后本地缓存缺失时仍保持原返回逻辑
4. 接受时若已是成员，应返回“您已是该清单成员”而不是报错

## 风险与注意事项

- `register.js` 当前成功后默认 `navigateBack()`，改成邀请回跳时要兼容普通资料编辑场景
- 邀请页和注册页之间若用本地缓存传递上下文，要注意消费后清除，避免下次误跳
- `list-invite-accept.js` 当前还保留 `privateMessageResult` 相关代码，实施时应一并收敛，避免继续制造误导
- 如果保留 `activityId` 字段但不再依赖它，不应再用“定向好友”类强文案描述能力
- `acceptInvite/rejectInvite/applyJoinList` 的调用方判断既有 `success` 也有 `code === 0` 风格，实施时保持兼容返回结构

---

## 最终建议

按当前仓库结构，最合适的实现方案是：

- 将“微信好友邀请”收敛为“微信分享的邀请码邀请”
- 好友点击后，未登录先完善资料
- 完善完成后回到邀请页
- 再由好友明确选择接受或拒绝
- 后端基于现有模板邀请复制机制更新用户级状态
- 若担心分享扩散，则通过 `needApproval` 控制风险，而不是继续尝试用私密消息做强鉴权
