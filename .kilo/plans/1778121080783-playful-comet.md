# 任务目标

修复“前端无法把清单任务改回个人任务”的交互缺口，在不改变现有后端数据模型的前提下，使用户可以在任务编辑页和任务详情页中将任务从某个清单移出并保存为个人任务。

# 现状结论

- 后端已支持把任务改回个人任务：`taskFunctions.updateTask()` 接收空 `listId`，并同步写回 `ownershipType = 1`。
- 当前缺口在前端：任务编辑页和任务详情页的“选择清单”弹窗只展示真实清单，没有“个人任务/不设置清单”选项。
- 当前占位文案 `请选择清单（可选）` 无法准确表达“当前任务已明确属于个人任务”的语义。

# 修改范围

1. `miniprogram/mixins/taskMixin.js`
2. `miniprogram/pages/task-edit/task-edit.wxml`
3. `miniprogram/pages/task-detail/task-detail.wxml`

可选检查项：

4. `miniprogram/pages/task-edit/task-edit.js`
5. `miniprogram/pages/task-detail/task-detail.js`

# 实施方案

## 1. 扩展清单选择逻辑

文件：`miniprogram/mixins/taskMixin.js`

修改 `onListSelect(e)`，让其支持空 `listId` 作为“个人任务”选项：

- 当 `e.currentTarget.dataset.id` 为空字符串时：
  - 设置 `task.listId = ''`
  - 设置 `task.listName = ''`
  - 关闭 `showListPopup`
- 当 `id` 非空时，保持现有逻辑：从 `availableLists` 中找到对应清单并写入 `task.listId`、`task.listName`

设计理由：

- `buildSaveParams()` 已会把空 `task.listId` 序列化为 `listId: ''`
- 与现有“分类弹窗里支持不设置分类”的模式一致
- 仅修改一个共享方法即可同时覆盖 `task-edit` 与 `task-detail`

## 2. 在两个清单弹窗中增加固定选项“个人任务”

文件：

- `miniprogram/pages/task-edit/task-edit.wxml`
- `miniprogram/pages/task-detail/task-detail.wxml`

在 `wx:for="{{availableLists}}"` 之前插入一个固定列表项：

- 文案：`个人任务`
- 选中态条件：`{{!task.listId}}`
- 点击行为：`bindtap="onListSelect"` 且 `data-id=""`
- 视觉结构复用现有 `list-option` / `list-icon-small` 样式，不新增样式类，尽量最小改动

建议结构：

- 使用与普通清单项一致的容器类 `list-option`
- 左侧图标使用现有图标体系，优先复用 `label-o` 或更贴近个人任务语义的单人图标；如果当前 Vant 图标集不确定，优先使用已存在的 `label-o` 以降低风险
- 右侧勾选图标沿用 `van-icon name="success"`

## 3. 调整表单中的默认显示文案

文件：

- `miniprogram/pages/task-edit/task-edit.wxml`
- `miniprogram/pages/task-detail/task-detail.wxml`

将当前展示逻辑：

- `{{task.listName || '请选择清单（可选）'}}`

调整为：

- `{{task.listName || '个人任务'}}`

同时将清单展示文本的占位样式判断从仅依赖 `task.listName` / `task.listId`，改为同时识别“当前为个人任务”的有效状态，避免 `个人任务` 被渲染成灰色占位文案：

- 将 `picker-value` 的 `placeholder` 条件改成基于 `hasListSelection = !!task.listId || !task.listName`
- 在当前页面语义下可直接写成 `{{task.listId ? '' : ''}}` 的反向等价最小形式，即去掉“无清单时必定加 placeholder”的旧判断
- 目标效果是：`task.listId=''` 且 `task.listName=''` 时，展示 `个人任务`，但视觉上按已选状态展示，而不是未填写占位态

设计理由：

- 当 `listId=''` 时，这不是“未选择完成”的中间态，而是明确的业务归属：个人任务
- 能让用户直观看到当前任务是否属于清单

## 4. 保持 JS 页面层不做额外业务改动

文件：

- `miniprogram/pages/task-edit/task-edit.js`
- `miniprogram/pages/task-detail/task-detail.js`

预期无需修改页面脚本，原因：

- 详情/编辑页面初始化时已经把后端 `listId` 映射到 `task.listId`
- 保存参数由 `taskMixin.buildSaveParams()` 统一构建
- 后端已支持空 `listId`

但实现时需要确认以下一点：

- 页面初始化若未显式提供 `task.listName`，显示层依赖新的默认文案 `个人任务` 即可，不需要额外把 `listName` 写死为该字符串

# 验证方案

## 功能验证

1. 新建个人任务
- 进入任务创建页
- 不选择清单
- 页面应显示 `个人任务`
- 保存后任务应成功创建为个人任务

2. 清单任务改回个人任务
- 打开一个已有清单任务
- 点击选择清单
- 选择 `个人任务`
- 保存后应成功，且任务不再属于原清单

3. 个人任务改为清单任务
- 打开一个个人任务
- 选择某个清单
- 保存后应成功归属到该清单

4. 任务详情页同样验证
- 在 `task-detail` 页面重复上面两个切换方向的验证

## 数据验证

保存后检查云函数请求参数与结果是否满足：

- 从清单移出时，提交值包含 `listId: ''`
- 后端返回成功
- 重新进入详情页时，页面展示为 `个人任务`

## 回归验证

1. 原有选择清单功能不受影响
2. 不设置分类功能不受影响
3. 任务保存、删除、附件上传相关逻辑不受影响
4. 列表页 / 首页 / 日历页对该任务的展示不出现异常

# 风险与规避

## 风险 1：空字符串 dataset 行为不稳定

规避：

- 在 `onListSelect` 中使用“空值即个人任务”的显式分支处理
- 不依赖 `find()` 返回值推导“空值”语义

## 风险 2：文案修改影响用户对“可选清单”的理解

规避：

- 保持弹窗标题为“选择清单”，用户仍可进入后选择某个清单
- 表单主文案显示当前状态“个人任务”，优先保证状态可读性

## 风险 3：样式不统一

规避：

- 复用现有 `list-option` 和图标容器样式
- 不新增 CSS，除非实际渲染后明显不一致

# 实施顺序

1. 修改 `taskMixin.js` 的 `onListSelect()` 逻辑
2. 修改 `task-edit.wxml`，加入“个人任务”选项并更新默认文案
3. 修改 `task-detail.wxml`，加入“个人任务”选项并更新默认文案
4. 手动验证创建、编辑、详情三条主要路径

# 预期结果

- 用户可以在前端显式把清单任务改回个人任务
- 页面文案能够正确表达任务当前归属
- 不需要修改后端接口与数据结构
- 以最小改动完成能力闭环
