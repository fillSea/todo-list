# 成员邀请页最近协作者头像修复方案

## 问题定位

成员邀请页 `miniprogram/pages/list-invite/list-invite.wxml` 的最近协作者头像直接使用后端返回的 `item.avatarUrl`：

```xml
<image class="recent-avatar" src="{{item.avatarUrl || '/images/default-avatar.png'}}" mode="aspectFill" />
```

真实环境中 `getRecentCollaborators` 云函数返回的是数据库原始 `avatarUrl`，其中用户注册头像通常是云存储 fileID，例如 `cloud://...`。该接口没有调用已有的 `resolveAvatarUrl` 转换为临时 HTTP URL，导致 `<image>` 无法正常展示，同时因为 `cloud://...` 是非空字符串，也不会回退到默认头像。

核心位置：

- 前端数据来源：`miniprogram/pages/list-invite/list-invite.js` 的 `loadRecentMembers`
- 后端接口：`cloudfunctions/listFunctions/index.js` 的 `getRecentCollaborators`
- 已有工具函数：`cloudfunctions/listFunctions/index.js` 的 `resolveAvatarUrl`

## 修改目标

1. 让最近协作者头像在真实云函数环境中正常显示。
2. 保持与清单成员、我的清单等其他接口的头像处理方式一致。
3. 尽量做最小改动，不改变邀请逻辑、筛选逻辑和权限逻辑。
4. 增加前端图片加载失败兜底，避免坏 URL 导致头像空白。

## 实施步骤

### 1. 修复云函数最近协作者头像返回

修改文件：`cloudfunctions/listFunctions/index.js`

在 `getRecentCollaborators` 末尾构造 `members` 的地方，将同步 `map` 改为 `Promise.all`，并对头像调用 `resolveAvatarUrl`。

当前逻辑：

```js
const members = uniqueUserIds
  .filter(id => userMap[id])
  .map(id => ({
    userId: id,
    nickname: userMap[id].nickname,
    avatarUrl: userMap[id].avatarUrl
  }));
```

建议改为：

```js
const members = await Promise.all(
  uniqueUserIds
    .filter(id => userMap[id])
    .map(async id => ({
      userId: id,
      nickname: userMap[id].nickname || '未知用户',
      avatarUrl: await resolveAvatarUrl(userMap[id].avatarUrl || '')
    }))
);
```

原因：

- `resolveAvatarUrl` 已经支持 HTTP URL 原样返回。
- `resolveAvatarUrl` 已经支持 `cloud://` 转临时链接。
- `resolveAvatarUrl` 对空值和异常会返回空字符串，前端可回退默认头像。
- 与 `getListMembers` 等接口现有处理方式一致。

### 2. 增加最近协作者头像加载失败兜底

修改文件：`miniprogram/pages/list-invite/list-invite.wxml`

为最近协作者头像增加 `binderror`，携带当前下标：

```xml
<image
  class="recent-avatar"
  src="{{item.avatarUrl || '/images/default-avatar.png'}}"
  mode="aspectFill"
  binderror="onRecentAvatarError"
  data-index="{{index}}"
/>
```

修改文件：`miniprogram/pages/list-invite/list-invite.js`

增加方法：

```js
onRecentAvatarError(e) {
  const index = e.currentTarget.dataset.index;
  if (index === undefined || index === null) return;

  this.setData({
    [`recentMembers[${index}].avatarUrl`]: '/images/default-avatar.png'
  });
}
```

原因：

- 后端修复后正常情况下不会触发。
- 如果临时链接过期、第三方 HTTP 图片失败、历史脏数据异常，前端仍能显示默认头像。
- 只影响最近协作者列表，不影响其他页面。

### 3. 可选统一搜索用户头像处理

同一个邀请页中搜索用户结果也展示头像：

```xml
<image class="result-avatar" src="{{item.avatarUrl || '/images/default-avatar.png'}}" mode="aspectFill" />
```

后端 `searchUser` 需要确认是否已经调用 `resolveAvatarUrl`。如果没有，建议同步修复为返回临时 URL，避免搜索结果头像也出现同类问题。

这个步骤不是最近协作者问题的必要修复，但属于同类隐患治理。

### 4. 验证方案

由于该项目主要通过微信开发者工具运行，没有可用的根目录测试脚本，验证以微信开发者工具和云函数调用为主。

验证点：

1. 部署 `listFunctions` 云函数后，打开成员邀请页。
2. 确认最近协作者列表中历史协作者头像正常显示。
3. 确认没有头像的用户显示 `/images/default-avatar.png`。
4. 确认点击最近协作者仍能弹出权限选择，并可正常发送邀请。
5. 确认待处理邀请、微信邀请、邀请链接、搜索用户功能没有回归。
6. 可在云函数日志里确认 `转换头像临时链接失败` 没有持续报错。

## 风险与注意事项

- 需要重新上传部署 `cloudfunctions/listFunctions`，否则前端仍会收到旧数据。
- 云存储临时链接有有效期，页面每次加载重新调用云函数获取即可。
- 如果用户数据库中的 `avatarUrl` 存的是无效 `cloud://` fileID，后端会返回空字符串，前端显示默认头像，这是预期兜底。
- 不建议前端直接处理 `cloud://`，因为云存储临时链接转换应统一放在云函数侧，避免各页面重复实现。

## 最小实施范围

必做：

- `cloudfunctions/listFunctions/index.js`：`getRecentCollaborators` 中 `avatarUrl` 调用 `resolveAvatarUrl`。

推荐一起做：

- `miniprogram/pages/list-invite/list-invite.wxml`：最近协作者头像增加 `binderror`。
- `miniprogram/pages/list-invite/list-invite.js`：新增 `onRecentAvatarError` 方法。

可选：

- 检查并统一 `searchUser` 的头像返回逻辑。
