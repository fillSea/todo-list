# 个人任务逻辑与删除功能改进方案

## 目标

1. 复核“个人任务界面中普通任务与周期任务”的现有逻辑，明确是否存在实现错误或设计缺陷。
2. 评估当前任务删除能力是否完整；若已存在，则提出改进方案并给出实现设计。

## 已确认决策

本方案按以下产品决策执行：
- 周期任务删除采用方案 1：先让用户选择“仅删除本次 / 删除整个周期”。

该决策在后续实现中视为已固定，不再二次确认。

## 最终实施决策

本次进入实现阶段时，以下范围与优先级一并锁定：

1. 按推荐方案 1 执行，不再评估其他删除交互方案
- 周期任务删除统一采用“先选范围，再最终确认”的双层交互。
- 适用页面固定为：`task-edit`、`task-detail`、`list-detail`。
- `calendar` 不新增删除入口，只在必要时跟随共用 helper 调整。

2. 本轮优先级固定为“后端规则收敛优先，前端交互跟进”
- 先修后端状态流转、周重复编码、系列删除语义。
- 再改前端各入口的确认弹窗、删除范围选择、刷新与缓存清理。
- 最后统一前端逾期派生逻辑，避免前端先改后又被后端响应形态打回。

3. 本轮不扩展需求
- 不新增新的删除 API，继续扩展现有 `deleteTask`。
- 不新增日历页删除按钮。
- 不处理统计页算法重做、不处理更大范围的数据模型调整。

4. 状态变更规则以“后端单一事实来源”为最终准则
- `toggleTaskStatus()` 与 `updateTask()` 的规则必须收敛到同一套校验函数。
- 前端不自行决定是否允许状态切换，只根据后端返回的 `needConfirm*` 信号继续确认或提示。

5. 删除能力以“语义完整且立即生效”为最终准则
- 删除单次周期实例时，只影响当前实例。
- 删除整个周期时，必须同步删除父任务、子实例、相关通知、相关附件引用，不依赖定时清理兜底。
- 从子实例发起系列删除也必须生效。

## 规划结论

规划阶段到此结束，下一会话应直接进入实现，不再重复确认以下事项：
- 删除方案采用推荐方案 1。
- 本轮范围包含后端规则修复、前端删除交互补齐、状态确认一致化、逾期 helper 统一。
- 本轮范围不包含日历页删除入口、统计重构、自动化测试建设。
- 验证方式以代码检查加 WeChat DevTools 手工回归为准。

## 现状结论

### 1. 个人任务逻辑并非完全正确，存在可确认的问题

当前实现整体方向是成立的：
- 普通任务是单条任务记录。
- 周期任务是“父任务 + 预生成实例”的模型。
- 首页个人任务页会对周期任务做“进行中仅显示每个系列最近一条实例”的折叠展示。

但代码层面至少存在以下明确问题：

#### A. 我之前的解释有一处不够准确

我之前把“状态变化规则”主要按 `toggleTaskStatus` 讲了，但实际编辑页 `task-edit` / `task-detail` 保存时走的是 `updateTask`，而 `updateTask` 会直接写入 `status` 字段：
- `cloudfunctions/taskFunctions/index.js:450-453`

这意味着：
- 列表勾选完成时，周期任务会被限制为“只能完成当天实例”；
- 但编辑页直接保存状态时，后端没有复用 `toggleTaskStatus` 的限制逻辑；
- 因而存在“通过编辑页绕过周期任务状态约束”的实现缺口。

这不是纯文档问题，而是实际业务规则不一致。

#### B. 周重复编码前后端存在不一致风险

前端周选择值：
- `taskMixin.js` 中 `weekdays` 用 `0-6`，其中 `0=周日`

后端周期推算：
- `calculateNextRepeatDate()` 里按 `1-7` 思路处理，且把 `getDay() || 7` 视作“周日=7”
- 位置：`cloudfunctions/taskFunctions/index.js:1670-1689`

这会导致：
- 前端存的 `repeatValue` 可能是 `0,1,3` 这种格式；
- 后端却按 `1-7` 理解；
- 周日和部分跨周推算可能出现偏差。

这是明确的逻辑错误，应修复。

#### C. 首页“过期”判断与日历页/后端约束存在粒度不一致

首页 `index` 页按“日期零点”判断是否过期：
- `miniprogram/pages/index/index.js:163-175`

日历页对当天任务有更细的处理注释；后端对周期任务完成限制则按 UTC+8 的“当天”来判断。

结果是：
- 首页显示层的“是否过期”与某些操作层规则并不完全同源；
- 用户可能看到某条任务还在“进行中”，但在其他页面/操作时触发不同判断。

这更偏设计不一致，不一定是致命 bug，但应统一规则来源。

#### D. 周期任务删除只删除当前记录，没有处理同系列实例

现有 `deleteTask`：
- 只删除当前 `taskId`
- 删除当前任务附件
- 删除 `notifications.relatedId = taskId`
- 不处理同系列周期实例
- 位置：`cloudfunctions/taskFunctions/index.js:510-589`

这会带来两个问题：
- 删除父周期任务后，未来实例仍保留，要等定时任务 `cleanupExpiredPeriodicTasks` 才清理孤儿实例；
- 删除某个周期实例时，没有区分“仅删本次”还是“删除整个周期系列”，用户语义不清晰。

这不是“没有删除功能”，而是“删除能力不完整”。

### 2. 当前程序已经存在删除任务功能

前端入口已存在：
- `miniprogram/pages/task-edit/task-edit.js`
- `miniprogram/pages/task-detail/task-detail.js`
- `miniprogram/pages/list-detail/list-detail.js`

后端接口已存在：
- `cloudfunctions/taskFunctions/index.js` -> `deleteTask(openid, data)`

因此本次不需要“从零编写删除功能”，而是应该“改进删除语义、删除范围与前后端一致性”。

## 设计目标

### 目标一：统一普通任务与周期任务的规则入口

把“状态修改约束”从页面分散判断，收敛为后端统一规则，前端只负责：
- 发起操作
- 接收 `needConfirm*` 类型响应
- 做对应确认与刷新

这样可以避免：
- 首页可用但详情页可绕过
- 日历页和首页规则漂移

### 目标二：明确周期任务的删除语义

把周期任务删除区分为两种：
- 仅删除当前实例
- 删除整个周期系列

普通任务仍保持单一删除。

### 目标三：修复周期规则中的编码与时间边界问题

重点修复：
- 周重复 `repeatValue` 编码不一致
- 过期/当天判断口径分裂

## 推荐方案

## 一、个人任务逻辑改进方案

### 1. 统一周期任务“每周”编码

推荐统一为：
- 前后端都使用 `0-6`
- `0=周日, 1=周一, ..., 6=周六`

原因：
- 前端已经是这个模型；
- WeChat/JS 原生 `Date.getDay()` 也是这个模型；
- 改动面更小，风险低于把前端全部改到 `1-7`。

需要修改：
- `cloudfunctions/taskFunctions/index.js` 中 `calculateNextRepeatDate()` 的周计算逻辑
- 相关注释
- 如有按 `1-7` 假设的其他页面逻辑，也一并统一核对

兼容策略：
- 后端解析周重复值时，优先支持标准 `0-6`；
- 若遇到历史异常值 `7`，按周日兼容；
- 兼容仅用于读取与推算，前端后续保存统一回写 `0-6`。

### 2. 统一状态修改规则，禁止通过编辑页绕过业务限制

推荐方案：
- `updateTask()` 不再直接“裸写” `status`。
- 当编辑页提交的 `status` 与旧状态不同：
  - 复用 `toggleTaskStatus()` 的校验逻辑，或提炼共享函数；
  - 保持过期确认、周期任务“只能完成当天”、取消完成确认等规则一致。

实现方式建议：
- 提炼一个后端内部函数，例如 `validateTaskStatusTransition(oldTask, targetStatus, options)`；
- `toggleTaskStatus()` 和 `updateTask()` 共同调用；
- `updateTask()` 在需要确认时返回与 `toggleTaskStatus()` 同结构的 `needConfirm*` 响应。

前端配套：
- `task-edit` 页需要补齐对 `needConfirmComplete`、`needConfirmCompleteNotToday`、`needConfirmUncheck` 的处理；
- `task-detail` 页需要补齐对 `needConfirmCompleteNotToday`、`needConfirmUncheck` 的处理；
- 详情页和编辑页都要在确认后重新发起保存，避免只改本地 UI。

### 3. 统一“过期”判断口径

建议抽象前端公共判断逻辑，至少在：
- `index/index.js`
- `calendar/calendar.js`
- `list-detail/list-detail.js`

使用相同的规则函数。

推荐口径：
- 状态持久化仍只保留 `0/1`；
- 逾期仍然是派生状态；
- 派生时统一以 UTC+8 的“日期边界”判断；
- 若产品上希望“当天过了具体截止时分也算过期”，则所有页面一起改，而不是首页/日历页各自为政。

本次建议优先做“统一但不改变产品语义”：
- 沿用当前主要模型：按日期判断逾期；
- 把实现抽成公共函数，减少多处漂移。

实现策略：
- 优先提取页面内共享 helper，若能放入现有公共模块则放入 `taskMixin` 或轻量 util；
- 不做大规模架构重构，只消除首页、日历页、清单页三处判断漂移。

### 4. 明确首页周期任务显示策略并补充注释/守护逻辑

当前首页策略本身可保留：
- 每个周期系列只展示最近的未过期未完成实例；
- 已过期实例全部展示；
- 已完成实例全部展示。

但建议补充：
- 注释明确“父任务本身也可能作为系列中的首个实例参与显示”；
- 如果删除父任务但实例残留，应在列表整理时避免出现不可达孤儿数据的异常展示。

具体守护策略：
- 首页与清单页在计算周期分组时，统一使用 `parentTaskId || _id`；
- 若发现子实例存在但父任务缺失，不阻断展示，但不得把其错误识别为新的独立系列；
- 系列删除完成后依赖主动刷新与缓存清理，不依赖孤儿兼容逻辑兜底。

## 二、删除功能改进方案

## 现状问题

当前已有删除，但不足点是：
- 普通任务和周期任务使用同一个确认文案，缺少语义区分；
- 删除周期任务时无法选择范围；
- 删除父周期任务不会立即联动删除系列实例；
- 附件删除仅覆盖当前任务本身；
- 通知删除仅覆盖当前 `taskId`；
- 各页面删除交互不统一。

## 推荐删除语义

### 普通任务

保持现状：
- 删除当前任务
- 删除当前任务附件
- 删除当前任务相关通知

### 周期任务

提供两种删除方式：

#### 方案 A：仅删除本次任务实例

适用：
- 用户只想跳过某一天/某一周/某一月

行为：
- 删除当前实例
- 不影响同系列其他实例
- 不影响父任务配置

#### 方案 B：删除整个周期系列

适用：
- 用户不再需要这个周期任务

行为：
- 删除父任务
- 删除全部子实例 `parentTaskId = parentId`
- 删除这些任务关联的附件与通知
- 立即完成，不依赖凌晨定时清理

## 推荐交互

### 1. 在详情页 / 编辑页

如果当前任务 `repeatType === 0`：
- 保持单个删除确认框

如果当前任务 `repeatType > 0`：
- 点击“删除任务”后先弹操作选择：
  - 删除本次
  - 删除整个周期
  - 取消
- 选择范围后，再弹最终确认框
- 最终确认文案按范围区分：
  - 删除本次：`确定删除本次任务吗？后续周期任务将保留。`
  - 删除整个周期：`确定删除整个周期任务吗？该周期下所有任务实例将一并删除。`

### 2. 在列表页

列表页中对周期任务也使用同样策略。

为了减少误删，建议：
- 周期任务在 action sheet 中点“删除任务”后，不直接进入最终确认框；
- 先二次选择删除范围；
- 再做最终确认。

## 后端接口设计

推荐扩展 `deleteTask` 入参，而不是新建多个接口。

### 入参建议

```js
{
  taskId: string,
  deleteScope?: 'single' | 'series'
}
```

默认行为：
- 普通任务默认 `single`
- 周期任务如果没传 `deleteScope`，前端应默认在调用前已选好范围；
- 后端仍需兜底：若周期任务未提供范围，则按 `single` 处理并记录日志，避免接口行为不确定。

### 后端删除逻辑建议

#### 普通任务 / 周期实例单次删除
- 删除当前任务记录
- 删除当前任务附件文件
- 删除 `notifications.relatedId = 当前taskId`
- 记录操作日志，附带删除范围 `single`

#### 周期系列整组删除
1. 根据当前任务找到系列父 ID：
   - `const parentTaskId = oldTask.parentTaskId || oldTask._id`
2. 查询：
   - 父任务本身 `_id = parentTaskId`
   - 所有子实例 `parentTaskId = parentTaskId`
3. 汇总所有任务：
   - 删除所有附件云文件
   - 删除所有相关通知
   - 删除所有任务记录
4. 记录一次系列级日志

### 额外注意

- 删除系列时必须按权限校验一次，但无需逐条重复校验系列内每个实例；
- 删除系列时不要依赖 `cleanupExpiredPeriodicTasks()` 异步收尾；
- 需要处理“从子实例发起删除整个系列”的场景；
- 删除系列时如父任务已不存在，仍应基于 `parentTaskId` 删除剩余实例，保证幂等；
- 删除附件云文件时需去重 `fileId`，避免重复删除同一附件导致无意义报错。

## 验证方案

由于仓库没有可用自动化测试脚本，本次主要采用代码级验证 + WeChat DevTools 手工验证。

### 逻辑验证清单

#### 任务状态
1. 普通未过期任务可正常完成/恢复
2. 普通过期任务完成时弹确认
3. 周期任务仅当天实例可直接完成
4. 未来周期实例不能通过首页、日历页、编辑页绕过完成限制
5. 周期任务取消完成时，所有入口行为一致
6. 编辑页修改其他字段但不改状态时，不应触发多余确认流程

#### 周重复
1. 选择周日重复后，下次生成日期正确
2. 选择多星期组合时，跨周推进正确
3. 旧数据若已经存为 `0-6`，兼容显示与生成
4. 异常旧值 `7` 若存在，也能被兼容解释为周日

#### 删除
1. 普通任务删除成功，列表刷新正常
2. 周期任务“删除本次”仅删除当前实例
3. 周期任务“删除整个周期”删除父任务和全部实例
4. 删除整个周期后，不再依赖定时清理才能恢复界面正确性
5. 相关附件和通知同步删除
6. 首页、详情页、编辑页、清单页删除体验一致
7. 从子实例发起“删除整个周期”时，系列能被完整删除
8. 父任务已缺失时，按系列删除剩余实例仍能成功

## 实施顺序

1. 修正后端周重复编码与状态变更共享校验
2. 扩展后端 `deleteTask` 支持系列删除
3. 改造 `task-edit` / `task-detail` 删除与状态确认逻辑
4. 改造 `list-detail` 删除范围选择
5. 统一前端逾期派生逻辑
6. 在 WeChat DevTools 做核心路径回归验证

### 顺序锁定说明

- 不先改前端删除弹窗，再回头补后端语义；否则容易出现前端文案已承诺“删除整个周期”，但后端仍只删当前记录。
- 不先做显示 helper 统一，再处理状态与删除语义；否则需要重复联调刷新与缓存行为。
- `task-edit` 与 `task-detail` 一起处理，避免一个页面支持完整确认、另一个页面仍可绕过规则。

## 具体执行任务

### 任务 1：后端状态与周期规则修复
- 在 `taskFunctions` 内提炼状态流转校验函数

## 风险与注意事项

1. 周重复编码修复可能影响历史已存数据
- 需要先假设当前库里多数数据来自现有前端，即 `0-6`
- 实现时应尽量兼容异常旧值，避免一次性破坏已有周期任务

2. 系列删除会涉及多条任务、多附件、多通知
- 要注意云函数执行时长与批量数量
- 若系列很长，后续可考虑分批删除，但当前预生成窗口有限，先做同步删除基本可接受

3. 缓存一致性
- 删除或状态变更后，要继续调用 `app.clearTaskCaches()`
- 相关页面必须刷新列表或重新拉取数据

4. 非本次范围内不改动
- 不重做数据库结构
- 不改周期统计页口径
- 不引入新的全局状态管理

## 建议的实施边界

本轮建议只做：
- 逻辑修正
- 删除能力增强
- 交互一致性提升

本轮不建议顺手改：
- 周期任务统计页算法
- 大规模重构页面公共模块
- 新增自动化测试基础设施

- 主要路径已完成 WeChat DevTools 手工验证
