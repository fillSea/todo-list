# 共享清单实时同步实施方案

## 目标

让共享清单成员在清单详情页内实时看到其他成员对任务产生的变化，包括任务完成/恢复、创建、编辑、删除，以及周期任务生成或清理导致的列表变化。避免要求用户手动刷新页面。

## 现状

- 共享清单详情页位于 `miniprogram/pages/list-detail/list-detail.js`。
- 页面通过 `listFunctions.getListDetail` 加载清单、成员和任务列表。
- 当前用户操作任务后只会在本机调用 `loadListDetail(false)` 刷新。
- 其他成员没有监听机制，只能通过重新进入页面或刷新看到变化。
- 任务增删改和状态切换集中在 `cloudfunctions/taskFunctions/index.js`：
  - `createTask`
  - `updateTask`
  - `deleteTask`
  - `toggleTaskStatus`
  - `batchUpdateTasks`
  - `batchDeleteTasks`
- 后端已存在 `recordOperation(type, targetId, userId, content, listId)`，可作为触发实时版本更新的相邻位置。

## 推荐技术方案

采用轻量版本集合触发前端实时监听，而不是直接监听 `tasks` 集合。

新增云数据库集合：

```txt
list_versions
```

每个清单一条版本记录：

```js
{
  listId: '清单ID',
  version: 1,
  updatedAt: Date,
  updatedBy: '用户ID',
  eventType: 'task_update'
}
```

后端每次成功修改共享清单任务后更新该集合对应记录。前端清单详情页只监听 `list_versions.where({ listId })`，收到变化后防抖调用现有 `loadListDetail(false)`。

优点：

- 不需要直接开放 `tasks` 客户端读取/监听权限。
- 保留现有 `listFunctions.getListDetail` 的云函数鉴权。
- 监听数据量小，成本和复杂度低。
- 可覆盖任务新增、编辑、删除、状态切换、批量操作和周期实例变化。
- 前端继续复用现有任务处理逻辑，包括 `processTaskData`、`filterPeriodicTasks`、`applyTaskFilter`、进度计算和权限状态。

## 后端修改计划

修改文件：

```txt
cloudfunctions/taskFunctions/index.js
```

### 1. 增加实时版本更新辅助函数

在 `recordOperation` 附近增加函数：

```js
async function touchListVersion(listId, userId, eventType) {
  if (!listId) return;

  try {
    const { data } = await db.collection('list_versions')
      .where({ listId })
      .limit(1)
      .get();

    const payload = {
      listId,
      updatedAt: db.serverDate(),
      updatedBy: userId,
      eventType
    };

    if (data.length > 0) {
      await db.collection('list_versions').doc(data[0]._id).update({
        data: {
          ...payload,
          version: _.inc(1)
        }
      });
      return;
    }

    await db.collection('list_versions').add({
      data: {
        ...payload,
        version: 1
      }
    });
  } catch (error) {
    console.error('更新清单实时版本失败:', error);
  }
}
```

说明：

- 该函数失败不应中断任务主流程，所以内部捕获异常。
- `listId` 为空时跳过，避免个人任务触发共享清单监听。
- 使用 `_.inc(1)` 保证并发更新时版本递增。

### 2. 任务创建后触发版本更新

在 `createTask` 中，当前已有：

```js
await recordOperation('task_create', result._id, userId, { taskTitle: taskData.title, task: taskData }, taskData.listId);
```

在其后或周期任务生成后触发：

```js
await touchListVersion(taskData.listId, userId, 'task_create');
```

如果希望周期任务生成完成后再刷新，放在 `generatePeriodicTasks` 之后更稳妥。

### 3. 任务编辑后触发版本更新

在 `updateTask` 成功更新并记录操作后触发：

```js
await touchListVersion(nextListIdForLog, userId, 'task_update');
```

如果支持任务从一个清单移动到另一个清单，需要同时触发旧清单和新清单：

```js
const affectedListIds = [...new Set([oldTask.listId || '', nextListId || ''].filter(Boolean))];
for (const affectedListId of affectedListIds) {
  await touchListVersion(affectedListId, userId, 'task_update');
}
```

### 4. 单个任务删除后触发版本更新

在 `deleteTask` 成功删除任务并记录操作后触发：

```js
await touchListVersion(oldTask.listId, userId, 'task_delete');
```

对于删除整个周期任务的分支，也应触发一次对应清单版本：

```js
await touchListVersion(oldTask.listId, userId, 'task_delete_repeat');
```

### 5. 任务状态切换后触发版本更新

在 `toggleTaskStatus` 中，数据库更新、操作日志记录、必要的周期任务补充生成之后触发：

```js
await touchListVersion(oldTask.listId, userId, 'task_status_change');
```

建议放在 `ensurePeriodicTasks` 之后，这样其他成员刷新时能拿到完整周期实例变化。

### 6. 批量更新后触发版本更新

`batchUpdateTasks` 当前逐个校验并更新任务，但没有操作日志。计划在每个成功结果中带上 `listId`，然后在 `Promise.all` 后汇总触发：

```js
return { taskId, success: true, listId: task.listId || '' };
```

汇总：

```js
const affectedListIds = [...new Set(results.filter(r => r.success && r.listId).map(r => r.listId))];
for (const listId of affectedListIds) {
  await touchListVersion(listId, userId, 'task_batch_update');
}
```

### 7. 批量删除后触发版本更新

同理让成功结果返回 `listId`：

```js
return { taskId, success: true, listId: task.listId || '' };
```

汇总：

```js
const affectedListIds = [...new Set(results.filter(r => r.success && r.listId).map(r => r.listId))];
for (const listId of affectedListIds) {
  await touchListVersion(listId, userId, 'task_batch_delete');
}
```

## 前端修改计划

修改文件：

```txt
miniprogram/pages/list-detail/list-detail.js
```

### 1. 页面实例保存监听器

不要把监听器对象放入 `data`。在页面实例上使用：

```js
this.realtimeWatcher = null;
this.realtimeRefreshTimer = null;
this.realtimeReady = false;
```

可在 `onLoad` 初始化，或在方法中惰性使用。

### 2. 页面显示时启动监听

当前 `onShow` 为：

```js
onShow: function () {
  if (this.data.listId) {
    this.loadListDetail(false);
  }
}
```

计划改为：

```js
onShow: function () {
  if (this.data.listId) {
    this.loadListDetail(false);
    this.startRealtimeWatcher();
  }
}
```

`onLoad` 仍保留首次 `loadListDetail()`。

### 3. 页面隐藏/卸载时关闭监听

新增：

```js
onHide: function () {
  this.stopRealtimeWatcher();
},

onUnload: function () {
  this.stopRealtimeWatcher();
}
```

避免重复监听、后台页面继续刷新和资源泄漏。

### 4. 增加监听启动方法

```js
startRealtimeWatcher() {
  if (DEBUG_MODE) return;

  const { listId } = this.data;
  if (!listId) return;

  this.stopRealtimeWatcher();

  const db = wx.cloud.database();
  this.realtimeReady = false;

  this.realtimeWatcher = db.collection('list_versions')
    .where({ listId })
    .watch({
      onChange: snapshot => {
        if (snapshot.type === 'init') {
          this.realtimeReady = true;
          return;
        }

        if (!this.realtimeReady) {
          this.realtimeReady = true;
          return;
        }

        this.scheduleRealtimeRefresh();
      },
      onError: err => {
        console.error('共享清单实时监听失败:', err);
        this.stopRealtimeWatcher();
      }
    });
}
```

说明：

- `init` 事件不刷新，避免页面刚进入时重复请求。
- 监听失败时仅关闭监听，不影响现有手动刷新和页面加载能力。

### 5. 增加防抖刷新方法

```js
scheduleRealtimeRefresh() {
  if (this.realtimeRefreshTimer) {
    clearTimeout(this.realtimeRefreshTimer);
  }

  this.realtimeRefreshTimer = setTimeout(() => {
    app.clearTaskCaches();
    this.loadListDetail(false);
  }, 300);
}
```

说明：

- 防止一次批量操作或周期任务生成触发多次刷新。
- 刷新仍走云函数 `getListDetail`，权限和数据加工保持一致。

### 6. 增加停止监听方法

```js
stopRealtimeWatcher() {
  if (this.realtimeWatcher) {
    this.realtimeWatcher.close();
    this.realtimeWatcher = null;
  }

  if (this.realtimeRefreshTimer) {
    clearTimeout(this.realtimeRefreshTimer);
    this.realtimeRefreshTimer = null;
  }

  this.realtimeReady = false;
}
```

## 数据库与权限配置

需要在云开发数据库中新建集合：

```txt
list_versions
```

建议权限策略：

- 写入：仅云函数写入。
- 读取/监听：允许小程序端读取必要字段以支持 `watch`。

如果数据库安全规则支持跨集合校验成员身份，建议限制为：当前用户是该 `listId` 的成员或创建者时可读。

如果当前环境无法配置精确规则，可临时让集合仅包含非敏感字段：

```js
{
  listId,
  version,
  updatedAt,
  eventType
}
```

注意：即使 `list_versions` 暴露，也不应包含任务标题、成员昵称、操作详情等敏感内容。真实任务数据仍通过 `listFunctions.getListDetail` 云函数鉴权返回。

## 兼容与边界处理

- `DEBUG_MODE` 下不启动实时监听。
- 个人任务 `listId` 为空时不触发版本更新。
- 前端监听失败不影响原有页面加载、下拉刷新、返回刷新逻辑。
- 如果某个用户自己操作任务，也可能收到版本变化事件；由于当前本地操作本来会刷新一次，额外监听刷新通过防抖降低影响。如需进一步优化，可在 `list_versions.updatedBy` 等于当前用户时跳过，但初版不建议跳过，避免周期任务或服务端副作用漏刷。
- 任务移动清单时，需要旧清单和新清单都触发版本，保证两个清单页面都能更新。
- 批量操作只需每个受影响清单触发一次版本更新。
- 定时触发器生成/清理周期任务如果影响共享清单，后续可复用 `touchListVersion`；本次优先覆盖用户在清单内的实时操作。

## 验证计划

由于项目主要通过微信开发者工具运行，验证以小程序真机/模拟器为主。

1. 在云开发数据库创建 `list_versions` 集合并配置读取/监听权限。
2. 部署更新后的 `taskFunctions` 云函数。
3. 使用两个账号或两个微信开发者工具会话进入同一个共享清单详情页。
4. 账号 A 完成/恢复任务，账号 B 应在不刷新页面的情况下看到任务状态和进度条变化。
5. 账号 A 新增任务，账号 B 应自动看到新任务。
6. 账号 A 编辑任务标题/时间/优先级，账号 B 应自动看到变化。
7. 账号 A 删除任务，账号 B 应自动移除该任务。
8. 对周期任务执行完成/恢复/删除，确认账号 B 的周期过滤后列表和进度同步。
9. 断开或禁止 `list_versions` 监听权限，确认页面仍可正常手动加载，只在控制台输出监听失败日志。
10. 返回上一页再进入清单详情，确认不会出现重复刷新、重复监听或控制台持续报错。

## 实施顺序

1. 后端增加 `touchListVersion`。
2. 后端在任务创建、编辑、删除、状态切换、批量更新、批量删除成功路径触发版本更新。
3. 前端 `list-detail.js` 增加监听启动、停止和防抖刷新方法。
4. 前端在 `onShow` 启动监听，在 `onHide`/`onUnload` 停止监听。
5. 云数据库创建并配置 `list_versions` 集合。
6. 部署云函数并在微信开发者工具/真机验证。

## 预期结果

共享清单内任一成员操作任务后，其余停留在同一清单详情页的成员会自动刷新并看到最新任务状态、列表和进度，不再需要手动刷新页面。
