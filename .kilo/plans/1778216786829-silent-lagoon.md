# 清单微信好友邀请改造方案

## 目标

将当前“微信好友邀请”从公开传播的邀请码机制，改造成尽可能接近“仅对特定微信好友有效”的机制，并在用户点击邀请后直接进入微信分享流程，而不是弹窗提示用户手动点右上角。

## 先决结论

### 平台能力边界

根据微信开放文档，当前小程序可用能力包括：

1. 通过 `button open-type="share"` 直接发起分享流程。
2. 通过 `wx.updateShareMenu({ isPrivateMessage: true, activityId })` 将本次分享声明为“私密消息”。
3. 接收方打开分享卡片后，可通过 `shareTicket + wx.authPrivateMessage` 验证该用户是否为这次私密消息的合法接收者。

### 明确不能实现的点

以下诉求无法由普通微信小程序实现：

1. 代码直接拉起“微信好友列表”并让用户从列表中选择某个好友。
2. 在分享前拿到目标好友的微信身份并提前写入邀请记录。
3. 在前端精确知道用户这次到底发给了哪个具体好友。

因此，用户提出的“点击邀请设置后直接拉起用户的好友列表让用户发送邀请”，在平台层面不能原样实现。

## 建议采用的可实施方案

### 方案定位

将“微信好友邀请”改为：

1. 用户点击“微信好友邀请”按钮后，直接进入微信原生分享面板。
2. 本次分享被声明为“私密消息”。
3. 接收者只有在属于本次私密消息合法接收范围内时，才允许继续处理邀请。
4. 私密消息不可二次转发，从而消除当前公开传播的问题。

这不是“分享前选择并绑定一个特定好友”，而是“分享后只允许本次直达接收者使用”。这是微信小程序范围内最接近需求、且可落地的方案。

## 现状问题归纳

### 前端问题

1. `miniprogram/pages/list-invite/list-invite.js`
   - 当前通过普通点击事件 `onWechatInvite` 创建邀请码。
   - 创建后仅弹出 `wx.showModal` 提示用户去右上角手动转发。
   - `onShareAppMessage` 依赖实例字段 `_pendingInviteCode`，流程脆弱。

2. `miniprogram/pages/list-invite/list-invite.wxml`
   - “微信好友”入口不是 `open-type="share"` 按钮，因此无法直接触发官方分享流程。

3. `miniprogram/pages/list-invite-accept/list-invite-accept.js`
   - 当前仅依赖 `code/scene` 获取邀请。
   - 未利用私密消息校验能力，无法判断进入者是不是原始接收人。

### 后端问题

1. `cloudfunctions/listFunctions/index.js#createWechatInvite`
   - 生成的是公开模板邀请：`inviteeId: ''`。
   - 本质与公开链接一致，只是渠道名叫 `wechat`。

2. `getInviteByCode / acceptInvite / rejectInvite / applyJoinList`
   - 对 `inviteType: 'wechat'` 没有额外约束。
   - 任何拿到卡片的人，只要能打开页面，就能处理邀请。

3. `getInviteList`
   - 当前邀请管理只展示用户级记录；公开模板邀请不可见。
   - 导致“发出去但管理页不见”的体验不一致。

## 改造原则

1. 保留现有邀请数据模型主干，避免大范围重写邀请体系。
2. 将微信邀请从“公开模板邀请”调整为“私密消息模板邀请 + 接收者校验”。
3. 让分享动作由 `open-type="share"` 按钮触发，提升交互确定性。
4. 把不可实现的“好友列表选择”需求降级为“微信原生分享面板 + 私密消息验证”。

## 详细改造方案

### 一、前端分享入口改造

#### 1. 将“微信好友”入口改为真正的分享按钮

修改 `miniprogram/pages/list-invite/list-invite.wxml`：

1. 将当前普通 `view` 点击块替换为 `button open-type="share"`。
2. 保留现有样式，必要时增加一个专门样式类避免按钮默认样式破坏布局。
3. 按钮点击前仍要先让用户选角色，因此需要拆成两段式交互。

#### 2. 重构交互为“先配置，再点分享按钮”

因为 `open-type="share"` 由用户点击按钮直接触发，不能在一次点击里先弹角色框再继续分享，因此建议：

1. 用户点“微信好友”卡片后，先打开角色设置弹窗。
2. 用户确认角色后：
   - 创建一条“待分享的微信私密邀请”记录。
   - 将页面切换到“可立即分享”的状态。
   - 在弹窗或按钮区显示一个明确的“发送给微信好友”分享按钮，使用 `open-type="share"`。
3. 用户点击该按钮时，直接拉起微信原生分享面板。

说明：
- 这仍然不是“好友列表选择器”，但会比现在的“右上角提示”更接近用户预期。
- 这是平台允许的最顺畅交互。

#### 3. onShareAppMessage 绑定到已准备好的私密邀请

改造 `miniprogram/pages/list-invite/list-invite.js`：

1. 不再依赖 `_pendingInviteCode` 这种临时字段作为唯一控制。
2. 使用页面 `data` 保存当前待分享微信邀请，例如：
   - `wechatInviteCode`
   - `wechatInviteActivityId`
   - `wechatInviteRole`
   - `wechatInviteReady`
3. 在 `onShareAppMessage` 中只读取已准备好的微信邀请数据。
4. 若未准备好邀请，返回兜底提示路径或阻止错误分享。

### 二、接入微信私密消息能力

#### 1. 创建活动 ID

要使用私密消息，必须为每次业务活动创建 `activityId`。

实现建议：

1. 在云函数中新增一个创建微信私密邀请的流程，替代当前 `createWechatInvite`。
2. 该流程需要调用微信后台接口 `createActivityId` 生成 `activityId`。
3. 将 `activityId` 与邀请记录一一绑定存储。

邀请记录建议新增字段：

1. `activityId`
2. `shareMode: 'private_message'`
3. `shareStatus: 'ready' | 'shared' | 'consumed' | 'expired'` 或保持现有 `status` 体系并补充字段
4. `shareTicketRequired: true`

注意：实现时需要核查当前云开发环境是否已封装该后台接口；若无现成 SDK，需要新增服务端调用方式。该点是本方案唯一可能增加接入复杂度的部分。

#### 2. 分享前声明私密消息

在前端准备好邀请后，调用：

```js
wx.updateShareMenu({
  withShareTicket: true,
  isPrivateMessage: true,
  activityId,
  menus: ['shareAppMessage']
})
```

用途：

1. 将本次分享声明为私密消息。
2. 禁止二次转发。
3. 使后续进入小程序时具备私密消息校验条件。

说明：
- `list-invite-link` 页面仍然保持普通公开链接逻辑。
- 只有“微信好友邀请”走私密消息能力。

### 三、后端邀请模型改造

#### 1. 停止把微信邀请当成公开模板邀请

替换 `createWechatInvite` 逻辑：

当前问题：
- `inviteeId: ''`
- 任意人可使用

改造后建议：

1. 仍可在分享前创建模板记录，但必须带上 `activityId`。
2. 模板记录仅作为“本次私密消息分享凭证”，不再视为公开可消费模板。
3. 接收者首次合法打开时，再固化为该接收者的用户级邀请记录。

#### 2. 新增私密消息校验分支

在 `getInviteInfo` / `acceptInvite` / `applyJoinList` / `rejectInvite` 中增加 `wechat` 私密消息校验。

前端需要在进入邀请接受页时传递或读取：

1. `shareTicket`
2. `activityId` 校验结果（前端校验后传后台，或后台再次校验）

建议采用更稳妥方案：

1. 前端在 `App.onLaunch/onShow` 或 `list-invite-accept` 加载阶段读取 `shareTicket`。
2. 前端调用 `wx.authPrivateMessage({ shareTicket })`。
3. 将返回的 `valid/encryptedData/iv` 连同 `inviteCode` 一起传给云函数。
4. 云函数解密校验 `activityId` 是否与邀请记录上的 `activityId` 匹配。
5. 匹配成功才允许查看和处理邀请。

这样可以避免仅依赖前端 `valid` 字段。

#### 3. 邀请记录状态流转调整

建议的微信私密邀请流转：

1. 创建者生成私密邀请模板
   - `inviteType: 'wechat'`
   - `inviteeId: ''`
   - `activityId: xxx`
   - `status: 0`

2. 接收者通过私密消息合法进入后
   - 若当前用户尚无用户级记录，则复制生成用户级邀请记录
   - `inviteeId = currentUserId`
   - `inviteCode` 可沿用原模板码

3. 用户接受/申请/拒绝时
   - 基于用户级记录流转
   - 模板记录保持“仅凭 activityId 可验证”的作用，或标记已消耗

#### 4. 限制同一私密邀请的消费范围

需要增加两层限制：

1. 若校验不是私密消息合法接收者，则直接拒绝。
2. 若已经有某用户基于该 `activityId` 生成了用户级记录，则后续其他非合法用户不得复用。

是否允许“同一个私密消息发到群里，群里所有成员都可加入”，取决于产品定义：

1. 若只允许单个好友：则要求分享仅允许单聊，并在校验后额外限制首次消费用户唯一。
2. 若允许分享给群：则通过 `authPrivateMessage` 验证“该用户属于该群”，群内成员可使用。

由于用户需求写的是“特定微信好友”，建议按单个好友场景设计，不以群聊为主流程。

### 四、接收页改造

修改 `miniprogram/pages/list-invite-accept/list-invite-accept.js`：

1. 页面加载时优先检查是否来自分享卡片。
2. 若是 `wechat` 邀请：
   - 尝试获取 `shareTicket`。
   - 调用 `wx.authPrivateMessage`。
   - 将校验数据传给后端换取邀请信息。
3. 若校验失败：
   - 展示“该邀请仅对指定微信好友有效”错误页。
   - 不显示接受/申请按钮。
4. 若校验成功：
   - 正常展示邀请信息与后续处理操作。

兼容性处理：

1. 对 `link` 邀请维持现状。
2. 对 `search/recent` 邀请维持现状。
3. 只有 `wechat` 邀请增加私密消息校验强约束。

### 五、邀请管理页改造

修改 `getInviteList` 与 `list-invite-manage`：

1. 对“微信好友邀请”展示模板记录的摘要状态，避免发送后列表为空。
2. 显示更准确的状态文案，例如：
   - 待发送
   - 已分享，待对方打开
   - 已打开，待处理
   - 已接受
   - 已拒绝
   - 已过期
3. 若继续沿用当前“只展示用户级记录”的思路，则至少要在邀请页局部展示“最近一条微信好友邀请已创建，等待分享/已分享”。

建议最小改法：

1. 不大改管理页筛选模型。
2. 在 `list-invite` 页面单独显示当前待分享/最近分享的微信私密邀请状态。
3. 管理页仍以用户级记录为主。

### 六、文案与产品语义修正

所有文案应从“公开传播式邀请”调整为“私密好友邀请”：

1. 将“发送给微信好友”保留。
2. 增加说明文案：
   - “仅本次接收该消息的微信好友可加入”
   - “不可二次转发”
3. 错误提示统一：
   - “该邀请仅对指定微信好友有效”
   - “该私密邀请已失效或不属于你”

## 推荐的最小可实施改动集

为了控制改动量，优先分两阶段实施。

### 阶段 1：修正交互与数据模型基础

1. 把分享入口改成 `open-type="share"` 的真实分享按钮。
2. 重构“先配置，再分享”的前端流程。
3. 后端微信邀请记录新增 `activityId` 等字段。
4. 接入 `wx.updateShareMenu({ isPrivateMessage: true })`。

阶段 1 完成后，用户体验会明显优于当前版本，但还依赖阶段 2 才能形成真正的“仅合法接收者可处理”。

### 阶段 2：完成私密消息校验闭环

1. 接入 `wx.authPrivateMessage`。
2. 后端增加 `activityId` 解密校验。
3. `getInviteInfo / acceptInvite / applyJoinList / rejectInvite` 增加微信私密邀请鉴权。
4. 增加错误页与状态展示。

## 风险点

1. `createActivityId` 的服务端接入方式需要确认当前云开发环境是否已有现成可用接口。
2. `shareTicket` 与私密消息验证依赖真机微信环境，开发工具中验证能力有限。
3. 若产品坚持“分享前必须选中某个具体好友”，该需求无法在小程序内实现，只能调整产品定义。
4. 若要严格禁止分享到群，需要在交互和校验上进一步限制，但微信分享面板本身不提供“只允许发单聊”的简单前端开关，需要通过验证失败兜底。

## 验证方案

### 前端验证

1. 点击“微信好友邀请”后，不再弹“请点右上角”提示。
2. 用户确认角色后，页面出现可直接点击的分享按钮。
3. 点击分享按钮后，直接拉起微信原生分享面板。

### 业务验证

1. A 分享给 B，B 打开后可看到邀请并可接受。
2. B 再尝试转发，不应形成二次可用邀请。
3. 非接收者 C 拿到路径后打开，应被拒绝并看到“仅对指定微信好友有效”。
4. 链接邀请功能不受影响。
5. 搜索邀请、最近协作邀请、通知跳转不受影响。

### 真机验证重点

1. 单聊分享场景。
2. 私密消息 `shareTicket` 获取与校验。
3. 不同账号交叉验证。
4. 过期后拒绝处理。

## 对用户诉求的最终对齐

### 可满足的部分

1. “只针对特定微信好友”可以在平台允许范围内，改造成“只对本次私密消息合法接收者有效”。
2. “点击后直接拉起发送流程”可以通过 `open-type="share"` 实现为直接拉起微信分享面板。

### 无法原样满足的部分

1. 无法直接拉起微信好友列表并选中具体好友。
2. 无法在分享前拿到好友身份并写成 `inviteeId=某微信好友`。

## 实施建议

按以下顺序落地：

1. 先改前端交互为真实分享按钮。
2. 再改后端微信邀请记录结构，接入 `activityId`。
3. 最后补齐私密消息校验闭环与错误页。

这样可以先消除当前最明显的错误实现，再逐步完成“定向有效”的能力闭环。
