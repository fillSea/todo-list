# 清单与邀请功能审查及修订计划

## 目标

基于 `doc/list.md` 与 `doc/list_invite_member.md`，审查当前清单界面与共享清单邀请实现，输出：

1. 现状与设计文档不一致之处
2. 已存在的逻辑错误与权限漏洞
3. 可以落地的优化方案
4. 后续实施顺序

## 代码范围

- 云函数：`cloudfunctions/listFunctions/index.js`
- 清单列表页：`miniprogram/pages/checklist/checklist.js`
- 清单详情页：`miniprogram/pages/list-detail/list-detail.js`
- 成员页：`miniprogram/pages/list-members/list-members.js`
- 邀请页：`miniprogram/pages/list-invite/list-invite.js`
- 邀请链接页：`miniprogram/pages/list-invite-link/list-invite-link.js`
- 接受邀请页：`miniprogram/pages/list-invite-accept/list-invite-accept.js`
- 邀请管理页：`miniprogram/pages/list-invite-manage/list-invite-manage.js`
- 我的清单页：`miniprogram/pages/profile/my-lists/my-lists.js`
- 编辑页：`miniprogram/pages/list-edit/list-edit.js`

## 主要发现

### P0 逻辑错误

1. `inviteMember` 语义错误，当前会直接把用户插入 `list_members`，并没有创建“待接受邀请”。
- 位置：`cloudfunctions/listFunctions/index.js:766-826`
- 与设计文档冲突：`doc/list_invite_member.md` 要求邀请流程为“发送邀请 -> 等待接受 -> 接受后加入成员”。
- 当前结果：
  - 搜索邀请、最近协作邀请实际上变成“直接加成员”
  - `list-invite` 页面刷新待处理邀请时，可能根本看不到对应记录
  - 成员数会被提前占用，和待处理邀请概念混淆

2. `getListDetail` 未做访问权限校验，知道 `listId` 的任意登录用户都可读取清单详情、任务、成员信息。
- 位置：`cloudfunctions/listFunctions/index.js:355-440`
- 风险：越权查看共享清单、私人清单详情。

3. `getListMembers` 未做成员身份校验，非成员也可读取成员列表。
- 位置：`cloudfunctions/listFunctions/index.js:707-758`
- 风险：成员信息泄露。

4. `updateMemberRole` 只校验 `listId` 与创建者关系，但不校验 `memberId` 是否属于该 `listId`。
- 位置：`cloudfunctions/listFunctions/index.js:1080-1115`
- 风险：如果拿到别的成员记录 `_id`，理论上可跨清单改角色。

5. `acceptInvite`、`rejectInvite`、`applyJoinList` 按 `inviteCode + status:0` 查第一条记录，没有强校验邀请对象、邀请类型、是否允许当前用户处理该邀请。
- 位置：`cloudfunctions/listFunctions/index.js:1566-1770`
- 风险：
  - 定向邀请与链接邀请边界模糊
  - 微信邀请未绑定具体被邀请人时，任何拿到 code 的人都能处理
  - 记录语义混乱，后续提醒/统计不可信

6. `joinList` 与 `acceptInvite` 存在重复职责且语义不一致。
- 位置：`cloudfunctions/listFunctions/index.js:877-946`, `1566-1674`
- 现状：一个按 `listId` 入群，一个按 `inviteCode` 接受邀请；加入方式、状态更新、权限校验不统一。

7. 邀请链接分享路径错误。
- 位置：`miniprogram/pages/list-invite-link/list-invite-link.js:214-219`
- 当前路径：`/pages/list-invite-accept/accept?code=...`
- 实际页面文件是 `pages/list-invite-accept/list-invite-accept`
- 结果：分享后的跳转很可能失效。

8. 小程序码 `scene` 参数解析错误。
- 位置：
  - 生成：`miniprogram/pages/list-invite-link/list-invite-link.js:147-149`
  - 接收：`miniprogram/pages/list-invite-accept/list-invite-accept.js:36-40`
- 当前生成 `scene: invite=${inviteCode}`，接收端却直接 `code || scene`，没有解析 `invite=` 前缀。
- 结果：扫码进入后云函数会用 `invite=xxx` 当邀请码查库，无法命中。

9. `getInviteList` 权限过宽，任意成员都能查看邀请列表。
- 位置：`cloudfunctions/listFunctions/index.js:1119-1177`
- 设计文档中“管理邀请”仅创建者可操作。
- 当前实现允许普通成员查看邀请记录，和权限矩阵不一致。

10. `list-members` 页面的待处理邀请数量在生产模式下从未加载。
- 位置：`miniprogram/pages/list-members/list-members.js:160-170`
- UI 中显示 `pendingInviteCount`，但真实环境没有查询更新。

11. `list-edit` 保存成功后尝试调用上一页 `loadData()`，但相关页面大多实现的是 `loadLists()` / `loadListDetail()`。
- 位置：`miniprogram/pages/list-edit/list-edit.js:224-231`, `261-267`
- 结果：返回后常常不会主动刷新。

12. `checklist.wxml` 绑定了 `onEditList` / `onDeleteList`，而 `checklist.js` 中没有对应方法，只有 `openEditDialog` / `onDeleteConfirm` 流程。
- 位置：`miniprogram/pages/checklist/checklist.wxml:102-103`
- 风险：直接点击卡片滑出操作时会报事件未定义或无响应。

### P1 设计不一致

1. 设计文档要求四种邀请方式：微信好友、链接、搜索用户、最近协作成员。
- 当前“搜索用户/最近协作成员”没有走邀请记录，只是直接加成员。
- 微信邀请和链接邀请才会创建 `list_invites`，系统行为不统一。

2. 设计文档中邀请记录应区分：`wechat/link/search`。
- 当前实现只有 `wechat` 和 `link` 明确落库，`search/recent` 不落邀请表。

3. 设计文档要求 `list_members` 新增 `inviteId` 关联邀请记录。
- 当前多数加入路径没有写 `inviteId`，`joinList` / `acceptInvite` / `approveApplication` 也不统一。

4. 设计文档要求“同一用户同一清单最多 3 次待处理邀请”。
- 当前没有任何重复邀请/频控机制。

5. 设计文档要求“已是成员提示、已有待处理邀请提示、拒绝后可重新邀请”。
- 当前只校验是否已是成员，没有系统性校验“待处理邀请已存在”。

6. 设计文档要求支持到期管理。
- 当前仅在读取/接受时临时判断 `expireAt`，没有自动过期清理或统一过期状态迁移逻辑。

7. 设计文档中链接邀请支持“需审核/直接加入”的差异。
- 当前前端接受页对 `needApproval` 做了分支，但后端邀请模型仍然混杂“定向邀请”和“公开申请入口”，抽象不清晰。

### P2 可优化点

1. `getListMembers` 和 `getInviteList` 存在 N+1 查询问题。
- 典型位置：按成员/邀请逐个查用户资料。
- 可改为一次性收集用户 ID 后批量查询并映射。

2. `getMyLists` 对每个清单单独统计任务与成员摘要，列表增大后性能会明显下降。
- 位置：`cloudfunctions/listFunctions/index.js:322-340`
- 可优化为：
  - 首屏保留轻量统计
  - 或使用聚合/批量查询减少循环内 I/O

3. 时间格式化逻辑重复散落在多个页面。
- 页面：`checklist`, `list-detail`, `list-invite`, `list-invite-manage`
- 可抽到公用 util，避免文案和阈值不一致。

4. 清单编辑入口重复实现。
- `checklist` 内嵌弹窗编辑
- `list-edit` 独立页面编辑
- 建议统一为一个入口，避免字段/校验/刷新行为分叉。

5. 页面加载职责过重。
- `getListDetail` 一次返回清单 + 全量任务 + 全量成员，详情页刷新成本高。
- 后续可考虑拆分：基础信息、成员、任务分页。

6. 部分云函数返回结构不统一。
- 有的返回 `{ code: 0 }`
- 有的返回 `{ success: true }`
- 前端因此混用两套判断逻辑，易出错。

## 建议的修订设计

### 一、统一邀请模型

把“邀请成员”严格定义为创建 `list_invites` 记录，不直接写 `list_members`。

建议统一字段语义：
- `inviteType`: `wechat | link | search | recent`
- `inviteeId`: 定向邀请时必填；公开链接时为空
- `inviteeInfo`: 用于冗余展示，不作为权限依据
- `role`: 目标角色
- `status`: `0待处理 1已接受 2已拒绝 3已过期 4待审批 5已取消` 或者继续沿用删除取消，但推荐显式状态
- `needApproval`: 仅链接邀请/申请流使用
- `inviteCode`: 微信卡片/链接场景使用

成员真正加入时统一由以下路径写入 `list_members`：
- 定向邀请接受
- 链接邀请直接接受
- 链接申请审核通过

并且必须补全：
- `joinType`
- `inviteId`

### 二、按邀请类型拆清业务流

1. 微信邀请
- 创建 `list_invites` 记录
- 生成 `inviteCode`
- 分享进入接受页
- 接受后再加入成员

2. 搜索用户邀请 / 最近协作成员邀请
- 创建定向邀请记录，`inviteeId` 为目标用户
- 发送站内通知
- 不直接加成员
- 对方在消息或邀请页接受后才加入

3. 链接邀请
- 创建链接邀请记录
- 若 `needApproval=false`：持码用户接受后直接加入
- 若 `needApproval=true`：持码用户申请后转为 `status=4`，由创建者审批

### 三、加强权限边界

所有读取和写入都应通过统一权限函数，例如：
- `verifyListAccess(userId, listId)`：能否查看
- `verifyListPermission(userId, listId, requiredRole)`：能否编辑/管理

至少覆盖：
- `getListDetail`
- `getListMembers`
- `getInviteList`
- `updateMemberRole`
- `removeMember`
- `generateInviteLink`
- `createWechatInvite`
- `approveApplication`
- `clearInvites`

邀请处理还应校验：
- 定向邀请只能由 `inviteeId` 处理
- 链接邀请才允许匿名持码接受/申请
- 已过期/已取消/已处理记录不能重复操作

### 四、前端页面改造建议

1. `list-invite`
- 搜索邀请/最近成员邀请改为“发送邀请成功”，不是“已添加成员”
- 页面顶部待处理邀请应真实反映 `status=0`
- 进入页面时应校验当前用户是否有管理成员权限，避免仅靠后端报错

2. `list-invite-link`
- 修正分享路径
- 修正 `scene` 解析协议
- 生成新链接前，明确“是否复用旧链接/是否废弃旧链接”的产品规则

3. `list-invite-accept`
- 正确解析 `scene`
- 页面根据邀请类型显示不同文案
- 对“你已是成员 / 邀请已处理 / 邀请已过期”做更细化状态展示

4. `list-members`
- 真实加载待处理邀请数量
- 旧版“手输 userId 直接添加成员”入口建议下线或改为“发送定向邀请”

5. `checklist`
- 修复 WXML 事件绑定与 JS 方法不匹配问题
- 如果保留内嵌编辑弹窗，需要与 `list-edit` 做统一；否则建议全部跳独立编辑页

6. `list-edit`
- 返回刷新改为基于页面栈调用存在的方法，或依赖 `onShow` 自动刷新
- 避免依赖不存在的 `loadData`

## 推荐实施顺序

### 阶段 1：先修安全与主逻辑

1. 给 `getListDetail` / `getListMembers` / `getInviteList` 增加严格访问校验
2. 重构 `inviteMember`，改为写 `list_invites`，不再直接加成员
3. 统一 `acceptInvite` / `applyJoinList` / `rejectInvite` 的校验与状态迁移
4. 修复 `updateMemberRole` 对 `memberId` 的归属校验
5. 修复链接分享路径与 `scene` 解析

### 阶段 2：修前端行为与文案

1. 修复邀请页“已添加成员”误导文案
2. 成员页接入真实待处理邀请数
3. 接受邀请页按 `needApproval`、`inviteType`、处理状态展示正确按钮
4. 修复 `checklist` 缺失事件处理
5. 修复 `list-edit` 返回刷新逻辑

### 阶段 3：性能与结构优化

1. 批量化查询成员与邀请用户资料
2. 优化 `getMyLists` 的统计查询
3. 提取公共时间格式化工具
4. 统一云函数返回格式

## 验证要点

1. 创建者发送搜索邀请后，被邀请人未接受前，不应出现在 `list_members`
2. 邀请管理页应能看到该条待处理邀请
3. 被邀请人接受后才加入成员，且 `joinType/inviteId` 正确
4. 链接邀请扫码与分享路径都能进入接受页
5. `needApproval=true` 时，申请后不直接入群，创建者可在待审批中通过/拒绝
6. 非成员不能获取清单详情、成员列表、邀请列表
7. 查看者/编辑者无法管理邀请
8. 清单列表、详情页、成员页在保存/删除后可正确刷新

## 预期交付

实施完成后，清单与邀请功能将与设计文档的核心流程对齐：
- 邀请和成员加入解耦
- 权限边界闭合
- 链接/微信/搜索/最近协作四类邀请行为一致
- 页面状态与后台数据一致
- 后续再扩展提醒、统计、频控会更稳定
