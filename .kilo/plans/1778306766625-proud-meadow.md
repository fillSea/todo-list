# 关于我们页面生产化修复方案

## 目标

修复 `miniprogram/pages/profile/about/about` 页面在生产环境中的展示和交互问题，使其达到可发布标准。范围包括图标渲染、图片资源缺失、无效跳转、占位信息、分享配置、检查更新稳定性，以及与帮助中心共用入口的一致性。

## 当前问题

1. `about.json` 未注册 `van-icon`，导致 `about.wxml` 中功能特点和底部按钮图标无法可靠显示。
2. 顶部 Logo 引用 `/images/logo.png`，但 `miniprogram/images/logo.png` 不存在。
3. 分享配置引用 `/images/share-cover.png`，但资源不存在。
4. `onViewAgreement` 和 `onViewPrivacy` 跳转到不存在的页面：
   - `/pages/profile/help/agreement/agreement`
   - `/pages/profile/help/privacy/privacy`
5. `pages/profile/help/help.js` 中也存在相同的无效协议/隐私跳转。
6. 客服邮箱 `support@example.com` 是模板占位，不符合生产环境。
7. “开发团队”“更新日志”“版权年份”等内容偏模板化，发布前应改成真实且可维护的信息。
8. `onCheckUpdate` 每次点击都会注册 updateManager 回调，多次点击可能造成重复回调或重复弹窗。

## 实施方案

### 1. 修复关于我们页面 Vant 图标配置

修改 `miniprogram/pages/profile/about/about.json`：

- 保留现有标题、背景色、下拉刷新配置。
- 增加局部组件注册：

```json
"usingComponents": {
  "van-icon": "@vant/weapp/icon/index"
}
```

验收标准：

- 功能特点区域 6 个图标正常显示。
- 底部“检查更新”“联系邮箱”“分享给朋友”按钮图标正常显示。
- 控制台不再出现 `van-icon` 未注册相关警告。

### 2. 处理 Logo 和分享封面资源

优先方案：补充真实设计资源。

新增资源：

- `miniprogram/images/logo.png`
- `miniprogram/images/share-cover.png`

建议规格：

- `logo.png`：方形透明底或纯色底，建议 512x512 或 256x256，压缩后小于小程序包体限制。
- `share-cover.png`：微信分享卡片常用比例建议 5:4，例如 500x400，主体文字和 Logo 居中。

降级方案：如果暂时没有正式分享封面：

- `onShareAppMessage` 中删除 `imageUrl` 字段，让微信使用默认截图/默认卡片。
- Logo 不建议继续缺失；如没有设计图，可临时使用现有 tabbar 的 `profile.png` 或用 WXML/CSS 绘制文字 Logo，但上线前仍应替换为正式品牌资源。

验收标准：

- 顶部白色 Logo 容器内显示正式应用图标。
- 分享给好友时卡片封面不为空、不报资源不存在。
- 开发者工具 Network/Console 无图片 404 或找不到文件错误。

### 3. 新增用户协议和隐私政策页面

新增页面目录：

- `miniprogram/pages/profile/help/agreement/agreement.js`
- `miniprogram/pages/profile/help/agreement/agreement.json`
- `miniprogram/pages/profile/help/agreement/agreement.wxml`
- `miniprogram/pages/profile/help/agreement/agreement.wxss`
- `miniprogram/pages/profile/help/privacy/privacy.js`
- `miniprogram/pages/profile/help/privacy/privacy.json`
- `miniprogram/pages/profile/help/privacy/privacy.wxml`
- `miniprogram/pages/profile/help/privacy/privacy.wxss`

更新 `miniprogram/app.json` 的 `pages`：

- 增加 `pages/profile/help/agreement/agreement`
- 增加 `pages/profile/help/privacy/privacy`

页面内容建议：

- 使用静态 WXML 展示正式协议文本。
- 包含生效日期、应用名称、服务范围、用户行为规范、数据处理说明、联系方式。
- 隐私政策需明确收集的信息、用途、云开发/微信能力使用、存储期限、用户权利、联系方式。

注意：协议和隐私政策文本涉及合规，最终内容应由业务/法务确认。技术实现可先放入结构完整的正式模板，但不应使用空白或“开发中”。

验收标准：

- 从“关于我们”点击“用户协议”可正常进入协议页。
- 从“关于我们”点击“隐私政策”可正常进入隐私政策页。
- 从“帮助中心”底部点击“用户协议”“隐私政策”也可正常进入。
- 所有新增页面已在 `app.json` 注册。

### 4. 统一生产联系方式

建议在 `about.js` 和 `help.js` 中统一使用同一个客服邮箱常量，避免两个页面不一致。

可选实现：

- 最小改动：分别替换 `support@example.com` 为真实邮箱。
- 更可维护：新增或复用小程序配置模块，例如 `miniprogram/utils/config.js`，导出 `supportEmail`，两个页面共用。

考虑到当前项目没有明显统一前端配置模块，优先采用最小改动；如果后续还有更多页面使用联系方式，再抽配置文件。

需用户提供：

- 正式客服邮箱。
- 可选：客服微信、公众号、工作时间、公司/团队名称。

验收标准：

- “关于我们”复制邮箱得到真实客服邮箱。
- “帮助中心”联系客服弹窗展示真实客服邮箱。
- 页面中不再出现 `support@example.com`。

### 5. 优化检查更新逻辑

修改 `about.js` 的更新检查逻辑，避免重复注册监听。

建议实现：

- 在 `onLoad` 中初始化 `this.updateManager = wx.getUpdateManager()`。
- 增加状态字段，例如 `isCheckingUpdate`，防止短时间重复点击。
- 将 `onUpdateReady` 和 `onUpdateFailed` 监听只注册一次。
- `onCheckUpdate` 只负责触发用户反馈；由于 `wx.getUpdateManager().onCheckForUpdate` 是监听式 API，需避免每次点击都叠加监听。
- 对不支持更新管理器的环境增加兜底提示。

验收标准：

- 连续多次点击“检查更新”不会重复弹窗。
- 无更新时提示“已是最新版本”。
- 有更新时仍能提示重启应用。
- 更新失败时有明确提示。

### 6. 调整关于我们页面内容为生产文案

建议修改 `about.js` 中以下内容：

- `appInfo.name`：保持正式小程序名，例如“协同待办清单”。
- `appInfo.slogan`：保持简洁、真实，不夸大功能。
- `appInfo.description`：与当前功能一致，避免宣传未实现能力。
- `team`：从“产品团队/前端团队/后端团队/UI设计”改为真实主体，例如“开发与维护：xxx团队”。如果不希望公开团队分工，可改为“协同待办清单团队”。
- `changelog`：更新到当前实际版本和日期。
- 版权年份：建议使用当前年份或动态年份。若不想新增逻辑，至少将 `© 2024` 更新为当前发布年份。

验收标准：

- 页面无明显模板占位内容。
- 展示内容与真实发布信息一致。
- 功能说明不包含未上线或不可用功能。

### 7. 修复分享行为

修改 `onShareAppMessage`：

- 保留 `title` 和 `path`。
- 如果已有正式 `share-cover.png`，保留 `imageUrl: '/images/share-cover.png'`。
- 如果没有正式图片，删除 `imageUrl`，避免引用不存在资源。

验收标准：

- 点击分享按钮可正常拉起分享。
- 分享标题、路径正确。
- 分享封面存在或安全回退，不产生资源错误。

### 8. 验证计划

由于本项目主要通过微信开发者工具运行，验证以 DevTools 为主。

手动验证：

1. 启动微信开发者工具，编译小程序。
2. 进入“我的”页面，点击“关于我们”。
3. 检查顶部 Logo、功能特点图标、底部按钮图标是否显示。
4. 点击“检查更新”，确认不会多次点击造成重复弹窗。
5. 点击“联系邮箱”，确认复制真实邮箱成功。
6. 点击“分享给朋友”，确认分享卡片正常。
7. 点击“用户协议”，确认页面可进入且内容完整。
8. 点击“隐私政策”，确认页面可进入且内容完整。
9. 进入“帮助中心”，重复验证底部“用户协议”“隐私政策”入口。
10. 查看开发者工具 Console，确认无组件未注册、页面不存在、资源不存在等错误。

静态检查：

- 搜索 `support@example.com`，应无残留。
- 搜索 `/images/logo.png` 和 `/images/share-cover.png`，确认对应资源存在或引用已移除。
- 搜索 `/pages/profile/help/agreement/agreement` 和 `/pages/profile/help/privacy/privacy`，确认路径已在 `app.json` 注册。
- 检查 `about.json` 是否注册 `van-icon`。

## 需要用户确认的信息

实施前建议确认以下生产信息：

1. 正式客服邮箱是什么。
2. 是否已有正式 Logo 和分享封面图片；如果已有，需要放入 `miniprogram/images/`。
3. 协议和隐私政策是否已有正式文本；如果没有，可先使用合规结构模板，但上线前应业务/法务确认。
4. “开发团队”和版权主体展示为哪个名称。

## 推荐执行顺序

1. 先修 `about.json` 的 `van-icon` 注册，立即解决图标不显示。
2. 新增协议/隐私政策页面并注册到 `app.json`，解决无效跳转。
3. 替换或移除缺失图片引用，补齐 Logo/分享封面。
4. 替换客服邮箱、团队信息、版权和更新日志。
5. 优化检查更新逻辑，避免重复监听。
6. 在微信开发者工具中完整回归“关于我们”和“帮助中心”相关入口。

## 风险与边界

- 图片资源和协议文本属于产品/合规素材，技术上可以接入，但最终内容需业务确认。
- `wx.getUpdateManager` 在开发者工具、体验版、正式版表现可能不同，需以真机和体验版验证为准。
- 如果新增图片导致包体增加，需要关注小程序主包大小。
- 如果后续要复用客服信息，建议再抽公共配置；本次为降低改动风险，优先最小化修复。
