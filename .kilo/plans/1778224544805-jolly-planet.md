# 共享清单搜索用户机制设计方案

## 目标

将“共享清单 -> 搜索用户”机制调整为以下规则：

1. 只根据用户昵称搜索，不支持用户ID搜索。
2. 当前清单内已存在的成员，不应出现在搜索结果中。
3. 搜索接口应明确属于“当前清单的邀请搜索”能力，而不是全局泛用搜索。
4. 保持现有邀请发送链路不变，继续由 `inviteMember` 做最终权限与重复邀请校验。

## 现状问题

### 1. 前后端定义不一致

- 前端 `miniprogram/pages/list-invite/list-invite.wxml` 当前提示文案为“输入用户ID或昵称”。
- 后端 `cloudfunctions/listFunctions/index.js` 的 `searchUser` 实际只按 `nickname` 正则搜索。
- 这会导致用户预期与真实能力不一致。

### 2. 搜索结果未排除当前清单成员

- 当前 `searchUser` 只过滤了当前登录用户自身。
- 已经属于该清单的成员仍会被搜出来，直到点击邀请时才在 `inviteMember` 中被拦截。
- 这会造成无效候选人进入列表，交互不正确。

### 3. 搜索接口缺少清单上下文

- 当前 `searchUser` 只接收 `keyword`。
- 它不知道当前在哪个清单中搜索，因此无法在搜索阶段过滤成员。
- 这使接口更像“全局搜索平台用户”，而不是“共享清单邀请搜索”。

## 设计原则

1. 最小改动优先，不重构邀请主链路。
2. 搜索阶段只解决“候选人可见性”问题。
3. 邀请阶段仍由 `inviteMember` 兜底校验，避免仅依赖前端或搜索结果。
4. 保留当前“昵称模糊搜索”的交互方式，不扩展新的搜索维度。

## 方案设计

### 一、后端：收口 `searchUser` 为带清单上下文的昵称搜索

修改位置：`cloudfunctions/listFunctions/index.js`

#### 1. 调整入参

将 `searchUser` 的入参从：

```js
{ keyword }
```

调整为：

```js
{ listId, keyword }
```

#### 2. 参数校验

在 `searchUser` 中新增：

- `listId` 非空校验
- `keyword` 非空校验
- `keyword.trim()` 后再参与搜索

错误提示建议：

- `清单ID不能为空`
- `搜索关键词不能为空`

#### 3. 权限校验

在搜索前调用现有能力：

```js
verifyListPermission(userId, listId, [1])
```

原因：

- 搜索用户是邀请流程的一部分。
- 当前 `inviteMember` 已要求只有创建者可邀请。
- 搜索能力应与邀请权限保持一致，避免无邀请权限的人仍能获取候选用户列表。

如果权限不足，直接返回现有错误语义，例如：

- `无权限执行该操作`
- 或沿用 `permission.message`

#### 4. 仅按昵称模糊搜索

保留当前按 `nickname` 搜索的主逻辑，不新增 `userId` / `_id` 搜索。

即：

- 不改成多字段 OR 搜索
- 不支持通过用户ID精确查找

这样与产品要求一致，也让接口语义保持单一。

#### 5. 过滤当前清单成员

在返回搜索结果前，增加当前清单成员过滤：

- 查询 `list_members` 中该 `listId` 下的所有 `userId`
- 构建成员ID集合
- 从 `users` 搜索结果中过滤：
  - 当前登录用户自己
  - 已在成员集合中的用户

结果上要保证：

- 清单创建者搜不到自己
- 清单其他已加入成员也搜不到
- 非成员候选用户才可显示

#### 6. 输出结构保持兼容

继续返回：

```js
{ success: true, users: [...] }
```

每个用户项仍保留：

```js
{
  userId,
  nickname,
  avatarUrl
}
```

这样前端结果渲染与邀请按钮逻辑无需大改。

### 二、前端：将搜索请求改为带 `listId` 的昵称搜索

修改位置：`miniprogram/pages/list-invite/list-invite.js`

#### 1. 调整云函数调用参数

当前请求：

```js
action: 'searchUser',
data: { keyword: searchKeyword.trim() }
```

调整为：

```js
action: 'searchUser',
data: {
  listId: this.data.listId,
  keyword: searchKeyword.trim()
}
```

这样后端才有足够上下文去过滤当前清单成员。

#### 2. DEBUG 模式同步语义

当前 DEBUG 模式会直接伪造一个结果。

建议在模拟数据层面也体现“只按昵称搜索”的语义：

- mock 结果字段仍保留 `userId`、`nickname`、`avatarUrl`
- 但不再暗示“输入用户ID也能搜到”
- 如需更严谨，可在注释上标注“模拟昵称搜索结果”

不需要在 DEBUG 模式实现完整成员过滤，只要避免与正式逻辑文案冲突即可。

### 三、前端：修正文案，避免错误预期

修改位置：`miniprogram/pages/list-invite/list-invite.wxml`

将输入框占位文案从：

```xml
输入用户ID或昵称
```

调整为：

```xml
输入用户昵称
```

必要时可同步检查页面中是否还有类似“搜索用户ID”的提示文案，统一改为昵称搜索口径。

## 是否需要修改 `inviteMember`

结论：本轮不需要调整主逻辑。

原因：

1. `inviteMember` 已具备关键兜底能力：
   - 权限校验
   - 已是成员校验
   - 重复待处理邀请校验
   - 目标用户存在性校验
2. 即使搜索阶段过滤有遗漏，邀请阶段仍能阻断错误写入。
3. 本次需求核心是“搜索机制正确性”，不需要扩大到邀请链路重构。

可保留的后续兜底价值：

- 若前端缓存滞后，某用户在搜索后刚好已加入清单，`inviteMember` 仍能正确报错。

## 影响范围

### 直接影响文件

1. `cloudfunctions/listFunctions/index.js`
   - 修改 `searchUser` 入参、权限校验、成员过滤逻辑

2. `miniprogram/pages/list-invite/list-invite.js`
   - 修改 `searchUser` 云函数调用参数

3. `miniprogram/pages/list-invite/list-invite.wxml`
   - 修改搜索输入框文案

### 间接影响

- 不影响 `inviteMember` 主逻辑
- 不影响 `list-invite-accept` 邀请接收逻辑
- 不影响最近协作成员、邀请链接、微信邀请等其他分享方式

## 实施步骤

1. 在 `listFunctions.searchUser` 中新增 `listId` 参数读取与校验。
2. 在 `searchUser` 内先获取当前用户ID，再做 `verifyListPermission(userId, listId, [1])`。
3. 查询当前清单全部成员 `list_members`，得到成员 `userId` 集合。
4. 按昵称模糊搜索 `users` 集合，限制结果数量保持现状（20条）。
5. 在结果中过滤当前用户和成员集合内用户。
6. 保持原返回结构不变。
7. 前端 `list-invite.js` 调用 `searchUser` 时补充 `listId`。
8. 前端 `list-invite.wxml` 将占位文案改为“输入用户昵称”。
9. 如有必要，微调 DEBUG 注释或 mock 文案，确保语义一致。

## 验证方案

### 功能验证

1. 创建者在共享清单邀请页输入某个非成员用户昵称。
   - 预期：能搜到匹配用户。

2. 创建者输入当前清单内某位成员昵称。
   - 预期：搜索结果中不出现该成员。

3. 创建者输入自己的昵称。
   - 预期：搜索结果中不出现自己。

4. 输入不存在的昵称。
   - 预期：显示“未找到相关用户”。

5. 输入与多名非成员昵称模糊匹配的关键词。
   - 预期：返回最多20条匹配结果，且都不是当前成员。

### 权限验证

1. 非创建者触发 `searchUser`。
   - 预期：返回无权限错误，与 `inviteMember` 权限口径一致。

2. 创建者在有权限的清单内搜索。
   - 预期：正常返回结果。

### 回归验证

1. 搜到某个非成员后点击邀请。
   - 预期：仍能正常走 `inviteMember` 并发送邀请。

2. 某用户在搜索后、点击邀请前被别人加入清单。
   - 预期：`inviteMember` 仍返回“该用户已是清单成员”。

3. 最近协作成员邀请、微信邀请、链接邀请。
   - 预期：不受本次改动影响。

## 风险与注意事项

### 1. 正则搜索仍然沿用现有实现

本轮需求重点不是正则安全或搜索算法优化，因此：

- 可先保持当前 `db.RegExp({ regexp: keyword, options: 'i' })`
- 不把本次任务扩大成搜索引擎重构

但需注意：后续若继续收敛搜索质量，可再单独处理关键词转义问题。

### 2. 成员查询的额外一次读操作

为了过滤当前清单成员，`searchUser` 会多一次 `list_members` 查询。

这是合理成本，因为：

- 当前需求必须依赖清单上下文过滤
- 结果上限仅20条，整体负担可接受

### 3. `list-members` 页的“按用户ID添加”逻辑不在本次范围

代码中 `miniprogram/pages/list-members/list-members.js` 仍存在直接输入 `newMemberId` 后调用 `inviteMember` 的路径。

这与本次“共享清单搜索用户机制”不是同一入口。

本轮建议：

- 不主动改这个页面
- 只修正 `list-invite` 页的搜索能力

若产品要求全站统一改成“只能按昵称邀请”，再单独处理 `list-members` 页交互。

## 最终实施结论

本次应采用“最小闭环修正”方案：

1. 后端 `searchUser` 强制带 `listId`，并先做创建者权限校验。
2. 搜索维度明确限定为 `nickname`。
3. 搜索结果中排除当前用户和当前清单成员。
4. 前端同步传入 `listId`，并把文案改成“输入用户昵称”。
5. `inviteMember` 保持现状，继续作为最终一致性兜底。
