# “我的”页面明显错误修复方案

## 目标
修复 `miniprogram/pages/profile/profile.*` 及关联云函数中已经确认的明显错误，优先保证：
1. 统计数据与用户实际可见页面语义一致
2. 页面跳转行为符合文案与注释语义
3. 首次进入页面不会重复触发全量加载
4. 图表统计与渲染逻辑都走低风险、可验证的实现路径

## 修复范围
- 前端页面：`miniprogram/pages/profile/profile.js`
- 前端模板：`miniprogram/pages/profile/profile.wxml`
- 可选样式清理：`miniprogram/pages/profile/profile.wxss`
- 云函数：`cloudfunctions/profileFunctions/index.js`

## 明显错误与修改方案

### 1. 统计卡片与落地页口径不一致
**问题**
`getUserStatistics` 当前三项统计与点击后的目标页语义并不完全一致：
1. `sharedLists` 直接统计 `list_members.count()`，会被重复成员记录放大。
2. 更关键的是，`/pages/profile/my-lists/my-lists?type=shared` 的实际语义是“当前用户可访问的共享清单”，既包含自己创建的共享清单，也包含作为成员加入的共享清单；仅统计 `list_members` 会漏掉“自己创建但未写成员记录”的共享清单。
3. `myLists` 当前统计的是 `creatorId = userId && isShared = false`，而点击后进入的是 `/pages/profile/my-lists/my-lists` 默认 `type=all`，该页面展示的是“全部可访问清单”，不是“仅个人清单”。

**修改方案**
以现有落地页行为为准，统一 `getUserStatistics` 的统计语义，避免再改动页面命名和路由：
1. 复用 `getMyLists` 的核心口径思路：
   - 查询当前用户创建的清单 ID。
   - 查询当前用户在 `list_members` 中的成员清单 ID。
   - 合并并去重为 `allAccessibleIds`。
2. `myLists` 改为统计 `allAccessibleIds` 对应的全部清单数量，和 `/pages/profile/my-lists/my-lists` 保持一致。
3. `sharedLists` 改为统计 `allAccessibleIds` 中 `isShared = true` 的清单数量，和 `/pages/profile/my-lists/my-lists?type=shared` 保持一致。

**实现要点**
- 不继续使用 `list_members.count()` 直接作为展示值。
- 共享清单数量必须基于“去重后的可访问清单 ID + `isShared` 过滤”。
- 个人/全部口径在 UI 文案上仍显示“我的清单”，但实现以实际点击页面为准，这是最低风险且最不容易产生新分歧的方案。
- 保持返回字段名不变：`myLists`、`sharedLists`。

**预期结果**
统计卡片数字与用户点击后看到的清单页内容一致，不再出现“数字和列表对不上”的业务错误。

### 2. “近7天完成趋势”统计口径错误
**问题**
`getDashboardData` 的柱状图当前使用 `status: 1 + updatedAt 当天范围` 计数：
1. 会把“完成后又被编辑”的任务错误归入后续日期。
2. 名义上是“完成趋势”，实际却是“最后更新时间趋势”。
3. 与饼图采用的周期任务统计口径不一致，容易出现前后矛盾。

**已核实事实**
任务状态切换链路已经写入稳定的 `completedAt` 字段，位置在 `cloudfunctions/taskFunctions/index.js` 的任务完成更新逻辑中，因此不需要扩展数据模型，也不需要补新字段。

**修改方案**
在 `cloudfunctions/profileFunctions/index.js` 的 `getDashboardData` 中，把柱状图统计从 `updatedAt` 切换为 `completedAt`。

**低风险实现路径**
1. 普通任务：查询 `status = 1` 且 `completedAt` 落在当天范围内的数据。
2. 周期任务：优先沿用当前统计中已有的“周期任务智能过滤”辅助逻辑，补一个基于 `completedAt` 日期窗口的过滤条件，避免把同一系列的展示口径做散。
3. 若直接把日期窗口塞进现有辅助函数改动过大，则退一步：
   - 先只把柱状图从 `updatedAt` 改为 `completedAt`；
   - 同时保持非周期任务与周期任务都使用 `completedAt` 作为完成日依据。
   这样至少先修复“完成后编辑导致趋势漂移”的明显业务错误。

**实现要点**
- 不再把 `updatedAt` 当成完成时间。
- 优先修正“日期归属错误”这个确定性 bug，再处理更深的周期任务口径统一。
- 保持返回结构不变：`barChart: [{ date, count, fullDate }]`。

**预期结果**
近 7 天趋势体现“任务在哪一天完成”，而不是“哪一天最后一次被修改”。

### 3. “已完成任务”点击行为与注释不一致
**问题**
`onStatisticTap` 中 `completedTasks` 仅 `switchTab('/pages/index/index')`，没有真正筛选已完成任务，但注释写的是“跳转到任务页面并筛选已完成”。

**已核实事实**
`pages/index/index.js` 目前没有现成的 tab/query/filter preset 入口来直接切到“已完成任务”区域，因此不能只改 profile 页而假设首页会自动响应。

**修改方案**
采用一次性跨页标记方案，避免重构首页结构：
1. 在 `profile.js` 点击“已完成任务”前写入一次性标记，例如 `wx.setStorageSync('indexTaskViewPreset', 'completed')`。
2. `index.js` 在 `onShow` 中登录态确认后读取该标记。
3. 如果命中 `'completed'`：
   - 将首页切换到“已完成任务”对应的展示区域；
   - 如当前页面没有显式 tab 状态，则补一个最小展示状态字段，例如 `taskSection = 'completed'`，只服务于默认滚动/聚焦逻辑，不重做整体筛选架构。
4. 标记消费后立即清理，确保它是一次性的，不污染正常进入首页的行为。

**实现要求**
- 行为必须与文案一致。
- 不引入事件总线，不改 `switchTab` 路由结构。
- 改动控制在 `profile.js` 和 `index.js` 两处最小闭环。

**预期结果**
点击“已完成任务”后，首页实际进入已完成任务视图，而不是默认任务列表。

### 4. 首次进入页面重复加载全部数据
**问题**
`onLoad` 和 `onShow` 都调用 `checkLoginStatus()`，首次进入常会触发两次 `loadAllData()`，造成四个云函数重复请求。

**修改方案**
在 `profile.js` 中增加一次性初始化控制，保持回到页面仍可刷新。

**推荐方案**
1. 增加页面实例字段，例如 `hasEnteredOnce` 或 `skipNextOnShowRefresh`，不要放到 `data` 里，避免无意义渲染。
2. `onLoad` 负责首次 `checkLoginStatus()`。
3. 首次紧接着触发的 `onShow` 直接跳过。
4. 后续 `onShow` 继续执行登录态同步和刷新，兼容从资料编辑页、通知页返回的场景。

**实现要点**
- 保留现有游客态重置逻辑，不动 `resetGuestView()` 的 badge/loading 清理副作用。
- 不改变 `checkLoginStatus()` 对过期 toast 的处理顺序。

**预期结果**
首次进入“我的”页面时，每类数据只请求一次；后续回到页面仍能刷新。

### 5. 扇形图在模板中做复杂计算，存在渲染风险
**问题**
`profile.wxml` 直接在 `style` 中做除法、角度换算和 `|| 0` 容错，表达式过重；`conic-gradient` 一旦拼接值异常，整块图表就容易失效。

**修改方案**
在 `profile.js` 中预计算最终 `pieStyle`，模板只做绑定。

**实施方式**
1. 在 `loadDashboardData()` 成功后，对 `dashboardData.pieChart` 做一次派生计算。
2. 生成最终样式字符串，例如 `background: conic-gradient(...)`，写入 `dashboardData.pieStyle` 或平级 `pieChartStyle`。
3. 当 `total === 0` 时，输出固定兜底样式，避免模板内除零与 `NaN`。
4. `profile.wxml` 改为只绑定 `style="{{pieChartStyle}}"` 或等价字段，不再内联数学表达式。

**实现要点**
- 这是纯前端低风险改动，不改变云函数返回结构。
- 保留 `maxBarValue` 现有兜底逻辑：`Math.max(...counts, 1)`。

**预期结果**
模板更简单，环形图更稳定，后续若需要替换实现方式也更容易。

## 非阻塞但建议顺手处理

### 6. 移除未使用的登录卡片样式
`profile.wxss` 中存在 `login-card`、`login-content`、`login-avatar`、`login-text` 等当前模板未使用样式。可在本次顺手清理，降低维护噪音。

### 7. 保持登录态字段命名不做半吊子统一
`profile.js` 当前页面态使用 `isLogin`，而应用级接口与多数页面使用 `isLoggedIn`。本轮不要顺手做跨页面命名统一，只在当前页局部修复时保持现状，避免引入新的条件分支错误。

## 实施顺序
1. 修复 `profileFunctions.getUserStatistics` 的清单统计口径，使卡片与落地页一致
2. 修复 `profile.js` + `index.js` 的“已完成任务”跳转联动
3. 修复 `profile.js` 的首次进入重复加载
4. 重构 `profile.js` + `profile.wxml` 的扇形图预计算逻辑
5. 修复 `profileFunctions.getDashboardData` 的近 7 天完成趋势口径，改用 `completedAt`
6. 可选清理 `profile.wxss` 未使用样式

## 实现阶段核查点
1. `pages/index/index.js` 需要补一个最小的一次性“已完成视图”入口，而不是假设已有筛选能力
2. `taskFunctions` 已存在 `completedAt` 写入，可直接作为近 7 天趋势统计依据
3. `getMyLists(type=shared)` 的定义是“可访问且 `isShared = true` 的清单”，统计口径必须对齐它
4. 周期任务柱状图若无法一次完全复用饼图辅助逻辑，优先先修正 `updatedAt` 误用问题

## 验证方案
### 前端验证
1. 未登录进入“我的”页，确认不报错，游客态正常
2. 登录后首次进入“我的”页，确认只触发一次全量加载
3. 点击“我的清单”“共享清单”，确认卡片数字与落地页列表数量口径一致
4. 点击“已完成任务”，确认首页实际进入已完成任务视图，且一次性标记会被消费清理
5. 饼图在 `total = 0` 和 `total > 0` 时都能正常显示

### 数据验证
1. 构造“自己创建的共享清单 + 作为成员加入的共享清单 + 普通个人清单”，确认 `myLists`、`sharedLists` 都与落地页一致
2. 构造重复 `list_members` 记录或 creator 未入成员表的场景，确认统计不失真
3. 构造周期任务和普通任务，确认饼图、总完成数、近 7 天趋势没有明显互相矛盾
4. 完成任务后再编辑任务，确认不会把趋势错误归到编辑日期

## 风险说明
1. `myLists` 文案本身更像“我的清单”，但当前点击目标页实际是“全部可访问清单”；本轮以现有页面行为为准修正统计，避免同时改文案和路由带来额外变更面
2. 周期任务的 7 天趋势若要做到和饼图完全同源，可能需要对现有辅助逻辑做小幅扩展；但即便如此，先改用 `completedAt` 仍然是低风险且必要的第一步
3. “已完成任务”跳转联动需要小范围改动 `pages/index/index.js`，但应控制在一次性标记消费，不扩展成全局筛选系统

## 建议的落地策略
优先落地统计口径对齐、已完成任务跳转联动、首次加载去重、扇形图预计算这四项确定性低风险修复；近 7 天趋势直接切到 `completedAt`，并在实现时尽量收敛周期任务口径差异，不扩大到任务模型改造。
