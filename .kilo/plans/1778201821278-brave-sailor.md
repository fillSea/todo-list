# 任务页面问题修复方案

## 目标

修复当前任务页面中已确认的功能错误与逻辑不一致问题，确保以下能力稳定一致：

- 提醒选择器不再绑定不存在的方法
- 截止时间的展示值、提交值、后端存储语义一致
- “仅日期”和“日期+时间”任务在编辑后不会发生隐式语义漂移
- 周重复值编码与文档、前后端规则统一
- `task-edit` 与 `task-detail` 的共享逻辑收敛，避免继续漂移

## 问题分组

### P0：必须立即修复的真实功能错误

1. `task-edit.wxml` 绑定了不存在的 `onReminderColumnChange`
2. 页面默认显示 `23:59`，但未主动选择时间时实际提交 `null`，后端落库为 `00:00`
3. 编辑已有“仅日期任务”时，页面会自动显示并提交 `00:00`，导致数据语义被悄悄改变

### P1：需要统一的语义/一致性问题

1. 周重复编码前端使用 `0-6`，文档使用 `1-7`
2. 两个页面对提醒选择器实现不一致
3. 周期任务判断仅依赖 `repeatType > 0`，实现脆弱

### P2：结构性风险

1. `task-edit` 与 `task-detail` 双实现并行，未来极易继续出现行为漂移

## 设计原则

1. 优先做最小正确修复，先解决用户可感知 bug。
2. 尽量不改动云函数接口形状，除非无法正确表达“仅日期 / 日期+时间”语义。
3. 若需要补充字段，优先使用显式字段表达，而不是继续依赖 `00:00` / `23:59` 猜测语义。
4. 前后端与文档中的周重复编码必须统一到单一标准。
5. 页面间能复用的逻辑尽量下沉到 `taskMixin`，避免复制粘贴维护。

## 推荐方案

### 一、提醒选择器修复

#### 方案

将提醒选择器收敛为单列选择器语义，移除无意义的多列联动绑定。

#### 修改点

1. `miniprogram/pages/task-edit/task-edit.wxml`
   - 删除 `bindcolumnchange="onReminderColumnChange"`
2. `miniprogram/pages/task-detail/task-detail.wxml`
   - 与 `task-edit` 保持一致，仅保留 `bindchange="onReminderChange"`
3. `miniprogram/mixins/taskMixin.js`
   - 保留现有 `onReminderChange`
   - 评估是否将 `reminderRanges` 从二维数组调整为更明确的单列结构

#### 推荐实现

短期保持 `picker mode="multiSelector"` 不变，只移除不存在的方法绑定，避免扩大改动面。

#### 后续可选优化

如果想进一步简化，可以改为单列 `selector`，但这不是本轮必要项。

### 二、截止时间语义修复

这是本轮最关键的设计点。

#### 当前问题

- 新建任务时页面显示 `23:59`，但若用户未点时间选择器，提交的是 `null`
- 后端把无时间解释为 `00:00`
- 编辑已有“仅日期任务”时，前端会从后端 `00:00` 反推为显式时间 `00:00`

#### 推荐目标语义

系统需要显式区分两种任务：

1. 仅日期任务：只关心哪一天截止，不关心具体时间
2. 日期+时间任务：关心精确到分钟的截止时间

#### 推荐实现方式

在前端页面状态中新增显式布尔标记，例如：

- `task.hasDueTime: boolean`

并让保存参数显式携带该信息，例如：

- `hasDueTime: true/false`

#### 前端设计

1. 新建任务默认：
   - `dueDate` 为空
   - `dueTime` 为空
   - `hasDueTime` 默认 `false`
2. 用户选择日期后：
   - 页面不再强制展示一个伪默认值作为真实已选时间
   - 截止时间区域可显示“未设置时间”或“全天截止”
3. 用户选择时间后：
   - `hasDueTime = true`
   - `dueTime` 为用户选择值
4. 若用户取消时间或点击“清除时间”：
   - `hasDueTime = false`
   - `dueTime = ''`

#### 页面交互建议

截止时间区域建议从“总显示一个时间值”改为“可选时间”：

- 未设置时间时显示：`未设置（按当天处理）` 或 `全天截止`
- 已设置时间时显示真实时间
- 提供“设置时间 / 清除时间”能力

#### 后端设计

云函数 `createTask` / `updateTask` 增加对 `hasDueTime` 的处理：

1. `hasDueTime === true`
   - 使用 `dueDate + dueTime` 构造精确时间
2. `hasDueTime === false`
   - 仍按日期入库，但需要保证后续展示层不会把它误判成“用户设置了 00:00”

#### 关于存储语义的推荐

推荐保留当前数据库只存一个 `dueDate` 的方式，但新增一个持久字段：

- `hasDueTime: boolean`

原因：

- 仅靠 `00:00` 无法区分“用户真设置了凌晨 00:00”还是“只是没有具体时间”
- 页面编辑回填必须依赖显式字段，否则语义会继续丢失
- 文档中已说明该字段当前未稳定持久化，本轮正好把它补齐

#### 需要同步修改的地方

1. `miniprogram/mixins/taskMixin.js`
   - 初始化 `task.hasDueTime`
   - `onDueTimeChange` 同步设置 `hasDueTime = true`
   - 增加清除时间的处理方法
   - `buildSaveParams` 传递 `hasDueTime`
   - `setTaskData` 相关页面回填时依据后端字段判断是否显示时间
2. `miniprogram/pages/task-edit/task-edit.js`
   - 初始化任务模型中的 `hasDueTime`
   - `setTaskData` 回填时不要无条件从 `dueDate` 推导 `dueTime`
3. `miniprogram/pages/task-detail/task-detail.js`
   - 同上
4. `miniprogram/pages/task-edit/task-edit.wxml`
   - 调整时间展示逻辑，不再用 `task.dueTime || '23:59'`
5. `miniprogram/pages/task-detail/task-detail.wxml`
   - 同上
6. `cloudfunctions/taskFunctions/index.js`
   - `createTask` 写入 `hasDueTime`
   - `updateTask` 更新 `hasDueTime`
   - 所有回传任务详情、任务列表的路径保留 `hasDueTime`
7. `doc/data_design.md`
   - 将 `hasDueTime` 从“未稳定持久化”更新为正式字段

#### 兼容旧数据策略

旧任务没有 `hasDueTime` 时，建议采用以下兼容规则：

1. 如果数据库记录中 `hasDueTime` 明确存在，按该字段为准
2. 如果旧数据没有该字段：
   - 默认按 `false` 处理更稳妥，避免把历史“仅日期任务”误判为显式时间任务
   - 但这会把少量真实设置为 `00:00` 的历史任务也当成“仅日期任务”

#### 推荐取舍

推荐“默认旧数据 `hasDueTime=false`”。

理由：
- 当前系统主问题是大量仅日期任务会被错误编辑为 `00:00`
- 与其保住极少数真正的 `00:00` 任务，不如优先保住大多数“无具体时间”任务语义

如果希望更稳，可以后续单独做一次历史数据迁移，但不作为本轮必做项。

### 三、周重复编码统一

#### 当前问题

- 前端周日使用 `0`
- 文档使用 `1-7` 表示周一到周日
- 后端兼容两套编码，但这会放大维护成本

#### 推荐标准

统一采用文档标准：`1-7` 表示周一到周日。

#### 前端修改点

1. `taskMixin.data.weekdays`
   - 改为：周一=1 ... 周日=7
   - 展示顺序建议也与该编码一致：一、二、三、四、五、六、日
2. `onWeekdayToggle`
   - 按新编码生成 `repeatValue`
3. `parseRepeat`
   - 不再依赖 `0 -> 周日` 特殊兼容

#### 后端修改点

1. `normalizeWeeklyRepeatDays`
   - 保留对旧 `0` 的兼容输入
   - 输出统一为内部标准表示
2. 所有构建周期说明、计算下一次周期日期、生成实例的逻辑
   - 检查是否以 `0-6` 为内部运算模型
   - 若内部依赖 JS `Date.getDay()`，可在函数入口做映射，内部保留 `0-6`，但外部协议统一 `1-7`

#### 推荐做法

采用“外部协议统一、内部计算兼容映射”的方式：

- 前端/文档/接口参数统一 `1-7`
- 后端进入核心计算前转换成 JS 习惯的 `0-6`

这样改动最小，也更稳。

### 四、周期任务判断条件增强

#### 当前问题

前端大量逻辑仅用 `task.repeatType > 0` 判断是否周期任务。

#### 风险

如果未来实例数据不稳定携带 `repeatType`，则：

- 删除按钮文案
- 删除范围选择
- 状态切换提示

都可能判断失真。

#### 推荐方案

抽出统一判断函数，例如：

- `isPeriodicTask(task)`

推荐规则：

- `repeatType > 0` 或
- `isPeriodicInstance === true` 或
- `parentTaskId` 存在

#### 修改点

1. `taskMixin.js` 或页面工具函数中新增统一判断方法
2. `task-edit.js` / `task-detail.js`
   - 删除流程使用统一判断函数，而不是手写 `repeatType > 0`

### 五、收敛 `task-edit` 与 `task-detail` 的双实现

#### 目标

不在本轮做大重构，但要把高风险重复逻辑继续下沉，减少两页漂移。

#### 本轮建议收敛范围

1. `setTaskData` 的日期/时间/提醒/重复回填逻辑抽成 mixin 公共方法
2. 保存响应处理逻辑抽成 mixin 公共方法
3. 删除弹窗文案构造逻辑抽成公共方法
4. 周期任务判断抽成公共方法

#### 不建议本轮做的事

- 不建议直接合并成单页面
- 不建议重写整个页面结构

原因：改动过大，超出本轮修 bug 范围。

## 验证方案

### 页面交互验证

1. 新建任务，仅选择日期、不选时间
   - 页面不显示伪 `23:59`
   - 提交后详情页仍显示为“未设置具体时间”
2. 新建任务，选择日期和时间
   - 保存后回显为同一时间
3. 编辑历史仅日期任务
   - 不应自动变成 `00:00`
4. 编辑显式时间任务
   - 不应丢失时间
5. 提醒选择器操作
   - 不报事件缺失错误
6. 周重复任务选择
   - 前端提交值符合 `1-7` 标准
7. 周期任务删除
   - 模板任务、实例任务都能正确出现删除范围选择

### 后端逻辑验证

1. `createTask`
   - `hasDueTime=false` 时正确落库
   - `hasDueTime=true` 时正确落库
2. `updateTask`
   - 可从有时间改为无时间
   - 可从无时间改为有时间
3. 周重复
   - 对旧 `0-6` 输入仍可兼容
   - 对新 `1-7` 输入生成实例正确

### 回归关注点

1. 首页任务列表的截止时间展示
2. 日历页任务日期落点
3. 逾期判断
4. 周期任务“只能完成当天任务”的提示逻辑
5. 附件编辑、保存、删除流程不受影响

## 风险与控制

### 风险 1：新增 `hasDueTime` 后影响现有查询/展示

控制方式：
- 仅作为附加字段引入
- 不改变已有 `dueDate` 主字段结构

### 风险 2：旧数据语义无法百分百还原

控制方式：
- 明确采用“旧数据默认无具体时间”的兼容策略
- 后续如有需要再单独做数据修复脚本

### 风险 3：两个页面继续漂移

控制方式：
- 本轮把关键回填/删除判断/弹窗文案抽入 mixin
- 后续再评估是否进一步合并页面

## 实施顺序

1. 修复提醒选择器无效绑定
2. 引入 `hasDueTime` 语义并贯通前后端
3. 修改时间展示与保存逻辑
4. 统一周重复编码
5. 补强周期任务统一判断
6. 下沉重复逻辑到 `taskMixin`
7. 更新文档并做回归验证

## 推荐决策

本轮实现时按以下决策执行：

1. 提醒选择器不做大改，只删除错误绑定
2. 正式引入并持久化 `hasDueTime`
3. 旧数据默认按 `hasDueTime=false` 兼容
4. 周重复对外统一为 `1-7`，后端内部继续兼容映射
5. 不合并页面，只做关键逻辑下沉
