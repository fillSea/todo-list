# 通知设置完整修复计划

## 目标
在开发环境下按完整方案修复通知设置功能，使 4 个设置项都能：
- 正确读取
- 正确保存
- 正确影响通知发送逻辑
- 在前端页面表现一致且可预期

本计划不考虑历史数据迁移，不做兼容旧结构的复杂兜底，只保留必要默认值。

## 数据模型
统一采用以下用户字段结构：

```js
{
  enableNotifications: true,
  notificationSettings: {
    taskReminder: true,
    listCollaboration: true,
    systemNotice: true
  }
}
```

说明：
- `enableNotifications` 保留在顶层，减少对现有代码的影响。
- 细分通知项统一放入 `notificationSettings`。
- 总开关关闭时，所有通知均不发送；子开关仅在总开关开启时生效。

- `miniprogram/pages/profile/notification-settings/notification-settings.js`

- `cloudfunctions/listFunctions/index.js`
- 可选同步文案：`miniprogram/pages/profile/help/help.js`

### 协作通知范围说明
本次只处理 `listFunctions/index.js` 中已确认的 5 处通知写入点，避免把权限校验、成员管理和业务返回值混在一起：

1. 邀请成员时写入邀请通知
2. 成员申请加入时写入申请通知
3. 邀请提醒时写入提醒通知
4. 申请通过时写入通过通知
5. 申请拒绝时写入拒绝通知

说明：
- `removeMember` 当前没有通知写入，不纳入本次修复。
- 本次只拦截 `notifications` 写入，不改变邀请、审批、加入等业务主流程。

## 实施步骤

### 1. 前端通知设置页整理状态模型
在 `notification-settings.js` 中引入单一默认设置常量，例如：

```js
const DEFAULT_SETTINGS = {
  enableNotifications: true,
  taskReminder: true,
  listCollaboration: true,
  systemNotice: true
};
```

调整点：
- 页面初始化、未登录态重置、读取本地缓存、读取云端返回时都基于 `DEFAULT_SETTINGS` 合并。
- 避免直接用本地缓存或云端对象整体覆盖 `settings`，防止字段缺失导致异常。
- `loadSettings()` 读取本地缓存后先渲染，再以云端结果覆盖，最终仍以云端为准。

### 2. 修复前端保存与失败回滚逻辑
在 `onSettingChange` 中改成以下流程：
- 先保存 `previousSettings`
- 乐观更新页面与本地缓存
- 调用 `updateNotificationSettings`
- 如果失败：
  - 回滚 `this.data.settings`
  - 回滚 `notificationSettings` 本地缓存
  - 弹出错误提示

这样可以解决当前“界面回滚了，但本地缓存仍是错的”的问题。

### 3. 实现总开关对子开关的联动交互
在 `notification-settings.wxml` 中为子开关增加禁用规则：
- `enableNotifications` 永远可操作
- 当 `settings.enableNotifications === false` 时，其余 3 个开关置灰不可点击

建议规则：
```xml
 disabled="{{!settings.enableNotifications && item.key !== 'enableNotifications'}}"
```

这样总开关语义清晰，同时保留子开关值，不会在关闭总开关时重置细分偏好。

### 4. 修复 `profileFunctions.getNotificationSettings`
在 `cloudfunctions/profileFunctions/index.js` 中：
- 从 `users` 表中读取 `enableNotifications`
- 从 `user.notificationSettings` 读取 `taskReminder`、`listCollaboration`、`systemNotice`
- 返回结构固定为 4 个字段

建议实现要点：
- 不再硬编码 3 个子开关为 `true`
- 对缺失字段用默认值兜底即可

目标返回值：

```js
{
  code: 0,
  message: 'success',
  data: {
    enableNotifications: user.enableNotifications !== false,
    taskReminder: user.notificationSettings?.taskReminder !== false,
    listCollaboration: user.notificationSettings?.listCollaboration !== false,
    systemNotice: user.notificationSettings?.systemNotice !== false
  }
}
```

### 5. 修复 `profileFunctions.updateNotificationSettings`
在 `cloudfunctions/profileFunctions/index.js` 中支持更新全部 4 个开关。

实现原则：
- 顶层字段只更新 `enableNotifications`
- 细分字段更新到 `notificationSettings`
- 只修改本次传入的字段，不覆盖未传入字段

建议流程：
1. 读取当前用户文档
2. 构造 `updateData`
3. 合并当前 `notificationSettings` 与本次变更
4. 一次 `update` 写回数据库

目标行为示例：
- 传 `{ taskReminder: false }` 时，只修改 `notificationSettings.taskReminder`
- 传 `{ enableNotifications: false }` 时，只修改顶层总开关

### 6. 修复新用户与资料更新的默认通知结构
在 `registerOrUpdateUser` 和 `getUserInfo` 的新建用户逻辑中补全默认通知设置：

```js
notificationSettings: {
  taskReminder: true,
  listCollaboration: true,
  systemNotice: true
}
```

要求：
- 新注册用户创建时就写入完整结构
- 更新用户资料时，不应无意覆盖已有 `notificationSettings`
- `getUserInfo` 自动创建新用户时也保持相同结构，避免数据源不一致

### 7. 任务提醒逻辑接入 `taskReminder`
在 `cloudfunctions/taskFunctions/index.js` 的提醒发送逻辑中，当前只判断：
- `enableNotifications`

需要改为同时判断：
- `user.enableNotifications !== false`
- `user.notificationSettings?.taskReminder !== false`

目标逻辑：
- 总开关关闭：不创建任务提醒通知，也不发订阅消息
- 任务提醒开关关闭：同样不创建任务提醒通知，也不发订阅消息

保持现有“跳过后标记 `reminderSent`”的处理方式不变，避免定时任务重复打扰。

### 8. 协作通知逻辑接入 `listCollaboration`
在 `cloudfunctions/listFunctions/index.js` 中，给以下通知写入点增加接收方通知偏好校验：

1. `list_invite` 邀请通知
2. `join_request` 申请加入通知
3. `invite_remind` 邀请提醒通知
4. `application_approved` 申请通过通知
5. `application_rejected` 申请拒绝通知

建议按“接收方是否可接收”做统一判断，调用位置固定为这 5 处写通知之前：
- 邀请通知：判断被邀请人
- 申请加入通知：判断列表创建者或审批人
- 邀请提醒通知：判断被提醒的被邀请人
- 申请通过通知：判断申请人
- 申请拒绝通知：判断申请人

建议抽一个本文件内私有小函数，避免重复查询，例如：

```js
async function canReceiveCollaborationNotification(userId) {
  // 查询 users 文档
  // 同时判断 enableNotifications 和 notificationSettings.listCollaboration
}
```

使用规则：
- 仅在允许接收时写入 `notifications` 集合
- 不影响业务主流程本身，例如邀请、审批、加入结果仍应成功
- 日志 `logOperation` 仍然保留，不受通知开关影响

### 9. 系统通知开关的本次处理边界
当前代码中未发现明确的系统通知发送逻辑接入点，因此本次完整修复先完成以下内容：
- 前端可读写 `systemNotice`
- `profileFunctions` 能持久化 `systemNotice`

说明：
- `systemNotice` 本次先修复为“数据闭环完成”
- 若后续新增系统公告写入 `notifications` 的逻辑，再按相同模式接入该开关

### 10. 文案同步
同步调整帮助文案与设置页说明，避免语义与真实行为不一致：
- 强调“接收通知”为总开关
- 强调“任务提醒/协作通知”为分类开关
- 若 `systemNotice` 暂无实际发送来源，可保持中性文案，不宣称已有完整系统通知能力

## 验证方案

### 前端页面验证
1. 登录后进入通知设置页，确认 4 个开关都能正常展示。
2. 关闭 `enableNotifications`，确认其余 3 个开关置灰不可操作。
3. 重新开启 `enableNotifications`，确认子开关恢复可操作且保留原值。
4. 修改任意子开关后退出再进入，确认云端值正确回显。
5. 模拟保存失败时，确认页面状态与本地缓存都能回滚。

### 云端数据验证
1. 新注册用户后检查 `users` 文档，确认包含：
   - `enableNotifications`
   - `notificationSettings.taskReminder`
   - `notificationSettings.listCollaboration`
   - `notificationSettings.systemNotice`
2. 分别调用 `getNotificationSettings` / `updateNotificationSettings`，确认读写一致。

### 任务提醒验证
1. 将 `enableNotifications=false`，触发提醒任务，确认不写入任务提醒通知。
2. 将 `enableNotifications=true` 且 `taskReminder=false`，确认同样不写入任务提醒通知。
3. 将两者都设为 `true`，确认会正常写入通知，并在配置存在时尝试发送订阅消息。

### 协作通知验证
分别验证以下场景的接收方：
1. 邀请成员
2. 申请加入
3. 邀请提醒
4. 审批通过
5. 审批拒绝

验证矩阵：
- `enableNotifications=false` 时不写通知
- `enableNotifications=true` 且 `listCollaboration=false` 时不写通知
- 两者都为 `true` 时正常写通知

## 风险与注意事项
- `systemNotice` 当前没有已确认的发送链路，本次只能完成存储闭环，不能伪造业务消费点。
- `listFunctions` 中通知发送点较多，需确保只拦截通知写入，不影响原业务返回值。
- 不要把通知偏好判断混入权限校验逻辑，二者应分离。
- 本仓库主要通过微信开发者工具验证，修改后需在开发者工具中联调云函数与小程序页面。
