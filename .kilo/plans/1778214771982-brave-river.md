# 修复“我的通知”残留问题方案

## 目标
修复上一轮核查中确认的两个残留问题，使“我的通知”页与当前纯站内通知方案完全对齐：
- 清理通知页中已基本失效的历史筛选项，避免出现长期无数据的分类。
- 统一 `profileFunctions.getNotifications` 在“用户不存在”和“正常成功”场景下的返回结构，消除接口形状不一致。

## 待修复问题

### 1. 通知页保留了历史筛选项
现状文件：`miniprogram/pages/profile/notifications/notifications.js`

当前 `typeOptions` 仍包含：
- `task_assigned`
- `task_updated`
- `list_shared`
- `deadline_reminder`

而当前代码检索到的实际通知写入来源主要为：
- `task_reminder`
- `list_invite`
- `invite_remind`
- `join_request`
- `application_approved`
- `application_rejected`

影响：
- 页面不会报错。
- 但部分筛选项会长期为空，和当前产品能力不一致。
- 会让用户误以为存在任务分配、任务更新、清单共享等通知链路。

### 2. `getNotifications` 返回结构不一致
现状文件：`cloudfunctions/profileFunctions/index.js`

当前实现：
- 用户存在时返回：`{ code, message, data: formattedNotifications, total }`
- 用户不存在时返回：`{ code, message, data: { list: [], total: 0 } }`

影响：
- 当前通知页主要走已登录已注册用户路径，所以通常不炸。
- 但接口契约不稳定，后续任何复用方若直接消费 `data` 为数组，都会在未注册边界出问题。
- 这和当前前端的消费方式也不一致。

## 修改原则
- 只做最小修复，不扩大到整套通知系统重构。
- 不修改通知集合数据结构。
- 不改“我的通知”页主流程：加载、分页、已读、删除、跳转保持现状。
- 不顺手改动更多通知类型映射，除非和当前两项问题直接相关。

## 方案细节

### 一、收敛通知筛选项
修改文件：`miniprogram/pages/profile/notifications/notifications.js`

#### 方案
将 `typeOptions` 收敛为当前真实有效的站内通知类型：
- `all`
- `list_invite`
- `task_reminder`

可选保留项：
- `join_request`
- `application_approved`
- `application_rejected`
- `invite_remind`

#### 推荐取舍
推荐保留“用户可感知且当前真实会出现”的全部类型，而不是只剩 2 个大类。理由：
- 当前前端已经支持这些类型的文案、图标、跳转。
- 这些类型确实仍会被后端写入。
- 只移除已确认失效的历史类型，可以做到最小改动且不损失现有可筛选能力。

因此推荐最终筛选项为：
- `all` 全部
- `list_invite` 清单邀请
- `invite_remind` 邀请提醒
- `join_request` 加入申请
- `application_approved` 申请通过
- `application_rejected` 申请被拒
- `task_reminder` 任务提醒

#### 不改内容
以下逻辑保持不动：
- `getNotificationTypeText`
- `getNotificationIcon`
- `onNotificationTap`

原因：
- 它们已经兼容当前真实通知类型。
- 即使保留少量历史映射也不会导致错误；本次目标是修复筛选冗余，而不是清空所有历史兼容分支。

### 二、统一 `getNotifications` 返回结构
修改文件：`cloudfunctions/profileFunctions/index.js`

#### 方案
将“用户不存在”分支返回改为与正常成功路径一致：

从：
```js
return {
  code: 0,
  message: 'success',
  data: { list: [], total: 0 }
};
```

改为：
```js
return {
  code: 0,
  message: 'success',
  data: [],
  total: 0
};
```

#### 说明
- 这是接口契约修复，不改变业务语义。
- 前端 `notifications.js` 现在已经按数组消费 `res.result.data`，统一后更安全。
- 不需要同步修改通知页前端逻辑，因为它已经兼容“正确结构”。

## 实施步骤
1. 阅读 `miniprogram/pages/profile/notifications/notifications.js`，确认当前筛选项数组与类型映射位置。
2. 修改 `typeOptions`，移除已确认失效的历史类型：
   - `task_assigned`
   - `task_updated`
   - `list_shared`
   - `deadline_reminder`
3. 保留当前仍在写入的通知类型筛选项：
   - `list_invite`
   - `invite_remind`
   - `join_request`
   - `application_approved`
   - `application_rejected`
   - `task_reminder`
4. 阅读 `cloudfunctions/profileFunctions/index.js` 中 `getNotifications`。
5. 修改“用户不存在”分支返回结构，使其与成功路径统一为：`data: []`、`total: 0`。
6. 自查通知页是否仍能按 `res.result.data` 数组做 `enrichNotifications`，无需额外兼容代码。
7. 自查筛选项是否完全移除历史失效通知类型，避免 UI 暗示不存在的通知能力。

## 验收点
- “我的通知”页筛选栏不再出现：
  - `任务分配`
  - `任务更新`
  - `清单共享`
  - `截止提醒`
- “我的通知”页筛选栏仍能筛选当前真实会出现的通知类型。
- `profileFunctions.getNotifications` 在用户不存在时返回：
  - `data` 为数组
  - `total` 为数字
- 通知页现有加载逻辑无需新增兼容分支即可正常工作。
- 已读、全部已读、删除、跳转逻辑不受影响。

## 风险与注意事项
- 如果数据库历史里仍残留旧类型通知，移除筛选项后这些通知只会在“全部”中出现，不能再按旧分类单独筛选。这是符合当前产品收敛方向的。
- 本次不建议进一步删除 `getNotificationTypeText` / `getNotificationIcon` / `onNotificationTap` 中的历史分支，避免扩大改动面；这些可留待后续独立清理。
- 本次不修改 `notificationTestData`，除非后续明确要求测试数据也同步收敛。

## 预期结果
修复完成后：
- 通知页 UI 与当前真实通知能力保持一致。
- `getNotifications` 接口契约在所有成功路径下保持一致。
- “我的通知”相关逻辑从产品语义到接口形状都更稳定。
