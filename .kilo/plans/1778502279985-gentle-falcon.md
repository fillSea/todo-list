# 协作邀请页邀请人头像不可见修复方案

## 背景

被邀请用户通过邀请链接进入 `list-invite-accept` 页面后，页面能显示邀请人的用户名，但头像显示为空或默认头像。

相关链路：

- 前端展示位置：`miniprogram/pages/list-invite-accept/list-invite-accept.wxml`
- 邀请信息接口：`cloudfunctions/listFunctions/index.js` 中的 `getInviteInfo`
- 邀请人数据来源：`users` 集合的 `nickname` 和 `avatarUrl`

前端当前使用：

```xml
<image class="inviter-avatar" src="{{inviteInfo.inviterAvatar || '/images/default-avatar.png'}}" mode="aspectFill" />
<text class="inviter-name">{{inviteInfo.inviterName || '未知用户'}}</text>
```

后端当前返回：

```js
inviterName: inviter ? inviter.nickname : '未知用户',
inviterAvatar: inviter ? inviter.avatarUrl : '',
```

## 根因

用户名能显示，说明 `users.nickname` 被正确查询并返回。

头像不可见的原因通常是 `users.avatarUrl` 不是页面可直接展示的有效图片地址：

1. `avatarUrl` 为空：邀请人历史资料中没有头像，或头像上传/保存失败。
2. `avatarUrl` 是 `cloud://...` 云文件 ID：注册页会上传头像到云存储并保存云文件 ID，但邀请信息接口没有把云文件 ID 转成临时 HTTPS URL。
3. `avatarUrl` 是本地临时路径：例如 `wxfile://...`，只能在当前设备临时访问，不能跨用户展示。

当前 `getInviteInfo` 直接把数据库里的 `avatarUrl` 原样返回给前端，导致 `<image>` 无法稳定展示邀请人头像。

## 解决目标

在邀请信息接口返回前，将头像地址规范化为前端可展示地址：

- 空值保持为空，由前端兜底默认头像。
- `http://` 或 `https://` 地址直接返回。
- `cloud://` 云文件 ID 转换为临时 HTTPS URL 后返回。
- 无法转换或非法地址返回空字符串，避免前端渲染异常。

## 实施方案

### 1. 在 `cloudfunctions/listFunctions/index.js` 增加头像 URL 解析辅助函数

新增函数建议放在辅助函数区域，靠近 `getUserBasicInfo` 或其他通用工具函数：

```js
async function resolveAvatarUrl(avatarUrl) {
  if (!avatarUrl || typeof avatarUrl !== 'string') {
    return '';
  }

  if (!avatarUrl.startsWith('cloud://')) {
    return avatarUrl;
  }

  try {
    const result = await cloud.getTempFileURL({
      fileList: [avatarUrl]
    });

    return result.fileList?.[0]?.tempFileURL || '';
  } catch (error) {
    console.error('转换头像临时链接失败:', error);
    return '';
  }
}
```

### 2. 修改 `getInviteInfo` 的邀请人头像返回逻辑

在查到 `inviter` 后解析头像：

```js
const inviterAvatar = await resolveAvatarUrl(inviter?.avatarUrl || '');
```

然后返回：

```js
inviterAvatar,
```

替换原来的：

```js
inviterAvatar: inviter ? inviter.avatarUrl : '',
```

### 3. 同步处理成员预览头像

`getInviteInfo` 同时返回 `members` 给邀请页展示已有成员头像。当前成员头像同样直接返回 `avatarUrl`：

```js
members: memberUsers.map(u => ({ avatarUrl: u.avatarUrl })),
```

建议一并转换，避免成员头像也出现相同问题：

```js
const resolvedMemberAvatars = await Promise.all(
  memberUsers.map(user => resolveAvatarUrl(user.avatarUrl || ''))
);
```

返回：

```js
members: resolvedMemberAvatars.map(avatarUrl => ({ avatarUrl })),
```

### 4. 可选优化：批量转换云文件 ID

如果担心成员头像增多后多次调用 `cloud.getTempFileURL`，可以实现批量转换函数：

```js
async function resolveAvatarUrls(avatarUrls = []) {
  const normalized = avatarUrls.map(url => typeof url === 'string' ? url : '');
  const cloudUrls = normalized.filter(url => url.startsWith('cloud://'));

  if (cloudUrls.length === 0) {
    return normalized;
  }

  try {
    const result = await cloud.getTempFileURL({ fileList: cloudUrls });
    const tempUrlMap = {};

    (result.fileList || []).forEach(item => {
      tempUrlMap[item.fileID] = item.tempFileURL || '';
    });

    return normalized.map(url => url.startsWith('cloud://') ? (tempUrlMap[url] || '') : url);
  } catch (error) {
    console.error('批量转换头像临时链接失败:', error);
    return normalized.map(url => url.startsWith('cloud://') ? '' : url);
  }
}
```

本次修复可优先采用单个 `resolveAvatarUrl`，改动更小；如果后续头像展示场景较多，再提取批量函数。

## 不建议的方案

不建议只在前端 `list-invite-accept.js` 中调用 `wx.cloud.getTempFileURL`：

- 邀请页只是头像消费端，其他页面也可能需要展示同类云头像。
- 后端接口本身知道哪些字段来自数据库，统一返回可展示 URL 更稳定。
- 前端处理会增加页面逻辑复杂度，并且多个页面容易重复实现。

不建议把数据库中的 `avatarUrl` 直接永久改成临时 HTTPS URL：

- 临时 URL 有有效期，不适合作为持久数据保存。
- 数据库应保留稳定的云文件 ID 或长期可访问 URL。

## 验证方案

### 数据验证

检查邀请人的 `users` 记录：

```js
{
  _id: invite.inviterId,
  nickname: '...',
  avatarUrl: '...'
}
```

确认测试覆盖：

- `avatarUrl` 为空。
- `avatarUrl` 为 `cloud://...`。
- `avatarUrl` 为 `https://...`。

### 功能验证

通过 WeChat DevTools 执行：

1. 使用有头像的用户生成协作邀请链接。
2. 使用另一个用户打开邀请链接。
3. 确认协作邀请页显示邀请人头像和昵称。
4. 确认成员预览头像仍能正常显示。
5. 使用头像为空的邀请人验证默认头像兜底正常。

### 回归验证

重点检查：

- 搜索邀请、微信邀请、邀请链接三种入口进入邀请接受页。
- 需要审核和直接加入两种邀请流程。
- 已处理邀请、已是成员状态下页面仍能正常加载。
- `getInviteInfo` 返回结构保持兼容，不影响前端字段名。

## 风险与注意事项

- 云函数修改后需要重新部署 `listFunctions`。
- 如果历史用户头像是 `wxfile://...` 或其他本地路径，无法通过云函数转换，需要用户重新保存头像或做数据清洗。
- 如果头像文件已被删除，`cloud.getTempFileURL` 可能返回空或失败，应保留默认头像兜底。
- `memberCount` 当前使用 `members.length`，只统计当前查询到的最多 5 个成员，不是全量成员数；这不是本次头像问题根因，可另行修复。

## 推荐执行顺序

1. 在 `listFunctions` 添加 `resolveAvatarUrl`。
2. 在 `getInviteInfo` 中转换 `inviterAvatar`。
3. 同步转换 `members` 中的头像。
4. 在 WeChat DevTools 重新部署云函数。
5. 使用邀请链接端到端验证头像展示。
