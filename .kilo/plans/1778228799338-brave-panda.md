# “我的通知”改进方案

## 目标

修复“我的通知”页面中通知类型不一致、点击通知后的跳转不稳定、已读操作与未读角标不同步的问题，保证通知列表展示、点击、已读、全部已读、删除后的状态在通知页、我的页和 TabBar 之间保持一致。

## 现状结论

### 已确认的行为

1. 通知列表通过 `profileFunctions.getNotifications` 拉取，分页逻辑可用。
2. 点击通知会先尝试调用 `markNotificationAsRead`，再按 `type` 执行跳转。
3. “全部已读”和“删除通知”后，通知页只更新当前页面本地列表。
4. 个人页未读数和 TabBar 角标仅在 `miniprogram/pages/profile/profile.js` 的 `loadUnreadCount()` 中刷新。
5. 任务相关通知类型在代码中确认到的是 `task_reminder` 和 `task_update`，而通知页前端当前使用的是 `task_updated`。

### 已识别的问题

1. 前后端任务通知类型命名不一致。
   - 后端实际通知类型包含 `task_update`
   - 前端文案、图标、点击跳转分支使用 `task_updated`
   - 结果是该类通知可能显示为“系统通知”且点击无跳转
2. 已读状态与未读角标同步不完整。
   - 单条已读、全部已读、删除通知后，通知页更新了本地列表
   - 但“我的”页角标和 TabBar 红点未立即同步
3. 通知点击分支可维护性较差。
   - 类型、文案、图标、跳转逻辑散落在多个 `switch/map` 中
   - 后续新增类型时容易遗漏某个映射

## 改进原则

1. 优先做最小正确修改，不改动无关业务流。
2. 前端先做兼容，避免依赖历史数据清洗。
3. 不假设线上通知数据全部统一，兼容旧值与现值并存。
4. 角标同步尽量复用现有云函数 `getUnreadNotificationCount`，避免本地推算和真实数据偏离。

## 实施方案

### 一、统一通知类型兼容逻辑

在 `miniprogram/pages/profile/notifications/notifications.js` 中统一收敛通知类型定义，至少兼容以下场景：

1. 在文案映射中同时支持：
   - `task_update`
   - `task_updated`
2. 在图标映射中同时支持：
   - `task_update`
   - `task_updated`
3. 在点击跳转分支中同时支持：
   - `task_update`
   - `task_updated`

建议做法：

1. 保持现有函数结构不大改。
2. 直接在 `getNotificationTypeText()`、`getNotificationIcon()`、`onNotificationTap()` 中补齐 `task_update` 分支。
3. 如需进一步降低维护成本，可在页面内增加一个很小的“类型归一化”辅助函数，例如将 `task_updated` 和 `task_update` 统一映射为同一内部类型后再展示和跳转。

推荐优先级：

1. 先做双值兼容，保证线上马上可用。
2. 不立即改后端历史数据，不做数据库迁移。

### 二、补齐通知页的未读数同步

目标是让以下动作执行成功后，三处状态一致：

1. 通知页列表项 `isRead`
2. 个人页“我的通知”角标
3. TabBar 红点/角标

建议在通知页增加一个专用同步方法，例如：

- `refreshUnreadCount()`

职责：

1. 调用 `profileFunctions.getUnreadNotificationCount`
2. 根据返回值更新当前页面可见的未读数量状态（如果需要）
3. 同步更新 TabBar badge
4. 将最新未读数通过全局状态或页面通信方式传递给“我的”页

推荐同步方式按侵入性从小到大排序：

1. 最小方案：
   - 通知页操作成功后仅刷新 TabBar badge
   - “我的”页继续依赖 `onShow -> checkLoginStatus -> loadAllData -> loadUnreadCount` 进行刷新
   - 优点：改动小
   - 缺点：如果用户未返回“我的”页，看不到该页列表角标即时变化
2. 中等方案，推荐：
   - 通知页操作成功后刷新 TabBar badge
   - 同时通过 `getApp()` 上挂一个轻量通知计数同步方法或共享字段
   - “我的”页在 `onShow` 读取该共享值并更新 `featureList` badge
   - 优点：改动仍较小，体验更完整
3. 重方案：抽公共通知状态管理
   - 目前项目没有明显的集中式状态层，不建议为此单独引入

本次建议采用第 2 种方案：

1. 在 `app.js` 中补一个轻量级通知未读数同步能力
   - 例如统一方法：设置 TabBar badge、缓存最新未读数
2. `profile.js` 的 `loadUnreadCount()` 复用该能力
3. `notifications.js` 在以下动作成功后调用该能力：
   - `markAsRead`
   - `onMarkAllRead`
   - `deleteNotification`

### 三、明确三类成功后的刷新策略

#### 1. 单条已读

当前行为：

1. 调云函数标记已读
2. 本地把对应项 `isRead` 设为 `true`

改进为：

1. 保持本地即时更新，保证手感
2. 成功后刷新未读数并同步角标
3. 不强制整页重载通知列表，避免额外闪烁

#### 2. 全部已读

当前行为：

1. 调 `markAllNotificationsAsRead`
2. 本地将当前列表全部设为已读

改进为：

1. 保持本地即时更新
2. 成功后刷新未读数并同步角标
3. 如果当前筛选未来扩展为“仅未读”，再考虑重新拉取列表；当前无需增加复杂度

#### 3. 删除通知

当前行为：

1. 删除成功后从当前列表移除

改进为：

1. 删除后从当前列表移除保持不变
2. 若删除的是未读通知，需要刷新未读数并同步角标
3. 为简化判断，可删除成功后统一刷新一次未读数，不依赖本地推断

### 四、整理通知配置，降低遗漏风险

为避免后续新增通知类型时漏掉文案、图标、跳转分支，建议在 `notifications.js` 内做一次小幅整理：

1. 提取一份页面级通知配置表，例如按 `type` 描述：
   - 类型文案
   - 图标
   - 跳转方式
2. `getNotificationTypeText()` 与 `getNotificationIcon()` 基于同一份配置读取
3. `onNotificationTap()` 只处理：
   - 标记已读
   - 查配置并执行跳转

注意：

1. 这是轻量重构，不要扩散到多个文件。
2. 保持现有页面结构和 WXML 不变。
3. 若实施过程中发现改动超出预期，则退回到“仅补 `task_update` 分支 + 加未读数同步方法”的最小方案。

## 涉及文件

### 必改

1. `miniprogram/pages/profile/notifications/notifications.js`
   - 补齐 `task_update` 兼容
   - 增加未读数同步逻辑
   - 视情况整理通知类型映射
2. `miniprogram/app.js`
   - 增加轻量通知 badge/未读数同步方法，供多个页面复用
3. `miniprogram/pages/profile/profile.js`
   - 复用统一 badge/未读数同步逻辑，避免页面间行为不一致

### 只读核对，不预期改动

1. `cloudfunctions/profileFunctions/index.js`
   - 当前接口已满足读取/已读/全部已读/删除需求
2. `cloudfunctions/taskFunctions/index.js`
   - 当前先不改后端类型，前端先兼容 `task_update`

## 详细执行步骤

1. 在 `app.js` 中确认是否已有适合复用的全局方法；若无则新增一个轻量方法用于：
   - 设置最新未读数缓存
   - 同步 TabBar badge
2. 修改 `profile.js`：
   - `loadUnreadCount()` 成功后调用统一同步方法
   - 保持当前 `featureList` badge 更新逻辑，但必要时抽成小函数复用
3. 修改 `notifications.js`：
   - 在文案、图标、点击逻辑中兼容 `task_update`
   - 为单条已读、全部已读、删除成功后补 `refreshUnreadCount()` 调用
   - 如需要，从 `app.js` 读取或写入共享未读数
4. 进行静态检查，确认以下类型至少都有一致处理：
   - `list_invite`
   - `invite_remind`
   - `join_request`
   - `application_approved`
   - `application_rejected`
   - `task_reminder`
   - `task_update`
5. 手工验证页面链路

## 验证方案

由于项目主要依赖微信开发者工具，本次以手工验证为主。

### 场景 1：通知列表展示

1. 打开“我的通知”
2. 确认列表正常加载
3. 确认 `task_update` 类通知不再显示为“系统通知”
4. 确认图标不再回退到默认图标

### 场景 2：点击任务更新通知

1. 点击一个 `task_update` 通知
2. 预期：
   - 通知转为已读
   - 正常跳转到任务详情页
   - 返回后该通知保持已读

### 场景 3：点击清单协作类通知

分别验证：

1. `list_invite` / `invite_remind` 跳转邀请接受页
2. `join_request` 跳转邀请管理页
3. `application_approved` 跳转清单详情页
4. `application_rejected` 弹出提示，不报错

### 场景 4：单条已读后的角标同步

1. 在“我的”页观察未读角标 > 0
2. 进入通知页点击一条未读通知
3. 返回“我的”页
4. 预期：
   - “我的通知”角标减少
   - TabBar badge 同步减少

### 场景 5：全部已读后的角标同步

1. 通知页点击“全部已读”
2. 预期：
   - 当前列表全部变为已读
   - 返回“我的”页后角标为 0
   - TabBar badge 清除

### 场景 6：删除未读通知

1. 删除一条未读通知
2. 预期：
   - 列表移除该项
   - 未读数减少
   - “我的”页角标和 TabBar badge 同步变化

## 风险与注意事项

1. `relatedId` 在不同通知类型下语义不同。
   - 任务通知通常是任务 ID
   - 邀请类通知通常是 `inviteCode`
   - 申请/审批类通知通常是 `listId`
   - 修改点击逻辑时不能做错误抽象，必须保留按类型路由
2. 不要仅修改文案映射，必须同时修改图标映射与点击分支。
3. 不建议本次直接调整云函数里的历史通知类型。
   - 会引入数据兼容和回归范围扩大
   - 前端兼容可先解决用户感知问题
4. 不建议把通知页的成功操作改成“每次都整页 reload”。
   - 体验更差
   - 额外云函数调用更多
   - 局部更新 + 单独刷新未读数更适合当前项目结构

## 预期产出

1. `task_update` 通知可正确展示与点击跳转。
2. 通知已读、全部已读、删除后，通知页、个人页、TabBar 的未读状态一致。
3. 通知类型映射集中度提升，后续再加类型时更不容易遗漏。

## 实施边界

本次不包含：

1. 数据库历史通知数据迁移
2. 新增通知类型
3. 重构云函数通知生成逻辑
4. 引入全局状态管理库
