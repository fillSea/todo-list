# 共享清单“最近协作者”第一阶段执行计划

## 1. 背景
当前“邀请成员”页中的“最近协作”入口，实际返回的是“当前用户参与过的共享清单中的其他成员”，并按 `list_members.joinedAt` 倒序去重展示。

这与“最近协作”的产品语义不一致，存在两个直接问题：

1. 候选人排序依据错误
- 当前使用成员加入时间 `joinedAt`，只能表达“最近加入过同一个共享清单的人”，不能表达“最近与我协作过的人”。

2. 候选人与当前邀请场景不匹配
- 当前接口未接收 `listId`
- 返回结果未排除当前清单已有成员
- 未排除当前清单已有待处理邀请/待审批记录
- 前端会展示一些“看起来可邀请、实际提交后会失败”的人

## 2. 本次目标
本次只执行第一阶段，把“最近协作”修正为：

> 当前清单可邀请的最近协作者候选人。

本次目标分为三点：

1. 列表与当前清单邀请场景对齐
- 最近协作候选必须基于当前 `listId` 生成

2. 尽量保证列表中的人可以直接邀请
- 在返回前排除当前清单中已不可再邀请的对象

3. 保持改动最小
- 不改邀请状态流转
- 不改角色选择逻辑
- 不做真实行为级协作排序

## 3. 第一阶段范围
第一阶段只包含以下交付范围：

1. 后端 `getRecentCollaborators` 改为基于当前 `listId` 生成候选
2. 返回前过滤当前清单不可再邀请的用户
3. 前端请求补传 `listId`
4. 前端文案统一为“最近协作者”
5. 邀请成功后补刷新 `recentMembers`，避免页面继续展示刚邀请过的人

## 4. 第一阶段不做
以下内容明确不纳入本次实现：

1. 不做第二阶段“基于真实行为日志的协作排序”
2. 不引入新的数据库字段或新的集合
3. 不改 `inviteMember`、`acceptInvite`、`applyJoinList` 的主流程语义
4. 不重构邀请页整体 UI
5. 不把“最近协作”扩展到搜索、微信邀请、邀请链接等其他入口
6. 不解决 `list_members` 来源清单查询可能受单次 `.get()` 条数限制导致的不全量问题
7. 不把最近协作者能力下沉为其他页面或公共模块

## 5. 现状链路

### 5.1 前端
文件：`miniprogram/pages/list-invite/list-invite.js`

当前逻辑：
1. 页面加载时调用 `getRecentCollaborators`
2. 调用参数为空对象 `{}`
3. 返回结果直接写入 `recentMembers`
4. 点击成员后仍走统一 `inviteMember`

### 5.2 后端
文件：`cloudfunctions/listFunctions/index.js`

当前 `getRecentCollaborators` 逻辑：
1. 查询当前用户所在的 `list_members`
2. 收集所有 `listId`
3. 查询这些清单中的其他成员
4. 按 `joinedAt desc` 去重后返回前 20 个

### 5.3 已有后端保护
虽然候选列表不准确，但真正邀请时后端已做校验：

1. 已是当前清单成员时会拦截
2. 已有待处理邀请时会拦截
3. 待审批冲突时会拦截

因此本次重点是修正“候选生成”，不是重写“邀请执行”。

## 6. 第一阶段方案定义

### 6.1 产品定义
第一阶段中，“最近协作”定义为：

> 当前用户在其他共享清单中接触过的协作者里，过滤掉当前清单不可再邀请的人后，得到的候选列表。

说明：
- 这是“当前清单可邀请候选人”方案
- 不是严格意义上的真实协作行为排序
- 但能够显著改善语义和可用性

### 6.2 数据来源
本次只使用已有 `list_members` 和 `list_invites` 数据。

不使用：
- `operations`
- `tasks.completedBy`
- 其他行为日志

原因：
- 降低风险
- 控制范围
- 保持实现简单可验证

## 7. 实现清单

### 7.1 后端实现
文件：`cloudfunctions/listFunctions/index.js`

1. 保留现有接口名 `getRecentCollaborators`，增加请求参数：

```js
{
  listId: string,
  limit?: number
}
```

2. 新增参数与访问校验：

1. `listId` 不能为空
2. 当前用户必须对该清单有访问权限
3. `limit` 可选，默认 20，最大限制 20
4. 当前用户若未注册或无法映射出 `userId`，直接返回空列表或失败，避免拿 `null` 查询 `list_members`

建议行为：
- 缺少 `listId` 时直接返回失败
- 无权限时返回现有风格错误结构

3. 按以下顺序生成候选列表，本次只做当前清单过滤，不做复杂行为打分：

1. 获取当前用户 `userId`
2. 校验其对目标 `listId` 的访问权限
3. 查询当前用户参与过的共享清单成员关系
4. 收集候选来源清单 ID
5. 排除当前 `listId` 本身
6. 查询这些来源清单中的其他成员
7. 基于 `userId` 去重，保留最近一条共现记录
8. 查询当前目标清单中不可再邀请的用户集合
9. 从候选中排除不可邀请对象
10. 查询用户基础资料并组装返回
11. 截断到 `limit`

4. 返回前必须排除：

1. 当前用户自己
2. 当前清单已有成员
3. 当前清单已有待接受邀请对象 `status = 0`
4. 当前清单已有待审批对象 `status = 4`
5. 用户资料缺失的记录
6. `inviteeId` 为空的邀请模板记录，避免误判为用户级邀请占用

5. 排序规则采用最小可落地方案：

1. 仍基于共享清单共现关系
2. 对同一候选人保留“最近一次共现记录”
3. 按该记录的 `joinedAt` 倒序排序

说明：
- 这不是行为级协作排序
- 但比当前“全局其他成员列表”更贴近当前邀请场景
- 可在不扩大范围的前提下完成本次需求

6. 返回结构保持兼容，不新增调试字段：

```js
{
  success: true,
  members: [
    {
      userId: string,
      nickname: string,
      avatarUrl: string
    }
  ]
}
```

原因：
- 前端当前不依赖这些字段
- 第一阶段以最小联调面为准
- 降低联调范围

### 7.2 前端实现
文件：`miniprogram/pages/list-invite/list-invite.js`
文件：`miniprogram/pages/list-invite/list-invite.wxml`

1. `loadRecentMembers` 调用 `getRecentCollaborators` 时传入当前 `listId`：

```js
{
  action: 'getRecentCollaborators',
  data: {
    listId: this.data.listId
  }
}
```

2. 保持现有结果写入方式：
- 成功时设置 `recentMembers`
- 失败时置空 `recentMembers`

不新增复杂前端过滤逻辑，过滤统一由后端完成。

3. 交互基本保持原样：

1. 点击“最近协作”入口提示用户从下方列表选择
2. 点击某个候选人
3. 弹出角色选择
4. 调用 `inviteMember`
5. 刷新待处理邀请和清单信息
6. 邀请成功后额外刷新 `recentMembers`

4. 将以下两处文案从：
- `最近协作`

调整为：
- `最近协作者`

原因：
- 比当前语义更准确
- 与第一阶段候选人定义更一致
- 不引入过强的技术意味

5. 空态策略保持不变。当前模板已通过 `wx:if="{{recentMembers.length > 0}}"` 控制展示。

本次建议：
1. 保持“无数据时不展示模块”的策略
2. 不新增额外空态 UI

原因：
- 最小改动
- 当前页面已有多个模块，无数据时隐藏即可

## 8. 验证点

### 8.1 功能验证
1. 当前用户没有其他共享清单协作者
- `recentMembers` 应为空
- 页面不显示该模块

2. 候选人已是当前清单成员
- 不应出现在最近协作者列表

3. 候选人已有待处理邀请
- 不应出现在最近协作者列表

4. 候选人已有待审批记录
- 不应出现在最近协作者列表

5. 候选人来自其他共享清单共现成员
- 应正常展示
- 点击后可走角色选择并发起邀请

6. 邀请成功后刷新当前页
- 刚刚成功邀请的用户不应继续留在最近协作者列表中

### 8.2 回归验证
需确认不影响：

1. 搜索用户邀请
2. 微信好友邀请
3. 邀请链接邀请
4. 邀请管理页的待处理/已接受/待审批展示
5. 邀请接受页的正常流转

### 8.3 失败场景验证
1. `listId` 缺失
- 后端应返回失败
- 前端应安全兜底为不展示最近协作者

2. 当前用户无权限访问目标清单
- 后端应返回失败
- 前端应安全兜底为不展示最近协作者

3. 最近协作者接口单独失败
- 不应阻断 `loadData()` 其他并行请求
- `recentMembers` 应安全置空

## 9. 风险与取舍

### 9.1 取舍
第一阶段不追求“真实协作”排序，只解决两个核心问题：

1. 列表必须与当前清单相关
2. 列表中的人必须尽量可直接邀请

这是当前最小、最稳、最容易验证的实现范围。

### 9.2 已知限制
第一阶段实现后，“最近协作者”仍然不是严格的行为协作排序，而是：

> 基于共享清单共现关系生成的、面向当前清单邀请场景的候选列表。

该限制在第一阶段可接受。

## 10. 预期结果
实施后，“最近协作者”模块将从：

> 一个全局候选、语义不准、且经常展示不可邀请对象的列表

变为：

> 一个基于当前清单生成、可直接用于邀请成员的最近协作者候选列表
