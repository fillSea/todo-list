# 清单模块修复方案

## 目标
修复清单模块中已确认的 6 类问题，优先保证：
1. 列表展示与统计数据正确
2. 删除操作不会残留脏数据
3. 清单类型变更后的权限语义一致
4. 前端调试/状态逻辑不再出现明显错误

## 修复范围
- `cloudfunctions/listFunctions/index.js`
- `miniprogram/pages/checklist/checklist.js`
- `miniprogram/pages/checklist/checklist.wxml`

## 方案概览

### 一、修复云函数中的全量查询/删除问题

#### 1. 抽取分页读取辅助函数
在 `cloudfunctions/listFunctions/index.js` 内新增最小化的通用辅助函数，避免在多个业务函数里重复写分页逻辑。

建议增加：
- `fetchAllByWhere(collectionName, where, options = {})`
- 可选参数支持：`orderByField`、`orderDirection`、`field`

行为约定：
- 使用 `skip + limit` 或项目当前更稳妥的批量读取方式循环拉取
- 单次批量大小固定为安全值，如 `100`
- 直到当前批次返回数小于批量大小时停止
- 返回完整数组

这样可以统一替换以下直接 `.get()` 的全量读取场景：
- `getListTaskStats()` 读取 `tasks`
- `getMyLists()` 读取用户创建的 `lists`
- `getMyLists()` 读取用户的 `list_members`
- `deleteList()` 读取待删除的 `tasks`
- `deleteList()` 读取待删除的 `list_members`
- `deleteList()` 读取待删除的 `list_invites`

#### 2. 修复 `getListTaskStats()` 统计不完整
把当前：
- `db.collection('tasks').where({ listId }).get()`

改为：
- 通过分页辅助函数拿到该清单下全部任务后，再执行现有周期任务过滤与统计逻辑

要求：
- 保持当前“周期任务只展示最近一个未完成且未过期实例”的业务规则不变
- 仅修正数据来源不完整的问题，不重写统计规则

#### 3. 修复 `getMyLists()` 可访问清单漏查
当前问题分两部分：
- 我创建的清单 `.get()` 可能截断
- 我的成员关系 `.get()` 可能截断

修复方案：
- 用分页辅助函数完整获取 `myLists`
- 用分页辅助函数完整获取 `memberships`
- 继续沿用现有 `allAccessibleIds` 去重逻辑
- 保持当前分页输出接口不变：返回 `{ list, hasMore }`

注意点：
- 清单列表对外仍按 `updatedAt desc` 分页
- 不修改前端调用协议，避免扩大改动面

#### 4. 修复 `deleteList()` 删除不彻底
当前删除逻辑是先 `.get()` 然后 `Promise.all(...remove())`，当数据量超过默认上限时会残留。

修复方案：
- 通过分页辅助函数拿到完整的 `tasks`、`list_members`、`list_invites`
- 再分批执行删除

建议增加一个最小辅助函数：
- `removeDocsByIds(collectionName, ids, batchSize = 50)`

执行顺序保持原意：
1. 校验清单归属与权限
2. 删除任务
3. 删除成员关系
4. 删除邀请记录
5. 写操作日志
6. 删除清单本身

注意：
- 不引入事务，保持当前最小改动
- 如项目云环境支持事务，后续可再独立优化；本次先解决“删不干净”的确定性问题

### 二、修复共享清单改为个人清单后的权限错乱

#### 5. 明确共享 -> 个人的处理策略
这是本次唯一需要做业务策略选择的地方。基于现有页面语义“个人清单”应只属于创建者，建议采用最小且一致的策略：

**推荐策略：允许修改为个人清单，并自动移除除创建者外的全部成员关系。**

原因：
- 与“个人清单”的语义一致
- 与当前前端可编辑行为一致，无需额外改 UI 流程
- 相比“禁止修改”改动更小，不需要增加新的前端拦截与提示文案链路

#### 6. 在 `updateList()` 中补全成员清理逻辑
在现有逻辑基础上保留：
- `个人 -> 共享` 时，确保创建者在 `list_members` 中

新增：
- `共享 -> 个人` 时，查询该清单全部成员关系
- 删除所有 `userId !== creatorId` 的成员记录
- 如创建者成员记录不存在，可不强制补建，因为个人清单访问本就依赖 `creatorId`

还需注意：
- 该清理逻辑应在更新清单后执行，确保最终状态一致
- 操作日志 `list_update` 可以继续沿用现有结构，不必扩大日志模型

#### 7. 可选补充：清理相关待处理邀请
共享改个人后，旧邀请理论上也失去意义。为避免后续仍能加入该清单，建议在 `共享 -> 个人` 时额外处理：
- 将该清单的待处理邀请记录删除，或
- 将其状态改为失效

**本次推荐：直接删除待处理邀请记录。**
原因：
- 当前删除清单逻辑已允许物理删除邀请，说明项目接受该处理方式
- 实现最简单，能立即阻断后续加入入口

如果需要更保守，可仅删除 `status` 为待处理/审批中的邀请。

### 三、修复前端 checklist 页的问题

#### 8. 修复 DEBUG 分支 `wx.showLoading()` 未关闭
在 `miniprogram/pages/checklist/checklist.js` 中修复两处：
- 创建/编辑成功的 DEBUG 分支
- 删除成功的 DEBUG 分支

要求：
- 成功路径补 `wx.hideLoading()`
- 保持异常路径现有 `catch` 中的 `wx.hideLoading()` 不变
- 不改变生产分支逻辑

#### 9. 消除搜索结果重复处理
当前 `performRemoteSearch()` 中对同一份 `lists` 调用了两次 `processListData()`。

修复方案：
- 先计算一次 `const processedLists = this.processListData(lists)`
- 再同时赋值给：
  - `remoteSearchResults`
  - `filteredLists`

收益：
- 降低重复计算
- 避免以后 `processListData` 出现副作用时产生双重处理问题

#### 10. 修正 `hasProfile` 状态判断
当前 `syncLoginState()` 里只要登录就直接设为 `hasProfile: true`，和页面“完善资料”分支语义不一致。

修复方案：
- 依据 `userInfo` 是否完成必要资料来判断 `hasProfile`
- 优先复用项目现有注册/资料逻辑中的真实字段，不自行发明新的状态字段

建议检查并采用以下思路之一：
- 若 `userInfo` 中已有显式字段，如 `nickname`、`avatarUrl`、`gender`、`phone`、`isRegistered` 之类，则沿用现有规则
- 若项目没有统一标记，则使用当前注册页真正必填项作为最小判断条件

实现要求：
- `isLoggedIn = true` 不等于 `hasProfile = true`
- 当已登录但资料不完整时：
  - 不加载清单数据
  - 展示“完善资料后查看你的清单”空状态
  - “创建清单”入口跳转注册页的行为保持不变

注意：
- 实施前需读取注册页/用户资料云函数，确认项目里“资料完成”的真实判定标准
- 这一条在执行阶段需要先做一次小范围代码核对，再落地改动

#### 11. 修正文案与实际搜索行为不一致
`checklist.wxml` 当前 placeholder 为“搜索清单名称”，但后端同时搜索 `name + description`。

修复方案：
- 将 placeholder 改为更准确的表述，例如：`搜索清单名称或描述`

这是低风险文案修正，可与前端其他修改一起提交。

## 执行顺序

### 第一阶段：后端正确性修复
1. 在 `listFunctions/index.js` 增加分页读取/批量删除辅助函数
2. 替换 `getListTaskStats()` 的任务读取方式
3. 替换 `getMyLists()` 中创建清单/成员关系读取方式
4. 替换 `deleteList()` 中任务/成员/邀请读取与删除方式
5. 在 `updateList()` 中补上“共享 -> 个人”的成员清理逻辑
6. 同步处理共享 -> 个人时的待处理邀请

### 第二阶段：前端 checklist 页修复
1. 修复 DEBUG 分支漏掉的 `wx.hideLoading()`
2. 合并搜索结果重复处理
3. 按真实资料字段修复 `hasProfile` 判定
4. 调整搜索框 placeholder 文案

### 第三阶段：回归验证
重点验证以下场景：
1. 我的清单分页加载正常
2. 拥有大量任务的清单，任务数/待完成数/进度正确
3. 删除大清单后，相关任务、成员、邀请全部清理
4. 共享清单改为个人清单后，原成员不再能在清单列表中看到该清单，也不能继续访问
5. 搜索结果正常显示，清空/取消搜索后状态恢复正确
6. 调试模式下创建、编辑、删除后加载遮罩正常关闭
7. 已登录但资料不完整时，页面展示“去完善资料”而不是直接尝试加载清单

## 风险与注意事项
- `hasProfile` 的真实判定标准必须以项目现有注册流程为准，不能凭空猜测字段
- `共享 -> 个人` 时是否删除所有邀请，需要保持和现有邀请流程语义一致；本方案推荐删除待处理邀请
- 大量使用 `skip` 在超大数据量下性能一般，但当前目标是修复正确性，暂不做架构级优化
- 本次应尽量保持云函数返回结构不变，避免影响其他页面如 `profile/my-lists`

## 预期产出
- 清单列表、搜索、统计、删除、类型编辑行为与数据模型一致
- 修复已确认的高优先级数据错误与权限错误
- 前端空状态、搜索、调试流程更稳定
