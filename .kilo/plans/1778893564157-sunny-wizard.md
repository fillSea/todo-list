# 实时监听设计方案

## 背景与问题

当前协作数据的实时刷新实现不完整：

- `pages/list-detail/list-detail.js` 已监听 `list_versions`，监听到当前 `listId` 变化后调用 `loadListDetail(false)`。
- `pages/checklist/checklist.js`、`pages/profile/my-lists/my-lists.js` 等清单列表页面没有实时监听，只在 `onShow`、下拉刷新、筛选/搜索等场景调用 `getMyLists`。
- 首页 `pages/index/index.js` 与日历 `pages/calendar/calendar.js` 有任务缓存，协作任务在其他用户操作后不会自动失效，只能依赖页面重新进入、缓存过期或手动刷新。
- 成员、邀请、申请、操作记录等页面没有实时监听，只在页面显示或本机操作后刷新。
- 后端 `taskFunctions` 已通过 `touchListVersion(listId, userId, eventType)` 更新 `list_versions`，但 `listFunctions` 中清单更新、成员变更、邀请/申请状态变更主要只写 `operations`，没有统一更新 `list_versions`。

根因：后端协作事件没有统一版本化，前端也没有为清单列表及相关协作页面建立统一订阅入口。

## 目标

- 让清单列表页在协作者变更清单任务、清单信息、成员摘要等后自动刷新。
- 让清单详情、成员、邀请管理、操作记录等与单个清单强相关的页面能根据相关事件自动刷新。
- 让首页和日历在协作任务变化后失效任务缓存并自动刷新当前可见数据。
- 降低前端直接监听业务集合的复杂度，避免对 `tasks`、`lists`、`list_members`、`list_invites` 分别建立多个 watcher。
- 保持现有云函数拉取接口为数据来源，实时监听只作为“变化通知”。

## 总体设计

采用统一版本集合方案：

- 继续使用并扩展 `list_versions` 作为清单级协作事件总线。
- 后端所有会影响协作视图的操作，都调用统一的 `touchListVersion(listId, userId, eventType, meta)`。
- 前端页面只监听 `list_versions`，监听到变化后根据 `eventType` 决定清缓存与刷新哪些接口。
- 对用户级聚合视图，例如清单列表页、首页、日历，监听当前用户可访问清单对应的 `list_versions` 记录。
- 对单清单页面，例如详情、成员、邀请、操作记录，监听当前 `listId` 的 `list_versions` 记录。

## 后端设计

### 1. 在 `listFunctions` 中补齐版本更新能力

在 `cloudfunctions/listFunctions/index.js` 新增与 `taskFunctions` 类似的 helper：

- `touchListVersion(listId, userId, eventType, meta = {})`
- 内部查询 `list_versions.where({ listId }).limit(1)`。
- 已存在时 `update`：
  - `updatedAt: db.serverDate()`
  - `updatedBy: userId`
  - `eventType`
  - `meta`
  - `version: _.inc(1)`
- 不存在时 `add`：
  - `listId`
  - `updatedAt`
  - `updatedBy`
  - `eventType`
  - `meta`
  - `version: 1`

建议不在 `listFunctions` 的 `touchListVersion` 中重算任务统计，任务统计仍由 `taskFunctions.touchListVersion` 负责，避免清单/成员/邀请操作无意义重算任务。

### 2. 后端事件类型规范

事件类型分层：

- 任务事件：
  - `task_create`
  - `task_update`
  - `task_delete`
  - `task_delete_repeat`
  - `task_status_change`
  - `task_batch_update`
  - `task_batch_delete`
- 清单结构事件：
  - `list_create`
  - `list_update`
  - `list_delete`
- 成员事件：
  - `member_add`
  - `member_remove`
  - `member_update`
- 邀请/申请事件：
  - `invite_send`
  - `invite_cancel`
  - `invite_reject`
  - `invite_remind`
  - `invite_clear`
  - `join_apply`
  - `application_approve`
  - `application_reject`

前端可按事件类型粗分刷新范围：

- `task_*`：任务缓存、详情任务列表、清单统计、首页、日历。
- `list_*`：清单列表、详情清单信息、操作记录。
- `member_*`：成员列表、清单列表成员摘要、详情成员、权限状态。
- `invite_*` / `join_apply` / `application_*`：邀请页、邀请管理页、成员页待处理数量、操作记录。

### 3. `listFunctions` 中触发 `touchListVersion` 的位置

需要在以下成功变更后调用：

- `createList`：创建后触发 `list_create`。
- `updateList`：更新后触发 `list_update`。
- `deleteList`：删除前或删除后触发 `list_delete`。
  - 推荐删除前先更新版本，避免清单删除后前端仍能收到事件；前端收到后重新加载列表，详情页若刷新失败则提示清单不存在并返回。
- `inviteMember`：邀请创建成功后触发 `invite_send`。
- `removeMember`：成员移除成功后触发 `member_remove`。
- `updateMemberRole`：角色更新成功后触发 `member_update`。
- `acceptInvite` / `ensureMemberJoinedFromInvite` 成功新增成员后触发 `member_add`。
- `applyJoinList` 成功提交申请后触发 `join_apply`。
- `rejectInvite` 成功后触发 `invite_reject`。
- `remindInvite` 成功后触发 `invite_remind`。
- `cancelInvite` 成功后触发 `invite_cancel`。
- `approveApplication` 成功后触发：
  - `application_approve`
  - 若实际新增成员，也触发或合并为 `member_add`。
- `rejectApplication` 成功后触发 `application_reject`。
- `clearInvites` 成功后触发 `invite_clear`。
- `recalculateListMemberSummary` 不建议自己触发版本，避免调用者无法控制事件类型；由调用者在成功完成业务流程后触发。

### 4. `taskFunctions.touchListVersion` 小幅扩展

在 `cloudfunctions/taskFunctions/index.js` 中：

- 保留现有重算清单任务统计逻辑。
- 可增加 `meta` 参数，写入更多信息，例如 `taskId`、`affectedListId`、`oldListId`。
- 不改变现有调用语义，避免任务逻辑回归。

### 5. 权限与数据库索引

`list_versions` 至少需要支持：

- 单清单监听：`where({ listId })`。
- 多清单监听：`where({ listId: _.in(listIds) })`。

建议索引：

- `listId`。
- `updatedAt`，如果后续需要按更新时间查询。

安全规则需要允许清单成员读取对应 `list_versions`。如果当前云开发权限不足导致 `watch` 报错，前端需要降级为定时/页面显示刷新，后端仍保留版本表。

## 前端通用监听设计

### 1. 新增通用工具模块

建议新增 `miniprogram/utils/realtimeWatcher.js`，封装 watcher 生命周期与节流刷新。

核心能力：

- `createListVersionWatcher(options)`：
  - `listIds` 或 `getListIds`。
  - `onChange(events)`。
  - `onError(error)`。
  - `debounceMs` 默认 `300`。
- 自动忽略 `snapshot.type === 'init'` 的初始化事件。
- 支持 `start()`、`stop()`、`restart(nextListIds)`。
- 多清单监听时对 listIds 去重，空数组时不启动。
- 如果 `listIds.length > 20`，分片建立多个 watcher，避免 `_.in` 数量限制风险。
- `stop()` 关闭所有 watcher 并清理 timer。

伪代码：

```js
function createListVersionWatcher({ listIds, onChange, onError, debounceMs = 300 }) {
  let watchers = [];
  let ready = false;
  let timer = null;

  function start(nextListIds = listIds) {
    stop();
    const ids = Array.from(new Set((nextListIds || []).filter(Boolean)));
    if (!ids.length) return;

    const db = wx.cloud.database();
    const _ = db.command;
    ready = false;

    chunk(ids, 20).forEach(chunkIds => {
      const watcher = db.collection('list_versions')
        .where({ listId: _.in(chunkIds) })
        .watch({
          onChange(snapshot) {
            if (snapshot.type === 'init') {
              ready = true;
              return;
            }
            if (!ready) {
              ready = true;
              return;
            }
            if (timer) clearTimeout(timer);
            timer = setTimeout(() => {
              timer = null;
              onChange(snapshot);
            }, debounceMs);
          },
          onError(err) {
            stop();
            if (onError) onError(err);
          }
        });
      watchers.push(watcher);
    });
  }

  function stop() {
    watchers.forEach(w => w && w.close && w.close());
    watchers = [];
    ready = false;
    if (timer) clearTimeout(timer);
    timer = null;
  }

  return { start, stop, restart: start };
}
```

### 2. 缓存失效策略

在实时变更刷新前：

- 任务事件：调用 `app.clearTaskCaches()`。
- 清单列表事件：清单列表当前没有专门缓存，无需清理；重新调用 `loadLists(false)`。
- 成员摘要事件：重新加载清单列表或成员页。
- 分类事件本方案不涉及；如分类未来要实时化，应另设 `category_versions` 或用户级版本。

首页/日历必须强制刷新，避免命中缓存：

- 首页调用 `loadTasks({ forceRefresh: true })`。
- 日历调用 `loadTasks({ forceRefresh: true })`。

### 3. 生命周期规范

所有实时 watcher 遵循：

- `onShow`：登录/有权限后启动监听。
- `onHide`：停止监听。
- `onUnload`：停止监听。
- 用户退出登录或资料不完整：停止监听并清空页面数据。
- 监听错误：停止监听，打印错误；页面仍保留手动刷新能力。

## 页面改造方案

### 1. 清单首页 `pages/checklist/checklist.js`

目标：协作者任务、清单信息、成员变化后，列表上的任务数、待完成数、进度、成员头像、更新时间自动刷新。

实现：

- 引入通用 watcher。
- 在 `data` 或实例字段中维护 `this.listVersionWatcher`。
- `onShow` 登录并有 profile 后调用 `refreshCurrentView(false)`，刷新完成后基于当前 `lists` 提取 listIds 启动监听。
- `loadLists` 成功后调用 `restartChecklistWatcher(nextLists)`。
- 监听变化后：
  - `app.clearTaskCaches()`。
  - 如果处于远程搜索模式且有关键字，调用 `performRemoteSearch(keyword, false)`。
  - 否则将 `page` 重置为 `1`，调用 `loadLists(false)`。
- `onHide` / `onUnload` 停止 watcher。
- 处理分页：只监听当前已加载页的 listIds。若需要更完整，可新增 `listFunctions.getMyListVersionIds` 获取全部可访问清单 id，但第一阶段可先监听已展示清单。

注意：

- 避免每次 `loadLists` 都造成重复 watcher，必须先 stop/restart。
- 搜索模式下刷新远程结果，避免 UI 与当前搜索上下文不一致。
- 如果收到 `list_delete`，刷新列表即可；被删除清单会自然消失。

### 2. 个人中心清单列表 `pages/profile/my-lists/my-lists.js`

目标：从“我的清单/共享清单”入口进入时，也能看到协作变化。

实现：

- 复用通用 watcher。
- `loadLists` 成功后监听当前页 listIds。
- 监听变化后重置 `page: 1` 并调用 `loadLists()`。
- `onHide` / `onUnload` 停止。

### 3. 清单详情 `pages/list-detail/list-detail.js`

目标：保留现有能力，改为复用通用 watcher，并按事件类型清缓存。

实现：

- 用 `realtimeWatcher` 工具替换页面内重复的 `startRealtimeWatcher` / `stopRealtimeWatcher` 逻辑，或保留现有逻辑但与工具行为一致。
- 监听当前 `listId`。
- 变化后：
  - 对 `task_*` 调用 `app.clearTaskCaches()`。
  - 调用 `loadListDetail(false)`。
- 对 `list_delete` 事件：刷新失败后提示“清单已删除或无权限访问”，然后 `navigateBack()`。

### 4. 成员页面 `pages/list-members/list-members.js`

目标：成员被添加、移除、角色更新、待处理邀请数量变化时自动刷新。

实现：

- 监听当前 `listId` 的 `list_versions`。
- 触发刷新事件：`member_*`、`invite_*`、`join_apply`、`application_*`、`list_update`、`list_delete`。
- 变化后调用 `loadMembers()`，该函数内部已经会根据角色加载 `pendingInviteCount`。
- 若当前用户被移除或权限变化导致云函数返回无权限，显示提示并返回上一页。

### 5. 邀请创建页 `pages/list-invite/list-invite.js`

目标：待处理邀请、最近成员、成员上限、当前用户权限变化时自动刷新。

实现：

- 监听当前 `listId`。
- 触发刷新事件：`member_*`、`invite_*`、`join_apply`、`application_*`、`list_update`、`list_delete`。
- 变化后调用 `loadData()`，但注意不要覆盖用户正在编辑的弹窗草稿：
  - 若 `showRolePopup`、`showSearchPopup`、`showExpirePopup` 为 true，可只刷新 `pendingInvites` 与 `recentMembers`，不重置分享草稿。
  - 若无弹窗，完整 `loadData()`。

### 6. 邀请管理页 `pages/list-invite-manage/list-invite-manage.js`

目标：申请/邀请被其他人处理后自动刷新当前列表。

实现：

- 监听当前 `listId`。
- 触发刷新事件：`invite_*`、`join_apply`、`application_*`、`member_add`、`member_remove`。
- 变化后重置 `page` 到 `1`，调用 `loadInvites()`。
- 避免在用户确认弹窗打开时强制关闭弹窗；只刷新列表数据。

### 7. 操作记录页 `pages/operations/operations.js`

目标：有新操作日志时自动刷新。

实现：

- 监听当前 `listId`。
- 任意 `list_versions` 变化都可能对应操作日志，重置 `page: 1` 后调用 `loadOperations()`。
- 如果 `listFunctions.logOperation` 与 `touchListVersion` 分别失败，不影响主流程；日志页最终仍以 `operations` 集合为准。

### 8. 首页任务页 `pages/index/index.js`

目标：协作清单任务被其他成员创建、更新、完成、删除后，首页任务列表自动刷新。

实现选项：

- 方案 A：监听用户当前可访问清单的 `list_versions`。
  - 进入页面后调用 `listFunctions.getMyLists` 或新增轻量接口获取可访问 listIds。
  - 监听这些 listIds。
  - 收到 `task_*`、`list_delete`、`member_*` 事件后 `app.clearTaskCaches()` 并 `loadTasks({ forceRefresh: true })`。
- 方案 B：新增 `user_versions`，由后端在影响用户任务视图时更新用户级版本。
  - 更精确，但改动更大。

推荐第一阶段采用方案 A，复用 `list_versions`。

注意：

- 首页还包含个人独立任务。个人任务由当前用户自己操作时现有本地刷新已覆盖；若未来有多端同账号实时需求，需要增加用户级任务版本。
- 监听清单列表只覆盖共享清单协作事件，不覆盖个人独立任务的跨设备事件。

### 9. 日历页 `pages/calendar/calendar.js`

目标：协作清单任务变化后日历任务点和选中日期任务自动刷新。

实现：

- 与首页相同，监听可访问 listIds。
- 仅对 `task_*`、`list_delete`、`member_*` 事件响应。
- 收到变化后：
  - `app.clearTaskCaches()`。
  - 调用 `loadTasks({ forceRefresh: true })`。
- 保留当前选中日期与月份，不重置用户正在看的日期。

### 10. 通知相关页面

现有通知页与 profile 页只是主动拉取未读数。

可选增强：

- 监听 `notifications.where({ userId, isRead: false })`，用于未读角标实时更新。
- 或新增 `user_versions` 中的 `notification_*` 事件。

本次清单协作实时刷新可以不纳入通知实时化，避免范围过大。

## 推荐实施顺序

1. 新增前端通用 watcher 工具 `miniprogram/utils/realtimeWatcher.js`。
2. 扩展 `listFunctions`，新增 `touchListVersion`，并在清单/成员/邀请/申请操作后调用。
3. 小幅扩展 `taskFunctions.touchListVersion` 支持 `meta`，不改变现有行为。
4. 改造 `list-detail` 复用/对齐 watcher 逻辑，确保现有详情实时能力不退化。
5. 改造 `checklist`，实现清单列表实时刷新。
6. 改造 `list-members`、`list-invite`、`list-invite-manage`、`operations`。
7. 改造 `profile/my-lists`。
8. 改造 `index` 与 `calendar`，监听共享清单任务变化并强制刷新任务缓存。
9. 在微信开发者工具中验证数据库 watch 权限、监听稳定性、页面隐藏后 watcher 是否关闭。

## 验证方案

### 手工验证场景

- 两个账号 A/B 同时进入同一个共享清单详情页，A 新增任务，B 自动看到任务。
- A 在详情页完成任务，B 的详情页任务状态、清单列表页待完成数和进度自动变化。
- A 修改清单名称/描述，B 的清单详情和清单列表自动变化。
- A 邀请 C，B 在邀请管理页或成员页待处理邀请数量自动变化。
- C 通过邀请加入，A/B 的成员页、详情页成员头像、清单列表成员摘要自动变化。
- A 移除 B，B 当前相关页面提示无权限或返回，A 的成员页自动刷新。
- A 删除清单，B 的清单列表自动移除；B 若在详情页则提示清单不存在并返回。
- A 创建/完成共享任务，B 的首页和日历在当前页面可见时自动刷新。

### 回归点

- 页面切到后台或其他页面后 watcher 已关闭，不持续刷新。
- 远程搜索模式下清单页仍刷新搜索结果，而不是强制退出搜索。
- 首页/日历实时刷新不会命中旧缓存。
- 分页列表刷新后页码、`hasMore`、加载状态正确。
- `DEBUG_MODE` 为 true 时不启动 watch。
- watch 报错后不影响手动刷新与页面基本使用。

## 风险与取舍

- 多清单监听需要 `_.in`，当用户清单很多时需要分片 watcher；第一阶段监听当前已加载列表可降低连接数，但未加载页的变化不会触发刷新。
- `list_versions` 是变化通知，不是数据源；如果 `touchListVersion` 调用失败，页面不会实时刷新，但下次手动刷新仍能看到真实数据。
- 直接监听业务集合数据更细，但会增加权限规则、连接数、数据处理复杂度；当前更推荐统一版本表。
- 首页/日历只监听共享清单协作事件，不解决个人任务多端实时同步；若有这个需求，应新增用户级 `user_versions`。

## 第一阶段验收标准

- 清单页 `checklist` 能在协作者修改共享清单任务后自动刷新统计数据。
- 清单详情、成员页、邀请管理页能在其他协作者操作后自动刷新。
