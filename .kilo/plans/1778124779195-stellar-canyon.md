# 个人任务界面标题简化方案

## 目标
解决个人任务相关页面在跳转后出现“顶部系统标题 + 页面内标题”重复显示的问题，统一导航层级，减少视觉噪音，提升页面信息密度与一致性。

## 现状结论
当前代码里存在两类标题来源同时生效：

1. 小程序原生导航栏标题
- 由页面 `.json` 中的 `navigationBarTitleText` 提供。
- 例如：`miniprogram/pages/periodic-stats/periodic-stats.json`

2. 页面内部自绘标题栏
- 由 `van-nav-bar` 或自定义 `header` 提供。
- 例如：
  - `miniprogram/pages/periodic-stats/periodic-stats.wxml`
  - `miniprogram/pages/task-edit/task-edit.wxml`
  - `miniprogram/pages/list-detail/list-detail.wxml`

因此用户在进入“周期任务统计”等页面时，会同时看到：
- 顶部原生导航栏标题
- 页面内容区上方再次出现的标题栏

这会造成信息重复，并压缩首屏内容空间。

## 设计原则
1. 一个页面只保留一层主标题。
2. 返回、保存、更多操作等导航动作应与主标题处于同一导航层。
3. 标题栏样式在“只读详情页”和“编辑页”之间保持一致，但不强制一次性重构所有页面视觉。
4. 优先用最小改动修复当前问题，避免影响页面业务逻辑。

## 方案对比

### 方案 A：保留原生导航栏，删除页面内标题栏
适用场景：纯展示页、导航动作简单、无需复杂右侧操作布局。

优点：
- 改动最小。
- 与小程序平台行为一致。
- 首屏空间更大。

缺点：
- 对有复杂操作按钮的页面不够灵活。
- 原生导航栏在样式统一性上受限。

### 方案 B：隐藏原生导航栏，保留页面内标题栏
适用场景：需要返回、保存、更多菜单、自定义标题、副标题等复杂交互页。

优点：
- 可完全控制标题栏布局与样式。
- 适合本项目已经广泛存在的 `header` / `van-nav-bar` 模式。
- 可统一不同业务页面的交互位置。

缺点：
- 需要每页处理安全区、顶部间距、沉浸式适配。
- 如果个别页面实现不一致，容易继续出现样式漂移。

## 推荐方案
推荐采用“按页面类型分层治理”的混合策略，但以“单页只保留一层标题”为硬规则：

1. `periodic-stats`：保留页面内 `van-nav-bar`，隐藏原生导航栏。
2. `task-edit`：保留页面内编辑头部，隐藏原生导航栏。
3. `list-detail`：保留页面内自定义头部，隐藏原生导航栏。
4. 其余没有页面内标题栏的页面，继续使用原生导航栏，不做改动。

推荐这个方向的原因：
- 这几个页面已经依赖自定义头部承载返回、保存、编辑、更多等操作。
- 如果反过来删除页面内头部，会牵涉按钮重排，改动面更大。
- 可以先解决重复显示问题，再视需要做导航组件统一。

## 页面级设计建议

### 1. 周期任务统计页 `pages/periodic-stats`
当前：
- 原生标题：`周期任务统计`
- 页面内 `van-nav-bar`：`周期任务统计`
- 卡片内首屏又展示任务名称 `taskInfo.title`

建议：
- 隐藏原生导航栏。
- 保留 `van-nav-bar`，但简化文案层级：
  - 顶部栏标题保留为通用页面名：`周期统计`
  - 首卡主标题保留具体任务名：`taskInfo.title`
- 这样用户先知道“这是统计页”，再看到“是哪条任务的统计”。

额外建议：
- `left-text="返回"` 可视情况去掉，仅保留箭头，减少占位。
- 页面首卡与导航栏之间的留白控制在轻量级，避免再次形成“双抬头”。

### 2. 任务编辑页 `pages/task-edit`
当前：
- 原生标题：`编辑任务`
- 页面内标题：`编辑任务 / 添加任务`
- 页面内右侧还有 `保存`

建议：
- 隐藏原生导航栏。
- 保留页面内头部，因为其承担：关闭、标题、保存。
- 将头部作为唯一导航层。

设计收益：
- 编辑页符合表单编辑心智，顶部操作集中。
- 解决“编辑任务”重复显示问题。

### 3. 清单详情页 `pages/list-detail`
当前：
- 原生标题：`清单详情`
- 页面内标题：`listInfo.name || 清单详情`
- 页面内头部还承担编辑、更多操作。

建议：
- 隐藏原生导航栏。
- 保留自定义头部。
- 当 `listInfo.name` 存在时，只展示清单名，不再在页面内容区重复写一遍主标题语义。

## 实施范围
本次实施建议先控制在“已确认存在重复标题的页面”，避免一次性扩大到所有页面：

1. `miniprogram/pages/periodic-stats/periodic-stats.json`
2. `miniprogram/pages/task-edit/task-edit.json`
3. `miniprogram/pages/list-detail/list-detail.json`

可能涉及的配套检查文件：
1. `miniprogram/pages/periodic-stats/periodic-stats.wxml`
2. `miniprogram/pages/periodic-stats/periodic-stats.wxss`
3. `miniprogram/pages/task-edit/task-edit.wxml`
4. `miniprogram/pages/list-detail/list-detail.wxml`

## 实现要点
核心实现方式：在上述页面的 `.json` 中增加或切换为自定义导航配置，避免原生导航栏继续显示。

执行时重点检查：
1. 页面顶部安全区是否正确。
2. 自定义头部是否与状态栏重叠。
3. 返回按钮点击区域是否足够。
4. 右侧按钮在 iPhone 小屏下是否挤压标题。
5. 从 `index` / `calendar` 跳转到 `periodic-stats` 后的视觉层级是否更简洁。

## 验收标准
1. 进入“周期任务统计”页面时，顶部只出现一处标题栏。
2. 页面仍保留返回能力，且不影响统计内容展示。
3. 进入“编辑任务”页面时，不再出现双重“编辑任务”标题。
4. 进入“清单详情”页面时，不再出现系统标题与页面头部并存。
5. 页面首屏可视内容增加，没有明显布局跳动或顶到状态栏的问题。

## 后续可选优化
如果后续要继续统一体验，可以做第二阶段整理：

1. 统一自定义头部的视觉规范
- 标题字号
- 左右按钮间距
- 是否显示文字型返回
- 安全区高度处理

2. 抽离公共导航组件
- 当前多个页面分别写了 `header` / `van-nav-bar`
- 后续可沉淀为统一的页面头部组件，但不建议在本次修复中一并进行，避免超出最小修复范围

## 推荐执行顺序
1. 先修复 `periodic-stats`，验证重复标题问题完全消失。
2. 再同步处理 `task-edit` 与 `list-detail`。
3. 在微信开发者工具中验证从“任务页 / 日历页 / 我的页”进入这些页面的显示效果。
