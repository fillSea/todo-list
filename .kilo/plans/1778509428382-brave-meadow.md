# 为编辑者增加需创建者审核的邀请权限设计方案

## 目标

在保留当前创建者邀请机制的基础上，为编辑者增加邀请成员能力，但编辑者发起的所有邀请都必须进入创建者审核流程，不能直接加入。查看者仍不可邀请。

最终权限规则：

| 邀请者角色 | 可邀请 | 是否可直接加入 | 是否可选择审核 | 审核人 |
|---|---:|---:|---:|---|
| 创建者 `role=1` | 是 | 是 | 是 | 创建者 |
| 编辑者 `role=2` | 是 | 否 | 否，强制审核 | 创建者 |
| 查看者 `role=3` | 否 | 否 | 否 | 无 |

## 现状摘要

核心后端在 `cloudfunctions/listFunctions/index.js`：

- `verifyListPermission(userId, listId, [1])` 当前限制邀请相关接口仅创建者可用。
- `generateInviteLink` 创建公开模板链接，支持 `needApproval`。
- `createWechatInvite` 创建公开微信邀请模板，目前不支持 `needApproval` 字段。
- `inviteMember` 创建指定用户邀请，目前固定 `status: 0`，不支持编辑者强制审核。
- `acceptInvite` 在 `invite.needApproval` 为真时会转入 `applyJoinList`。
- `applyJoinList` 当前把审核通知发给 `invite.inviterId`。
- `approveApplication` / `rejectApplication` 已强制只有清单创建者可审批。
- `getInviteList` 仅创建者可查看，且审核页读取 `status=4`。

核心前端：

- `miniprogram/pages/list-invite/list-invite.js` 在 `loadListInfo` 中非创建者直接返回并提示“仅创建者可管理邀请”。
- `list-invite` 的链接高级设置允许选择是否需要审核。
- `list-detail` 和 `list-members` 的成员/邀请入口基本只暴露给创建者。
- `list-invite-manage` 已有“申请审核”页签，适合继续承载创建者审核。
- 通知页 `profile/notifications/notifications.js` 的 `join_request` 会跳转到邀请管理页。

## 设计原则

1. 后端强制权限，不依赖前端隐藏选项。
2. 最小改造现有 `needApproval + status=4` 审核流，避免新增独立申请集合。
3. 创建者现有体验保持不变。
4. 编辑者只获得“发起邀请/推荐加入”能力，不获得成员管理、审核、清空、改角色、移除成员能力。
5. 审核列表和通知中必须让创建者知道申请来源，尤其是“通过编辑者某某的邀请申请加入”。

## 后端设计

### 1. 增加邀请权限判定

在 `listFunctions/index.js` 增加或复用权限判断：

- 创建邀请类接口允许 `[1, 2]`。
- 审核、清空、成员管理仍只允许 `[1]` 或 `creatorId === userId`。
- 搜索用户、最近协作者用于邀请，也允许 `[1, 2]`。

建议受影响接口：

- `inviteMember`: `[1, 2]`
- `generateInviteLink`: `[1, 2]`
- `createWechatInvite`: `[1, 2]`
- `searchUser`: `[1, 2]`
- `getRecentCollaborators`: 当前只要求可访问，可保持不变
- `getInviteList`: 保持 `[1]`，避免编辑者查看全量邀请/审核信息
- `cancelInvite` / `remindInvite`: 需要按邀请人权限处理，详见下文

### 2. 强制编辑者邀请必须审核

对所有创建邀请记录的入口统一计算：

```js
const inviterRole = permission.role;
const forceApproval = inviterRole === 2;
const finalNeedApproval = forceApproval ? true : !!needApproval;
```

创建记录时保存：

```js
inviterRole,
needApproval: finalNeedApproval,
approvalTargetId: permission.list.creatorId,
approvalSource: inviterRole === 2 ? 'editor_invite' : undefined
```

字段说明：

- `inviterRole`: 邀请创建时邀请人的角色，用于展示和审计。
- `approvalTargetId`: 应审批人，当前为清单创建者。
- `approvalSource`: 审核来源，建议值包括：
  - `direct_invite`: 创建者指定用户邀请且需审核，兼容现有逻辑。
  - `public_link`: 创建者公开链接申请，兼容现有逻辑。
  - `editor_invite`: 编辑者发起的邀请。
  - `editor_link`: 如需要细分编辑者公开链接。
  - `editor_wechat`: 如需要细分编辑者微信分享。

最小实现可只新增 `inviterRole` 和统一使用 `approvalSource: 'editor_invite'`，前端根据 `inviteType` 再展示“微信/链接/搜索”。

### 3. `inviteMember` 指定用户邀请

当前流程是生成 `status: 0` 指定用户邀请，用户接受后直接加入。

调整后：

- 创建者：保持 `status: 0`，`needApproval` 默认 false。
- 编辑者：生成 `status: 0`，但 `needApproval: true`。
- 被邀请人点击接受时，`acceptInvite` 发现 `needApproval` 为 true，调用 `applyJoinList`，流转为 `status: 4`，等待创建者审批。

这能复用现有“用户确认后才进入审核”的体验，而不是编辑者一邀请就把申请塞给创建者。

### 4. `generateInviteLink` 公开链接邀请

当前公开模板邀请会在用户处理时复制为用户级记录。

调整后：

- 权限从 `[1]` 改为 `[1, 2]`。
- 创建者传入的 `needApproval` 继续生效。
- 编辑者传入的 `needApproval` 被后端强制改为 `true`。
- 公开模板记录中保存 `inviterRole` 和 `needApproval: true`。
- 被邀请人打开并接受时，现有 `acceptInvite -> applyJoinList -> createInviteRecordFromTemplate(... status=4 ...)` 流程继续生效。

注意现有 `expireActiveLinkTemplates(listId)` 会把同清单所有有效公开 link 模板都置过期。若编辑者也可生成链接，这会导致编辑者重生成链接时使创建者的公开链接失效。建议调整为只过期“同一邀请人”的有效 link 模板，或按产品口径保留“每个清单仅一个公开链接”。推荐改为同一邀请人维度：

```js
expireActiveLinkTemplates(listId, inviterId)
```

这样创建者和多个编辑者的链接互不覆盖。

### 5. `createWechatInvite` 微信分享邀请

当前微信邀请没有 `needApproval` 字段。

调整后：

- 权限从 `[1]` 改为 `[1, 2]`。
- 创建者生成的微信邀请保持无需审核。
- 编辑者生成的微信邀请保存 `needApproval: true`。
- 用户接受编辑者微信邀请时走申请审核。

### 6. `applyJoinList` 通知应发给创建者

当前申请通知发送给 `invite.inviterId`，这在编辑者邀请时不符合目标机制。

调整后通知对象：

```js
const approvalTargetId = invite.approvalTargetId || list.creatorId;
await createNotificationIfEnabled(approvalTargetId, ...)
```

文案建议：

- 创建者自己邀请需审核：`张三确认加入清单"家庭待办"，等待您审核`
- 创建者公开链接需审核：`张三申请加入清单"家庭待办"`
- 编辑者邀请：`张三通过编辑者李四的邀请申请加入清单"家庭待办"`

可选：同时通知编辑者“你邀请的成员已提交申请，等待创建者审核”，但不是首期必须。

### 7. 审核时重新校验编辑者权限

在 `approveApplication` 中，加入前增加检查：

- 申请记录仍为 `status: 4`。
- 申请人尚未是成员，已有则只收敛状态。
- 审核人仍是当前清单创建者。
- 若 `inviterRole === 2` 或 `approvalSource` 为编辑者邀请：检查 `invite.inviterId` 当前仍是该清单编辑者或创建者。

推荐安全口径：编辑者若已被移出清单或降为查看者，相关待审核申请不可通过，提示“邀请人已无邀请权限”。

拒绝申请可以不强校验邀请人当前角色，因为拒绝不会新增成员。

### 8. 取消与提醒权限

当前：

- `remindInvite` 只要求 `invite.inviterId === userId`。
- `cancelInvite` 先要求创建者权限，再要求邀请人本人。

调整建议：

- `remindInvite`: 邀请人本人可提醒，但若是编辑者邀请，提醒只针对被邀请人接受/申请，不涉及审批。
- `cancelInvite`: 创建者可取消任意待处理邀请；邀请人本人可取消自己发起且仍 `status=0` 的邀请。
- 对 `status=4` 的申请，创建者通过拒绝处理，不建议让编辑者取消申请。

### 9. 邀请列表补充邀请人信息

`getInviteList` 目前只补充 `inviteeInfo`。为了审核展示来源，增加邀请人信息：

```js
inviterInfo: {
  nickname,
  avatarUrl
}
```

同时返回：

- `inviterId`
- `inviterRole`
- `approvalSource`
- `inviteType`
- `needApproval`

## 前端设计

### 1. `list-invite` 页面允许编辑者进入

`miniprogram/pages/list-invite/list-invite.js`：

- `loadListInfo` 中 `detail.myRole !== 1` 改为只拒绝非创建者/编辑者。
- 保存 `myRole` 到页面 data。
- 文案从“仅创建者可管理邀请”调整为“仅创建者和编辑者可邀请成员”。

新增数据：

```js
myRole: null,
canChooseApproval: false,
forceApproval: false
```

规则：

- `myRole === 1`: `canChooseApproval=true`, `forceApproval=false`
- `myRole === 2`: `canChooseApproval=false`, `forceApproval=true`, `linkDraftNeedApproval=true`

### 2. 编辑者隐藏或禁用“无需审核”设置

`list-invite.wxml` 高级设置中：

- 创建者看到开关，保持现状。
- 编辑者看到固定提示：`编辑者邀请需由创建者审核，通过后成员才能加入`。
- 分享面板标签对编辑者始终展示“需要创建者审核”。

`generateLinkInviteAndPrepare` 和 `createInviteAndPrepare` 前端传参也设置 `needApproval: true`，但以后端为准。

### 3. 入口开放给编辑者

`list-detail.js`：

- 保持 `canManageMembers: myRole === 1`，不扩大成员管理能力。
- 新增 `canInviteMembers: myRole === 1 || myRole === 2`。
- 更多操作中：
  - 创建者显示“管理成员”。
  - 编辑者可显示“邀请成员”，直接跳转 `list-invite`。

`list-members.js/wxml`：

- 如果编辑者能进入成员页，则底部“添加成员”可按 `canInviteMembers` 显示为“邀请成员”。
- “邀请管理”入口仍只给创建者，因为它包含审核、清空、全量邀请记录。
- 成员角色修改、移除仍只给创建者。

最小方案可以只改 `list-detail` 的更多操作，让编辑者直接进 `list-invite`，不扩大 `list-members` 的可见入口。

### 4. 审核列表展示邀请来源

`list-invite-manage.js` 的 `formatApplications` 增强：

- `approvalSource === 'editor_invite'` 或 `inviterRole === 2`：`sourceLabel = '编辑者邀请'`
- `approvalSource === 'direct_invite'`：`sourceLabel = '受邀后待审核'`
- `approvalSource === 'public_link'`：`sourceLabel = '主动申请加入'`

`list-invite-manage.wxml` 增加一行来源：

```text
通过编辑者 李四 的链接邀请申请加入
```

或压缩展示：

```text
来源：李四（编辑者） · 链接邀请
```

### 5. 邀请接受页文案

`list-invite-accept.js`：

- `onApplyJoin` 当前文案是“等待邀请人审核”，编辑者邀请时不准确。
- 根据后端返回 `inviteInfo.approvalTargetName` 或 `inviteInfo.inviterRole` 调整：
  - 创建者邀请：`等待创建者审核`
  - 编辑者邀请：`等待清单创建者审核`

`getInviteInfo` 后端可返回：

```js
inviterRole,
approvalTargetId: list.creatorId,
approvalTargetName: creator.nickname
```

最小前端文案也可统一改为“等待创建者审核”。

## 数据兼容

无需迁移已有数据。

兼容策略：

- 旧邀请缺少 `inviterRole` 时，默认按创建者邀请处理。
- 旧邀请缺少 `approvalTargetId` 时，审批目标回退到 `list.creatorId`。
- 旧邀请缺少 `approvalSource` 时，按现有逻辑显示。
- `needApproval` 缺失时按 `false` 处理。

## 操作日志

建议扩展现有日志内容，方便后续排查：

- `invite_send`: 增加 `inviterRole`, `needApproval`, `approvalTargetId`。
- `join_apply`: 增加 `inviterId`, `inviterRole`, `approvalSource`。
- `application_approve` / `application_reject`: 增加 `inviterId`, `inviterRole`, `approvalSource`。

不需要新增日志类型。

## 验证清单

### 后端验证

1. 创建者生成无需审核链接，用户接受后直接加入。
2. 创建者生成需审核链接，用户接受后进入 `status=4`，创建者可审批。
3. 编辑者生成链接，即使前端传 `needApproval:false`，数据库仍为 `needApproval:true`。
4. 编辑者微信邀请，用户接受后进入 `status=4`，不会直接加入。
5. 编辑者搜索/最近协作者邀请指定用户，用户确认后进入 `status=4`。
6. 查看者调用任一邀请创建接口失败。
7. 编辑者不能调用 `getInviteList` 查看审核列表。
8. 编辑者不能调用 `approveApplication` / `rejectApplication`。
9. 创建者可审批编辑者邀请产生的申请。
10. 编辑者被降级/移除后，其待审批申请不可通过或按设计明确处理。
11. 创建者和编辑者各自生成链接不会互相过期，若采用“按邀请人过期模板”方案。

### 前端验证

1. 创建者进入邀请页体验不变，可选择是否审核。
2. 编辑者可从清单详情进入邀请页。
3. 编辑者邀请页不显示“无需审核”选择，或显示固定说明。
4. 编辑者分享面板展示“需要创建者审核”。
5. 被邀请人打开编辑者邀请，按钮/确认文案为申请加入，等待创建者审核。
6. 创建者审核页能看到申请人、邀请人、邀请人角色、邀请方式。
7. 通知点击 `join_request` 可进入审核页。
8. 查看者无法看到邀请入口，直接访问邀请页也被拦截。

## 实施顺序

1. 后端先改权限和强制审核：`inviteMember`、`generateInviteLink`、`createWechatInvite`、`searchUser`。
2. 后端改 `applyJoinList` 通知目标为创建者，并补充邀请人/审批信息返回。
3. 后端改 `getInviteList` 返回 `inviterInfo`，审核通过时校验编辑者当前权限。
4. 前端改 `list-invite` 允许编辑者进入并强制审核 UI。
5. 前端改 `list-detail` 增加编辑者邀请入口。
6. 前端改 `list-invite-manage` 展示邀请来源。
7. 前端改 `list-invite-accept` 审核文案。
8. 按验证清单在微信开发者工具中手动验证云函数路径。

## 风险与取舍

- 若保留“每个清单仅一个公开链接”，编辑者生成链接会覆盖创建者链接，体验风险较高；推荐改为“每个邀请人一个有效链接”。
- 如果允许编辑者搜索用户，会扩大用户昵称搜索能力；但编辑者本身已有协作权限，风险可接受。后端仍需限制为当前清单编辑者。
- 如果编辑者邀请后被移除，推荐审批时禁止通过，安全性高于便利性。
- 邀请管理页仍只给创建者，避免编辑者看到其他成员邀请记录或审核申请。
