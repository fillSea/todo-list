# 设计方案：邀请接受/拒绝后通知邀请者

## 目标

当被邀请者处理清单邀请后，邀请者能在通知中心收到处理结果：

- 被邀请者直接接受邀请并加入清单后，通知邀请者。
- 被邀请者拒绝邀请后，通知邀请者。
- 需要审批的邀请在被邀请者点击接受时现有逻辑会转为加入申请，并已通过 `join_request` 通知邀请者，本方案不额外重复发送“已接受”通知。

## 当前行为

后端入口位于 `cloudfunctions/listFunctions/index.js`：

- `acceptInvite`：
  - `invite.needApproval === true` 时调用 `applyJoinList`，已有 `createNotificationIfEnabled(invite.inviterId, { type: 'join_request', ... })`。
  - `invite.needApproval === false` 时通过 `ensureMemberJoinedFromInvite` 加入清单，只写 `member_add` 操作日志，不通知邀请者。
- `rejectInvite`：
  - 更新 `list_invites.status = 2`。
  - 写 `invite_reject` 操作日志。
  - 不通知邀请者。
- 通知展示位于 `miniprogram/pages/profile/notifications/notifications.js`，目前类型支持 `list_invite`、`invite_remind`、`join_request`、`application_approved`、`application_rejected`、`task_reminder`。

## 通知类型设计

新增两个通知类型，避免复用含义不准确的现有类型：

- `invite_accepted`
  - 接收人：邀请者 `invite.inviterId`
  - 触发场景：无需审批邀请被接受并完成加入流程后
  - `relatedId`：`invite.listId`
  - 内容建议：`{userName} 已接受邀请并加入清单"{listName}"`
  - 点击行为：跳转清单详情 `/pages/list-detail/list-detail?id=${relatedId}`

- `invite_rejected`
  - 接收人：邀请者 `invite.inviterId`
  - 触发场景：邀请被拒绝并更新邀请记录为 `status: 2` 后
  - `relatedId`：`invite.listId`
  - 内容建议：`{userName} 已拒绝加入清单"{listName}"的邀请`
  - 点击行为：跳转邀请管理页 `/pages/list-invite-manage/list-invite-manage?listId=${relatedId}`，便于邀请者查看/清理邀请记录

## 后端实现方案

文件：`cloudfunctions/listFunctions/index.js`

1. 增加一个小工具函数，集中创建邀请处理结果通知。

建议函数名：`notifyInviterInviteHandled(invite, inviteeUserId, type)`。

职责：

- 校验 `invite.inviterId` 存在。
- 若 `invite.inviterId === inviteeUserId`，不发送，避免自邀请或异常数据产生自通知。
- 查询清单名称：`lists` collection by `invite.listId`。
- 查询被邀请者昵称：优先用 `getUserBasicInfo(inviteeUserId)`，兜底 `invite.inviteeInfo?.nickname`，再兜底 `未知用户`。
- 根据 `type` 构造通知：
  - `accepted` -> `invite_accepted`
  - `rejected` -> `invite_rejected`
- 调用已有 `createNotificationIfEnabled(invite.inviterId, notificationData)`，遵守邀请者的通知开关。
- 捕获并记录错误，不阻断接受/拒绝主流程。通知失败不应导致用户处理邀请失败。

2. 在 `acceptInvite` 的直接加入成功路径发送通知。

位置：`ensureMemberJoinedFromInvite(...)` 返回之后，`return` 之前。

建议条件：

- 仅当 `!invite.needApproval` 路径。
- 仅当 `joinResult.memberCreated === true` 时发送 `invite_accepted`。
- `joinResult.alreadyMember === true` 时不发送，避免用户重复处理历史邀请时造成误导。

说明：

- 对需要审批的邀请，`acceptInvite` 会走 `applyJoinList` 并发送 `join_request`，不发送 `invite_accepted`。
- 对公开链接邀请，`createInviteRecordFromTemplate` 会保留 `inviterId`，因此也能通知创建该链接的邀请者。

3. 在 `rejectInvite` 的拒绝成功路径发送通知。

位置：更新 `list_invites.status = 2` 后，返回成功前。

建议条件：

- 拒绝操作只会在可处理邀请上执行，成功更新后发送 `invite_rejected`。
- 发送通知放在 `logOperation('invite_reject', ...)` 附近。
- 可复用已查询的 `targetUserInfo`，或由工具函数查询；为了最小改动，可让工具函数独立查询并处理兜底。

4. 保持现有操作日志不变。

- `invite_reject` 操作日志继续用于清单操作记录。
- `member_add` 操作日志继续用于成员加入记录。
- 新通知只补充通知中心行为，不替换操作日志。

## 前端实现方案

文件：`miniprogram/pages/profile/notifications/notifications.js`

1. 扩展 `NOTIFICATION_TYPE_CONFIG`。

新增：

- `invite_accepted`: 文案 `邀请已接受`，图标建议 `success`
- `invite_rejected`: 文案 `邀请被拒绝`，图标建议 `close`

2. 扩展 `typeOptions` 筛选项。

新增：

- `{ value: 'invite_accepted', label: '邀请接受' }`
- `{ value: 'invite_rejected', label: '邀请拒绝' }`

3. 扩展通知点击行为。

- `invite_accepted`：跳转清单详情页，使用 `relatedId` 作为 `listId`。
- `invite_rejected`：跳转邀请管理页，使用 `relatedId` 作为 `listId`。

## 数据结构影响

`notifications` 集合无需结构变更，继续使用现有字段：

- `userId`
- `type`
- `relatedId`
- `content`
- `isRead`
- `createdAt`

仅新增 `type` 枚举值：

- `invite_accepted`
- `invite_rejected`

## 边界与去重

- 邀请者关闭通知：`createNotificationIfEnabled` 会自动跳过。
- 需要审批邀请：接受动作视为“提交/确认申请”，已有 `join_request`，不额外发 `invite_accepted`。
- 已是成员：不发 `invite_accepted`，避免“已接受并加入”与实际无新增成员不一致。
- 已处理邀请：现有逻辑提前返回错误，不会重复发送通知。
- 通知发送失败：不影响邀请状态更新和成员加入，避免主流程被通知系统耦合。

## 验证方案

由于该项目主要通过微信开发者工具验证，建议用云函数联调验证以下场景：

1. 直接邀请用户，用户接受：
   - `list_invites.status` 变为 `1`。
   - `list_members` 新增成员。
   - 邀请者 `notifications` 新增 `type = invite_accepted`。
   - 邀请者通知中心显示“邀请已接受”，点击进入清单详情。

2. 直接邀请用户，用户拒绝：
   - `list_invites.status` 变为 `2`。
   - 邀请者 `notifications` 新增 `type = invite_rejected`。
   - 邀请者通知中心显示“邀请被拒绝”，点击进入邀请管理页。

3. 需要审批邀请，用户点击接受：
   - `list_invites.status` 变为 `4`。
   - 邀请者只收到现有 `join_request`，不产生 `invite_accepted`。

4. 邀请者关闭通知后重复验证：
   - 主流程成功。
   - 不新增通知记录。

## 实施顺序

1. 修改 `cloudfunctions/listFunctions/index.js`，添加通知辅助函数。
2. 在 `acceptInvite` 直接加入成功路径调用该辅助函数。
3. 在 `rejectInvite` 成功更新状态后调用该辅助函数。
4. 修改 `miniprogram/pages/profile/notifications/notifications.js`，补充类型配置、筛选项和点击跳转。
5. 在微信开发者工具中上传/部署 `listFunctions` 云函数并联调以上场景。
