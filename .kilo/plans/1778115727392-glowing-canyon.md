# 任务计划：补充 AGENTS.md 与项目设计文档

## 目标

基于当前仓库的可验证事实，完成两项文档产出：
1. 在仓库根目录创建 `AGENTS.md`，只保留未来代理最容易踩坑、且必须从仓库中归纳出的高信号说明。
2. 在 `doc/` 下补充本项目设计文档，回应用户“阅读本项目，并给出本项目的设计文档”的要求。

## 已确认的仓库事实

### 项目形态
- 这是一个微信小程序 + 云开发项目，不是常规 Node Web 应用。
- 微信开发者工具配置位于 `project.config.json`，其中：
  - `miniprogramRoot` 为 `miniprogram/`
  - `cloudfunctionRoot` 为 `cloudfunctions/`
  - `packNpmManually: true`，且 `package.json` 的 npm 产物会被打包到 `miniprogram/`
- 根 `package.json` 只有占位 `test` 脚本，没有可用的 lint / test / build 自动化命令。
- 根依赖只有 `@vant/weapp`，其组件通过 `miniprogram/app.json` 的 `usingComponents` 全局注册。

### 运行与验证方式
- 主要开发入口不是命令行，而是微信开发者工具。
- 小程序云环境在 `miniprogram/app.js` 中硬编码：`cloud1-0g144inb6530ffb6`。
- 多数验证应通过微信开发者工具完成，而不是尝试运行仓库根的 `npm test`。
- 由于 `project.config.json` 启用了手动 npm 打包，依赖变更后需要在微信开发者工具中执行“构建 npm”，否则 `miniprogram/miniprogram_npm/` 不会同步。

### 前端结构
- 应用入口：`miniprogram/app.js`
- 页面注册中心：`miniprogram/app.json`
- 4 个 tab 页分别是：
  - `pages/index/index` 任务首页
  - `pages/calendar/calendar` 日历页
  - `pages/checklist/checklist` 清单页
  - `pages/profile/profile` 我的页
- 任务、分类、用户信息存在明显缓存策略：
  - `cachedTasks`
  - `cachedTasksTime`
  - `cachedCategories`
  - 日历页按月缓存 `calendarTasks_${year}_${month}`
- 登录状态依赖本地存储：`userInfo`、`isLoggedIn`、`loginTime`，并在 `app.js` 里做 30 天过期判断。

### 云函数边界
- `cloudfunctions/taskFunctions/index.js`
  - 负责任务 CRUD、筛选、搜索、批量操作、周期任务统计、提醒与定时清理。
  - 支持 timer trigger，且定时触发不要求 openid。
- `cloudfunctions/listFunctions/index.js`
  - 负责清单 CRUD、成员管理、邀请/申请/审批、协作者与操作日志等。
- `cloudfunctions/profileFunctions/index.js`
  - 负责用户注册/更新、个人统计、看板、通知、通知设置。
  - 暴露测试数据入口：`insertTestData`、`clearTestData`、`insertNotificationTestData`，且这些 action 不要求 openid。
- `cloudfunctions/categoryFunctions/index.js`
  - 负责分类 CRUD。
- `cloudfunctions/quickstartFunctions/`
  - 仍保留模板遗留云函数，不属于当前核心业务。

### 主要数据库集合
从云函数代码可验证的核心集合：
- `users`
- `tasks`
- `lists`
- `list_members`
- `categories`
- `notifications`
- `operations`
- 邀请相关逻辑还涉及邀请记录集合，实施时需要在 `listFunctions` 中再精确确认其实际集合名与字段，再写入设计文档。

### 关键业务约束
- 周期任务不是简单单条记录，`taskFunctions` 会预生成未来实例，并在前后端同时执行“每个周期系列只展示最近一个未完成未逾期实例”的过滤逻辑。
- 任务截止时间按 `+08:00` 解析后再转 `Date` 入库，属于明确的时区约定。
- 清单权限采用角色数字：1 创建者、2 编辑者、3 查看者；部分云函数通过 `requiredRole` 控制可见/可编辑范围。
- 多个页面保留 `DEBUG_MODE` / `MOCK_DATA` 分支，如 `checklist`、`list-detail`、`list-invite-accept`，修改真实数据流时不能只改 mock 分支。
- 注册页会先把头像上传到云存储，再调用 `profileFunctions.registerOrUpdateUser`；网络失败时还有本地降级保存逻辑。
