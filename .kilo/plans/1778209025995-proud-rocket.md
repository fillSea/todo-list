# 方案 B：补齐订阅消息完整链路的修改计划

## 目标
按“方案 B”补齐前后端订阅消息链路，使前端“开启订阅消息”按钮请求的模板，和后端真实发送的微信订阅消息模板保持一致，至少覆盖以下两类场景：

- 任务提醒类订阅消息
- 协作通知类订阅消息

同时修复当前“前端声明了 3 个模板、后端只实际发送 1 个模板”的不一致问题。

## 当前问题归纳

### 1. 前端模板配置与后端发送链路不一致
当前 `miniprogram/pages/profile/notification-settings/notification-settings.js` 中声明了：
- `TASK_REMINDER`
- `DEADLINE_REMINDER`
- `LIST_SHARED`

但当前云函数里只有 `cloudfunctions/taskFunctions/index.js` 存在一处真实 `cloud.openapi.subscribeMessage.send(...)`，且只使用：
- `process.env.REMINDER_TEMPLATE_ID`

`listFunctions` 中虽已写入应用内 `notifications`，但尚未发送微信订阅消息。

### 2. “截止提醒”与当前任务提醒实现未拆分
`taskFunctions` 当前只有单一订阅模板发送逻辑，未发现独立的“截止提醒模板”与“普通提醒模板”双通道实现。

### 3. 当前前端按钮的模板配置是占位结构，不能直接联调真实能力
`SUBSCRIBE_TEMPLATE_CONFIGS` 目前是占位符，且 `placeholder: true` 会拦截请求。这对防错是正确的，但在方案 B 下必须替换为可实际配置、可扩展、与后端一致的真实配置结构。

## 修改范围
- `miniprogram/pages/profile/notification-settings/notification-settings.js`
- `cloudfunctions/taskFunctions/index.js`
- `cloudfunctions/listFunctions/index.js`
- 可选同步文案：`miniprogram/pages/profile/notification-settings/notification-settings.wxml`
- 可选同步帮助文案：`miniprogram/pages/profile/help/help.js`

## 实施原则
- 只在已有通知写入成功的基础上，补充微信订阅消息发送，不改变原业务返回值。
- 继续保留用户通知偏好判断：
  - 总开关 `enableNotifications`
  - 任务提醒 `taskReminder`
  - 协作通知 `listCollaboration`
- 若用户未订阅或模板未配置，记录日志并跳过，不让主流程失败。
- 前端请求订阅的模板集合必须严格对应后端真实可能发送的模板集合。

## 设计方案

### 一、统一订阅模板配置模型
前端和云函数都要围绕同一组模板语义组织，但不强行抽公共文件，避免扩大改动范围。

建议定义三类模板能力：

1. `taskReminder`
   - 用于任务提醒相关订阅消息
   - 对应 `taskFunctions` 发送

2. `taskDeadline`
   - 如果要保留“截止提醒”这个前端概念，则必须补一条独立后端发送链路
   - 若当前业务无法明确区分与 `taskReminder` 的发送时机，需要在实现前做一次规则收敛

3. `listCollaboration`
   - 用于清单邀请、加入申请、邀请提醒、审批通过、审批拒绝等协作事件
   - 对应 `listFunctions` 发送

### 二、前端调整方案
文件：`miniprogram/pages/profile/notification-settings/notification-settings.js`

#### 1. 将订阅模板配置改为“真实模板配置结构”
把当前的占位配置：

```js
const SUBSCRIBE_TEMPLATE_CONFIGS = [
  { key: 'taskReminder', label: '任务提醒', templateId: 'TASK_REMINDER', placeholder: true },
  ...
];
```

调整为适合联调的结构，例如：

```js
const SUBSCRIBE_TEMPLATE_CONFIGS = [
  {
    key: 'taskReminder',
    label: '任务提醒',
    templateId: 'REPLACE_WITH_REAL_TASK_TEMPLATE_ID'
  },
  {
    key: 'taskDeadline',
    label: '截止提醒',
    templateId: 'REPLACE_WITH_REAL_DEADLINE_TEMPLATE_ID'
  },
  {
    key: 'listCollaboration',
    label: '协作通知',
    templateId: 'REPLACE_WITH_REAL_LIST_TEMPLATE_ID'
  }
];
```

注意：
- `listShared` 命名建议改为 `listCollaboration`，与通知设置项和后端偏好字段一致。
- 若短期内无法提供真实模板 ID，可继续保留“未配置时不请求”的保护逻辑，但配置结构要与后端模板用途一一对应。

#### 2. 订阅按钮请求模板集合时支持按通知偏好过滤
建议 `onRequestSubscribe` 请求模板时：
- 仅请求当前设置允许的分类模板
- 例如：
  - `taskReminder=false` 时，不请求任务类模板
  - `listCollaboration=false` 时，不请求协作类模板

这样可以让“开启订阅消息”与通知设置页自身语义一致。

#### 3. 订阅结果提示继续保留细分状态
保留当前对 `accept / reject / ban / filter` 的区分，但把提示文案与最终模板分类对应起来，必要时在日志中带上模板 key。

### 三、`taskFunctions` 调整方案
文件：`cloudfunctions/taskFunctions/index.js`

#### 1. 将单模板发送扩展为显式模板选择
当前逻辑：

```js
const TEMPLATE_ID = process.env.REMINDER_TEMPLATE_ID || '';
```

建议改为按模板类型读取，例如：

```js
const TASK_REMINDER_TEMPLATE_ID = process.env.TASK_REMINDER_TEMPLATE_ID || '';
const TASK_DEADLINE_TEMPLATE_ID = process.env.TASK_DEADLINE_TEMPLATE_ID || '';
```

#### 2. 明确“任务提醒”和“截止提醒”的触发规则
这是方案 B 的关键设计点，需在实现中明确：

方案建议：
- 若当前 `checkReminders()` 覆盖的是“到提醒时间/临近截止”的统一入口，可先在同一入口内根据任务字段决定模板类型。
- 优先采用最小可实现规则：
  - 普通提醒：有显式 reminderTime 的任务，走 `TASK_REMINDER_TEMPLATE_ID`
  - 截止提醒：无单独 reminderTime、或当前提醒是由 dueDate 临近触发时，走 `TASK_DEADLINE_TEMPLATE_ID`

如果现有代码无法稳定区分这两种来源，需要在实现阶段先阅读该函数更完整上下文，再做小范围判断抽取。

#### 3. 保持失败不影响任务主流程
- 模板未配置：跳过并记录日志
- 用户未订阅（47003）：继续视为正常情况
- 其他异常：记录日志，不阻断 `reminderSent=true`

### 四、`listFunctions` 调整方案
文件：`cloudfunctions/listFunctions/index.js`

#### 1. 在已确认的 5 个协作通知写入点补充微信订阅消息发送
当前 5 处应用内通知写入点：
- `list_invite`
- `join_request`
- `invite_remind`
- `application_approved`
- `application_rejected`

方案 B 需要在这些点上，在“允许接收协作通知”的前提下：
- 保留 `notifications` 集合写入
- 追加调用统一的协作订阅消息发送辅助函数

#### 2. 建议新增本文件内私有发送辅助函数
例如：

```js
async function sendCollaborationSubscribeMessage({
  userId,
  page,
  thing1,
  thing2,
  phrase3
}) {
  // 查询接收者 openid
  // 读取 process.env.LIST_COLLABORATION_TEMPLATE_ID
  // 调用 cloud.openapi.subscribeMessage.send
  // 捕获 47003 并记录其余异常
}
```

建议同时抽出“获取接收用户 openid”的小函数，避免每个触发点重复查 `users`。

#### 3. 模板字段映射保持统一
实现前要根据实际微信模板字段名调整，但计划建议统一承载以下信息：
- 协作类型：邀请 / 申请 / 提醒 / 通过 / 拒绝
- 清单名称
- 发起人或申请人昵称
- 跳转页面：
  - 优先跳清单详情或邀请处理页

#### 4. 保持业务主流程不受影响
- 邀请、申请、审批成功返回不依赖订阅消息发送成功
- 订阅消息发送失败仅记日志
- 偏好判断继续在通知写入前统一控制

### 五、模板与环境变量约定
建议统一新增以下环境变量：
- `TASK_REMINDER_TEMPLATE_ID`
- `TASK_DEADLINE_TEMPLATE_ID`
- `LIST_COLLABORATION_TEMPLATE_ID`

兼容建议：
- 实现阶段可以临时保留对旧变量 `REMINDER_TEMPLATE_ID` 的回退读取，仅用于任务提醒，减少部署切换风险。
- 但最终目标应是用新命名取代旧命名，避免语义混乱。

## 具体实施步骤

### 1. 先整理前端模板配置结构
- 把 `listShared` 改为 `listCollaboration`
- 将模板配置改成与后端一一对应的结构
- 请求订阅时按当前通知偏好过滤模板
- 完善未配置模板和部分模板缺失时的提示日志

### 2. 扩展 `taskFunctions` 的双模板发送逻辑
- 引入 `TASK_REMINDER_TEMPLATE_ID` / `TASK_DEADLINE_TEMPLATE_ID`
- 在现有提醒发送点选择模板类型
- 继续保留 `taskReminder` 偏好判断

### 3. 扩展 `listFunctions` 的协作订阅消息发送逻辑
- 增加统一发送辅助函数
- 在 5 个协作通知触发点追加发送订阅消息
- 继续沿用 `canReceiveCollaborationNotification(userId)` 结果

### 4. 同步前端说明文案
建议在通知设置页或帮助页同步说明：
- 订阅消息需单独授权微信模板消息
- 协作通知授权后可接收邀请、申请、提醒、审批结果等消息

## 验证方案

### 一、前端验证
1. 进入“通知设置”页，点击“开启订阅消息”时：
   - 未配置任一真实模板 ID：提示“订阅消息模板未配置”
   - 部分模板配置缺失：日志中能看到缺失模板分类
   - 已配置模板：正常发起微信订阅授权
2. 关闭 `taskReminder` 后，再点“开启订阅消息”，确认不请求任务类模板。
3. 关闭 `listCollaboration` 后，再点“开启订阅消息”，确认不请求协作类模板。

### 二、任务提醒订阅验证
1. 配置 `TASK_REMINDER_TEMPLATE_ID` 后，触发普通提醒任务，确认：
   - 应用内 `task_reminder` 通知仍写入
   - 微信订阅消息发送成功或用户未订阅时有正确日志
2. 配置 `TASK_DEADLINE_TEMPLATE_ID` 后，触发截止提醒场景，确认发送的是截止提醒模板。
3. `taskReminder=false` 时，确认既不写应用内提醒，也不发微信订阅消息。

### 三、协作通知订阅验证
分别验证以下场景：
1. 邀请成员
2. 申请加入
3. 邀请提醒
4. 审批通过
5. 审批拒绝

验证点：
- 应用内通知正常
- 微信订阅消息在模板配置存在时发送
- `enableNotifications=false` 或 `listCollaboration=false` 时不发送
- 用户未订阅时主流程不报错

## 风险与注意事项
- 方案 B 不是单纯修提示文案，而是补齐真实发送链路，改动范围会从前端扩展到两个云函数。
- “截止提醒”的定义当前代码里未完全显式区分，实现时必须基于 `checkReminders()` 的完整提醒来源做一次小范围规则确认。
- 微信订阅模板字段名受模板内容限制，实际实现时要按真实模板字段调整 `data` 结构，不能机械照搬计划中的占位字段名。
- 订阅消息发送属于增强链路，必须保证失败不影响现有邀请、审批、提醒等主流程。

## 建议实施顺序
1. 先确认真实模板 ID 与模板字段设计。
2. 先改前端模板配置结构和请求逻辑。
3. 再改 `taskFunctions` 的多模板发送。
4. 再改 `listFunctions` 的协作订阅发送。
5. 最后联调前端订阅授权与云函数发送结果。
