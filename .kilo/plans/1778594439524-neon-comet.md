# 性能优化方案

## 目标

提升小程序首屏加载、清单页加载、首页任务渲染、日历页切换和云函数查询性能，同时降低云数据库调用次数、`setData` 数据传输量和基础包压力。

## 优先级总览

1. 优先优化清单页 `getMyLists` 的 N+1 查询和统计计算。
2. 减少首页、日历页重复请求和重复计算。
3. 降低前端 `setData` 大数组传输和重复更新。
4. 为日历页建立任务索引，避免每次生成日历全量扫描任务。
5. 补齐云数据库索引。
6. 优化搜索策略和包体积。

## 方案一：优化清单页云函数查询

### 问题

`cloudfunctions/listFunctions/index.js` 的 `getMyLists` 对每个清单分别调用：

- `getListTaskStats(list._id)`
- `getListMemberSummary(list._id)`
- `getUserRole(list._id, userId)`

其中 `getListTaskStats` 会拉取该清单所有任务后在云函数中统计。清单数量和任务数量增多后，数据库调用次数和云函数耗时会明显上升。

### 优化方向

1. 将统计字段冗余到 `lists` 集合：
   - `taskCount`
   - `pendingCount`
   - `completedCount`
   - `progress`
   - `statsUpdatedAt`

2. 在任务变更时维护统计：
   - 创建任务时增加 `taskCount` 和 `pendingCount`。
   - 删除任务时减少对应计数。
   - 切换完成状态时更新 `pendingCount`、`completedCount`、`progress`。
   - 批量删除、周期任务生成、清单删除等流程同步处理。

3. 将成员摘要冗余到 `lists` 集合：
   - `memberCount`
   - `memberPreview`，最多保留 3 个成员头像和昵称。

4. 在成员变更时维护成员摘要：
   - 加入清单。
   - 移除成员。
   - 邀请接受。
   - 成员资料头像变化时，如要求严格实时，可通过懒更新或下次访问时刷新。

5. 批量获取当前用户角色：
   - `getMyLists` 已经查询 `list_members`，可构造 `roleMap`。
   - 避免每个清单再调用 `getUserRole`。

### 实施步骤

1. 梳理所有会改变任务数量和状态的云函数入口。
2. 新增统一的清单统计更新函数，例如 `recalculateListStats(listId)` 或增量更新函数。
3. 先使用全量重算保证正确性，再视需要改为增量更新。
4. 修改 `getMyLists`，优先直接读取 `lists` 上的统计和成员摘要字段。
5. 保留兜底逻辑：如果老数据缺少统计字段，可按需回填或懒计算。

### 验收标准

- 清单页一次加载不再对每个清单分别拉取全量任务。
- 一页 20 个清单时，数据库查询次数显著下降。
- 清单卡片上的任务数量、待完成数量、进度、成员数和头像显示正确。

## 方案二：减少首页和日历页重复请求

### 问题

首页 `onLoad` 和 `onShow` 都可能触发登录检查和任务加载，首次进入时可能重复调用 `loadTasks`。

日历页 `onLoad` 会调用 `loadTasks`、`generateCalendar`、`loadTasksForSelectedDate`，而 `loadTasks` 内部也会调用后两者，存在重复计算。

### 优化方向

1. 增加页面级请求状态：
   - `_isLoadingTasks`
   - `_lastTaskLoadTime`
   - `_loadedOnce`

2. 增加缓存 TTL：
   - 首页任务缓存：30 秒到 2 分钟。
   - 日历月任务缓存：1 到 5 分钟。
   - 分类缓存：5 到 30 分钟。

3. 统一加载流程：
   - `onLoad` 负责初始化。
   - `onShow` 只在登录状态变化、缓存过期、从编辑页返回等需要刷新时加载。

4. 对分类加载增加 `cachedCategoriesTime`。

### 实施步骤

1. 修改 `app.js` 或新增缓存工具函数，统一判断缓存是否过期。
2. 调整首页 `checkLoginStatus` 与 `onShow` 的调用关系，避免首次重复加载任务。
3. 调整日历页 `onLoad`，避免手动调用 `generateCalendar` 和 `loadTasksForSelectedDate` 与 `loadTasks` 内部重复。
4. 在任务创建、编辑、删除、状态切换后明确调用 `app.clearTaskCaches()` 或设置刷新标记。

### 验收标准

- 首次进入首页不会连续发起两次相同任务列表请求。
- 首次进入日历页不会重复生成日历和筛选选中日期任务。
- 修改任务后仍能及时刷新数据。

## 方案三：减少 `setData` 数据量和次数

### 问题

首页和日历页将完整任务数组放在 `data` 中，并多次 `setData`。小程序中 `setData` 是高成本操作，大数组会影响渲染性能。

### 优化方向

1. 原始任务数组放到页面实例字段：
   - `this._rawTasks`
   - `this._processedTasks`

2. `data` 只保留 WXML 直接渲染需要的数据：
   - `inProgressTasks`
   - `overdueTasks`
   - `completedTasks`
   - `selectedDateTasks`
   - `calendarDays`

3. 合并连续 `setData`：
   - 先计算完整 nextState。
   - 最后一次性 `setData(nextState)`。

4. 列表项字段裁剪：
   - 首页列表只保留渲染字段。
   - 详情页通过 taskId 再读取完整任务信息。

### 实施步骤

1. 从首页 `loadTasks` 开始改造：不再先 `setData({ tasks })`，改为 `processTasks(tasks)` 返回渲染分组。
2. 将 `processTasks` 改为纯计算函数，减少对 `this.data.tasks` 的依赖。
3. 日历页同样将原始任务放实例字段，`data.tasks` 只在确实被 WXML 使用时保留。
4. 检查 WXML 中对 `tasks` 的引用，避免移除后页面异常。

### 验收标准

- 首页任务加载期间 `setData` 次数减少。
- `data` 中不再长期保存不参与渲染的大型原始任务数组。
- 任务列表渲染内容保持一致。

## 方案四：优化日历页任务计算

### 问题

`generateCalendar` 为 42 个日历格逐个调用 `calculatePeriodicCount`，而 `calculatePeriodicCount` 内部会重新扫描任务、计算选中日期展示任务和周期系列。周期任务较多时会放大计算成本。

### 优化方向

1. 加任务索引：
   - `this._calendarTaskIndex`
   - key 为日期字符串，例如 `2026-05-12`。
   - value 包含当天任务、周期数量等。

2. 加周期系列索引：
   - 按 `parentTaskId || _id` 分组。
   - 预先解析 `repeatValue`，避免每天重复 split 和 parse。

3. `generateCalendar` 只负责生成 42 个格子，并从索引 O(1) 读取数量。

### 实施步骤

1. 新增 `buildCalendarTaskIndex(tasks, currentYear, currentMonth, selectedDate, currentCategory)`。
2. 在 `loadTasks`、切换月份、切换分类、切换选中日期时重建索引。
3. 修改 `generateCalendar` 使用索引结果。
4. 保留现有周期任务规则，避免改变业务行为。

### 验收标准

- 日历切月和切日期明显减少卡顿。
- 周期任务显示数量和选中日期任务列表与改造前一致。

## 方案五：补齐云数据库索引

### 微信云开发索引创建说明

以下索引均按微信云开发数据库控制台的“索引管理”录入。微信云开发常用索引类型为“普通索引”和“唯一索引”。本项目中的业务字段大多允许重复，因此默认使用“普通索引”。只有明确要求全局唯一的字段才使用“唯一索引”。

复合索引字段顺序按下表从左到右添加。等值筛选字段放前面，排序或范围字段放后面。排序方向需要与代码中的 `orderBy` 保持一致；没有排序要求的等值字段统一使用升序。

控制台录入时可直接按“字段与排序”列逐项添加字段：升序对应控制台中的 `asc` 或“升序”，降序对应 `desc` 或“降序”。创建复合索引时不要调整字段顺序，例如 `creatorId` 升序，`createdAt` 降序必须先添加 `creatorId`，再添加 `createdAt`。

本节列出的索引名称为建议名称，便于后续排查慢查询和核对配置；如果控制台限制名称格式，可使用同等语义的英文名，但索引类型、字段顺序和排序方向应保持一致。

### 建议索引清单

`tasks` 集合：

| 索引名称建议 | 索引类型 | 字段与排序 | 适用查询 |
| --- | --- | --- | --- |
| `idx_creator_dueDate` | 普通索引 | `creatorId` 升序，`dueDate` 升序 | 首页、日历页按用户和日期范围查询任务 |
| `idx_creator_createdAt_desc` | 普通索引 | `creatorId` 升序，`createdAt` 降序 | 用户任务列表按创建时间倒序分页 |
| `idx_creator_status_dueDate` | 普通索引 | `creatorId` 升序，`status` 升序，`dueDate` 升序 | 用户未完成/已完成任务按日期筛选 |
| `idx_list_createdAt_desc` | 普通索引 | `listId` 升序，`createdAt` 降序 | 清单详情任务列表按创建时间倒序 |
| `idx_list_dueDate` | 普通索引 | `listId` 升序，`dueDate` 升序 | 清单内任务按日期范围或日期排序 |
| `idx_list_status` | 普通索引 | `listId` 升序，`status` 升序 | 清单任务统计、待完成/已完成筛选 |
| `idx_category_dueDate` | 普通索引 | `categoryId` 升序，`dueDate` 升序 | 分类任务筛选和日期排序 |
| `idx_parentTaskId` | 普通索引 | `parentTaskId` 升序 | 周期任务系列实例查询 |
| `idx_related_list_status_dueDate` | 普通索引 | `listId` 升序，`status` 升序，`dueDate` 升序 | 如果后续将清单统计改为数据库聚合，可优先使用 |

控制台优先创建：`idx_creator_dueDate`、`idx_creator_createdAt_desc`、`idx_list_createdAt_desc`、`idx_list_status`。其余索引可在对应页面或云函数出现慢查询后分批创建。

`lists` 集合：

| 索引名称建议 | 索引类型 | 字段与排序 | 适用查询 |
| --- | --- | --- | --- |
| `idx_creator_updatedAt_desc` | 普通索引 | `creatorId` 升序，`updatedAt` 降序 | 我创建的清单按更新时间倒序分页 |
| `idx_isShared_updatedAt_desc` | 普通索引 | `isShared` 升序，`updatedAt` 降序 | 个人/共享清单筛选后排序 |
| `idx_updatedAt_desc` | 普通索引 | `updatedAt` 降序 | 清单默认排序 |

说明：`_id` 是系统字段，不需要手动创建索引。代码中大量使用 `_id in [...]`，这部分依赖系统索引即可。

控制台优先创建：`idx_creator_updatedAt_desc`、`idx_updatedAt_desc`。`idx_isShared_updatedAt_desc` 仅在共享/个人清单筛选成为高频慢查询后创建。

`list_members` 集合：

| 索引名称建议 | 索引类型 | 字段与排序 | 适用查询 |
| --- | --- | --- | --- |
| `idx_userId` | 普通索引 | `userId` 升序 | 查询用户加入的所有清单 |
| `idx_listId` | 普通索引 | `listId` 升序 | 查询某个清单的成员列表 |
| `idx_list_user` | 普通索引 | `listId` 升序，`userId` 升序 | 校验用户是否属于某清单、读取用户角色 |
| `idx_user_list` | 普通索引 | `userId` 升序，`listId` 升序 | 从用户维度批量构造清单角色映射 |

可选增强：如果业务要求同一用户在同一清单中只能有一条成员记录，可将 `listId + userId` 建为“唯一索引”。启用前必须先清理历史重复数据，否则索引创建会失败。

控制台优先创建：`idx_userId`、`idx_listId`、`idx_list_user`。如果后续 `getMyLists` 批量构造角色映射改为以 `userId` 为主查询，再补充 `idx_user_list`。

`notifications` 集合：

| 索引名称建议 | 索引类型 | 字段与排序 | 适用查询 |
| --- | --- | --- | --- |
| `idx_user_createdAt_desc` | 普通索引 | `userId` 升序，`createdAt` 降序 | 通知列表分页 |
| `idx_user_isRead_createdAt_desc` | 普通索引 | `userId` 升序，`isRead` 升序，`createdAt` 降序 | 未读通知列表和未读数相关查询 |
| `idx_relatedId` | 普通索引 | `relatedId` 升序 | 任务删除时清理关联通知 |

控制台优先创建：`idx_user_createdAt_desc`、`idx_user_isRead_createdAt_desc`。`idx_relatedId` 可在任务删除清理关联通知出现慢查询后创建。

`list_invites` 集合：

| 索引名称建议 | 索引类型 | 字段与排序 | 适用查询 |
| --- | --- | --- | --- |
| `idx_inviteCode` | 普通索引 | `inviteCode` 升序 | 通过邀请码查邀请记录 |
| `idx_list_status_createdAt_desc` | 普通索引 | `listId` 升序，`status` 升序，`createdAt` 降序 | 清单邀请管理页按状态查询 |
| `idx_target_status_createdAt_desc` | 普通索引 | `targetUserId` 升序，`status` 升序，`createdAt` 降序 | 查询发给某用户的邀请或申请 |
| `idx_expiresAt` | 普通索引 | `expiresAt` 升序 | 过期邀请扫描或过期判断 |

可选增强：如果邀请码必须全局唯一，可将 `inviteCode` 建为“唯一索引”。当前代码已经尝试避免重复邀请码，唯一索引可以作为数据库层兜底；启用前需要确认历史 `inviteCode` 没有重复值。

控制台优先创建：`idx_inviteCode`、`idx_list_status_createdAt_desc`。如果存在面向用户的邀请/申请列表，再创建 `idx_target_status_createdAt_desc`。

`categories` 集合：

| 索引名称建议 | 索引类型 | 字段与排序 | 适用查询 |
| --- | --- | --- | --- |
| `idx_user_sortOrder` | 普通索引 | `userId` 升序，`sortOrder` 升序 | 获取用户分类并按排序字段展示 |
| `idx_user_createdAt` | 普通索引 | `userId` 升序，`createdAt` 升序 | 没有排序字段时按创建时间展示分类 |

控制台优先创建：`idx_user_sortOrder`。如果代码路径会回退到创建时间排序，再补充 `idx_user_createdAt`。

`operations` 集合：

| 索引名称建议 | 索引类型 | 字段与排序 | 适用查询 |
| --- | --- | --- | --- |
| `idx_list_createdAt_desc` | 普通索引 | `listId` 升序，`createdAt` 降序 | 清单操作日志分页 |
| `idx_operator_createdAt_desc` | 普通索引 | `operatorId` 升序，`createdAt` 降序 | 用户维度操作记录查询 |

控制台优先创建：`idx_list_createdAt_desc`。`idx_operator_createdAt_desc` 仅在需要按用户查看操作记录时创建。

### 创建顺序建议

1. 第一批：`list_members` 的 `idx_userId`、`idx_listId`、`idx_list_user`，以及 `tasks` 的 `idx_creator_dueDate`、`idx_list_createdAt_desc`。
2. 第二批：`lists` 的 `idx_creator_updatedAt_desc`、`idx_updatedAt_desc`，以及 `notifications` 的通知列表索引。
3. 第三批：`list_invites`、`categories`、`operations` 和搜索、统计相关补充索引。

### 注意事项

1. 不要盲目把所有建议索引一次性全建，索引会增加写入成本和存储成本。
2. 优先创建与慢查询、主流程页面相关的索引。
3. 复合索引不是字段集合，字段顺序会影响命中率。比如 `listId + userId` 和 `userId + listId` 适用场景不同。
4. 正则模糊搜索通常不能充分利用普通索引，搜索性能优化不能只依赖索引，需要配合搜索字段改造。
5. 唯一索引创建前必须确认历史数据无重复，否则微信云开发会创建失败。

### 实施步骤

1. 根据微信云开发控制台的慢查询或实际页面加载耗时确认第一批索引。
2. 在云开发控制台进入对应集合，打开“索引管理”，按上表添加索引类型、字段顺序和排序方向。
3. 先创建普通索引；唯一索引只在确认无重复数据且业务确实要求唯一时创建。
4. 分批创建索引，优先 `tasks`、`list_members`、`lists`。
5. 索引构建完成后再验证页面和云函数耗时，避免在索引仍在构建时误判效果。
6. 观察查询性能、写入性能和索引存储成本。

### 验收标准

- 任务列表、清单列表、成员权限查询耗时下降。
- 云开发控制台不再频繁出现无索引慢查询。

## 方案六：搜索性能优化

### 问题

任务搜索和清单搜索使用正则模糊匹配。数据量增大后，普通索引难以充分利用。

### 优化方向

1. 前端限制搜索触发：
   - 小于 2 个字符不发起远程搜索。
   - 保留 300ms 防抖。
   - 请求期间避免并发覆盖，可加搜索 token。

2. 后端增加搜索字段：
   - `titleLower`
   - `descriptionLower`
   - `searchText`
   - `searchTokens`

3. 可接受前缀搜索时，改为范围查询。

### 实施步骤

1. 先增加前端最小输入长度限制和并发保护。
2. 新增写入任务、更新任务时维护搜索字段。
3. 对旧数据提供一次性回填脚本或懒回填。
4. 将搜索接口逐步切换到可索引字段。

### 验收标准

- 短关键词不会频繁触发云函数。
- 搜索结果正确。
- 数据量较大时搜索耗时可控。

## 方案七：包体积和加载优化

### 问题

`app.json` 已启用 `lazyCodeLoading: requiredComponents`，但仍有全局 Vant 组件。非首屏页面较多，适合分包。

### 优化方向

1. 将非全局必须的 Vant 组件移到页面级 `.json`。
2. 对非 tabBar 页面分包：
   - 注册页。
   - 清单邀请相关页。
   - 帮助、协议、隐私、关于页。
   - 周期统计页。
   - 通知设置页。

### 注意事项

- tabBar 页面不能放入分包。
- 分包后需要检查页面跳转路径和组件引用路径。
- `packNpmManually: true`，如调整 npm 构建产物，需要在微信开发者工具中重新构建 npm。

### 验收标准

- 主包体积下降。
- 首次打开耗时下降。
- 所有分包页面可正常跳转和渲染。

## 推荐实施顺序

1. 先加缓存 TTL 和避免重复请求，风险低、收益快。
2. 改造首页 `setData` 和任务处理，控制客户端渲染成本。
3. 优化日历页索引计算，解决周期任务多时的卡顿。
4. 改造 `getMyLists`，逐步引入清单统计和成员摘要冗余。
5. 补齐云数据库索引。
6. 优化搜索字段。
7. 做分包和组件级引入优化。

## 风险与回归点

1. 统计字段冗余可能出现一致性问题，需要覆盖任务创建、删除、状态切换、批量操作、周期任务生成等路径。
2. 缓存 TTL 可能导致用户短时间看到旧数据，需要在写操作后主动清缓存或设置刷新标记。
3. 日历周期任务规则前后端有重复实现，改造时必须保持现有显示规则一致。
4. 分包会影响页面路径和组件加载，需要用微信开发者工具完整检查。

## 验证方式

1. 使用微信开发者工具观察首页、日历页、清单页网络请求次数。
2. 对比优化前后云函数耗时和数据库读取次数。
3. 在任务数量较多、清单数量较多、周期任务较多的账号上验证。
4. 手动覆盖以下场景：
   - 登录后首次进入首页。
   - 从任务编辑页返回首页。
   - 日历切月、切日期、切分类。
   - 清单页加载更多、筛选、搜索。
   - 任务完成、取消完成、删除、批量删除。
   - 邀请成员、移除成员。

## 不建议立即做的优化

1. 不建议一开始就引入复杂状态管理，当前项目规模用页面实例字段和缓存工具即可。
2. 不建议先改搜索底层，除非已有明显慢查询；可先限制输入和补索引。
3. 不建议大范围重写页面结构，应按上述顺序小步改造，保证业务行为稳定。
