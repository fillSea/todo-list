# 清单模块问题修复方案

## 目标

整理清单页两个已确认问题的实施方案，覆盖改动点、原因、验证方式和影响范围，并以最小改动为原则约束实现边界：

1. 创建者权限识别不稳定，可能导致创建者在长按菜单中缺少应有操作。
2. 骨架屏使用 `wx:for="{{3}}"` 的写法不规范，存在兼容性和渲染稳定性风险。

本方案只针对 `checklist` 页面前端消费逻辑，不调整后端接口契约，不扩散到无关页面。

## 事实依据

### 已确认的权限链路

- `miniprogram/pages/checklist/checklist.js` 的 `openEditDialog()` 已通过 `const canEditStructure = list.myRole === 1;` 控制是否允许修改类型和可见性。
- `onDialogConfirm()` 也依赖相同语义决定是否提交 `isShared` 和 `visibility`。
- `miniprogram/pages/list-members/list-members.js` 同样以 `myRole === 1` 控制管理权限。
- `cloudfunctions/listFunctions/index.js` 中 `getMyLists` 与 `searchLists` 都会显式返回 `myRole`，并已将创建者约定为 `1`。

### 已确认的问题根因

- `checklist` 页长按菜单仍在使用 `creatorId === (userInfo._id || userInfo.openid)` 推导创建者身份。
- `app.getLoginState()` 读取的是本地缓存 `userInfo`，并不保证字段结构与后端实时一致。
- 后端 `creatorId` 来自 `users._id`，本地登录态可能只有 `openid`，两者并不保证同构。
- 因此当前问题不是后端没返回权限，而是前端在 `checklist` 页混用了两套权限来源。

### 已确认的骨架屏上下文

- 骨架屏显示条件是 `isLoading && !lists.length`，不是 `filteredLists.length`。
- `applyFilter()` 当前并不做额外本地筛选：非远程搜索时 `filteredLists = lists`，远程搜索时 `filteredLists = remoteSearchResults`。
- 因此骨架屏修正只需替换遍历数据源，不会引入搜索态或筛选态联动复杂度。

## 问题一：创建者权限判断不稳定

### 现状

`miniprogram/pages/checklist/checklist.js` 的 `onListLongPress` 当前同时依赖：

- `list.creatorId === (userInfo._id || userInfo.openid)` 判断创建者
- `list.myRole === 2` 判断编辑者

这导致清单页内部存在两套权限语义：

- 页面菜单展示依赖本地登录态与 `creatorId` 比对
- 编辑弹窗和后端接口消费依赖 `myRole`

一旦本地 `userInfo` 只有 `openid` 而 `creatorId` 存的是 `users._id`，创建者就会在长按菜单中被误判为普通成员。

### 修改方案

#### 1. 统一长按菜单权限来源为 `myRole`

在 `miniprogram/pages/checklist/checklist.js` 的 `onListLongPress` 中，统一先做角色归一化，再基于角色决定菜单项：

```js
const role = Number(list.myRole);
const isCreator = role === 1;
const canEdit = role === 1 || role === 2;
const canManageMembers = role === 1 && list.isShared;
const canDelete = role === 1;
```

调整原则：

- 长按菜单展示只依赖 `myRole`，不再使用 `userInfo._id || userInfo.openid` 与 `creatorId` 的比对结果。
- 保持已有 action `type` 不变，只调整菜单项是否展示，避免影响 `onActionSheetSelect()` 的后续分发逻辑。
- `openEditDialog()` 与 `onDialogConfirm()` 已经使用 `myRole` 语义，无需重做结构权限设计，只需对齐菜单入口判断。

#### 2. 是否增加兜底的取向

优先推荐直接使用 `myRole`，原因如下：

- 后端 `getMyLists` / `searchLists` 已明确回传 `myRole`。
- 页面内其他权限相关逻辑已经以 `myRole` 为准。
- 当前已知误判根因正是继续扩散 `creatorId` 与本地登录态比对。

仅在需要兼容历史异常数据时，才考虑最小兜底：

- 仅在 `list.myRole` 为 `null` 或 `undefined` 时，退回 `creatorId === currentUserId`。
- 不建议把兜底写成新的常态主逻辑，避免再次引入双重语义。

### 修改原因

- 与当前页面现有的 `openEditDialog()` / `onDialogConfirm()` 语义保持一致。
- 与 `list-members` 页的权限模型保持一致。
- 避免本地缓存登录态字段差异导致的误判。
- 不需要改后端接口，也不需要在 `checklist` 页继续扩散 `creatorId` 推导。

### 预期结果

- 创建者始终能看到“编辑清单”。
- 创建者在共享清单中始终能看到“管理成员”。
- 创建者始终能看到“删除清单”。
- 编辑者只看到允许的编辑入口，不会错误出现删除或成员管理入口。
- 查看者只看到“查看详情”。

### 验证方式

#### 场景 1：创建者

- 使用创建者账号进入清单页。
- 长按个人清单，应出现“查看详情 / 编辑清单 / 删除清单”。
- 长按共享清单，应出现“查看详情 / 编辑清单 / 管理成员 / 删除清单”。

#### 场景 2：编辑者

- 使用共享清单编辑者账号进入清单页。
- 长按共享清单，应出现“查看详情 / 编辑清单”。
- 不应出现“管理成员 / 删除清单”。

#### 场景 3：查看者

- 使用查看者账号进入清单页。
- 长按共享清单，只应出现“查看详情”。

#### 场景 4：登录态字段差异

- 本地 `userInfo` 只有 `openid`。
- 后端列表数据仍返回 `myRole === 1`。
- 创建者长按菜单仍应完整显示创建者操作。

## 问题二：骨架屏 `wx:for="{{3}}"` 写法不规范

### 现状

`miniprogram/pages/checklist/checklist.wxml` 当前骨架屏写法为：

```xml
<view class="skeleton-card" wx:for="{{3}}" wx:key="index">
```

`wx:for` 规范上应接收数组或可迭代列表，直接传数字属于不规范写法，可能在不同基础库、设备或开发者工具环境下表现不一致。

### 修改方案

#### 1. 在页面 `data` 中新增固定骨架数组

在 `miniprogram/pages/checklist/checklist.js` 的 `data` 中增加固定原始值数组，例如：

```js
skeletonList: [0, 1, 2]
```

推荐使用固定简单值数组，方便配合 `wx:key="*this"`，避免无意义对象 key。

#### 2. WXML 改为遍历骨架数组

将 `miniprogram/pages/checklist/checklist.wxml` 中骨架屏部分改为遍历 `skeletonList`：

```xml
<view class="skeleton-card" wx:for="{{skeletonList}}" wx:key="*this">
```

如需显式声明循环变量，也可写成：

```xml
<view class="skeleton-card" wx:for="{{skeletonList}}" wx:for-item="skeletonItem" wx:key="*this">
```

该改动只修正渲染数据源，不影响样式、加载状态条件或搜索筛选逻辑。

### 修改原因

- 消除对非标准数字遍历行为的依赖。
- 提升不同环境下骨架屏的渲染稳定性。
- 该问题只涉及模板循环数据源，不需要改动 `applyFilter()` 或其他筛选逻辑。

### 预期结果

- 初次加载时稳定显示 3 条骨架卡片。
- 不再依赖基础库对数字遍历的兼容实现。
- 避免不同设备或开发者工具环境出现首屏骨架空白。

### 验证方式

- 首次进入清单页，网络较慢时应看到 3 条骨架卡片。
- 切换搜索或筛选并触发列表重新请求时，骨架逻辑不报错。
- 开发者工具控制台不应出现与 `wx:for` 数据类型相关的警告。

## 实施步骤

1. 修改 `miniprogram/pages/checklist/checklist.js`，统一 `onListLongPress` 的权限判定逻辑为 `myRole`。
2. 在 `miniprogram/pages/checklist/checklist.js` 的 `data` 中增加 `skeletonList`。
3. 修改 `miniprogram/pages/checklist/checklist.wxml`，将骨架屏遍历从数字改为数组。
4. 静态复查 `openEditDialog()`、`onDialogConfirm()` 与长按菜单权限语义是否一致。
5. 在微信开发者工具中按创建者、编辑者、查看者三类角色执行手动验证。

## 影响范围

### 直接影响

- `miniprogram/pages/checklist/checklist.js`
- `miniprogram/pages/checklist/checklist.wxml`

### 间接关联但原则上无需修改

- `cloudfunctions/listFunctions/index.js`
- `miniprogram/pages/list-members/list-members.js`
- `miniprogram/pages/list-detail/list-detail.js`
- `miniprogram/app.js`

### 明确不改动

- 不新增或调整云函数 action。
- 不修改 `listFunctions` 中 `getMyLists`、`searchLists`、`createList`、`updateList`、`deleteList` 的字段契约。
- 不顺手改 `my-lists`、`list-detail` 等其他页面，仅修正 `checklist` 页前端消费方式。

## 风险与控制

### 风险 1：历史数据中 `myRole` 缺失或类型不稳定

控制方式：

- 实施时统一先做 `const role = Number(list.myRole)`，避免数字字符串影响判断。
- 若确认线上存在 `myRole` 缺失数据，再增加仅针对 `null/undefined` 的最小兜底。
- 若未发现异常数据，保持只使用 `myRole` 的单一语义。

### 风险 2：调整菜单判断后影响现有 action 分发

控制方式：

- 不修改 action `type` 常量和 `onActionSheetSelect()` 的分支。
- 仅调整菜单项展示条件，保证后续交互链路不变。

### 风险 3：骨架列表 key 选择不当引发警告

控制方式：

- 使用固定原始值数组作为骨架数据源。
- 使用 `wx:key="*this"`，避免多余对象或索引 key 带来的警告和不稳定性。

## 推荐实现取向

推荐采用以下取向并作为实际实施的默认方案：

1. 清单页权限逻辑统一以后端返回的 `myRole` 为准，必要时仅做一次最小兜底。
2. 骨架屏改为 `data` 中固定数组驱动，保持现有显示条件和样式不变。
3. 本次只修复 `checklist` 页前端消费问题，不扩展到后端接口和其他页面。

这是最小、最稳、与现有代码语义最一致的方案。
