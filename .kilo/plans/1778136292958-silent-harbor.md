# 清单页面问题修复方案

## 目标
针对 `miniprogram/pages/checklist/checklist` 相关代码中已确认的问题，制定最小可行且一致性较好的修复方案。优先修复会直接影响用户认知或界面表现的问题，其次处理搜索语义和健壮性问题，最后清理残留实现与布局硬编码风险。

## 修复范围
- 前端页面：`miniprogram/pages/checklist/checklist.js`
- 前端模板：`miniprogram/pages/checklist/checklist.wxml`
- 可选参考页：`miniprogram/pages/list-edit/list-edit.js`
- 后端云函数：`cloudfunctions/listFunctions/index.js`（仅当采用服务端搜索方案时需要修改）

## 问题与修改方案

### 1. 资料未完善状态的命名与引导语义不清
问题：结合现有设计，用户身份识别依赖微信 `openId`，`users` 集合仅用于保存资料。因此这里的核心不是“未登录”，而是“尚未建立或完善用户资料”。当前页面通过 `isLoggedIn` 表达这个状态，并在空状态中显示“去完善资料”，点击后走 `onCreateList()` 跳转 `/pages/register/register`。从业务上看这个跳转未必错误，但状态命名、按钮行为入口和文案语义混在了一起，容易让维护者误解为传统账号注册/登录流程。

方案 A：保留当前引导页路径，澄清语义（推荐）
- 保留空状态按钮文案“去完善资料”。
- 保留未建档用户点击后进入 `/pages/register/register` 的行为，将该页面视为“用户资料初始化/完善页”，而不是传统注册页。
- 不再把此问题定义为“文案错误”，而是修正页面内部状态表达与入口命名，避免后续继续混淆。

具体改法
- 在 `checklist.js` 中将这类状态从“未登录”语义中拆出来，至少在注释和空状态文案层面表述为：
  - 已识别微信身份，但尚未完善资料
  - 完善资料后可创建和查看清单
- 如果改动允许稍大一点，建议新增一个更贴近业务语义的状态位，例如：
  - `hasProfile` 或 `isProfileReady`
- `syncLoginState()` 可继续兼容现有 `app.getLoginState()`，但页面层使用更准确的文案，不把它解释为传统登录态。
- 为未完善资料的按钮单独使用一个方法，例如 `onCompleteProfile()`，内部仍跳转 `/pages/register/register`。这样 `onCreateList()` 只负责真正的“创建清单”入口，职责更清晰。

方案 B：维持现状，仅补充注释
- 不改按钮和方法，仅在代码中注明 `/pages/register/register` 实际承担的是资料初始化职责。
- 不推荐。虽然改动最小，但仍然保留了 `onCreateList()` 同时承担“创建清单”和“资料引导”两种职责的问题。

实施建议
- 采用方案 A。
- 优先目标不是改文案，而是把“资料未完善”与“创建清单”入口解耦。
- 修改位置：`checklist.js` 和 `checklist.wxml`。

验收标准
- 未建档/未完善资料用户进入清单页时，空状态文案明确表达“需要完善资料”，不误导为普通登录失败。
- 点击按钮后进入 `/pages/register/register`，且代码层面该入口命名能体现“完善资料”而不是“创建清单”。

### 2. 进度计算缺少边界保护
问题：`progress` 直接由 `taskCount - pendingCount` 计算，若后端数据异常，可能出现负数或超过 100 的百分比。

方案
- 在 `processListData()` 中增加数值归一化：
  - `taskCount = Math.max(Number(list.taskCount) || 0, 0)`
  - `pendingCount = Math.max(Number(list.pendingCount) || 0, 0)`
- 计算 `completedCount` 时做边界收敛：
  - `completedCount = Math.min(Math.max(taskCount - pendingCount, 0), taskCount)`
- 计算 `progress` 后再强制限制在 `0-100`：
  - `progress = Math.min(Math.max(progress, 0), 100)`
- 如需保留排查能力，可在发现 `pendingCount > taskCount` 时打印一次 `console.warn`。

实施建议
- 只在前端做保护即可，先确保 UI 不异常。
- 不在本轮扩大到后端统计口径调整，避免超出当前任务范围。

验收标准
- `progress` 永远不会小于 0 或大于 100。
- 进度条宽度与文案始终合法。

### 3. 搜索仅在本地已加载分页数据中生效，结果不完整
问题：当前搜索只是对 `lists` 做本地过滤；当用户数据较多时，未加载页不会参与搜索。

方案 A：接入后端 `searchLists`（推荐）
- 在 `searchKeyword` 非空时，不再使用 `applyFilter()` 的本地搜索逻辑作为唯一来源。
- 在 `onSearchConfirm()` 中调用 `listFunctions.searchLists`。
- 搜索结果单独写入 `filteredLists`，并设置一个状态位，例如：
  - `isRemoteSearching`
  - `searchMode: 'local' | 'remote' | 'none'`
- 搜索清空或取消时，恢复常规列表数据展示。

前端改动建议
- 保留 `applyFilter()`，但职责收敛为：
  - 无搜索词时：显示当前 `lists`
  - 有搜索词但尚未执行确认搜索时：可选择继续本地预过滤，或仅等待确认后执行远程搜索
- 更稳妥的做法：
  - `bindinput` 只更新输入值
  - `bindconfirm` 执行远程搜索
  - 清空关键词后恢复默认列表

后端改动建议
- 复用现有 `searchLists`。
- 可选增强：支持 `filter` 参数，使搜索结果能遵循当前筛选标签（`all/personal/shared/created`）。
- 若不增强后端，则前端在搜索结果返回后再按当前筛选做一次本地过滤。

方案 B：保持本地搜索，但在 UI 明示“仅搜索已加载内容”
- 不推荐。
- 虽然实现最小，但功能体验明显偏弱。

实施建议
- 采用方案 A。
- 最小实现版本：
  - `onSearchConfirm()` 调用后端搜索
  - `onClearSearch()` 与 `onCancelSearch()` 恢复默认列表
  - `bindinput` 不再频繁本地过滤

验收标准
- 搜索结果覆盖用户有权限访问的全部清单，而不是仅限当前页已加载数据。
- 搜索结果与当前数据量无关，行为稳定可解释。

### 4. 搜索状态下继续上拉加载，语义混乱
问题：搜索时仍允许 `onLoadMore()` 触发普通列表分页加载，会造成“搜索结果”和“普通列表分页”混在一起。

方案
- 在 `onLoadMore()` 中增加搜索态拦截：
  - 若 `isSearching && searchKeyword.trim()` 为真，则直接 return。
- 若采用远程搜索且未来要支持“搜索结果分页”，应单独维护搜索分页状态；但本轮不建议扩展。
- 在搜索状态下只展示一次性搜索结果，不展示常规 `hasMore` 的加载更多逻辑。

模板建议
- `no-more` 和 `load-more` 文案可以根据搜索态决定是否显示。
- 最小改动下，只要在搜索态禁止继续加载即可。

验收标准
- 输入搜索词并确认后，上拉不会继续混入普通列表数据。
- 搜索结果集在一次搜索周期内保持稳定。

### 5. `formatTime()` 缺少非法时间值保护
问题：若 `updatedAt` 无法解析，可能生成 `NaN` 相关显示结果。

方案
- 在 `formatTime()` 中增加合法性判断：
  - `const date = new Date(timestamp)`
  - `if (Number.isNaN(date.getTime())) return ''`
- 可选：返回兜底文本“时间未知”，但从现有页面风格看，空字符串更稳妥。
- 如果 `diff < 0`，说明时间可能来自未来，也建议兜底返回格式化日期或“刚刚”。

实施建议
- 最小处理：非法时间返回空字符串。
- 稍强处理：未来时间统一显示“刚刚”，避免出现负时间差。

验收标准
- 无论 `updatedAt` 是否异常，页面都不会显示 `NaN分钟前`、`NaN-NaN-NaN`。

### 6. 右滑操作代码为残留死代码
问题：WXML/WXSS 中保留了 `swipe-actions` 结构，但 `showActions` 始终为 `false`，没有任何触发逻辑，当前真实交互是长按弹出 ActionSheet。

方案 A：删除残留右滑代码（推荐）
- 删除 `processListData()` 里固定注入的 `showActions: false`。
- 删除 `checklist.wxml` 中 `swipe-actions` 结构。
- 删除 `checklist.wxss` 中与 `.swipe-actions / .action-edit / .action-delete` 相关样式。
- 保留长按菜单作为唯一入口。

方案 B：补全右滑手势逻辑
- 不推荐，改动大且不属于本轮问题修复的最小集合。

实施建议
- 采用方案 A，清理死代码，减少误导和维护成本。

验收标准
- 页面行为与代码一致，不再存在未启用但看似可用的右滑实现。

### 7. 顶部滚动区域高度使用硬编码
问题：`104` 和 `156` 为写死值，可能在不同机型上出现遮挡或留白。

方案
- 将头部高度改为运行时测量，而不是硬编码。
- 可行做法：
  - 为 `.header-fixed` 增加选择器标识，例如 `id="header-fixed"`
  - 在首次渲染后通过 `wx.createSelectorQuery().select('#header-fixed').boundingClientRect()` 获取真实高度
  - 根据测量值计算 `scrollMarginTop` 与 `scrollHeight`
- 搜索框显隐后，在 `setData` 回调里重新测量头部高度。

实现建议
- 提供一个统一方法，例如 `updateScrollLayout()`：
  - 读取 `windowHeight`
  - 测量头部高度
  - 更新 `scrollMarginTop`、`scrollHeight`
- 在以下时机调用：
  - `onLoad`
  - `onSearch`
  - `onCancelSearch`
  - 可能还包括页面首次完成列表渲染后

风险说明
- 这是本轮相对稍大的前端改动，但仍限定在单页面内部，风险可控。
- 若希望本轮继续保持最小修改，也可以先不做，列为后续优化项。

实施建议
- 如果本轮只修核心问题，可先暂缓。
- 如果希望一次性提升稳定性，则纳入本轮。

验收标准
- 不同机型下列表顶部不会被固定头部遮挡。
- 搜索框显示/隐藏时滚动区域高度同步正确更新。

## 推荐实施顺序
1. 修正资料未完善状态的命名与入口职责。
2. 为进度与时间格式化增加前端防御。
3. 禁止搜索态的普通分页加载。
4. 将搜索改为远程搜索，恢复完整结果语义。
5. 清理残留右滑代码。
6. 视本轮时间决定是否纳入头部高度动态测量。

## 建议的最小修复集
若希望控制改动面，建议本轮先做以下 5 项：
- 将资料完善入口与创建清单入口解耦
- 修复 `progress` 边界
- 修复 `formatTime()` 非法值处理
- 搜索态禁止 `onLoadMore()`
- 搜索接入 `searchLists`

这组改动能解决最主要的用户可见问题，同时避免把任务扩大成完整 UI 重构。

## 可选增强
- 后端 `searchLists` 支持 `filter` 参数，保证搜索结果与当前筛选标签一致。
- 搜索结果增加独立空状态文案，例如“未找到匹配的共享清单”。
- 对创建/编辑成功后清空搜索态进行统一处理，避免刚创建后列表仍停留在旧搜索结果中。

## 验证建议
实现后至少验证以下场景：
- 资料未完善空状态文案与跳转语义一致。
- `taskCount=0`、`pendingCount>taskCount`、缺失字段时进度展示正常。
- `updatedAt` 为 `null`、非法字符串、未来时间时不出现 `NaN` 文案。
- 搜索能命中不在第一页的清单。
- 搜索结果状态下上拉不再混入普通列表。
- 长按菜单仍可查看详情、编辑、管理成员、删除。
- 清理右滑代码后页面样式无回归。

## 决策建议
推荐本轮按以下范围实施：
- 必做：问题 1、2、3、4、5、6
- 选做：问题 7

理由：前 6 项都属于低到中等改动量、收益直接、风险较低；第 7 项偏布局优化，可在本轮时间允许时再纳入。
