# 对 `doc/data_design.md` 中“个人任务管理”设计的评估与改进方案

## 目标
评估当前数据设计里“个人任务管理”是否存在可改进之处，并给出与现有代码实现相匹配、可逐步落地的改进方案。

## 结论
存在改进空间，且优先级较高。主要问题不是“完全不可用”，而是：

1. 文档设计与当前实现已经出现多处不一致。
2. 个人任务的归属模型过于含混，`task` 与 `list` 的关系不够清晰。
3. 周期任务采用“模板任务 + 预生成实例”的实现，但文档没有完整表达该模型。
4. 任务状态字段设计与前端实际展示逻辑不一致，容易造成后续统计、筛选和索引设计失真。
5. 本地缓存与分页/范围查询存在设计空白，个人任务规模增长后会影响一致性和性能。

## 已核对到的现状依据

### 文档现状
`doc/data_design.md` 中：

1. `tasks` 只正式列出了 `repeatType` / `repeatValue`，但后文示例和索引又使用了 `repeatStartDate` / `repeatEndDate`，主表定义不完整。
2. 权限部分写了“个人任务：仅创建者和负责人可读写”，但 `tasks` 结构中并没有正式定义 `assigneeId` 或“负责人”字段。
3. 索引部分写了 `(assigneeId, dueDate)`、`(repeatType, repeatStartDate)`、`(creatorId, categoryId)`、`(userId, sortOrder)`，其中部分字段并未在正式表结构中定义，或并非当前代码主路径所依赖。
4. 状态字段写为 `0-未完成, 1-已完成, 2-逾期`，但文档没有说明“逾期是持久化状态还是展示态”。

### 代码现状
从当前实现可确认：

1. `taskFunctions.createTask` 允许 `listId` 为空字符串，说明个人任务目前可以脱离任何清单独立存在。
2. `taskFunctions.getTaskList` 在未指定 `listId` 时，按“`creatorId = 当前用户` 或者 `listId in 我参与的共享清单`”查询任务，说明“个人任务”和“共享任务”当前混合存储在同一 `tasks` 集合。
3. 前端首页和日历页都会基于 `dueDate` + `status===0` 计算 `isOverdue`，也就是“逾期”当前主要是展示态，不是数据库里稳定写入的 `status=2` 主路径。
4. 周期任务已经存在 `parentTaskId`、`isPeriodicInstance` 等字段，说明当前真实模型是“源任务/模板 + 未来实例任务”，而不是单条规则型任务。
5. 前端缓存真实依赖 `cachedTasks`、`cachedCategories`、`calendarTasks_${year}_${month}`，但文档只笼统写了“本地缓存”，未定义缓存失效策略。
6. 任务编辑页支持选择清单，但没有“个人任务必须绑定默认个人清单”的硬约束，说明当前个人任务模型更接近“可选挂清单”。

## 主要问题与改进方案

### 1. 个人任务归属模型不清晰

#### 问题
当前设计中，个人任务既可以理解为：

1. 属于某个“个人清单”的任务；
2. 也可以理解为“没有 `listId` 的独立任务”。

代码实际上采用第 2 种为主：`listId` 可为空。这样虽然灵活，但会带来几个问题：

1. 文档层面很难说清“个人任务”和“个人清单”的边界。
2. 查询和统计需要同时兼容“挂清单”和“不挂清单”两种个人任务。
3. 后续如果要做任务迁移、归档、默认视图、批量权限判断，会出现大量 `listId` 为空的特殊分支。

#### 改进建议
推荐在设计层明确二选一，并优先采用更稳妥的方案：

**推荐方案 A：保留 `listId` 可空，但显式增加 `ownershipType` 字段。**

建议新增字段：

- `ownershipType: Number`
- 取值建议：`1=个人独立任务`，`2=清单任务`

配套约束：

1. `ownershipType=1` 时，`listId=''`。
2. `ownershipType=2` 时，`listId` 必填且必须指向存在的清单。
3. 查询个人任务时，优先按 `creatorId + ownershipType=1` 过滤，而不是通过 `listId` 是否为空来隐式判断。

优点：

1. 对现有代码侵入较小。
2. 可以兼容当前“无清单个人任务”的行为。
3. 后续统计、缓存、索引都更稳定。

**备选方案 B：强制所有个人任务挂在一个默认个人清单下。**

优点是模型统一；缺点是需要补数据迁移、默认清单生命周期管理、老数据修复，实施成本更高。当前项目更适合先做方案 A。

### 2. 任务状态设计混淆了“持久状态”和“派生状态”

#### 问题
文档把 `status=2` 定义成“逾期”，但当前前端真实逻辑是：

1. 数据库存的主状态主要只有 `0=未完成`、`1=已完成`；
2. “逾期”是由 `dueDate < today && status===0` 动态推导出来的展示态。

这会导致：

1. 文档误导后续开发直接写入 `status=2`；
2. 索引和查询条件设计会围绕一个不稳定状态展开；
3. 周期任务在跨天后如果不批量更新数据库，也无法天然变成 `status=2`。

#### 改进建议
建议将状态拆分为“持久状态 + 派生状态”：

1. `status` 仅保留：`0=待办`，`1=已完成`。
2. `isOverdue` 不入库，作为查询结果或前端计算字段。
3. 文档中明确：逾期是基于 `dueDate` 与业务时区比较得出的派生状态。

如果后续确实需要统计“曾经过期后又完成”的业务数据，再新增独立字段，而不是复用 `status`：

- `completedAt`
- `completedAfterDue: Boolean`

这样比 `status=2` 更能支撑统计分析。

### 3. 周期任务模型文档不完整

#### 问题
文档只描述了 `repeatType` / `repeatValue`，但代码里已经存在：

- `parentTaskId`
- `isPeriodicInstance`
- 周期任务预生成未来实例
- 完成实例后继续生成后续实例

这意味着当前真实模型不是“单条任务按规则重复展示”，而是“规则模板 + 实例化记录”。如果文档不补齐，会导致：

1. 后续开发误删或误改实例数据；
2. 索引无法覆盖实例查询；
3. 统计时无法区分“源任务”和“实例任务”。

#### 改进建议
在 `tasks` 集合设计中正式补齐以下字段：

- `parentTaskId: String`，所属周期系列根任务 ID；非实例任务为空或指向自身策略需统一。
- `isPeriodicInstance: Boolean`，是否为预生成实例。
- `repeatStartDate: Timestamp`，周期规则生效开始日期。
- `repeatEndDate: Timestamp | null`，周期规则结束日期。
- `generatedUntil: Timestamp | null`，已生成实例的最远日期，可选。

并明确规则：

1. 周期模板任务负责保存规则。
2. 实例任务负责承载“某一天是否完成、何时完成”的执行记录。
3. 首页/日历默认只展示每个系列最近的有效未完成实例，这是当前前端已有的业务事实，应该进入设计文档。

### 4. 个人任务权限描述不准确

#### 问题
文档写“个人任务：仅创建者和负责人可读写”，但当前实现中：

1. 个人任务没有正式 `assigneeId` 字段；
2. 权限实质上是“创建者可管理；如果挂在共享清单下，则按清单权限管理”。

这会导致文档与实现矛盾。

#### 改进建议
将权限说明改为：

1. `ownershipType=1` 的个人独立任务：仅 `creatorId` 可读写。
2. `ownershipType=2` 的清单任务：
   - 创建者可读写；
   - 所属清单成员根据角色读写；
   - 角色规则以 `list_members.role` 为准。

如果未来需要“负责人”能力，再单独引入：

- `assigneeId`
- `assignedBy`
- `assignedAt`

并同步改造通知、权限和索引，而不是先在文档里提前声明一个未实现字段。

### 5. 个人任务索引设计需要按真实查询路径重做

#### 问题
当前文档索引中有多项与真实字段或查询路径不完全匹配，例如：

1. `(assigneeId, dueDate)` 当前无实际字段支撑。
2. `(repeatType, repeatStartDate)` 文档字段未正式定义。
3. `(creatorId, categoryId)` 有价值，但无法覆盖首页常见的“用户 + 状态/时间排序”场景。
4. 当前 `getTaskList` 的主路径是：
   - 按 `creatorId` 查个人任务；
   - 按 `listId` 查清单任务；
   - 按 `dueDate` 做日历范围过滤；
   - 按 `categoryId` / `status` / `priority` 做二次筛选。

#### 改进建议
围绕当前真实查询补充或替换为：

1. `tasks(creatorId, ownershipType, status, dueDate)`
   - 用于个人任务首页、按状态筛选、日期排序。
2. `tasks(listId, status, dueDate)`
   - 用于清单详情页和清单内任务排序。
3. `tasks(creatorId, categoryId, status, dueDate)`
   - 用于个人任务按分类筛选。
4. `tasks(parentTaskId, dueDate)`
   - 用于周期实例系列查询。
5. `tasks(reminderAt, reminderSent, status)`
   - 用于提醒扫描。
6. `tasks(creatorId, updatedAt)`
   - 用于缓存刷新、最近修改列表。

如果短期不引入 `ownershipType`，则退而求其次用：

- `tasks(creatorId, status, dueDate)`
- `tasks(listId, status, dueDate)`
- `tasks(parentTaskId, dueDate)`

### 6. 个人任务时间语义需要标准化

#### 问题
代码里创建任务时显式按 `+08:00` 处理 `dueDate`，说明该系统实际上依赖“业务时区固定为 UTC+8”。但文档没有明确：

1. `dueDate` 是否总带时间；
2. 全天任务与定时任务如何区分；
3. 逾期判断到底按自然日还是精确时间；
4. 周期规则按哪一个时区计算。

#### 改进建议
建议文档补齐以下字段或语义：

1. `dueDate: Timestamp`，统一存 UTC 时间。
2. `hasDueTime: Boolean`，区分“仅日期”和“日期+时间”。
3. 业务比较统一按 `Asia/Shanghai` 自然日进行逾期与周期判断。
4. 周期任务实例生成也按同一业务时区执行。

这样可以减少日历页、首页、云函数之间的边界误差。

### 7. 个人任务缓存设计过于粗放

#### 问题
当前缓存模式：

1. 首页使用 `cachedTasks` 存整批任务。
2. 日历页使用 `calendarTasks_${year}_${month}` 分月缓存。
3. 分类使用 `cachedCategories`。

但文档没有定义：

1. 缓存 TTL。
2. 创建/编辑/删除任务后的缓存失效规则。
3. 周期任务实例生成后的跨月缓存刷新机制。

对于个人任务而言，这会导致用户经常看到旧数据，尤其是在：

1. 编辑任务分类后首页与日历页不一致；
2. 周期任务完成后，下个月缓存仍旧保留旧实例；
3. 不同页面分别维护各自派生结果，难以统一。

#### 改进建议
建议在设计文档中明确缓存策略：

1. `cachedTasks` 改为带元信息的结构，例如 `{ updatedAt, source, list }`。
2. 月缓存记录查询范围和生成时间，例如 `{ rangeStart, rangeEnd, fetchedAt, list }`。
3. 任务新增、编辑、删除、切换状态后，至少失效：
   - `cachedTasks`
   - 该任务 `dueDate` 所在月份缓存
   - 若修改了日期，还要失效新旧月份缓存
4. 分类变更时失效：
   - `cachedCategories`
   - `cachedTasks` 的前端派生分类展示

### 8. 统计字段缺失，个人任务列表页容易出现 N+1 或前端重算过重

#### 问题
当前首页和日历页都在前端重算：

1. 逾期数
2. 已完成数
3. 周期系列最近实例
4. 分类分组

随着个人任务增加，这些都放在前端做会变重，且多个页面重复计算。

#### 改进建议
短期不建议大改表结构，但可以在设计上预留轻量统计思路：

1. 为 `lists` 补充可异步维护的统计字段，例如：
   - `taskCount`
   - `pendingCount`
   - `completedCount`
2. 为个人任务首页增加单独云函数聚合接口，而不是每次拉全量 100 条再前端拆组。
3. 若后续规模继续增长，再考虑用户维度汇总表，例如 `user_task_stats`，但这一项优先级低于前面几项。

## 推荐的数据结构修订方向

### 推荐修订版 `tasks` 关键字段
建议文档中将 `tasks` 修订为更接近当前实现的结构：

- `_id: String`
- `title: String`
- `description: String`
- `ownershipType: Number`，`1=个人独立任务`，`2=清单任务`
- `listId: String`，`ownershipType=2` 时必填
- `creatorId: String`
- `categoryId: String`
- `dueDate: Timestamp`
- `hasDueTime: Boolean`
- `priority: Number`
- `status: Number`，仅 `0=待办`，`1=已完成`
- `repeatType: Number`
- `repeatValue: String`
- `repeatStartDate: Timestamp | null`
- `repeatEndDate: Timestamp | null`
- `parentTaskId: String | null`
- `isPeriodicInstance: Boolean`
- `reminderAt: Timestamp | null`
- `reminderSent: Boolean`
- `attachments: Array`
- `completedAt: Timestamp | null`
- `createdAt: Timestamp`
- `updatedAt: Timestamp`

其中：

1. `isOverdue` 不入库。
2. 若不想立即引入 `ownershipType`，至少要在文档中明确“个人任务通过 `listId` 为空来表示”，但这只是过渡方案。

## 落地顺序建议

### 第一阶段：先修正文档和语义
目标：先让设计文档与真实系统一致。

1. 修正 `tasks` 字段定义，补齐 `parentTaskId`、`isPeriodicInstance`、`repeatStartDate`、`repeatEndDate` 等真实字段。
2. 把 `status` 改写为“持久状态 0/1”，逾期改为派生状态说明。
3. 删除或更正文档中未实现的 `assigneeId` / “负责人”表述。
4. 明确个人任务与清单任务的归属规则。
5. 重写索引章节，使之匹配当前查询路径。

### 第二阶段：最小代码改造
目标：降低后续演进成本。

1. 在创建/更新任务处补 `ownershipType`。
2. 在 `getTaskList`、筛选和统计接口中改为基于 `ownershipType` 判断任务归属。
3. 统一前端个人任务、清单任务的展示过滤逻辑。

### 第三阶段：缓存与聚合优化
目标：提升个人任务数据一致性与性能。

1. 统一任务缓存失效规则。
2. 为首页/日历页增加更窄的查询接口或轻量统计接口。
3. 如有必要，再引入汇总统计表或异步统计字段。

## 风险与权衡

1. 如果直接改成“所有个人任务必须绑定默认清单”，需要一次性做数据迁移，风险高于引入 `ownershipType`。
2. 如果继续保留 `status=2`，短期表面简单，但会持续放大前后端逻辑不一致问题。
3. 周期任务是当前最容易出错的部分，文档若不先修正，后续任何个人任务优化都容易误伤该逻辑。

## 建议执行策略
优先采用“文档对齐 + 最小结构补强”的路线：

1. 先把 `doc/data_design.md` 修正为与真实实现一致。
2. 再引入 `ownershipType`，解决个人任务归属歧义。
3. 最后处理缓存和统计接口，不先做激进重构。

## 面向用户问题的直接回答
对于“个人任务的管理，是否存在可以改进的地方”这一问题，答案是：

存在，而且主要集中在以下五点：

1. 个人任务归属不清晰。
2. 状态字段把“逾期”设计成持久状态，不符合当前实现。
3. 周期任务的数据模型没有完整文档化。
4. 权限与索引设计引用了未落地字段。
5. 缓存与统计设计不足，规模增大后会影响一致性和性能。

对应的改进方案优先级建议为：

1. 修正文档字段和状态语义。
2. 为任务新增 `ownershipType` 明确个人任务/清单任务归属。
3. 正式补齐周期任务实例字段。
4. 依据真实查询重建索引。
5. 补充缓存失效与统计接口设计。
